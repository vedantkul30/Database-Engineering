Database Practical Patterns

Double Booking Prevention | Cursor Pagination | Connection Pooling

1. Big Picture

Why These Patterns Exist

These three patterns appear in nearly every production backend system:

PatternProblem SolvedWhere It Appears
Double Booking PreventionRace conditions on shared resources under concurrent loadTicketing, cinema, Airbnb, airline, hospital scheduling
Cursor-Based PaginationOffset pagination is O(n) and gets slower with each pageAny API with paginated lists: news feed, search results, order history
Connection PoolingTCP connection setup/teardown is expensive per requestEvery backend application connecting to any database

All three share a common theme: naive implementations work in development but collapse under production load. Understanding the failure mode is as important as knowing the fix.

2. Pattern 1 — Double Booking Prevention

2.1 — The Problem: Race Condition / Double Booking

A race condition occurs when two concurrent transactions both read a resource as "available" and both proceed to book it — because neither was aware of the other's in-progress reservation.

Timeline (without locking):

T=0ms:  User A requests seat 15
T=0ms:  User B requests seat 15 (concurrent)

T=1ms:  Transaction A: SELECT * FROM seats WHERE id=15 AND is_booked=0
         → Returns: seat available ✅
T=1ms:  Transaction B: SELECT * FROM seats WHERE id=15 AND is_booked=0
         → Returns: seat available ✅  (reads same uncommitted state)

T=2ms:  Transaction A: UPDATE seats SET is_booked=1, name='Ali' WHERE id=15
         → 1 row updated ✅
T=3ms:  Transaction A: COMMIT

T=4ms:  Transaction B: UPDATE seats SET is_booked=1, name='Edmund' WHERE id=15
         → 1 row updated ✅  ← OVERWRITES Transaction A!
T=5ms:  Transaction B: COMMIT

Result: Seat belongs to Edmund. Ali paid but lost the seat. ❌

This is called "last write wins" — whichever transaction commits last owns the row, regardless of who checked availability first.

2.2 — The Naive (Broken) Fix

Most systems implement an availability check before the update:

// BROKEN — Race condition still exists between SELECT and UPDATE
@Transactional
public BookingResult bookSeat(long seatId, String userName, JdbcTemplate jdbc) {
    // Step 1: Check if seat is available
    List<Map<String, Object>> rows = jdbc.queryForList(
        "SELECT * FROM seats WHERE id = ? AND is_booked = 0",
        seatId
    );

    // Step 2: If available, book it
    if (!rows.isEmpty()) {
        jdbc.update(
            "UPDATE seats SET is_booked = 1, name = ? WHERE id = ?",
            userName, seatId
        );
        return BookingResult.SUCCESS;
    }
    return BookingResult.ALREADY_BOOKED;

    // PROBLEM: Another transaction can slip between Step 1 and Step 2.
    // Both transactions pass the check. Both execute the UPDATE. Double booking.
}

Why the check is not enough: MVCC means each transaction reads its own consistent snapshot. At the moment Transaction B executes the SELECT, Transaction A's UPDATE hasn't committed yet — so B's snapshot still shows the seat as available. Both transactions pass the availability check.

2.3 — Solution 1: SELECT FOR UPDATE (Explicit Pessimistic Locking)

Acquire an exclusive row lock during the SELECT, before checking availability. This serializes access to the row.

@Transactional
public BookingResult bookSeat(long seatId, String userName, JdbcTemplate jdbc) {
    // Phase 1 of 2PL: Acquire exclusive lock via FOR UPDATE
    // Any concurrent transaction attempting FOR UPDATE on the same row BLOCKS here
    List<Map<String, Object>> rows = jdbc.queryForList(
        "SELECT * FROM seats WHERE id = ? FOR UPDATE",
        // ↑ Exclusive row lock acquired immediately
        // Any other SELECT FOR UPDATE / UPDATE / DELETE on this row will BLOCK
        seatId
    );

    if (rows.isEmpty()) {
        return BookingResult.NOT_FOUND;
    }

    boolean isBooked = (Boolean) rows.get(0).get("is_booked");

    if (isBooked) {
        // Seat taken — no update needed
        // Phase 2 of 2PL: ROLLBACK releases the exclusive lock
        return BookingResult.ALREADY_BOOKED;
    }

    // Safe to book — no concurrent transaction can be here simultaneously
    jdbc.update(
        "UPDATE seats SET is_booked = true, name = ? WHERE id = ?",
        userName, seatId
    );

    // Phase 2 of 2PL: COMMIT releases the exclusive lock
    // Concurrent transactions waiting on this row are now unblocked
    return BookingResult.SUCCESS;
}

What happens under concurrency:

Timeline (with SELECT FOR UPDATE):

T=0ms:  Transaction A: SELECT * FROM seats WHERE id=15 FOR UPDATE
         → Acquires exclusive lock on row 15 ✅

T=0ms:  Transaction B: SELECT * FROM seats WHERE id=15 FOR UPDATE
         → BLOCKED: row 15 is exclusively locked by Transaction A

T=1ms:  Transaction A: seat is available → UPDATE seats SET is_booked=1, name='Ali'
T=2ms:  Transaction A: COMMIT → exclusive lock RELEASED

T=2ms:  Transaction B: UNBLOCKED → reads CURRENT committed state
         → Sees: is_booked=1, name='Ali' (already taken)
         → Returns ALREADY_BOOKED

Result: Seat correctly belongs to Ali. Edmund gets "already booked" error. ✅

SQL Version

-- Transaction 1 (User A - Ali):
BEGIN;
SELECT * FROM seats WHERE id = 15 FOR UPDATE;
-- Exclusive lock acquired on row 15

UPDATE seats SET is_booked = TRUE, name = 'Ali' WHERE id = 15;
COMMIT;
-- Lock released

-- Transaction 2 (User B - Edmund) - concurrent:
BEGIN;
SELECT * FROM seats WHERE id = 15 FOR UPDATE;
-- BLOCKED until Transaction 1 commits

-- After Transaction 1 commits:
-- Unblocked. Reads current state: is_booked=TRUE, name='Ali'
-- Application returns "seat already booked"
ROLLBACK;

2.4 — Solution 2: Conditional UPDATE (Implicit Locking)

Skip the SELECT. Directly UPDATE with the availability condition in the WHERE clause. Check rows_affected to determine if the booking succeeded.

@Transactional
public BookingResult bookSeat(long seatId, String userName, JdbcTemplate jdbc) {
    // Attempt to book ONLY if seat is currently unbooked
    // The WHERE clause is atomic with the UPDATE — no race window
    int rowsUpdated = jdbc.update(
        "UPDATE seats SET is_booked = TRUE, name = ? WHERE id = ? AND is_booked = FALSE",
        userName, seatId
    );

    if (rowsUpdated == 1) {
        return BookingResult.SUCCESS;   // We got the seat
    } else {
        return BookingResult.ALREADY_BOOKED;  // Someone else got it first
    }
}

Why this works in PostgreSQL:

Transaction A executes:
  UPDATE seats SET is_booked=TRUE, name='Ali' WHERE id=15 AND is_booked=FALSE

  Internally:
  1. Index scan on id=15 → finds tuple
  2. Postgres checks: is this row locked? No → proceed
  3. Postgres detects WHERE is_booked=FALSE → true (row is unbooked)
  4. Acquires implicit exclusive lock on the row
  5. Writes new version: is_booked=TRUE, name='Ali'
  6. Returns: 1 row updated

Transaction B executes (concurrent):
  UPDATE seats SET is_booked=TRUE, name='Edmund' WHERE id=15 AND is_booked=FALSE

  Internally:
  1. Index scan on id=15 → finds tuple
  2. Postgres checks: is this row locked? YES (Transaction A holds it)
  3. Transaction B BLOCKS until Transaction A commits

  After Transaction A commits:
  4. Transaction B re-evaluates the WHERE clause against committed data
     (READ COMMITTED isolation: re-reads current committed values)
  5. is_booked is now TRUE → WHERE clause fails
  6. Returns: 0 rows updated → ALREADY_BOOKED

The critical nuance: PostgreSQL, under READ COMMITTED, re-evaluates the WHERE clause against the latest committed data after being unblocked. This is PostgreSQL-specific behavior. Other databases may behave differently.

Why Solution 2 Works but Solution 1 is Preferable

AspectSELECT FOR UPDATEConditional UPDATE
Explicit control✅ Full control over what happens between read and write❌ Limited — just one update
Multi-step operations✅ Can do many things before committing❌ Tied to single UPDATE
Portability✅ Behavior is explicit and clear❌ Depends on database's re-evaluation behavior
Implicit reliance❌ None❌ Relies on READ COMMITTED + heap re-read
PerformanceSlightly more overhead (explicit lock management)Slightly less overhead (implicit lock)
Preferred forMulti-step booking flows, complex logicSimple atomic check-and-set

2.5 — Optimistic vs Pessimistic Locking

PESSIMISTIC LOCKING (SELECT FOR UPDATE):
  "Something WILL conflict. I'll lock preemptively."
  Lock → Check → Update → Commit/Rollback → Unlock
  Good for: HIGH contention (many users competing for same resource)
  Bad for: Low contention (locking overhead wasted if rarely contested)

OPTIMISTIC LOCKING (version counter):
  "Conflicts are RARE. I'll try without locking and detect conflicts at write time."
  Read + version → Process → Update WHERE version=read_version → Check rows_affected
  Good for: LOW contention
  Bad for: High contention (many retries, wasted work, user frustration)

For cinema/concert booking: HIGH contention → PESSIMISTIC (SELECT FOR UPDATE)
For editing a rarely-touched config record: LOW contention → OPTIMISTIC
// Optimistic locking with version column
@Transactional
public BookingResult bookSeatOptimistic(long seatId, String userName, JdbcTemplate jdbc) {
    // Read without locking
    Map<String, Object> seat = jdbc.queryForMap(
        "SELECT id, is_booked, version FROM seats WHERE id = ?", seatId
    );

    if ((Boolean) seat.get("is_booked")) {
        return BookingResult.ALREADY_BOOKED;
    }

    int version = (Integer) seat.get("version");

    // Update ONLY if version hasn't changed since we read
    int updated = jdbc.update(
        "UPDATE seats SET is_booked=TRUE, name=?, version=version+1 WHERE id=? AND version=?",
        userName, seatId, version
    );

    if (updated == 0) {
        // Someone else changed the row between our read and write
        // Must RETRY — application responsibility
        throw new OptimisticLockException("Retry required");
    }
    return BookingResult.SUCCESS;
}

3. Pattern 2 — Cursor-Based Pagination (Keyset Pagination)

3.1 — The Problem with OFFSET

OFFSET N instructs the database to fetch and then discard the first N rows. As N grows, the database does increasingly more work for no benefit.

Query: SELECT title FROM news ORDER BY id DESC OFFSET 100000 LIMIT 10

What PostgreSQL actually does:
  1. Scan the index backward
  2. Fetch 100,010 rows (100,000 to discard + 10 to return)
  3. Discard the first 100,000 rows
  4. Return the 10 remaining rows

Work done: O(OFFSET + LIMIT) = O(100,010)
Work useful: O(LIMIT)         = O(10)
Efficiency: 0.01%

Performance degradation:

OFFSET 0:       Fetch 10 rows.      ~0.2ms
OFFSET 1,000:   Fetch 1,010 rows.   ~1ms
OFFSET 100,000: Fetch 100,010 rows. ~79ms
OFFSET 1,000,000: Fetch 1,000,010.  ~620ms (first run: ~6 seconds!)

Pattern: latency grows linearly with offset value.
Page 1 is fast. Page 10,000 is painfully slow.

The duplicate record problem:

User reads page 10 (offset 90, limit 10) → gets rows 91-100
Meanwhile: new record inserted at position 91
User reads page 11 (offset 100, limit 10) → gets rows 101-110
BUT: old row 100 is now at position 101 → returned again!

Result: User sees duplicate records across pages.

3.2 — Solution: Cursor-Based (Keyset) Pagination

Instead of skipping N rows, use the last-seen ID as a "cursor" to tell the database exactly where to start. The database can jump directly to that position via the index.

Query: SELECT id, title FROM news WHERE id < :last_seen_id ORDER BY id DESC LIMIT 10

What PostgreSQL actually does:
  1. Use the id index to jump directly to last_seen_id position
  2. Scan backward from that position
  3. Return the first 10 rows found

Work done: O(LIMIT) = O(10)
Work useful: O(LIMIT) = O(10)
Efficiency: 100%

Performance:

Any page, any depth:  Fetch exactly 10 rows.  ~0.05ms (index-only)
No degradation with page depth.
Page 1 and page 10,000 take the same time. ✅

Java Implementation

/**
 * Cursor-based pagination for news articles.
 * Returns 10 articles older than the given cursor ID.
 *
 * API contract:
 *   GET /news               → first page (no cursor)
 *   GET /news?cursor=12345  → next page after ID 12345
 */
@GetMapping("/news")
public NewsPage getNews(
        @RequestParam(required = false) Long cursor,
        JdbcTemplate jdbc) {

    List<Map<String, Object>> rows;

    if (cursor == null) {
        // First page: no cursor → get the most recent 10 articles
        rows = jdbc.queryForList(
            "SELECT id, title FROM news ORDER BY id DESC LIMIT 10"
        );
    } else {
        // Subsequent pages: only return articles older than the cursor
        // WHERE id < cursor uses the index to jump directly to cursor position
        rows = jdbc.queryForList(
            "SELECT id, title FROM news WHERE id < ? ORDER BY id DESC LIMIT 10",
            cursor
        );
    }

    // Extract the last ID from results — this becomes the cursor for the next page
    Long nextCursor = null;
    if (!rows.isEmpty()) {
        nextCursor = (Long) rows.get(rows.size() - 1).get("id");
    }

    return new NewsPage(rows, nextCursor);
}

public record NewsPage(List<Map<String, Object>> articles, Long nextCursor) {}

// Client usage:
// Page 1: GET /news              → returns articles with ids: [200, 199, 198, ..., 191]
//                                   nextCursor = 191
// Page 2: GET /news?cursor=191   → returns articles with ids: [190, 189, ..., 181]
//                                   nextCursor = 181
// Page N: GET /news?cursor=181   → continues backward in time

SQL Examples

-- OFFSET approach (bad):
SELECT id, title FROM news
ORDER BY id DESC
OFFSET 100000 LIMIT 10;
-- Fetches 100,010 rows, discards 100,000. Slow.

-- Keyset approach (good) — first page:
SELECT id, title FROM news
ORDER BY id DESC
LIMIT 10;
-- Fetches exactly 10 rows. Fast.

-- Keyset approach — subsequent pages:
SELECT id, title FROM news
WHERE id < 199991    -- cursor from previous page's last item
ORDER BY ID DESC
LIMIT 10;
-- Fetches exactly 10 rows regardless of page depth. Fast.

-- Verify with EXPLAIN ANALYZE:
EXPLAIN ANALYZE
SELECT id, title FROM news
WHERE id < 50000
ORDER BY id DESC
LIMIT 10;

-- Expected output:
-- Index Scan Backward using news_pkey on news
-- Index Cond: (id < 50000)
-- Rows Removed by Filter: 0
-- Actual rows: 10    ← Only 10 rows processed, regardless of cursor value
-- Execution time: 0.05ms

3.3 — Keyset Limitations and When to Use OFFSET

Limitations of Keyset Pagination

LimitationExplanation
No random page accessCannot jump to "page 500" — must traverse pages sequentially
Single sort columnWorks best with a single unique indexed column (ID, timestamp)
Complex multi-column cursorsSorting by (date, id) requires a compound cursor
No total page countCannot say "Page 3 of 47" — only "next/previous"
Cursor invalidationIf the cursor row is deleted, the cursor still works (skips the gap)

When OFFSET Is Acceptable

✅ Use OFFSET when:
  - Total dataset is small (< 10,000 rows)
  - Users rarely navigate beyond page 3-5
  - Random page access is required ("jump to page 200")
  - Admin interfaces where performance is secondary

✅ Use Keyset/Cursor when:
  - Infinite scroll (Twitter, Instagram, news feeds)
  - Large datasets where deep pagination is common
  - High-traffic APIs where p99 latency matters
  - Any list where pages are navigated sequentially

Multi-Column Cursor (Advanced)

-- Sort by date DESC, then id DESC (tie-breaker)
-- Cursor needs both: (last_date, last_id)

SELECT id, title, published_at
FROM news
WHERE (published_at, id) < (:last_date, :last_id)
-- ← Row-value comparison: works as a compound cursor
ORDER BY published_at DESC, id DESC
LIMIT 10;
// Java multi-column cursor
public List<Map<String, Object>> getNewsPage(
        Timestamp lastDate, Long lastId, JdbcTemplate jdbc) {
    if (lastDate == null) {
        // First page
        return jdbc.queryForList(
            "SELECT id, title, published_at FROM news ORDER BY published_at DESC, id DESC LIMIT 10"
        );
    }
    // Subsequent pages using row-value comparison
    return jdbc.queryForList(
        "SELECT id, title, published_at FROM news " +
        "WHERE (published_at, id) < (?, ?) " +
        "ORDER BY published_at DESC, id DESC LIMIT 10",
        lastDate, lastId
    );
}

4. Pattern 3 — Connection Pooling

4.1 — The Problem: Per-Request Connection Creation

The naive approach: open a database connection per request, execute query, close connection.

Without pooling:
GET /users → open TCP connection → 3-way handshake → TLS negotiation →
             PostgreSQL auth → SSL → execute query → get results →
             close connection → 4-way teardown

Cost breakdown (local DB):
  TCP 3-way handshake:     ~0.1ms
  PostgreSQL auth:         ~5-10ms  ← expensive
  SSL negotiation:         ~2-5ms
  Query execution:         ~1-2ms
  Connection teardown:     ~0.5ms
  ──────────────────────────────
  Total overhead:          ~9-17ms per request

vs query execution alone: ~1-2ms (5-17× faster without connection overhead)

For remote DB (cloud, different datacenter):
  Network RTT alone:       20-100ms per round trip
  Multiple round trips needed for auth: 60-300ms overhead per request!

The per-connection problem in PostgreSQL:

PostgreSQL spawns a NEW PROCESS per connection.
  Each process: ~5-10MB RAM
  100 connections: ~500MB-1GB RAM just for connection management
  PostgreSQL default max_connections = 100

When max_connections is reached:
  → New connection attempt: FATAL: sorry, too many clients already
  → Application crashes or returns 500 to users

4.2 — Solution: Connection Pooling

Pre-create N database connections at application startup. Reuse them for every request. Return connections to the pool after each query.

With pooling (pool size = 10):

App startup: establish 10 persistent TCP connections to PostgreSQL
             (one-time cost: ~90ms total for all 10)

GET /users  → borrow connection from pool (0ms) → execute query (1ms) → return to pool
GET /orders → borrow connection from pool (0ms) → execute query (2ms) → return to pool
GET /profile→ borrow connection from pool (0ms) → execute query (1ms) → return to pool

Connection overhead per request: ~0ms (already established)
Query execution time: ~1-2ms (same as before)

Performance improvement:
  Without pool (local):  ~10-18ms per request
  With pool (local):      ~1-2ms per request  → 5-10× faster
  
  Without pool (remote): ~60-300ms per request
  With pool (remote):     ~2-5ms per request  → 30-100× faster

4.3 — Java Implementation (HikariCP)

HikariCP is the standard connection pool library for Java Spring Boot applications.

import com.zaxxer.hikari.HikariConfig;
import com.zaxxer.hikari.HikariDataSource;
import org.springframework.jdbc.core.JdbcTemplate;

/**
 * Connection pool configuration using HikariCP.
 *
 * HikariCP is:
 *   - Default connection pool in Spring Boot
 *   - Fastest Java connection pool available
 *   - Benchmark leader across all major connection pools
 */
@Configuration
public class DatabaseConfig {

    /**
     * Configure HikariCP connection pool.
     * Equivalent to Node.js: new Pool({ max: 20, idleTimeoutMillis: 30000 })
     */
    @Bean
    public HikariDataSource dataSource() {
        HikariConfig config = new HikariConfig();

        // JDBC connection URL
        config.setJdbcUrl("jdbc:postgresql://localhost:5432/postgres");
        config.setUsername("postgres");
        config.setPassword("postgres");

        // ─── Pool Size Configuration ─────────────────────────────────────────

        // Maximum number of connections in the pool.
        // Rule of thumb: (2 × CPU cores) + effective_spindle_count
        // For a 4-core machine with SSD: 8-10 connections
        // Counterintuitive: more connections ≠ better performance (context switching)
        config.setMaximumPoolSize(20);

        // Minimum idle connections maintained in the pool.
        // Set equal to maximumPoolSize for fixed pool size (no scale-up/down overhead)
        config.setMinimumIdle(5);

        // ─── Timeout Configuration ────────────────────────────────────────────

        // Max time (ms) a request waits for a connection from the pool.
        // If all 20 connections are busy and a 21st request comes in,
        // it waits up to this long before throwing SQLException.
        // Node.js equivalent: connectionTimeoutMillis
        config.setConnectionTimeout(30_000);  // 30 seconds

        // Max time (ms) a connection can sit idle in the pool before being closed.
        // Prevents holding idle connections open indefinitely.
        // Node.js equivalent: idleTimeoutMillis
        config.setIdleTimeout(600_000);  // 10 minutes

        // Max lifetime (ms) of a connection regardless of idle/active state.
        // Prevents stale connections from accumulating.
        // Should be less than database's connection_timeout setting.
        config.setMaxLifetime(1_800_000);  // 30 minutes

        // ─── Health Check ─────────────────────────────────────────────────────

        // Query used to validate a connection is still alive before use.
        // PostgreSQL supports the lightweight "SELECT 1" or connection validation.
        config.setConnectionTestQuery("SELECT 1");

        // Pool name — appears in monitoring/metrics
        config.setPoolName("MainPool");

        return new HikariDataSource(config);
    }

    @Bean
    public JdbcTemplate jdbcTemplate(HikariDataSource dataSource) {
        return new JdbcTemplate(dataSource);
    }
}

Using the Pool for Queries

@Service
public class EmployeeService {

    private final JdbcTemplate jdbcTemplate;

    public EmployeeService(JdbcTemplate jdbcTemplate) {
        this.jdbcTemplate = jdbcTemplate;
        // JdbcTemplate uses HikariCP automatically via the configured DataSource
    }

    /**
     * Stateless query — pool handles connection selection automatically.
     * No explicit connect/disconnect needed.
     *
     * Node.js equivalent:
     *   pool.query("SELECT * FROM employees")
     */
    public List<Map<String, Object>> getAllEmployees() {
        // HikariCP:
        // 1. Borrows an available connection from the pool
        // 2. Executes the query
        // 3. Returns the connection to the pool automatically
        // Connection borrow/return: ~microseconds
        return jdbcTemplate.queryForList("SELECT id, name, salary FROM employees");
    }

    /**
     * For transactions, explicitly borrow a dedicated connection.
     * @Transactional ensures the same connection is used throughout.
     *
     * Node.js equivalent:
     *   const client = await pool.connect();
     *   await client.query('BEGIN');
     *   await client.query(...);
     *   await client.query('COMMIT');
     *   client.release();
     */
    @Transactional
    public void transferSalary(long fromId, long toId, BigDecimal amount) {
        // Spring's @Transactional borrows ONE connection from the pool
        // and uses it for all queries in this method.
        // The connection is returned to the pool after the method completes.

        jdbcTemplate.update(
            "UPDATE employees SET salary = salary - ? WHERE id = ?",
            amount, fromId
        );
        jdbcTemplate.update(
            "UPDATE employees SET salary = salary + ? WHERE id = ?",
            amount, toId
        );
        // COMMIT happens automatically at method exit
        // Connection returned to pool
    }
}

Pool Configuration Rules of Thumb

Pool Size Formula (PostgreSQL's recommendation):
  pool_size = (core_count × 2) + effective_spindle_count

  4-core CPU, SSD (spindle_count ≈ 1): pool_size = 9
  8-core CPU, SSD:                      pool_size = 17
  8-core CPU, HDD (spindle_count = 4):  pool_size = 20

Why NOT huge pools?
  100 connections → PostgreSQL spawns 100 processes → ~500MB-1GB RAM overhead
  Context switching between 100 processes is SLOWER than 10 processes taking turns
  HikariCP benchmark: optimal pool size is usually 10-20, not 100+

Common mistake: Setting max pool size = max PostgreSQL connections
  PostgreSQL default: max_connections = 100
  If you have 10 app servers × 20 pool size = 200 connections → EXCEEDS LIMIT
  Solution: Use PgBouncer as a connection pooler BETWEEN app servers and PostgreSQL

Monitoring Pool Health

@Component
public class PoolHealthMonitor {

    private final HikariDataSource dataSource;

    @Scheduled(fixedRate = 60_000)  // Every minute
    public void logPoolStats() {
        HikariPoolMXBean pool = dataSource.getHikariPoolMXBean();

        log.info("Connection Pool Stats: " +
            "active={}, idle={}, waiting={}, total={}",
            pool.getActiveConnections(),   // Currently executing queries
            pool.getIdleConnections(),     // Available for use
            pool.getThreadsAwaitingConnection(),  // Requests waiting for a connection
            pool.getTotalConnections()     // Total in pool (active + idle)
        );

        // Alert if requests are waiting for connections — pool may be undersized
        if (pool.getThreadsAwaitingConnection() > 5) {
            log.warn("Pool pressure: {} threads waiting for connection. " +
                     "Consider increasing pool size.", pool.getThreadsAwaitingConnection());
        }
    }
}

PgBouncer: Connection Pooler Between App and PostgreSQL

For high-scale systems where multiple application servers need more connections than PostgreSQL can handle:

Without PgBouncer:
  10 app servers × 20 pool size = 200 connections to PostgreSQL
  PostgreSQL max_connections = 100 → EXCEEDS LIMIT, connections rejected

With PgBouncer (transaction mode):
  10 app servers → PgBouncer pool (200 virtual connections)
  PgBouncer → PostgreSQL (20 real connections)
  PostgreSQL sees only 20 connections → well within limits
  App servers see 200 connections → no waiting

Architecture:
  App Server 1 ─┐
  App Server 2 ─┤
  App Server 3 ─┼──→ PgBouncer (200 virtual) ──→ PostgreSQL (20 real)
  App Server 4 ─┤
  App Server 5 ─┘

5. Complete System: Booking API with All Three Patterns

/**
 * Complete booking system demonstrating all three patterns:
 * 1. Double booking prevention (SELECT FOR UPDATE)
 * 2. Cursor-based pagination for seat listing
 * 3. Connection pooling (HikariCP via Spring Boot)
 */
@RestController
@RequestMapping("/seats")
public class SeatBookingController {

    private final JdbcTemplate jdbcTemplate;

    // ── PATTERN 3: Connection Pooling ──────────────────────────────────────
    // JdbcTemplate uses HikariCP pool automatically.
    // No explicit connect/disconnect anywhere in this class.
    public SeatBookingController(JdbcTemplate jdbcTemplate) {
        this.jdbcTemplate = jdbcTemplate;
    }

    // ── PATTERN 2: Cursor Pagination ──────────────────────────────────────
    /**
     * GET /seats?cursor=50&limit=10
     * Returns available seats starting after the cursor seat ID.
     *
     * No OFFSET — uses keyset pagination for O(1) performance at any depth.
     */
    @GetMapping
    public Map<String, Object> listSeats(
            @RequestParam(required = false) Long cursor,
            @RequestParam(defaultValue = "10") int limit) {

        List<Map<String, Object>> seats;

        if (cursor == null) {
            seats = jdbcTemplate.queryForList(
                "SELECT id, row_name, seat_number, is_booked, name " +
                "FROM seats WHERE is_booked = FALSE ORDER BY id ASC LIMIT ?",
                limit
            );
        } else {
            seats = jdbcTemplate.queryForList(
                "SELECT id, row_name, seat_number, is_booked, name " +
                "FROM seats WHERE id > ? AND is_booked = FALSE ORDER BY id ASC LIMIT ?",
                cursor, limit
            );
        }

        Long nextCursor = seats.isEmpty() ? null
            : (Long) seats.get(seats.size() - 1).get("id");

        return Map.of("seats", seats, "nextCursor", nextCursor);
    }

    // ── PATTERN 1: Double Booking Prevention ──────────────────────────────
    /**
     * PUT /seats/{seatId}/book
     * Books a seat for a user. Prevents double booking via SELECT FOR UPDATE.
     */
    @PutMapping("/{seatId}/book")
    @Transactional
    public ResponseEntity<Map<String, String>> bookSeat(
            @PathVariable long seatId,
            @RequestBody Map<String, String> body) {

        String userName = body.get("name");

        // Acquire exclusive row lock BEFORE checking availability
        List<Map<String, Object>> rows = jdbcTemplate.queryForList(
            "SELECT id, is_booked, name FROM seats WHERE id = ? FOR UPDATE",
            seatId
        );

        if (rows.isEmpty()) {
            return ResponseEntity.notFound().build();
        }

        if ((Boolean) rows.get(0).get("is_booked")) {
            return ResponseEntity.badRequest()
                .body(Map.of("error", "Seat already booked by " + rows.get(0).get("name")));
        }

        jdbcTemplate.update(
            "UPDATE seats SET is_booked = TRUE, name = ? WHERE id = ?",
            userName, seatId
        );

        return ResponseEntity.ok(Map.of(
            "message", "Seat " + seatId + " booked successfully for " + userName
        ));
    }
}

6. Important Interview Questions

Basic

Q: What is a race condition in a booking system?
A: A race condition occurs when two concurrent requests both read a shared resource as "available" simultaneously and both proceed to book it. Because each transaction operates on its own MVCC snapshot, both see the same "unbooked" state. Both execute the update — the last commit wins, causing double booking. One user pays but loses the seat.

Q: What does SELECT FOR UPDATE do?
A: It acquires an exclusive row-level lock on the selected rows as part of the SELECT itself. Any concurrent transaction attempting to SELECT FOR UPDATE, UPDATE, or DELETE the same rows is blocked until the locking transaction commits or rolls back. This serializes access to the row, preventing race conditions between concurrent writers.

Q: Why is OFFSET pagination slow for large pages?
A: OFFSET N LIMIT M requires the database to fetch N+M rows, discard the first N, and return M. The database does O(N+M) work but returns only M rows. For large N (deep pages), this is extremely wasteful — e.g., OFFSET 1,000,000 fetches 1,000,010 rows to return 10. Performance degrades linearly with page depth.

Q: What is connection pooling and why is it needed?
A: Connection pooling pre-creates a fixed set of database connections at startup and reuses them for every request. Without pooling, each request creates a new TCP connection (3-way handshake, TLS, PostgreSQL auth) and closes it after the query — 10-300ms overhead per request. With pooling, borrowing a connection from the pool takes microseconds. Pooling also limits the total number of connections to PostgreSQL, preventing connection exhaustion.

Q: What is keyset/cursor-based pagination?
A: Instead of skipping N rows (OFFSET), cursor-based pagination uses the last-seen record's unique identifier (cursor) as a WHERE clause filter: WHERE id < :cursor ORDER BY id DESC LIMIT 10. The database uses the index to jump directly to the cursor position and fetches exactly 10 rows — O(1) regardless of page depth.

Intermediate

Q: Why does SELECT FOR UPDATE prevent double booking but a plain SELECT followed by UPDATE does not?
A: Under MVCC, a plain SELECT reads from a consistent snapshot — it doesn't prevent another transaction from reading the same snapshot and also seeing the resource as available. Both transactions pass the availability check and both execute the UPDATE, causing double booking. SELECT FOR UPDATE acquires an exclusive lock at SELECT time, blocking any concurrent transaction from accessing the same row until the first transaction commits or rolls back. When the second transaction unblocks, it re-reads the current committed state (seat is now taken) and correctly rejects.

Q: Explain the "conditional UPDATE" approach to double booking prevention. Why is it database-specific?
A: UPDATE seats SET is_booked=TRUE WHERE id=? AND is_booked=FALSE — check rows_updated to determine outcome. When two concurrent transactions execute this: the first gets the implicit exclusive lock; the second blocks. When the second is unblocked after the first commits, PostgreSQL (under READ COMMITTED) re-evaluates the WHERE clause against current committed data, finding is_booked=TRUE → 0 rows updated → booking rejected. This works in PostgreSQL because of its specific READ COMMITTED behavior of re-checking WHERE on unblock. MySQL and SQL Server handle this differently and may not re-evaluate the WHERE clause in the same way. The explicit SELECT FOR UPDATE approach is more portable and gives the developer explicit control.

Q: What are the limitations of cursor-based pagination?
A: (1) No random page access — cannot jump to "page 500" without traversing all previous pages. (2) No total page count — cannot display "Page 3 of 47." (3) Multi-column sort requires a compound cursor with row-value comparison, which is more complex to implement. (4) Works best with a unique indexed column; non-unique columns require careful handling to avoid missing records.

Q: What happens when all connections in a pool are busy and a new request arrives?
A: The request waits for up to connectionTimeout milliseconds. During this wait, it's queued. If a connection becomes available before the timeout, the request proceeds. If the timeout expires, a SQLException is thrown — the request fails. This is monitored via pool.getThreadsAwaitingConnection(). Persistent queuing indicates the pool is undersized for the current load and should be increased (if PostgreSQL can handle more connections) or a connection pooler like PgBouncer should be added.

Advanced

Q: Design a system for booking 50,000 concert seats released simultaneously to 500,000 concurrent users. Explain the full database concurrency strategy.
A:
- Connection layer: PgBouncer in front of PostgreSQL. App servers have HikariCP pools (20 connections each). PgBouncer limits total connections to PostgreSQL to ~200 regardless of app server count.
- Lock strategy: SELECT FOR UPDATE on individual seat rows. Lock granularity is row-level — 50,000 seats means 50,000 independent locks; only users competing for the SAME seat block each other.
- Queue management: Users not in contention for the same seat proceed in parallel (no cross-seat blocking). Set lock_timeout = '5s' so users aren't stuck indefinitely.
- Retry: Application retries on timeout (limited retries with backoff).
- Seat listing: Cursor-based pagination for displaying available seats — no OFFSET at this scale.
- Caching: Pre-warm a Redis set of available seat IDs; do a quick SISMEMBER check before hitting the DB. Invalidate on booking. Reduces DB hits by 80%+ for the "is this seat available?" check.
- Partition: Seats table partitioned by venue/event so lock contention is isolated per partition.

Q: What is the optimal HikariCP pool size and why is bigger not always better?
A: The formula is (core_count × 2) + effective_spindle_count. For an 8-core machine with SSD: ~17 connections. The reason bigger pools hurt: PostgreSQL spawns a new OS process per connection (~5-10MB RAM each). With 200 connections, PostgreSQL spends significant CPU on process scheduling and context switching between processes. A smaller number of connections each processing queries sequentially outperforms a large number of connections thrashing the OS scheduler. Furthermore, database I/O (disk seeks, page reads) is the bottleneck, not CPU — more connections don't make the disk faster.

7. Common Mistakes

Double Booking

  • Checking availability without locking: SELECT then UPDATE without FOR UPDATE — classic race condition
  • Not handling the lock-wait timeout: Indefinite blocking on FOR UPDATE destroys UX; always set lock_timeout
  • Using optimistic locking for high-contention resources: At peak concert ticket release, every transaction conflicts → constant retries → thundering herd

Pagination

  • Using OFFSET for infinite scroll: Page depth increases as users scroll → query gets slower → page 50 loads in 5 seconds
  • Not indexing the cursor column: WHERE id < :cursor without an index on id becomes a full table scan — as slow as OFFSET
  • Returning total page count with keyset pagination: Cannot know total pages without a COUNT(*) query — which itself may be expensive; accept that keyset pagination is "next/previous" only

Connection Pooling

  • Opening/closing connections per request: The antipattern — 10-300ms overhead per request; crushes performance at scale
  • Setting pool size = PostgreSQL max_connections: 10 app servers × 100 pool size = 1,000 connections >> PostgreSQL's max_connections = 100 → connection errors under load
  • Not monitoring pool wait time: threadsAwaitingConnection > 0 for sustained periods = pool is a bottleneck; silent until requests start failing with timeout errors
  • Using pool connections for long-running batch jobs: One connection held for minutes blocks other requests; use a separate pool or direct connection for batch operations

8. Performance Quick Reference

Double Booking

No locking:          Race condition possible. Wrong result.
SELECT FOR UPDATE:   Correct. Lock overhead: ~0.1ms (row-level).
Conditional UPDATE:  Correct (PostgreSQL). Slightly less overhead.
Table-level LOCK:    Correct but severe: all 50,000 seats serialized.
                     Only 1 booking at a time across ALL seats. ❌

For 50,000 seats: SELECT FOR UPDATE is ideal — each seat is an independent lock.

Pagination

OFFSET pagination:
  Page 1 (offset=0):         0.2ms
  Page 100 (offset=1000):    1ms
  Page 10000 (offset=100k):  79ms
  Page 100000 (offset=1M):   620ms-6s
  Grows: O(offset)

Keyset pagination:
  Any page:                  0.05ms
  Constant: O(limit)
  100,000× faster at page 100,000

Connection Pooling

Per-request connections (local DB):    10-18ms overhead
Per-request connections (remote DB):   60-300ms overhead
With HikariCP pool (any):             ~0ms overhead
                                       
1000 requests, local DB:
  Without pool: 1000 × 15ms = 15 seconds of connection overhead
  With pool:    1000 × 0ms  = 0 seconds of connection overhead
  + Query time: 1000 × 2ms  = 2 seconds
  Total with pool: ~2 seconds vs ~17 seconds without
