Locking & Concurrency Control

Production Engineering Notes

1. Big Picture

Why Locking Exists

Databases are concurrent systems. Thousands of clients read and write simultaneously. Without coordination, concurrent access to the same data leads to:

  • Lost updates: Two transactions read a value, both compute a new value, one overwrites the other's change
  • Dirty reads: A transaction reads data written by another transaction that hasn't committed yet
  • Non-repeatable reads: A transaction reads a value twice; another transaction changes it between the reads
  • Phantom reads: A transaction re-runs a range query and gets different rows because another transaction inserted/deleted rows
  • Double booking: Two users book the same resource simultaneously

Locking is the mechanism databases use to serialize access to shared data, ensuring that concurrent transactions produce results equivalent to some serial (sequential) execution.

Where Locking Matters in Production

SystemLocking Concern
BankingBalance updates must be atomic; no dirty reads of account balance
E-commerceInventory must not oversell; two buyers can't buy the last item
Cinema/Airline bookingSame seat can't be booked by two users simultaneously
Configuration systemsConfig readers must not see partial updates
Ticketing (Ticketmaster)Seat allocation under extreme concurrent load
Trading systemsOrder execution must not double-fill at same price

2. Core Concepts

2.1 — Shared Lock (Read Lock / S-Lock)

A shared lock is acquired when a transaction wants to read a value and needs to guarantee that nobody modifies it for the duration of that read.

Properties:

  • Multiple transactions can hold shared locks on the same resource simultaneously
  • A transaction holding a shared lock is saying: "I am reading this. Don't let anyone write to it."
  • Blocks exclusive locks — prevents writers from modifying the resource
  • Does NOT block other readers — multiple shared locks coexist
Resource: account_balance (row for user Bob)

Transaction A: Shared Lock ──────────────────────────────→ Released
Transaction B: Shared Lock ──────────────────────────────→ Released
Transaction C: Exclusive Lock attempt ──── BLOCKED ────→ (waits for A and B to release)

Both A and B can read simultaneously.
C cannot write until BOTH A and B release their shared locks.

When to use:

-- Explicitly acquire shared lock in PostgreSQL:
SELECT * FROM accounts WHERE id = 42 FOR SHARE;

-- This ensures nobody can UPDATE/DELETE this row
-- while your transaction is reading it.

2.2 — Exclusive Lock (Write Lock / X-Lock)

An exclusive lock is acquired when a transaction wants to write (INSERT, UPDATE, DELETE) a value and needs to guarantee that nobody reads or writes it simultaneously.

Properties:

  • Only ONE transaction can hold an exclusive lock on a resource at a time
  • A transaction holding an exclusive lock is saying: "I am modifying this. Don't let anyone read or write it."
  • Blocks all other shared locks AND exclusive locks
  • No other transaction can read or write the resource until the exclusive lock is released
Resource: account_balance (row for user Bob)

Transaction A: Exclusive Lock ──────────────────────────→ Released
Transaction B: Shared Lock attempt ──── BLOCKED ───────→ (waits)
Transaction C: Exclusive Lock attempt ── BLOCKED ───────→ (waits)

Only A can access the resource.
Nobody else can even read it.

When to use:

-- Explicitly acquire exclusive lock in PostgreSQL:
SELECT * FROM seats WHERE id = 14 FOR UPDATE;

-- This row is now exclusively locked.
-- Any other SELECT FOR UPDATE or UPDATE/DELETE on this row blocks.
-- The lock is released when this transaction commits or rolls back.

2.3 — Lock Compatibility Matrix

                    Requested Lock
                  │ Shared  │ Exclusive │
Held Lock ────────┼─────────┼───────────┤
None              │    ✅   │    ✅     │  Both succeed
Shared            │    ✅   │    ❌     │  Exclusive blocked by existing shared
Exclusive         │    ❌   │    ❌     │  Both blocked by existing exclusive

Rules:

  1. Multiple shared locks on same resource → all succeed
  2. Any exclusive lock on a resource → all others (shared or exclusive) blocked
  3. To get exclusive → zero existing locks (shared or exclusive) must exist
  4. To get shared → zero exclusive locks must exist

2.4 — Deadlock

A deadlock occurs when two or more transactions are each waiting for the other to release a lock, creating a circular dependency where none can proceed.

How Deadlock Happens

Transaction A holds:  Lock on row 20
Transaction B holds:  Lock on row 21

Transaction A wants:  Lock on row 21  → BLOCKED (B holds it)
Transaction B wants:  Lock on row 20  → BLOCKED (A holds it)

Neither can proceed. Circular wait. DEADLOCK.

Timeline:
T1: A inserts row 20 → A acquires exclusive lock on row 20
T2: B inserts row 21 → B acquires exclusive lock on row 21
T3: A tries to insert row 21 → BLOCKED (B holds row 21)
T4: B tries to insert row 20 → BLOCKED (A holds row 20)
    → DEADLOCK DETECTED
    → PostgreSQL rolls back the transaction that ENTERED DEADLOCK LAST (B)
    → A is unblocked and continues

Deadlock Detection in PostgreSQL

PostgreSQL uses a background lock monitor that periodically checks for circular dependencies in the lock graph. When detected:

  • The transaction that "entered the deadlock last" (caused the cycle to close) is chosen as the victim
  • The victim's transaction is automatically rolled back
  • The victim receives: ERROR: deadlock detected
  • The other transaction is unblocked and continues
-- Demo: Two terminals, same PostgreSQL instance

-- Terminal 1:
BEGIN;
INSERT INTO test VALUES (20);  -- Acquires exclusive lock on row 20

-- Terminal 2:
BEGIN;
INSERT INTO test VALUES (21);  -- Acquires exclusive lock on row 21

-- Terminal 1:
INSERT INTO test VALUES (21);  -- BLOCKED: Terminal 2 holds row 21
                                -- (waiting...)

-- Terminal 2:
INSERT INTO test VALUES (20);  -- BLOCKED: Terminal 1 holds row 20
                                -- PostgreSQL detects deadlock cycle
                                -- ERROR: deadlock detected
                                -- Terminal 2's transaction is ROLLED BACK
                                -- Terminal 1 is UNBLOCKED

Three Possible Outcomes After Blocking

Scenario 1 — Other transaction COMMITS:
  Transaction A blocked waiting for row 20.
  Transaction B commits, releasing lock on row 20.
  Transaction A unblocks → tries to INSERT 20 → FAILS with DUPLICATE KEY
  (because B already committed the value)

Scenario 2 — Other transaction ROLLS BACK:
  Transaction A blocked waiting for row 20.
  Transaction B rolls back, releasing lock on row 20.
  Transaction A unblocks → INSERT 20 → SUCCEEDS
  (row 20 was never committed, so it's free to insert)

Scenario 3 — DEADLOCK:
  Both transactions waiting for each other.
  Database detects cycle → rolls back one transaction.

2.5 — Two-Phase Locking (2PL)

Two-Phase Locking is a concurrency control protocol that determines when locks can be acquired and released to guarantee serializability — the strongest isolation level.

The Two Phases

Phase 1: GROWING (Lock Acquisition Phase)
  Transaction can ACQUIRE locks.
  Transaction CANNOT release any lock.
  ─────────────────────────────────────────→ Lock Point (max locks held)

Phase 2: SHRINKING (Lock Release Phase)
  Transaction can RELEASE locks.
  Transaction CANNOT acquire any new locks.

Once you release even ONE lock, you cannot acquire any more.
Timeline visualization:

Phase 1 (Growing):    ACQUIRE A  ACQUIRE B  ACQUIRE C
                      ─────────────────────────────────┐ Lock Point
Phase 2 (Shrinking):                                   └──RELEASE A  RELEASE B  RELEASE C

Why 2PL Prevents Double Booking

The double booking problem occurs when two transactions read a seat as "available" and both book it:

WITHOUT 2PL (the problem):

T1: SELECT seat 14 → available  ┐
T2: SELECT seat 14 → available  ┘ Both see available simultaneously

T1: UPDATE seat 14 SET booked=1, name='Hussein'
T2: UPDATE seat 14 SET booked=1, name='Edmund'  ← BLOCKED (T1 holds exclusive lock)

T1: COMMIT → seat belongs to Hussein
T2: UNBLOCKED → COMMIT → seat now belongs to Edmund ← OVERWRITE!

Result: Seat booked to Edmund, Hussein paid but lost the seat.
WITH 2PL (SELECT FOR UPDATE):

T1: SELECT * FROM seats WHERE id=14 FOR UPDATE  → Acquires EXCLUSIVE lock
T2: SELECT * FROM seats WHERE id=14 FOR UPDATE  → BLOCKED (T1 holds exclusive lock)

T1: UPDATE seats SET booked=1, name='Hussein' WHERE id=14
T1: COMMIT → releases exclusive lock (Phase 2 begins)

T2: UNBLOCKED → reads seat 14 → sees booked=1, name='Hussein'
T2: Application logic: "Seat already booked" → rejects booking
T2: ROLLBACK

Result: Seat correctly belongs to Hussein. No double booking.

Strict 2PL (Most Common in Practice)

In Strict 2PL, all exclusive locks are held until transaction end (COMMIT or ROLLBACK). This is the most common variant implemented in production databases including PostgreSQL.

Strict 2PL rule:
  - Exclusive locks released ONLY at transaction end (commit/rollback)
  - Shared locks may be released earlier (in some implementations)
  - Prevents cascading aborts (dirty reads from uncommitted data)

3. Deep Dive — Internal Workings

3.1 — Lock Granularity

Databases support locking at multiple levels. Coarser granularity = less overhead but more contention. Finer granularity = more overhead but less contention.

Database lock levels (coarsest to finest):

DATABASE     → lock the entire database (rare, maintenance operations)
TABLESPACE   → lock an entire tablespace
TABLE        → lock an entire table (DDL operations, LOCK TABLE)
PAGE         → lock a disk page (~8KB, multiple rows)
ROW (TUPLE)  → lock a single row (most common in OLTP)
COLUMN       → not natively supported in most RDBMS (application-level)
VALUE        → predicate lock (used for phantom read prevention)

PostgreSQL row-level locking modes:

Lock ModeSQLConflicts With
FOR KEY SHAREWeakest. Allows other FKS, FS, and FNK.FOR UPDATE, FOR NO KEY UPDATE
FOR SHAREShared read lock.FOR NO KEY UPDATE, FOR UPDATE
FOR NO KEY UPDATEUpdate that doesn't affect PK/FK.FOR SHARE, FOR UPDATE, FNKU
FOR UPDATEExclusive lock. Strongest row lock.All other row locks

PostgreSQL lock manager data structures (in shared memory):

┌──────────────────────────────────────────────────────────────────┐
│  Lock Table (hash table in shared memory)                        │
│  Key: (relation_id, block_num, tuple_offset, transaction_id)     │
│                                                                  │
│  Each entry:                                                     │
│   ┌────────────────────────────────────────────────────────────┐ │
│   │  Resource identifier                                       │ │
│   │  Granted locks list: [(txn_A, SHARE), (txn_B, SHARE)]     │ │
│   │  Waiting locks list: [(txn_C, EXCLUSIVE)]                 │ │
│   └────────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────────┘

Deadlock detection:
  Background process checks lock table periodically.
  Builds a "wait-for graph": A waits for B, B waits for A → cycle → deadlock.
  Selects victim (transaction that closed the cycle).
  Sends SIGINT to victim process → transaction rolls back.

3.3 — MVCC and Locking (PostgreSQL)

PostgreSQL uses MVCC (Multi-Version Concurrency Control) for read-write concurrency. Under MVCC:

  • Readers do not block writers
  • Writers do not block readers
  • Each transaction sees a consistent snapshot of the database at a point in time
  • Locking (FOR UPDATE, FOR SHARE) is still needed for write-write concurrency
Without explicit locking (MVCC alone):

T1: BEGIN; reads row A (balance=1000) via snapshot
T2: BEGIN; reads row A (balance=1000) via snapshot
T1: UPDATE balance = balance - 200 → writes new version (balance=800)
T1: COMMIT
T2: UPDATE balance = balance - 300 → writes new version based on T2's SNAPSHOT (balance=700)
    ← T2 read balance=1000, computed 1000-300=700, wrote 700
    ← T1's deduction of 200 is LOST
T2: COMMIT
Final: balance=700 (should be 500)

This is the "Lost Update" problem — MVCC alone doesn't prevent it.
SELECT FOR UPDATE is needed to prevent lost updates.
With SELECT FOR UPDATE (2PL + MVCC):

T1: BEGIN; SELECT * FROM accounts WHERE id=1 FOR UPDATE → exclusive lock
T2: BEGIN; SELECT * FROM accounts WHERE id=1 FOR UPDATE → BLOCKED

T1: balance=1000. UPDATE SET balance=800. COMMIT → releases lock.
T2: UNBLOCKED → reads CURRENT value (800). UPDATE SET balance=500. COMMIT.

Final: balance=500 ✅ Correct.

4. The Example — Banking Scenario (Alice, Bob, Charlie)

Timeline Walkthrough

Resource: account_balance

Time ──────────────────────────────────────────────────────────────────→

ALICE: T1 (deposit $200)
  BEGIN
  UPDATE account SET balance = balance + 200 WHERE user='alice'
  → Acquires EXCLUSIVE lock on Alice's balance row
  COMMIT → releases exclusive lock
  
ALICE: T2 (long reporting job)
  BEGIN
  SELECT balance FROM account WHERE user='alice'  [shared lock acquired]
  SELECT ... (long running reads)
  → Holds SHARED lock on Alice's balance row
  
BOB: T3 (reporting job)
  BEGIN
  SELECT balance FROM account WHERE user='bob'  [shared lock acquired]
  → Holds SHARED lock on Bob's balance row

CHARLIE: T4 (transfer $300 to Bob)
  BEGIN
  UPDATE account SET balance = balance + 300 WHERE user='bob'
  → Attempts EXCLUSIVE lock on Bob's balance row
  → BLOCKED: Bob's T3 holds a SHARED lock
  → Charlie WAITS

BOB T3: completes, COMMIT → releases shared lock on Bob's row
CHARLIE T4: UNBLOCKED
  → Acquires EXCLUSIVE lock on Bob's row
  UPDATE account SET balance = balance + 300 WHERE user='bob'
  COMMIT → releases exclusive lock

Result: Bob gets his $300. No data inconsistency.

Real-World Analogy

Bank of America "Service Unavailable at 2 AM":

Reason: Nightly batch jobs run at midnight.
These jobs acquire shared locks on millions of account rows for reporting.
While those shared locks are held, nobody can acquire exclusive locks.
Therefore: no deposits, no withdrawals, no transfers.
The website grays out transaction options during this window.

This is NOT a bug — it's the correct behavior of locking.
Eventual consistency vs strong consistency trade-off in action.

5. Practical Java Implementation

5.1 — Shared Lock (SELECT FOR SHARE)

/**
 * Acquire a SHARED lock on an account row for a read operation.
 * Use when: you need the value to remain stable during a multi-step read process.
 * Effect: Prevents anyone from updating this row while you hold the lock.
 *         Multiple shared locks can coexist.
 *         Released when transaction ends.
 */
@Transactional(isolation = Isolation.READ_COMMITTED)
public BigDecimal getAccountBalanceStable(long accountId, JdbcTemplate jdbc) {
    // FOR SHARE = acquire shared lock on this row
    // Nobody can UPDATE/DELETE this row until our transaction ends
    Map<String, Object> result = jdbc.queryForMap(
        "SELECT balance FROM accounts WHERE id = ? FOR SHARE",
        accountId
    );
    // Safe to do complex calculations here — balance won't change
    return (BigDecimal) result.get("balance");
}

5.2 — Exclusive Lock via SELECT FOR UPDATE (2PL)

/**
 * Two-Phase Locking implementation for seat booking.
 * Prevents double booking by acquiring exclusive lock BEFORE checking availability.
 *
 * Phase 1 (GROWING): Acquire lock via SELECT FOR UPDATE
 * Phase 2 (SHRINKING): Release lock via COMMIT or ROLLBACK
 *
 * The critical insight: acquire the lock BEFORE checking the condition.
 * Reading without locking first allows two transactions to both see "available"
 * and both proceed to book.
 */
@Transactional
public BookingResult bookSeat(long seatId, String userName, JdbcTemplate jdbc) {
    // Phase 1: ACQUIRE EXCLUSIVE LOCK (growing phase)
    // FOR UPDATE = exclusive row lock
    // This will BLOCK if another transaction already has a lock on this row
    List<Map<String, Object>> rows = jdbc.queryForList(
        "SELECT id, is_booked, name FROM seats WHERE id = ? FOR UPDATE",
        seatId
    );

    if (rows.isEmpty()) {
        return BookingResult.NOT_FOUND;
    }

    Map<String, Object> seat = rows.get(0);
    boolean isBooked = (Boolean) seat.get("is_booked");

    if (isBooked) {
        // Seat is already taken — return failure
        // Phase 2 begins: ROLLBACK releases the exclusive lock
        // TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
        return BookingResult.ALREADY_BOOKED;
    }

    // Seat is available — book it
    jdbc.update(
        "UPDATE seats SET is_booked = true, name = ? WHERE id = ?",
        userName, seatId
    );

    // Phase 2: COMMIT releases the exclusive lock (shrinking phase)
    // The @Transactional annotation triggers commit here
    return BookingResult.SUCCESS;
}

public enum BookingResult {
    SUCCESS, ALREADY_BOOKED, NOT_FOUND
}

5.3 — Deadlock Prevention Pattern

/**
 * Deadlock prevention: always acquire locks in a consistent order.
 *
 * Deadlock scenario (wrong):
 *   T1: lock account 100, then lock account 200
 *   T2: lock account 200, then lock account 100
 *   → Circular wait → DEADLOCK
 *
 * Prevention: always lock in ascending ID order.
 *   T1: lock account 100, then lock account 200
 *   T2: lock account 100 (blocks), then lock account 200
 *   → No circular wait → No deadlock
 */
@Transactional
public void transferFunds(long fromAccountId, long toAccountId,
                          BigDecimal amount, JdbcTemplate jdbc) {

    // CRITICAL: Always lock in ascending ID order to prevent deadlock
    long firstLock  = Math.min(fromAccountId, toAccountId);
    long secondLock = Math.max(fromAccountId, toAccountId);

    // Acquire locks in consistent order
    jdbc.queryForList(
        "SELECT id, balance FROM accounts WHERE id IN (?, ?) ORDER BY id FOR UPDATE",
        firstLock, secondLock
    );

    // Now safely perform the transfer
    // Deduct from source
    int updated = jdbc.update(
        "UPDATE accounts SET balance = balance - ? WHERE id = ? AND balance >= ?",
        amount, fromAccountId, amount
    );

    if (updated == 0) {
        throw new InsufficientFundsException("Insufficient balance in account " + fromAccountId);
    }

    // Add to destination
    jdbc.update(
        "UPDATE accounts SET balance = balance + ? WHERE id = ?",
        amount, toAccountId
    );

    // COMMIT: releases both locks atomically
}

5.4 — Deadlock Retry Pattern

/**
 * Since deadlocks can still occur despite prevention attempts,
 * always implement retry logic for transactional operations.
 *
 * PostgreSQL deadlock error: SQLState "40P01"
 * Generic serialization failure: SQLState "40001"
 */
public BookingResult bookSeatWithRetry(long seatId, String userName,
                                        JdbcTemplate jdbc, int maxRetries) {
    int attempt = 0;

    while (attempt < maxRetries) {
        try {
            return bookSeat(seatId, userName, jdbc);

        } catch (DataAccessException e) {
            String sqlState = ((SQLException) e.getCause()).getSQLState();

            if ("40P01".equals(sqlState)) {
                // Deadlock detected — retry
                attempt++;
                log.warn("Deadlock on attempt {} for seat {}. Retrying...", attempt, seatId);

                try {
                    // Brief pause before retry (exponential backoff is better in production)
                    Thread.sleep(50L * attempt);
                } catch (InterruptedException ie) {
                    Thread.currentThread().interrupt();
                    throw new RuntimeException("Interrupted during retry", ie);
                }

            } else {
                // Non-deadlock error — don't retry
                throw e;
            }
        }
    }

    throw new RuntimeException("Failed to book seat after " + maxRetries + " attempts");
}

5.5 — Monitoring Lock Contention in Production

/**
 * Query PostgreSQL's pg_locks view to monitor lock contention.
 * Run this when queries are slower than expected.
 */
public void monitorLockContention(JdbcTemplate jdbc) {
    // Find currently blocked queries
    List<Map<String, Object>> blockedQueries = jdbc.queryForList("""
        SELECT
            blocked.pid               AS blocked_pid,
            blocked_activity.query    AS blocked_query,
            blocked_activity.wait_event_type,
            blocked_activity.wait_event,
            blocking.pid              AS blocking_pid,
            blocking_activity.query   AS blocking_query,
            now() - blocked_activity.query_start AS wait_duration
        FROM pg_locks blocked
        JOIN pg_stat_activity blocked_activity
            ON blocked_activity.pid = blocked.pid
        JOIN pg_locks blocking
            ON blocking.relation = blocked.relation
            AND blocking.granted = true
            AND blocked.granted = false
        JOIN pg_stat_activity blocking_activity
            ON blocking_activity.pid = blocking.pid
        WHERE NOT blocked.granted
        ORDER BY wait_duration DESC
        """);

    for (Map<String, Object> row : blockedQueries) {
        log.warn("Query blocked: pid={}, wait={}s, blocked_by=pid={}, blocking_query={}",
            row.get("blocked_pid"),
            row.get("wait_duration"),
            row.get("blocking_pid"),
            row.get("blocking_query")
        );
    }
}

/**
 * Check lock wait timeout configuration.
 * Prevents indefinite blocking in production.
 */
public void configureLockTimeout(JdbcTemplate jdbc) {
    // If lock cannot be acquired within 5 seconds, fail with ERROR
    // Prevents cascading slow queries in high-contention scenarios
    jdbc.execute("SET lock_timeout = '5s'");

    // If a statement is blocked for more than 10 seconds total, cancel it
    jdbc.execute("SET statement_timeout = '10s'");
}

6. Practical SQL Examples

Shared Lock

-- Acquire shared lock — prevents writes, allows concurrent reads
BEGIN;
SELECT balance
FROM accounts
WHERE user_id = 42
FOR SHARE;

-- Do complex multi-step calculations here...
-- balance row is guaranteed unchanged throughout this transaction

COMMIT;  -- Releases shared lock

Exclusive Lock (Two-Phase Locking)

-- Phase 1: ACQUIRE — Get exclusive lock first, then check condition
BEGIN;

-- SELECT FOR UPDATE acquires exclusive row lock immediately
-- If another transaction holds any lock on this row: BLOCKED here
SELECT id, is_booked, occupant_name
FROM seats
WHERE seat_id = 14
FOR UPDATE;

-- Now safely check and update
-- No other transaction can read (with lock) or write this row
UPDATE seats
SET is_booked = true,
    occupant_name = 'Hussein'
WHERE seat_id = 14
  AND is_booked = false;  -- Double-check in UPDATE for extra safety

-- Phase 2: RELEASE — COMMIT releases all locks
COMMIT;

Skip Locked (Non-blocking Queue Pattern)

-- SKIP LOCKED: Skip rows that are already locked instead of blocking
-- Useful for job queues: multiple workers process different jobs simultaneously

-- Worker 1: picks up next available job (skips any already being processed)
BEGIN;
SELECT id, payload
FROM job_queue
WHERE status = 'pending'
ORDER BY created_at
LIMIT 1
FOR UPDATE SKIP LOCKED;

UPDATE job_queue SET status = 'processing' WHERE id = <fetched_id>;
COMMIT;

-- Worker 2: runs simultaneously, gets a DIFFERENT job (not blocked by Worker 1)
-- SKIP LOCKED skips rows locked by Worker 1 and finds next available

Lock Timeout Configuration

-- Set per-session lock timeout (fail if lock not acquired in 3 seconds)
SET lock_timeout = '3s';

-- Now any lock acquisition attempt that blocks for > 3 seconds will fail:
-- ERROR: canceling statement due to lock timeout

-- Better than waiting indefinitely in production

Check Lock Status

-- See all current locks in the system
SELECT
    pg_locks.pid,
    pg_locks.locktype,
    pg_locks.relation::regclass AS table_name,
    pg_locks.mode,
    pg_locks.granted,
    pg_stat_activity.query,
    pg_stat_activity.state
FROM pg_locks
JOIN pg_stat_activity ON pg_locks.pid = pg_stat_activity.pid
WHERE pg_locks.relation IS NOT NULL
ORDER BY pg_locks.granted, pg_locks.pid;

-- granted=true  → lock is held
-- granted=false → transaction is waiting for this lock

7. Important Interview Questions

Basic

Q: What is a shared lock?
A: A shared lock (read lock) is acquired by a transaction that wants to read a value and ensure nobody modifies it during that read. Multiple transactions can hold shared locks on the same resource simultaneously. A shared lock blocks exclusive locks — no one can write to a resource that has any shared lock. It does not block other readers.

Q: What is an exclusive lock?
A: An exclusive lock (write lock) is acquired by a transaction that wants to write to a resource. Only one transaction can hold an exclusive lock at a time. An exclusive lock blocks both shared locks (readers) and other exclusive locks (writers). No other transaction can read or write the resource while an exclusive lock is held.

Q: Can two transactions hold shared locks on the same row simultaneously?
A: Yes. Shared locks are compatible with each other — multiple readers can coexist. This is by design: concurrent reads do not conflict with each other. Only when a writer wants an exclusive lock does it conflict with existing shared locks.

Q: What is a deadlock?
A: A deadlock occurs when two or more transactions are each waiting for the other to release a lock, creating a circular dependency. Transaction A holds a lock on resource X and wants resource Y; Transaction B holds a lock on resource Y and wants resource X. Neither can proceed. PostgreSQL's background deadlock detector identifies the cycle and rolls back the transaction that "closed" the cycle (the last one to create the dependency), allowing the other to continue.

Q: What is two-phase locking?
A: Two-phase locking (2PL) is a concurrency control protocol with two phases: (1) Growing phase — acquire locks, cannot release any; (2) Shrinking phase — release locks, cannot acquire any new ones. The constraint "once you release, you cannot acquire" ensures that all locks are held simultaneously at the peak, which guarantees serializability — results equivalent to some serial execution order.

Intermediate

Q: What is the double booking problem and how does SELECT FOR UPDATE solve it?
A: Double booking occurs when two transactions concurrently read a resource as "available" and both proceed to book it. Under MVCC, both reads succeed simultaneously, both see "available," and both commit updates — the second commit overwrites the first. SELECT FOR UPDATE solves this by acquiring an exclusive row lock during the SELECT. The second transaction's SELECT FOR UPDATE blocks until the first transaction commits or rolls back. When unblocked, the second transaction reads the current (post-commit) state — seeing the resource is already booked — and correctly rejects the booking.

Q: What is the difference between MVCC and locking, and when do you need both?
A: MVCC prevents readers from blocking writers and vice versa by maintaining multiple row versions (each transaction sees a snapshot). However, MVCC alone does not prevent lost updates (write-write conflicts). Two transactions can both read the same value via their snapshot, compute a new value, and both write — the second write overwrites the first (lost update). Explicit locking (SELECT FOR UPDATE) is needed for write-write conflicts. In PostgreSQL: MVCC handles read-write concurrency; explicit locking handles write-write concurrency.

Q: How does PostgreSQL decide which transaction to kill during a deadlock?
A: PostgreSQL's lock monitor builds a "wait-for graph" — a directed graph where an edge from A to B means "A is waiting for a lock held by B." A deadlock is a cycle in this graph. PostgreSQL identifies the cycle and selects the transaction that "closed" the cycle (the most recent transaction to enter the wait state that created the cycle). This transaction is the victim — it receives a deadlock error and is rolled back. The choice is deterministic and based on which transaction most recently became part of the circular dependency.

Q: What is the difference between FOR UPDATE and FOR SHARE?
A: FOR UPDATE acquires an exclusive row lock — no other transaction can read (with any row lock) or write the row. FOR SHARE acquires a shared row lock — other transactions can also acquire shared locks (multiple readers), but writers (FOR UPDATE, UPDATE, DELETE) are blocked. Use FOR UPDATE when you intend to update the row. Use FOR SHARE when you need the value stable for reading but don't intend to update it yourself.

Advanced

Q: Explain why acquiring locks in a consistent order prevents deadlocks. What happens if you don't?
A: Deadlock requires a circular wait: A holds X and wants Y; B holds Y and wants X. If all transactions always acquire locks in the same canonical order (e.g., ascending primary key), circular waits are impossible. Suppose A needs accounts 100 and 200: it acquires 100 first, then 200. Suppose B also needs 100 and 200: it also acquires 100 first — but A already holds 100, so B blocks. A then gets 200, completes, releases 100. B gets 100, then 200. No cycle. Without ordering: A gets 100, B gets 200, A wants 200 (blocked by B), B wants 100 (blocked by A) → deadlock.

Q: A cinema booking system processes 10,000 concurrent seat booking requests for a sold-out concert release. Walk through the complete locking strategy.
A: (1) Connection pooling: Limit connections to the DB — 10,000 HTTP requests do not mean 10,000 DB connections. Use a connection pool (HikariCP, PgBouncer). (2) SELECT FOR UPDATE: Each booking attempt acquires an exclusive row lock on the target seat. (3) SKIP LOCKED alternative: For a queue-based approach, workers use FOR UPDATE SKIP LOCKED to grab available seats without blocking each other. (4) Lock timeout: Set lock_timeout = '5s' to fail fast rather than pile up waiting connections. (5) Retry logic: Implement exponential backoff with retry for deadlock errors (SQLState 40P01). (6) Optimistic locking alternative: Use a version counter column; the UPDATE includes WHERE version = read_version; if 0 rows updated, retry (optimistic concurrency without blocking locks). (7) Horizontal partitioning: Partition seats table by venue/event so lock contention is isolated per event, not table-wide.

Q: What is SKIP LOCKED and when would you use it over standard FOR UPDATE?
A: SELECT ... FOR UPDATE SKIP LOCKED acquires an exclusive lock on rows that are immediately lockable and silently skips any row that is already locked. Standard FOR UPDATE blocks until all requested locks are available. SKIP LOCKED is ideal for work queues where multiple workers process jobs in parallel — each worker gets a different subset of available work without any worker blocking another. With standard FOR UPDATE, all workers would block on the same "next available" row, serializing what should be parallel work. Important caveat: SKIP LOCKED is not suitable when you need to process specific rows — it may result in rows being skipped indefinitely if other transactions hold long locks.

8. Common Mistakes

Beginner Mistakes

  • Reading without locking when you plan to write: Reading a value without FOR UPDATE, computing a new value, then updating — allows another transaction to change the value between your read and your write (lost update / race condition)
  • Confusing MVCC with locking: Assuming PostgreSQL's MVCC prevents all concurrency issues — MVCC prevents dirty reads but NOT lost updates in write-write scenarios
  • Not handling deadlock errors in application code: Letting a deadlock error bubble up to the user without retry logic

Backend Engineer Mistakes

  • Inconsistent lock acquisition order: Different code paths lock rows in different orders (e.g., sometimes account A then B, sometimes B then A) — guarantees eventual deadlock under concurrent load
  • Long transactions holding locks: Starting a transaction, acquiring a lock via SELECT FOR UPDATE, then doing expensive external operations (HTTP calls, file I/O) before committing — lock held for seconds or minutes, blocking all other transactions on that row
  • Over-locking: Using FOR UPDATE when FOR SHARE is sufficient, or locking the entire table when row-level locking suffices — unnecessarily reduces concurrency
  • Not setting lock_timeout: In production, a transaction waiting indefinitely for a lock causes cascading slowdowns (connections pile up, timeouts propagate)

Production Mistakes

  • No lock timeout: A runaway transaction holds a lock indefinitely. All other transactions waiting on that lock pile up. Connection pool exhausted. Service unavailable.
  • Not monitoring pg_locks: Lock contention is invisible until queries slow down dramatically. Regular monitoring of pg_locks and pg_stat_activity catches problems before they become incidents.
  • Batch operations in a single transaction: UPDATE 10 million rows in one transaction holds exclusive locks on 10 million rows for potentially minutes — blocks all other access to those rows. Use batched updates (10,000 rows per transaction).
  • Deadlock in application code that should have been prevented: Two code paths that update the same two resources in opposite orders — a time bomb that deadlocks under concurrent load. Always define and enforce a canonical lock acquisition order.

9. Performance & Scalability Notes

Lock Contention Impact on Throughput

Scenario: 100 concurrent users booking cinema seats

WITHOUT proper locking (double booking allowed, no FOR UPDATE):
  All 100 requests proceed simultaneously.
  Throughput: 100 req/sec
  Correctness: ❌ Multiple people book same seat

WITH FOR UPDATE (correct 2PL):
  For a single popular seat:
    First request acquires lock.
    Other 99 requests block waiting.
    Serialized: ~1 req/sec for that seat
    Correctness: ✅ One person gets the seat

For different seats (no contention):
  100 requests for 100 different seats proceed simultaneously.
  Throughput: 100 req/sec (no contention)
  Correctness: ✅

Conclusion: Locking overhead is proportional to contention.
            Low-contention systems: negligible performance impact.
            High-contention (same row): throughput limited to 1/lock_duration.

Lock Duration = Everything

Lock duration minimization rules:

❌ Bad pattern:
BEGIN;
SELECT * FROM seats WHERE id=14 FOR UPDATE;  ← lock acquired
// Do HTTP call to payment service (2 seconds)
// Parse response
UPDATE seats SET booked=true WHERE id=14;
COMMIT;  ← lock released after 2+ seconds

✅ Good pattern:
// Do payment service call BEFORE transaction
paymentResult = paymentService.charge(userId, amount);

BEGIN;
SELECT * FROM seats WHERE id=14 FOR UPDATE;  ← lock acquired
UPDATE seats SET booked=true WHERE id=14;
COMMIT;  ← lock released in < 10ms

Difference: 2000ms lock vs 10ms lock.
At 100 req/sec: 200 queued requests vs 1 queued request.

Optimistic vs Pessimistic Locking

Pessimistic (FOR UPDATE)Optimistic (Version Counter)
Lock acquisitionAt read time (SELECT FOR UPDATE)None — no locks taken
Conflict detectionBlocking (waits for lock)At write time (UPDATE fails if version changed)
Best forHigh contention (many conflicts expected)Low contention (conflicts rare)
Throughput under no contentionSameSame
Throughput under high contentionBetter (one waits, others proceed)Worse (many retries, wasted work)
ImplementationSimple (database handles)More complex (version column, retry logic)
 
-- Optimistic locking with version counter
CREATE TABLE seats (
    id INTEGER PRIMARY KEY,
    is_booked BOOLEAN,
    version INTEGER DEFAULT 0  ← version column
);

-- Read
SELECT id, is_booked, version FROM seats WHERE id = 14;
-- Returns: is_booked=false, version=5

-- Update (fails if someone else updated since our read)
UPDATE seats
SET is_booked = true, version = version + 1
WHERE id = 14
  AND version = 5;  ← Must match what we read

-- If 0 rows updated: conflict → retry
-- If 1 row updated: success

10. System Design Connections

Locking + Distributed Systems

In a distributed (sharded/microservice) setup, traditional DB locks don't work
across services. Alternatives:

1. Distributed Lock (Redis Redlock):
   Client acquires Redis lock before accessing shared resource.
   Lock has TTL to prevent indefinite hold on crash.
   
   try:
     lock = redis.set("seat:14:lock", uuid, NX=True, EX=5)  # 5 second TTL
     if lock:
       process_booking()
     else:
       return "Seat already being processed"
   finally:
     redis.delete("seat:14:lock")  # Release

2. Database-level advisory locks (PostgreSQL):
   pg_try_advisory_lock(key) → application-managed lock
   Not tied to a specific row, flexible resource identification

3. Optimistic locking with versioning:
   No distributed lock needed.
   Each service reads version, writes conditionally.
   Conflicts result in retry.

Locking + ACID

Locking is how databases implement the "I" in ACID (Isolation):

Isolation levels and what locks they use:

READ UNCOMMITTED: Barely any locks. Can read uncommitted data (dirty read).
READ COMMITTED:   Shared lock released immediately after each read.
                  Exclusive lock held until commit.
REPEATABLE READ:  Shared lock held for entire transaction duration.
                  Exclusive lock held until commit.
SERIALIZABLE:     Full 2PL or predicate locks. No phantoms possible.
                  Most restrictive, least concurrent.

PostgreSQL default: READ COMMITTED
Banking/finance:    SERIALIZABLE or REPEATABLE READ
Most web apps:      READ COMMITTED (MVCC handles most cases)
