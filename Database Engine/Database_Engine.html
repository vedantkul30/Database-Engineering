Production Engineering Notes

1. Big Picture

What Is a Database Storage Engine?

A database storage engine (also called an embedded database or storage layer) is a library responsible for:

  • How data is physically stored on disk (file format, page layout)
  • How data is read from disk (index structures, scan strategies)
  • CRUD operations at the lowest level
  • Whether ACID transactions are supported
  • Lock granularity (row-level vs table-level)
Without separating concerns (monolithic database):

┌──────────────────────────────────────────────────────────┐
│  DBMS (Everything mixed together)                        │
│  Network/client protocol + Query parser + Optimizer +    │
│  Transaction manager + Storage + Index management        │
└──────────────────────────────────────────────────────────┘
  ↑ Cannot reuse storage layer. Cannot swap implementations.

With separation of concerns:

┌─────────────────────────────────────────────────┐
│  DBMS Layer (Database Management System)         │
│  - Client/server protocol (TCP, ports)           │
│  - SQL parser and query optimizer                │
│  - Replication (master/follower)                 │
│  - Stored procedures, foreign keys              │
│  - Connection pooling                            │
└──────────────────────┬──────────────────────────┘
                       │ uses
┌──────────────────────▼──────────────────────────┐
│  Storage Engine Layer                            │
│  - Physical file layout on disk                 │
│  - Index data structures (B-tree, LSM)          │
│  - ACID transactions (or not)                   │
│  - Lock management                              │
│  - Read/write optimizations                     │
└─────────────────────────────────────────────────┘
  ↑ Can be swapped. Reusable. Purpose-built for workload.

Why This Matters for Engineers

  • MySQL lets you swap storage engines per table — different tables can have different engines
  • PostgreSQL does not — always B-tree based, cannot change
  • The engine choice determines: insert speed, read speed, ACID support, lock granularity, SSD friendliness
  • Building a new database? Pick an existing engine and build your DBMS layer on top — don't start from scratch

The Two Fundamental Storage Paradigms

B-Tree based (read-optimized):
  MySQL InnoDB, MyISAM, PostgreSQL, SQLite, Oracle, SQL Server
  Best for: Mixed read/write, OLTP, strong consistency requirements

LSM-Tree based (write-optimized):
  LevelDB, RocksDB, Cassandra, HBase, ScyllaDB, InfluxDB, Elasticsearch
  Best for: High-insert workloads, time-series, log storage, SSD-optimized writes

3. Core Concepts — Each Engine in Depth

3.1 — MyISAM (Indexed Sequential Access Method)

MyISAM was the default MySQL storage engine until version 5.5. It was eventually replaced by InnoDB as the default.

Architecture

MyISAM file structure per table:
  tablename.MYD  → actual row data (heap file)
  tablename.MYI  → index file (B-tree index structure)
  tablename.frm  → table definition / schema

Every index (including secondary indexes) points DIRECTLY to the
physical row location on disk (byte offset in .MYD file).

Index → (file offset 700) → row data at byte 700 in .MYD

Why Inserts Are Fast

INSERT appends to the END of the .MYD heap file.
The file's end position is always known → O(1) append.
New index entries are added pointing to the new offset.

No tree rebalancing required for sequential inserts.
No transaction log overhead.

Why Updates and Deletes Are Slow

UPDATE changes a row → row may grow or shrink in size.
Changed row size → byte offset of ALL subsequent rows changes.
Changed offsets → ALL indexes pointing to ANY row after this one must be updated.

This is called FRAGMENTATION.

Example:
  Row at offset 700 (50 bytes). Update adds 10 bytes.
  Row is now at offset 700 (60 bytes).
  Next row was at offset 750 → now at offset 760.
  Index pointing to next row: was 750 → must update to 760.
  CASCADE: every subsequent row's offset shifts.
  → Update ALL affected index entries → SLOW.

No Transactions

-- MyISAM demo (MySQL):
BEGIN;  -- Starts a "transaction"
INSERT INTO employees_myisam (name) VALUES ('Hussein');
-- At this point, without committing:

-- Another client immediately sees the row!
SELECT * FROM employees_myisam;
-- → Returns Hussein. Already visible. No isolation.

-- ROLLBACK is meaningless:
ROLLBACK;
-- → Row is STILL there. Cannot undo.
-- MyISAM ignores BEGIN/COMMIT/ROLLBACK entirely.

Why no transactions: MyISAM was designed for speed and simplicity. The overhead of transaction logging and rollback support was intentionally omitted.

Table-Level Locking

MyISAM locking:
  Any write (INSERT/UPDATE/DELETE) locks the ENTIRE table.
  Concurrent readers on the same table: BLOCKED.
  Concurrent writers on the same table: BLOCKED.

InnoDB locking:
  Any write locks only the SPECIFIC ROW being written.
  Other rows: fully accessible to concurrent readers and writers.

Impact on concurrent booking system (100 users):
  MyISAM: User 1 books a seat → all other 99 users BLOCKED from reading the table
  InnoDB: User 1 books seat 15 → other 99 users booking other seats proceed normally

Database Corruption

Crash during MyISAM write → corrupt table possible:
  Reason: Indexes store physical file offsets.
          If a write updates the data file but crashes before updating indexes,
          indexes now point to wrong locations.
          Data and indexes are out of sync.

Recovery: REPAIR TABLE tablename;
          (Rebuilds indexes from scratch — can take hours on large tables)

InnoDB: Uses WAL (Write-Ahead Log). On crash, replay WAL to restore consistency.
        No corruption. Automatic recovery.

When to Use MyISAM

✅ Appropriate when:
  - Read-heavy workload with almost no writes
  - Full-text search (MyISAM historically had better FTS, though InnoDB 5.6+ improved)
  - No concurrent writers needed
  - Absolute simplest possible storage needs
  - Legacy systems that cannot be migrated

❌ Never use when:
  - Transactions required (banking, inventory, anything financial)
  - Concurrent writes needed
  - Row-level lock control needed
  - Crash safety matters (production systems)

3.2 — InnoDB

InnoDB is the default MySQL/MariaDB storage engine since MySQL 5.5. It is a B+ Tree clustered index engine with full ACID support.

Architecture

InnoDB file structure:
  ibdata1       → shared tablespace (system tables, undo log, insert buffer)
  tablename.ibd → per-table data + index file (when innodb_file_per_table=ON)
  ib_logfile0   → WAL (redo log file)
  ib_logfile1   → WAL (redo log file, second file)

Primary Index = The Table (Clustered Index / Index-Organized Table):
  Table IS the B+ Tree ordered by primary key.
  Leaf nodes contain the FULL ROW (all columns).
  Physical row order = primary key order.

Secondary indexes:
  Do NOT point to physical row location.
  Point to the PRIMARY KEY value instead.
  To get full row: secondary index → PK → primary B+ Tree → full row.
  (Double lookup — but no offset invalidation on row movement)

Full ACID Support

-- InnoDB transaction demo (MySQL):
BEGIN;
INSERT INTO employees_innodb (name) VALUES ('Hussein');

-- Another client cannot see this yet:
-- SELECT * FROM employees_innodb → empty (isolation working correctly)

-- Within same transaction, you CAN see your own changes:
SELECT * FROM employees_innodb;  -- → Returns Hussein (your own view)

-- ROLLBACK works:
ROLLBACK;
-- → Row is GONE. Transaction fully rolled back.

-- Or COMMIT makes it visible to all:
-- COMMIT;  → all clients now see Hussein

Row-Level Locking

InnoDB: Lock granularity = individual row
  Transaction A writes row 15 → only row 15 is locked
  Transaction B writes row 20 → proceeds immediately (different row)
  Transaction C reads row 15 → blocked (row 15 is exclusively locked)

MyISAM: Lock granularity = entire table
  Transaction A writes row 15 → ENTIRE table is locked
  Transaction B writes row 20 → BLOCKED (table locked, even though different row)

Foreign Key Support

-- InnoDB supports foreign key constraints
CREATE TABLE orders (
    id INT PRIMARY KEY,
    customer_id INT,
    FOREIGN KEY (customer_id) REFERENCES customers(id)
        ON DELETE CASCADE
        ON UPDATE CASCADE
) ENGINE=InnoDB;

-- MyISAM: FOREIGN KEY syntax is parsed but SILENTLY IGNORED
CREATE TABLE orders (
    id INT PRIMARY KEY,
    customer_id INT,
    FOREIGN KEY (customer_id) REFERENCES customers(id)  -- IGNORED in MyISAM!
) ENGINE=MyISAM;

Crash Safety

InnoDB crash recovery:
  Every write goes to WAL (redo log) BEFORE the data file.
  On crash: replay WAL from last checkpoint.
  Uncommitted transactions: rolled back using undo log.
  Result: Database is always in a consistent state after restart.
  No manual repair needed.

The Uber Migration: PostgreSQL → MySQL
One cited reason: PostgreSQL's secondary indexes point to the physical tuple ID (ctid). When a row is updated, it moves to a new physical location — ALL secondary indexes must be updated (write amplification). InnoDB's secondary indexes point to the stable primary key. Row movement doesn't require updating secondary indexes (except the primary key B-tree). For Uber's write-heavy trip update workload, this reduced write amplification.

3.3 — XtraDB (Percona's InnoDB Fork)

Created by Monty Program / Percona as a fork of InnoDB to add enterprise features Oracle didn't open-source.

History:
  Oracle acquired Sun (which owned MySQL + InnoDB)
  Percona: "We don't want to depend on Oracle for storage engine features"
  Created XtraDB: InnoDB + additional performance patches + backup tools

Features added over InnoDB:
  - Better I/O performance under certain workloads
  - More tunable parameters
  - Better hot backup (XtraBackup tool)
  - Enhanced statistics

Problem:
  Oracle kept adding features to InnoDB faster than Percona could keep up.
  XtraDB started falling behind InnoDB in features.

Result:
  MariaDB 10.2: switched DEFAULT back from XtraDB to InnoDB.
  XtraDB now optional in Percona Server.
  "Sometimes it's better to just bite the bullet and go with the mainstream."

3.4 — Aria (MariaDB's MyISAM Replacement)

Created by Monty Widenius (creator of MySQL, founder of MariaDB) as a crash-safe alternative to MyISAM after Oracle acquired MySQL.

Why created:
  Monty forked MySQL → MariaDB
  Needed MyISAM-like engine (fast, simple) but NOT owned by Oracle
  MyISAM had one critical flaw: crashes could corrupt tables
  → Created Aria: MyISAM + crash safety

Key difference from MyISAM:
  Crash safe: Aria uses a transaction log for crash recovery.
              On restart, replays the log to fix corrupted tables automatically.
              No manual REPAIR TABLE needed.

Similar to MyISAM:
  No full ACID transactions (table-level atomicity only)
  Table-level locking (no row-level)
  B-Tree index structure

Current usage:
  MariaDB SYSTEM TABLES use Aria by default (not MyISAM, not InnoDB)
  Reason: System tables don't need transactions; Aria is lightweight and fast
  User tables: InnoDB is the default

3.5 — LevelDB (Google, 2011)

Created by Jeff Dean and Sanjay Ghemawat at Google. A completely different paradigm from B-Trees: the Log-Structured Merge Tree (LSM).

The SSD Problem with B-Trees

B-Tree on SSD — the write amplification problem:

INSERT row → B-Tree may need to rebalance → move existing nodes →
  UPDATE existing data on SSD → SSD writes bits in-place

SSD physics:
  - SSD cells have limited write cycles (P/E cycles: ~3,000-100,000 depending on NAND type)
  - Writing to an already-written cell: must ERASE then WRITE
  - Erase is slow (~1-5ms) and degrades the cell
  - Fragmented writes scatter data across pages → garbage collection overhead
  - In-place updates = worst case for SSD lifespan and performance

Ideal for SSD: Sequential writes only. Never update existing data.

LSM Tree Architecture

LevelDB write path (APPEND ONLY — never updates existing data):

Step 1: WRITE to WAL (Write-Ahead Log) on disk → immediate durability
Step 2: WRITE to MemTable in RAM (sorted, in-memory buffer)

MemTable full? → Step 3: FLUSH to disk as immutable SSTable file (Level 0)

Compaction: Background process merges and compacts SSTables
  Level 0 → Level 1 → Level 2 → ... → Level 6
  Each level is 10× larger than previous
  Compaction merges sorted SSTables, discards obsolete versions
  Output is a new, larger sorted SSTable

NOTHING IS EVER UPDATED IN PLACE. Every write is a new file/entry.
LevelDB levels:

Level 0 (MemTable flushed):    ~4 files, ~4MB each
Level 1:                        10 files, total ~10MB
Level 2:                        total ~100MB
Level 3:                        total ~1GB
Level 4:                        total ~10GB
Level 5:                        total ~100GB
Level 6:                        total ~1TB

As levels fill → background compaction merges them upward

Why "Level" DB?

Named because data flows through levels as it gets compacted from fast/small (memory) to slow/large (disk). Each level is progressively larger and on slower storage.

LevelDB Properties

✅ Pros:
  Fast sequential writes: O(1) — always append to MemTable
  SSD-friendly: no in-place updates, sequential I/O only
  High write throughput

❌ Cons:
  NO ACID transactions: reads see potentially stale data
  Read amplification: reading a key may need to check MemTable + all SSTable levels
  Write amplification from compaction: data rewritten multiple times as it flows down levels
  No SQL: key-value only

Used in: Bitcoin Core, Minecraft Pocket Edition, some AutoCAD versions

3.6 — RocksDB (Facebook, 2012)

Facebook forked LevelDB and added everything that enterprise production systems need.

Why Facebook Built It

Facebook's workload:
  - Billions of users, hundreds of billions of rows
  - Extreme write throughput (status updates, likes, messages)
  - SSD storage (cost-effective at Facebook scale)
  - LevelDB was single-threaded compaction → bottleneck at scale

Facebook's solution: Fork LevelDB, add what's missing.

RocksDB Improvements Over LevelDB

FeatureLevelDBRocksDB
Transactions❌ None✅ Full ACID
Compaction threadsSingle thread✅ Multi-threaded
Column families❌✅ (logical namespaces)
CompressionBasic✅ Per-level (Snappy, Zlib, LZ4, ZSTD)
Bloom filters❌✅ (reduces read amplification)
Rate limiting❌✅ (throttle compaction I/O)
Merge operators❌✅ (atomic read-modify-write)
Statistics & monitoringMinimal✅ Extensive
Backup❌✅ Checkpoint API

MyRocks — RocksDB as a MySQL Storage Engine

Stack:
  MySQL (SQL layer, connection handling, query parsing)
  ↓
  MyRocks (MySQL-RocksDB adapter)
  ↓
  RocksDB (LSM storage engine)
  ↓
  SSD/NVMe storage

Benefits demonstrated by Facebook:
  - ~2× compression vs InnoDB (LSM packs data better)
  - Reduced write amplification vs InnoDB B-tree
  - SSD endurance dramatically improved
  - Similar read performance with bloom filters

Supported by: MySQL (via Percona), MariaDB, Percona Server
Not supported by: Vanilla MySQL (Oracle's distribution)

3.7 — SQLite

Created by D. Richard Hipp in 2000. The most widely deployed database in the world by number of instances.

Design Philosophy

Problem Hipp solved:
  "I want a local database. Why do I need a server?"
  Traditional databases: client → TCP → server → storage
  SQLite: application → library API → storage file (directly)

No server process needed.
No TCP connection needed.
Entire database = a single file.
Import as a library in your code.

SQLite Architecture

Traditional RDBMS:              SQLite:
┌──────────────┐                ┌─────────────────────────────┐
│  Application │                │  Application                │
└──────┬───────┘                │  + SQLite library (linked)  │
       │ TCP connection         │  + Single .db file          │
┌──────▼───────┐                └─────────────────────────────┘
│  DB Server   │                       ↑
│  (separate   │                 No network. No server.
│   process)   │                 Zero configuration.
└──────────────┘

Where SQLite Is Used

Everywhere:
  - Every iOS device (system databases, app data)
  - Every Android device (same)
  - Every Chrome browser (history, bookmarks, web SQL)
  - Every Firefox browser
  - Every macOS/Windows installation
  - Python's built-in database module (import sqlite3)
  - Many embedded systems
  - Development/testing (no database setup needed)

SQLite Properties

✅ Strengths:
  Full ACID transactions (even locally!)
  Zero configuration (no setup, no server)
  Zero dependencies (one C file / library)
  Cross-platform (same .db file works on all OSes)
  B-Tree based (familiar, stable)
  Concurrent reads: multiple readers simultaneously ✅
  WAL mode: concurrent reads while one writer writes ✅

❌ Limitations:
  Table-level locking (no row-level locks)
  Not designed for high-concurrency multi-user write access
  No built-in network server (by design — it's embedded)
  Maximum DB size: 281 TB (theoretical); practical limit ~1TB
  Cannot handle thousands of concurrent writers

3.8 — Berkeley DB (Sleepycat / Oracle)

One of the oldest embedded database engines, predating MySQL. Created by Sleepycat Software in 1994, now owned by Oracle.

Historical significance:
  Influenced many subsequent database designs
  Bitcoin Core used Berkeley DB for blockchain storage until 2012-2014
  Then switched to LevelDB (faster for blockchain's append-heavy workload)

Properties:
  Key-value store (not SQL)
  Embedded (library, no server)
  ACID transactions
  Multiple access methods (B-Tree, Hash, Recno, Queue)
  Locking support

Current status:
  Oracle owns it (same theme — Oracle owns many database foundations)
  Less popular than RocksDB/LevelDB for new projects
  Still used in legacy systems, some Memcached DB implementations

4. B-Tree vs LSM-Tree Deep Comparison

Write Path

B-Tree (InnoDB, MyISAM, PostgreSQL):
  INSERT → find position in B-Tree → insert key into page →
    if page full: page split → update parent → WAL record →
    flush dirty pages to disk

  Characteristics:
    Random writes (page splits scatter across disk)
    In-place updates (bad for SSD P/E cycles)
    Write amplification: 1 logical write → multiple physical writes (splits, parent updates)

LSM-Tree (LevelDB, RocksDB):
  INSERT → WAL append (sequential) → MemTable (in-memory) →
    MemTable flush to SSTable (sequential write) →
    Background compaction (merge sorted files)

  Characteristics:
    Sequential writes only (excellent for SSD throughput)
    No in-place updates (new version written, old version marked obsolete)
    Write amplification from compaction, but sequential I/O

Read Path

B-Tree read:
  SELECT → B-Tree traversal → O(log n) → leaf page → heap fetch
  Fast for point lookups and range scans on indexed columns
  Index in buffer pool → often 1-2 disk I/Os

LSM read:
  SELECT → check MemTable → check Level-0 SSTables → check Level-1 →
           ... → check Level-6

  Worst case: check ALL levels (read amplification)
  Mitigation: Bloom filters per SSTable (eliminate ~99% of unnecessary level checks)
  With bloom filters: point lookup ~3-5 disk I/Os (vs B-Tree ~2-3)
  Range scans: B-Tree faster (contiguous index; LSM must merge sorted levels)

Summary Table

DimensionB-Tree (InnoDB)LSM-Tree (RocksDB)
Write throughputMedium (random I/O for splits)Very High (sequential I/O always)
Read throughputHigh (index traversal + heap)Medium (bloom filters help)
Range scansExcellent (linked leaves)Good (merge scan across levels)
Space amplificationLowHigher (old versions until compaction)
Write amplificationMedium (page splits)Medium-high (compaction rewrites)
SSD friendlinessMedium (in-place updates wear SSD)High (sequential only, no in-place)
ACID (in major engines)✅ InnoDB✅ RocksDB

5. MySQL Engine Switching Demo

MySQL's key differentiator: you can specify different storage engines per table.

Demo SQL

-- Show all available storage engines:
SHOW ENGINES;
-- → InnoDB (default), MyISAM, MEMORY, CSV, ARCHIVE, BLACKHOLE, FEDERATED, etc.
-- → If MyRocks installed: ROCKSDB also appears

-- Create database
CREATE DATABASE test;
USE test;

-- Create MyISAM table
CREATE TABLE employees_myisam (
    id   INT AUTO_INCREMENT PRIMARY KEY,
    name TEXT
) ENGINE=MyISAM;

-- Create InnoDB table
CREATE TABLE employees_innodb (
    id   INT AUTO_INCREMENT PRIMARY KEY,
    name TEXT
) ENGINE=InnoDB;

-- Create RocksDB table (if MyRocks installed)
CREATE TABLE employees_rocksdb (
    id   INT AUTO_INCREMENT PRIMARY KEY,
    name TEXT
) ENGINE=ROCKSDB;

-- Verify engine per table:
SELECT table_name, engine
FROM information_schema.tables
WHERE table_schema = 'test';
-- → employees_myisam → MyISAM
-- → employees_innodb → InnoDB

Transaction Behavior Comparison

-- ─── MyISAM transaction test ───────────────────────────────────────────────
BEGIN;
INSERT INTO employees_myisam (name) VALUES ('Hussein');
-- MyISAM ignores BEGIN. Row is IMMEDIATELY visible to ALL clients.
-- No isolation. No rollback possible.
ROLLBACK;  -- Does NOTHING. Row is still there.

SELECT * FROM employees_myisam;
-- → Hussein is there. Rollback had no effect.


-- ─── InnoDB transaction test ────────────────────────────────────────────────
BEGIN;
INSERT INTO employees_innodb (name) VALUES ('Hussein');
-- Row is NOT visible to other clients yet.
-- Within this connection, you CAN see your own change:
SELECT * FROM employees_innodb;  -- → Returns Hussein (your own view)

-- From another client/connection:
-- SELECT * FROM employees_innodb; → empty (not committed yet)

ROLLBACK;
-- Row is GONE. Fully rolled back.
SELECT * FROM employees_innodb;  -- → empty. Clean.

-- Now with COMMIT:
BEGIN;
INSERT INTO employees_innodb (name) VALUES ('Edmund');
COMMIT;  -- Now visible to all clients.

Java Implementation — Switching Engines per Table

@Component
public class StorageEngineDemo {

    private final JdbcTemplate jdbcTemplate;

    /**
     * InnoDB: transactional data (orders, accounts, inventory)
     * MyISAM: legacy read-only reference data (rarely updated lookups)
     * MEMORY: session data that doesn't need persistence
     * ARCHIVE: historical audit logs (insert-only, heavily compressed)
     */
    public void createTables() {
        // Critical transactional data → InnoDB (ACID, row-level locks)
        jdbcTemplate.execute("""
            CREATE TABLE IF NOT EXISTS orders (
                id BIGINT AUTO_INCREMENT PRIMARY KEY,
                customer_id BIGINT NOT NULL,
                total DECIMAL(10,2) NOT NULL,
                status VARCHAR(20) NOT NULL,
                created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
            ) ENGINE=InnoDB
            """);

        // Read-heavy reference data with rare updates → MyISAM
        jdbcTemplate.execute("""
            CREATE TABLE IF NOT EXISTS country_codes (
                code CHAR(2) PRIMARY KEY,
                name VARCHAR(100) NOT NULL,
                region VARCHAR(50)
            ) ENGINE=MyISAM
            """);

        // High-speed session storage in RAM → MEMORY engine
        jdbcTemplate.execute("""
            CREATE TABLE IF NOT EXISTS active_sessions (
                session_id VARCHAR(64) PRIMARY KEY,
                user_id BIGINT NOT NULL,
                expires_at TIMESTAMP NOT NULL
            ) ENGINE=MEMORY
            """);

        // Audit log (insert-only, must compress well) → ARCHIVE
        jdbcTemplate.execute("""
            CREATE TABLE IF NOT EXISTS audit_log (
                id BIGINT AUTO_INCREMENT PRIMARY KEY,
                action VARCHAR(100) NOT NULL,
                user_id BIGINT NOT NULL,
                details TEXT,
                occurred_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
            ) ENGINE=ARCHIVE
            """);
    }

    /**
     * Alter an existing table to use a different engine (online DDL).
     * WARNING: This copies the entire table — can be slow on large tables.
     * Use pt-online-schema-change (Percona toolkit) for large tables.
     */
    public void migrateTableEngine(String tableName, String newEngine) {
        jdbcTemplate.execute(
            "ALTER TABLE " + tableName + " ENGINE=" + newEngine
        );
    }
}

6. The History of MySQL's Default Engine

Timeline:

2000:   MySQL default engine → MyISAM
        Reason: Fast inserts, simple, lightweight.
        Problem: No transactions, no row locks, crashes corrupt tables.

2001-2010: InnoDB available as option.
           Manually specified: ENGINE=InnoDB
           Many developers didn't know they needed it.
           Legacy apps used MyISAM by default → transaction bugs in production.

MySQL 5.5 (2010): DEFAULT engine changed to InnoDB.
           Reason: MyISAM's limitations caused too many production issues.
           "Most applications NEED transactions, even if developers don't realize it."

MariaDB history:
  2009:   Monty forks MySQL → creates MariaDB
  2009-2017: MariaDB default = XtraDB (Percona's InnoDB fork)
  2017 (MariaDB 10.2): Switched back to InnoDB
          Reason: XtraDB couldn't keep pace with Oracle's InnoDB development.

The XtraDB lesson:
  "Fighting against a well-funded company on their core product is hard.
   Sometimes you have to accept the mainstream option and focus elsewhere."

  Result: Oracle owned InnoDB before owning MySQL

The irony: Oracle owned the best MySQL storage engine (InnoDB)
           before owning MySQL itself.

8. When to Use Which Engine (Decision Guide)

USE InnoDB when:
  ✅ Any application requiring ACID transactions
  ✅ Banking, e-commerce, inventory, booking systems
  ✅ Concurrent users writing to same tables
  ✅ Foreign key constraints needed
  ✅ Default choice for MySQL (safe, battle-tested)

USE MyISAM when:
  ✅ Legacy compatibility (existing codebase uses it)
  ✅ Read-only reference tables updated rarely via batch jobs
  ✅ Full-text search on MySQL < 5.6 (InnoDB FTS added in 5.6)
  ❌ NEVER for new transactional applications

USE RocksDB / MyRocks when:
  ✅ Extreme write throughput (social media, logging, time-series)
  ✅ SSD infrastructure (want to extend drive lifespan)
  ✅ Space efficiency critical (LSM compresses better)
  ✅ Insert-heavy workloads with acceptable read amplification

USE SQLite when:
  ✅ Mobile apps (iOS, Android)
  ✅ Local desktop application data storage
  ✅ Development/testing (no server setup)
  ✅ Embedded devices and single-user applications
  ❌ Multi-user server applications

USE LevelDB when:
  ✅ Simple key-value storage without SQL
  ✅ Embedded application storage (Bitcoin Core pattern)
  ✅ Write-heavy with eventual reads
  ❌ Need SQL or transactions

USE RocksDB directly (not via MySQL) when:
  ✅ Building a custom database or storage layer
  ✅ Key-value access patterns
  ✅ Need ACID in an embedded/key-value context

9. B-Tree vs LSM — The Right Mental Model

B-Tree databases (InnoDB, PostgreSQL):
  Think: librarian who keeps books in perfect alphabetical order.
  Every new book immediately goes to its exact spot.
  Shelves occasionally reorganized (page splits) to maintain order.
  Finding a book: instant (it's always in the right spot).

  Real-world analogy:
    Like a filing cabinet with alphabetical dividers.
    New file → find the right section → insert → done.
    Occasionally a section gets too full → move some files to a new section.

LSM-Tree databases (LevelDB, RocksDB, Cassandra):
  Think: receiving dock that sorts mail in batches.
  Every new letter goes to the inbox first (fast append).
  Periodically sort and merge the inbox into the main filing system.
  Finding a letter: check inbox first, then filed letters.

  Real-world analogy:
    Like a newspaper archive.
    New papers stacked daily (fast write).
    Monthly: sort and archive all papers in chronological order.
    Finding an article: check recent stack, then sorted archive.

Neither is universally better:
  OLTP (lots of reads and writes, must be fast for both) → B-Tree
  Write-heavy (IoT sensors, logging, social media inserts) → LSM
  Time-series data → LSM (Cassandra, InfluxDB)
  General purpose web app → B-Tree (InnoDB/PostgreSQL)

10. Important Interview Questions

Basic

Q: What is a database storage engine?
A: A library that handles the lowest level of database operations — how data is physically stored on disk, index data structures, and CRUD at the file level. It may or may not support ACID transactions, row-level locking, and foreign keys. The DBMS layer sits on top and adds features like SQL parsing, replication, and connection handling.

Q: What is the difference between MyISAM and InnoDB?
A: MyISAM is older, has no ACID transaction support, no row-level locking (table-level only), and indexes point directly to physical row locations. It's fast for simple reads but crashes can corrupt tables and concurrent writes are serialized. InnoDB has full ACID transactions, row-level locking, foreign key support, crash recovery via WAL, and secondary indexes point to the primary key (not physical offset). InnoDB has been the MySQL default since 5.5.

Q: What is the key architectural difference between B-Tree and LSM-Tree storage engines?
A: B-Tree engines maintain data in sorted order on disk at all times, requiring random writes and in-place updates when the tree rebalances — causing write amplification but enabling fast reads. LSM-Tree engines write everything sequentially — new data always appended (never updated in place) — then compacted in background. This makes writes fast and SSD-friendly but introduces read amplification (must check multiple levels for a key). B-Tree is better for mixed workloads; LSM is better for high-insert, write-heavy workloads.

Q: Why can't PostgreSQL switch storage engines while MySQL can?
A: PostgreSQL's storage layer is tightly integrated with its MVCC implementation, query planner, and index access methods. The storage is not cleanly abstracted as a separate pluggable component. MySQL was designed from the start with a pluggable storage engine API (the "Handler" interface), allowing completely different storage implementations behind the same SQL interface.

Q: What is SQLite and where is it used?
A: SQLite is an embedded, serverless, zero-configuration SQL database — a single C library that writes to a single file. No server process, no TCP connection, no setup. It supports full ACID transactions, B-Tree storage, and concurrent reads (with WAL mode). Used in every iOS and Android device, all major web browsers, Python's standard library, and hundreds of embedded applications. The most widely deployed database engine in the world.

Intermediate

Q: Why does LevelDB (and LSM in general) benefit SSD drives compared to B-Tree engines?
A: SSD NAND cells have limited write cycles (P/E cycles). Updating existing data requires: erase the cell → write new value. Erasing is slow and degrades the cell. B-Tree engines frequently update existing pages during rebalancing — random writes that erase and rewrite SSD cells repeatedly. LSM engines NEVER update existing data — every write is a sequential append to a new location. Old versions are discarded during compaction (sequential reads, sequential writes). This dramatically reduces random writes, extending SSD lifespan and improving throughput.

Q: What is compaction in LSM trees and why is it necessary?
A: As LSM trees accumulate SSTables across levels, multiple versions of the same key may exist across different levels. Compaction is a background process that merges sorted SSTables from one level into the next, discarding obsolete versions of keys. Without compaction: read amplification grows, space amplification grows, and performance degrades. Compaction is the LSM tree's "maintenance cost" — the trade-off for its write efficiency.

Q: Why did Facebook build RocksDB instead of using LevelDB?
A: LevelDB had key limitations at Facebook's scale: (1) Single-threaded compaction — at Facebook's write volume, one thread couldn't keep up, causing write stalls. (2) No ACID transactions — Facebook needed atomic multi-key operations. (3) No built-in bloom filters for read optimization. (4) Limited tuning options. RocksDB added multi-threaded compaction, full ACID transactions, bloom filters, extensive monitoring, compression per level, column families, and more — while keeping the LSM append-only write model.

Q: Explain why MyISAM index corruption is more likely than InnoDB corruption on crashes.
A: MyISAM indexes store physical byte offsets (direct pointers) to row locations in the .MYD heap file. When a write crashes mid-operation, the data file and index files may be out of sync — indexes pointing to wrong offsets or missing entries for inserted rows. There's no WAL to replay for consistency. InnoDB uses a Write-Ahead Log — every change is written to the WAL first. On crash recovery, InnoDB replays the WAL to restore a consistent state. InnoDB's design makes it structurally crash-safe; MyISAM's direct-offset indexing makes it crash-vulnerable.

Advanced

Q: A startup is building a system that ingests 1 million IoT sensor readings per second and needs to query the last hour's data in real time. Design the storage engine strategy.

  • Write path: LSM-based engine (RocksDB, Cassandra, InfluxDB) — append-only writes, no B-tree rebalancing, sequential I/O, SSD-friendly. Time-series data is perfectly suited to LSM (always-increasing timestamps → no random position inserts).
  • Data model: Partition key = device_id, clustering key = timestamp. Write path goes to MemTable → compacted to SSTable levels.
  • Read path: Last hour = recent data, likely still in MemTable or Level-0 SSTables. Bloom filters eliminate unnecessary SSTable checks.
  • Compaction strategy: TWCS (Time Window Compaction Strategy in Cassandra) — compact SSTables within same time window, then merge windows.
  • Retention: After 30 days, SSTables are simply dropped (LSM advantage — no expensive DELETE needed).

Q: Walk through what happens physically on disk when you INSERT a row in MyISAM vs InnoDB.
A: MyISAM: (1) Append row bytes to end of .MYD heap file. (2) Update .MYI index file — add a new B-Tree entry with (key, byte_offset). (3) If multiple secondary indexes exist, update each one. (4) No WAL written — on crash here, row may exist in data file but not in indexes.

InnoDB: (1) Write WAL entry (redo log) to ib_logfile — BEFORE touching data or index. (2) Write row to InnoDB buffer pool page. (3) Update primary B+ Tree in buffer pool — leaf node contains full row. (4) Update any secondary indexes in buffer pool — each stores (secondary_key, primary_key). (5) Buffer pool pages marked dirty, eventually flushed to .ibd file asynchronously. (6) On crash at any step: WAL replay restores consistency; uncommitted transactions rolled back via undo log.

11. Common Mistakes

Beginner Mistakes

  • Using MyISAM for any application that needs data integrity: "I'll just add transactions at the application layer" — this is fragile, error-prone, and doesn't protect against crashes
  • Not specifying ENGINE when creating MySQL tables: Relying on the default (which is InnoDB now, but wasn't always — legacy codebases may use MyISAM on older MySQL)
  • Using SQLite for a multi-user server application: SQLite has no row-level locking; one writer blocks all readers. Unacceptable for concurrent web apps

Backend Engineer Mistakes

  • Choosing a storage engine without understanding the access pattern: Selecting RocksDB/LSM for a system that does mostly reads with occasional writes — LSM's read amplification degrades read performance; B-Tree is better for balanced workloads
  • Ignoring compaction overhead in production for LSM: Compaction consumes I/O and CPU. Under sustained write load, compaction can fall behind, causing write stalls. Must monitor compaction queue and tune accordingly
  • ALTER TABLE to change engine on large tables without a tool: ALTER TABLE big_table ENGINE=InnoDB takes an exclusive lock and copies the entire table — potentially hours of downtime. Use pt-online-schema-change (Percona) or gh-ost (GitHub) for online engine migration

Production Mistakes

  • Mixing MyISAM tables in a system that uses transactions: A transaction touching both InnoDB and MyISAM tables — InnoDB changes can be rolled back, MyISAM changes cannot. Silent data inconsistency.
  • Not monitoring RocksDB compaction lag: If compaction can't keep up with write rate, Level-0 fills up → write stalls → latency spikes. Must alert on compaction pending bytes.
  • Not running REPAIR TABLE after MyISAM crash: Corrupt MyISAM indexes cause random query failures. After any MySQL crash involving MyISAM tables, run CHECK TABLE and REPAIR TABLE before returning to production.
