Single-Leader Failure Handling & Replication Lag

📖 Engineering Mastery Notes (Source: DDIA Chapter 6 Extract)

1. Big Picture

Why Do We Care About Failures and Lag?

In production distributed systems, node failure is not an "if," it is a "when." Machines reboot for kernel patches, power supplies blow, and network cables are unplugged.

  1. Fault Tolerance: The system must survive these individual node outages without dropping the overall availability of the cluster.
  2. Read Scalability: To handle massive traffic (e.g., millions of reads per second), we fan out queries across many read replicas.
  3. The Core Tradeoff: Read scaling fundamentally requires Asynchronous Replication. If you try to synchronously replicate to 50 nodes, one slow node halts all writes. However, async replication introduces Replication Lag, creating a window where users read stale data. As a Senior Engineer, your job isn't to eliminate lag entirely, but to mask it from the user through clever system design.

2. Core Concepts

  • Catch-up Recovery: The process a recovering follower uses to download missed replication logs from the leader and apply them.
  • Failover: The orchestration required to promote a follower to leader, re-route client traffic, and demote the dead leader.
  • Split Brain: A catastrophic state where two nodes simultaneously act as the leader, leading to divergent and corrupted data.
  • Eventual Consistency: If writes stop, all replicas will eventually process the backlog and match the leader.
  • Read-After-Write (Read-Your-Writes) Consistency: A guarantee that if a user updates data, their subsequent reads will reflect that update.
  • Monotonic Reads: A guarantee that users will never see time "move backward" (e.g., reading a fresh replica, then hitting a stale one and seeing data disappear).
  • Consistent Prefix Reads: A guarantee that causally related events are read in the exact order they were written.
  • Availability Zone (AZ): A single physical datacenter (power/cooling).
  • Region: A geographical grouping of multiple AZs connected via ultra-low-latency networks.

3. Deep Dive: Failure Handling Operations

A. Follower Failure: Catch-Up Recovery

When a follower disconnects, it doesn't lose its mind. It has a persistent WAL (Write-Ahead Log) on disk.

  • The Process: Upon restart, the follower reads its log, finds the last processed transaction (via LSN/GTID), connects to the leader, and requests all missing data.
  • Production SRE Risk: Log Retention vs. Disk Space. If a follower is down for 3 days, the leader must keep 3 days of replication logs on disk. If the leader runs out of disk, the whole database crashes. SREs must configure "max WAL retention" policies. If the follower is down too long, the leader deletes the old logs, and the follower must be fully rebuilt from a fresh snapshot.

B. Leader Failure: Failover Mechanics

Failover is notoriously dangerous. Most top-tier companies prefer manual or semi-automated failovers heavily gated by human operators because false positives are so destructive.

  1. Detection: Usually via a heartbeat timeout (e.g., 30 seconds).
  2. Election: Replicas use consensus (Raft/ZooKeeper) to agree on a new leader. The replica with the highest LSN (most up-to-date) is chosen to minimize data loss.
  3. Reconfiguration: Proxies (PgBouncer, Envoy) and clients are updated to point to the new leader IP. The old leader is fenced off.

4. Distributed Systems Perspective: Failover Pitfalls

What happens when automated failovers go wrong?

  • The GitHub Outage (Cross-System Data Corruption):
    • Context: GitHub used MySQL (async replication) and Redis.
    • Failure: Leader crashed. Follower promoted. Follower's DB auto-increment counter lagged behind the old leader.
    • Result: The new leader reused primary keys that the old leader had already issued. Redis, caching data based on those IDs, now served completely wrong user profiles/private data to the wrong users.
    • Lesson: Never blindly discard unreplicated writes if external systems (Caches, Search Indexes, Message Queues) have already observed those writes.
  • Timeout Tuning Dilemma:
    • Set timeout to 10 seconds? A JVM Garbage Collection pause or a brief network spike triggers a failover. You just caused database downtime to "fix" a 10-second blip.
    • Set timeout to 60 seconds? Actual hardware failures mean you have 1 full minute of total write unavailability.
  • Split Brain: If the old leader was just partitioned (not dead), it will keep accepting writes. If you don't use Fencing Tokens, the data will diverge permanently.

5. CAP and Consistency Tradeoffs: Replication Lag Anomalies

When you use read-scaling, you embrace eventual consistency. This leads to three major UX anomalies.

Anomaly 1: Reading Your Own Writes (The "My Post Vanished" Bug)

  • Problem: User submits a comment → Write hits Leader → App redirects user to "View Comment" page → Read hits a Follower → Follower is 500ms behind. The user sees an empty page and submits the comment again (creating a duplicate).
  • Solutions:
    1. Profile/Ownership Routing: Always route queries for a user's own data to the Leader. Route other users' data to Followers.
    2. Timestamp Tracking: Client passes a Last-Write-Timestamp. If the follower's log is older than this timestamp, wait or proxy the read to the leader.

Anomaly 2: Monotonic Reads (The Time Travel Bug)

  • Problem: User hits refresh. Request 1 hits Replica A (lag 100ms) → User sees new comment. User hits refresh again. Request 2 hits Replica B (lag 3 seconds) → Comment vanishes. Time went backward.
  • Solution: Sticky Routing. Hash the UserID. User 123 is always routed to Replica B. They may see delayed data, but they will never see data disappear.

Anomaly 3: Consistent Prefix Reads (The Causality Bug)

  • Problem: Typically happens in sharded databases. Event A happens, causing Event B. (Mr. Poons asks a question; Mrs. Cake answers it). Because shards replicate at different speeds, an observer might see the answer (Event B) replicated before the question (Event A).
  • Solution: Write causally related events to the exact same shard, or use advanced causality-tracking algorithms (Version Vectors).

6. Practical Backend Engineering Perspective

How top tech companies handle these tradeoffs:

  • Uber/Stripe (High Consistency Needs): They avoid relying on eventually consistent reads for core business logic (like payment state). State machines strictly read from the Primary/Leader.
  • Twitter/Reddit (High Scale Needs): Reading your own tweet uses "Read-Your-Writes" consistency (usually via a local client cache or primary read), but reading someone else's timeline is strictly Eventually Consistent.
  • Mobile/Cross-Device Challenges: If a user updates their profile on their iPhone, then opens their iPad on the same WiFi, the iPad might read stale data because the "last write timestamp" was local to the iPhone. SREs fix this by centralizing write-timestamps in a fast distributed cache (Redis).

7. Important Interview Questions

Basic

Q: Why do we use asynchronous replication for read replicas?
A: Because synchronous replication across many nodes destroys write availability. If just one of 10 read replicas has a network blip, the leader would be forced to block all incoming writes.

Intermediate

Q: How do you prevent a user from experiencing the "Read Your Own Writes" anomaly?
A: (1) Force all reads of a user's own editable data to hit the leader. (2) Temporarily cache the user's data on the client side. (3) Track the exact timestamp/LSN of the write, and when routing the read, wait for the follower to catch up to that LSN before responding.

Advanced

Q: You have an auto-failover mechanism for your Single-Leader DB. Walk me through the exact dangers of this and how to mitigate them.
A: The primary dangers are Split Brain and False Positives. To mitigate false positives, use a robust timeout (e.g., 30s) and require multiple cluster nodes to agree the leader is dead. To mitigate Split Brain, implement Fencing (STONITH - Shoot The Other Node In The Head), physically killing power to the old leader, or revoking its AWS IAM roles to write to EBS. Finally, avoid relying on DB auto-increment keys if external systems (like Redis) cache them, to prevent the "GitHub cross-system corruption" scenario.

8. Common Mistakes

  • Beginner Mistake: Pretending async replication is synchronous. Throwing a Thread.sleep(500) after a database write hoping the replica catches up before reading. (It will fail under load).
  • Backend Engineer Mistake: Using random Load Balancing (Round Robin) for database read replicas. This guarantees users will hit the Monotonic Read (Time Travel) anomaly. Always use consistent hashing (Sticky Sessions) based on User ID.
  • Production SRE Mistake: Overly aggressive auto-failovers. If a switch misbehaves and causes 5 seconds of packet loss, failing over the primary DB causes massive cache invalidations, proxy re-routing, and connection pooling storms.

9. Performance & Scalability Notes

  • Catch-Up Load Spike: When a heavily lagging follower reconnects, it pulls gigabytes of WAL. This saturates the Leader's network outbound interface and disk I/O, potentially impacting live production traffic. SREs cap the replication bandwidth using QoS (Quality of Service) network rules.
  • AZ vs Region Networking: Placing a replica in another AZ adds ~1ms latency. Placing it in another Region adds ~50-150ms. Regional replication must be asynchronous or your DB will grind to a halt.

10. Distributed Systems Internals (Beyond the Lecture)

  • Logical Clocks (Lamport Timestamps): Mentioned briefly under consistent prefixes. In complex systems, physical clocks are untrustworthy (NTP skew). Distributed systems use logical sequence numbers (counters) to definitively prove Event A happened before Event B.
  • NewSQL (Spanner, CockroachDB): The text mentions NewSQL. These databases avoid these replication lag anomalies entirely by providing strict Linearizability across distributed nodes, using specialized hardware (Atomic Clocks in Spanner) or complex consensus protocols (Raft over multiple partitions).

11. Practical Code / System Design Integration

Implementing Read-Your-Writes via Redis & LSN (Java/Spring Model):

// 1. User updates profile
@Transactional
public void updateProfile(User profile) {
    // Write goes to Leader. Returns the Log Sequence Number (LSN)
    long writeLsn = primaryDb.saveAndReturnLsn(profile);
    
    // Store LSN in Redis with a short TTL (e.g., 10 seconds)
    redis.setEx("user:" + profile.getId() + ":last_lsn", 10, writeLsn);
}

// 2. User reads profile
public User getProfile(String userId) {
    Long requiredLsn = redis.get("user:" + userId + ":last_lsn");
    
    if (requiredLsn != null) {
        // Query a replica, BUT tell the DB driver it MUST wait 
        // until the replica has applied up to 'requiredLsn'
        return replicaDb.queryWithLsnWait("SELECT * FROM users WHERE id = ?", userId, requiredLsn);
    } else {
        // Safe to read from any replica instantly
        return replicaDb.query("SELECT * FROM users WHERE id = ?", userId);
    }
}

Why this is master-level: It maximizes read scaling (reads still go to replicas) while mathematically guaranteeing no stale reads for the user who just wrote data.

13. Failure Analysis Section

Component Failure Risk Level SRE Recovery / System Action
Follower Node restarts after 2 mins Low Follower streams WAL from Leader. Catches up safely.
Follower Node offline for 4 days Medium Leader deleted WAL to save disk space. SRE must rebuild follower from a clean snapshot.
Leader Crash / Network Partition High Auto/Manual failover. Fencing tokens block the old leader. App route updates.
Old Leader Comes back online post-failover Critical (Split Brain) Old leader realizes its epoch/fencing token is invalid, steps down to become a Follower.

14. Lecture Summary

  • Key Takeaway 1: Automated failover is a distributed systems minefield. Between split-brain, timeout tuning, and discarded writes, human-in-the-loop failover is often preferred unless you have massive SRE tooling.
  • Key Takeaway 2: If you decouple systems (DB + Redis), asynchronous failover in the DB can corrupt the cache.
  • Key Takeaway 3: Eventual consistency creates UX anomalies. As a backend engineer, you must handle Read-Your-Writes (use leader/LSN routing), Monotonic Reads (use consistent hashing), and Consistent Prefix (use causal partitioning).

15. Revision Notes (2-Minute Review Sheet)

  • Catch-up: Follower uses log to rebuild. Watch out for Leader disk exhaustion.
  • Failover Steps: Detect (timeout) → Elect (consensus) → Reconfigure (routing).
  • Fencing: Mechanism to brutally shut down old leaders to prevent Split Brain.
  • Lag Anomalies & Fixes:
    • Read Your Writes: Read own data from primary.
    • Monotonic Reads: Hash routing (Sticky sessions).
    • Consistent Prefix: Put causally related data on the same shard.
  • AZ vs Region: AZ = low latency (sync possible). Region = high latency (async mandatory).
  • The SRE Rule: Never let an external system (Redis/Search) observe a DB state that hasn't been safely replicated, otherwise a DB failover will permanently desync the infrastructure.
