Database Partitioning

Production Engineering Notes

1. Big Picture

Why Partitioning Exists

Every table is a file (or set of files) on disk. As a table grows to hundreds of millions or billions of rows, two things happen that kill performance:

  1. Index size grows — a B+ Tree index on a 1-billion-row table is large. At some point it no longer fits in the buffer pool (RAM). Every index traversal starts causing disk I/Os instead of in-memory lookups.
  2. Sequential scans become unacceptable — when the query planner decides a scan is cheaper than an index lookup (e.g., low-selectivity queries), scanning a billion-row table is catastrophic.

Partitioning solves both by physically splitting the table into smaller sub-tables — each with its own smaller index that is more likely to fit in memory, and each covering only a fraction of the total rows.

The quickest way to query a table with a billion rows is to avoid querying a table with a billion rows.

Where It Is Used

SystemPartitioning Use Case
E-commerce (Amazon, Flipkart)Orders partitioned by date; recent orders on fast SSD, old orders on cold storage
IoT / TelemetrySensor readings partitioned by month/year; old partitions archived to tape/cheap disk
BankingTransactions partitioned by account_id range or date; compliance archive in cold tier
Social MediaEvents/logs partitioned by date; partition pruning eliminates old data from queries
SaaS Multi-TenantCustomers partitioned by ID range; per-partition indexes stay cache-friendly

Why Backend Engineers Must Care

  • Partition key selection is an irreversible schema decision — changing it later requires rebuilding the entire table
  • Queries that don't include the partition key in their WHERE clause silently scan all partitions — worse than a single table
  • Index creation, schema changes, and VACUUM all behave differently on partitioned tables
  • Partitioning is frequently confused with sharding — they are fundamentally different

2. Core Concepts

2.1 — What Is Partitioning?

Partitioning splits one large table into multiple smaller physical sub-tables (partitions), while the database presents them as a single unified table to the client.

BEFORE partitioning:
┌─────────────────────────────────────────────┐
│             customers (1B rows)             │
│  id=1 to id=1,000,000,000                  │
│  One giant file on disk                     │
│  One giant B+ Tree index                   │
└─────────────────────────────────────────────┘

AFTER partitioning (by ID range, 10M rows each):
┌──────────────────┐
│   customers      │  ← Master/parent table (EMPTY — no actual data)
│  (virtual)       │     Acts as query router
└──────┬───────────┘
       │
  ┌────┴──────────────────────────────────────────┐
  │                                               │
┌─▼──────────────────┐   ┌──────────────────────┐
│customers_0_10M     │   │customers_10M_20M      │  ... 100 partitions
│ id: 1 → 9,999,999  │   │ id: 10M → 19,999,999  │
│ 10M rows           │   │ 10M rows              │
│ Own B+ Tree index  │   │ Own B+ Tree index     │
└────────────────────┘   └──────────────────────┘

Key point: The client always queries the master table (customers). The database routes to the correct partition transparently. The client is completely unaware of which partition is hit.

2.2 — Horizontal vs Vertical Partitioning

Horizontal Partitioning (Most Common — What People Mean by "Partitioning")

Split by rows. Rows are distributed across partitions based on a partition key. Each partition has the same columns as the master table but a subset of the rows.

Horizontal: slice the table like cutting a deck of cards

customers (1M rows)
├── customers_200k   → rows where id BETWEEN 1 AND 200,000
├── customers_400k   → rows where id BETWEEN 200,001 AND 400,000
├── customers_600k   → rows where id BETWEEN 400,001 AND 600,000
├── customers_800k   → rows where id BETWEEN 600,001 AND 800,000
└── customers_1m     → rows where id BETWEEN 800,001 AND 1,000,000

Vertical Partitioning

Split by columns. Rarely-accessed columns (especially large BLOBs) are moved to a separate table/tablespace, keeping the main table compact and fast.

BEFORE vertical partitioning:
┌────────────────────────────────────────────────────────┐
│ id │ name │ email │ profile_picture_blob (10MB avg)    │
└────────────────────────────────────────────────────────┘
     ↑ Every row read loads the 10MB blob — wastes I/O

AFTER vertical partitioning:
┌──────────────────────────┐    ┌────────────────────────────┐
│ id │ name │ email        │    │ id │ profile_picture_blob  │
│ (fast SSD tablespace)    │    │ (cold/slow tablespace)     │
└──────────────────────────┘    └────────────────────────────┘
     ↑ Hot columns compact            ↑ Blob only read when needed
       and cache-friendly

Use case: Large BLOB columns, JSON documents, rarely-accessed audit fields, GDPR-sensitive data that needs separate storage policy.

2.3 — Partition Types

Range Partitioning

Rows are assigned to partitions based on a column value falling within a defined range.

-- Partitioned by ID range (the demo example)
CREATE TABLE customers (id SERIAL, name TEXT)
PARTITION BY RANGE (id);

CREATE TABLE customers_0_10m PARTITION OF customers
    FOR VALUES FROM (0) TO (10000000);

-- Partitioned by date (common for time-series data)
CREATE TABLE orders (id BIGINT, created_at TIMESTAMPTZ, total NUMERIC)
PARTITION BY RANGE (created_at);

CREATE TABLE orders_2024_q1 PARTITION OF orders
    FOR VALUES FROM ('2024-01-01') TO ('2024-04-01');

CREATE TABLE orders_2024_q2 PARTITION OF orders
    FOR VALUES FROM ('2024-04-01') TO ('2024-07-01');

Best for: Sequential IDs, timestamps, dates. Any monotonically increasing value.

List Partitioning

Rows are assigned based on discrete, known values of a column.

-- Partitioned by country/region
CREATE TABLE customers (id BIGINT, name TEXT, country TEXT)
PARTITION BY LIST (country);

CREATE TABLE customers_us PARTITION OF customers
    FOR VALUES IN ('US', 'CA', 'MX');

CREATE TABLE customers_eu PARTITION OF customers
    FOR VALUES IN ('DE', 'FR', 'GB', 'IT', 'ES');

CREATE TABLE customers_asia PARTITION OF customers
    FOR VALUES IN ('IN', 'CN', 'JP', 'KR', 'SG');

Best for: Known, low-cardinality categorical columns: country, status, region, category.

Hash Partitioning

A hash function is applied to the partition key. Rows are distributed based on the hash result modulo the number of partitions.

-- Evenly distribute customers across 8 partitions
CREATE TABLE customers (id BIGINT, name TEXT)
PARTITION BY HASH (id);

CREATE TABLE customers_p0 PARTITION OF customers
    FOR VALUES WITH (MODULUS 8, REMAINDER 0);
CREATE TABLE customers_p1 PARTITION OF customers
    FOR VALUES WITH (MODULUS 8, REMAINDER 1);
-- ... through p7

Best for: Even distribution when no natural range or list grouping exists. Common in Cassandra-style consistent hashing.

3. Deep Dive — How Partition Routing Works

3.1 — Query Routing: Partition Pruning

When you query the master table, the query planner checks the WHERE clause against partition definitions to eliminate irrelevant partitions before execution.

-- Query:
SELECT name FROM customers WHERE id = 701;

-- Planner logic:
-- id = 701: which partition contains values where 0 ≤ id < 10M? → customers_0_10m
-- Skip all other 99 partitions
-- Execute: Index scan on customers_0_10m only

-- Execution steps:
-- 1. Read partition metadata (in memory, near-zero cost)
-- 2. Identify: customers_0_10m covers id 0 → 9,999,999
-- 3. Traverse B+ Tree index on customers_0_10m → find id=701
-- 4. Fetch row from partition's heap
-- 5. Return result

The savings: Instead of traversing a B+ Tree over 1 billion entries, you traverse one over 10 million entries. The index is ~100× smaller. More of it fits in the buffer pool. Fewer I/Os.

3.2 — Partition Pruning Must Be Enabled (PostgreSQL)

-- Check current setting
SHOW enable_partition_pruning;

-- If OFF: all partitions are scanned regardless of WHERE clause
-- This is DANGEROUS — can make partitioning worse than no partitioning

-- Ensure it's ON (it's ON by default, but verify)
SET enable_partition_pruning = ON;

-- Verify with EXPLAIN:
EXPLAIN SELECT count(*) FROM grades_parts WHERE g = 30;

-- With pruning ON:
--   Index Scan on grades_parts_g_0_35 (only 1 partition)
--
-- With pruning OFF:
--   Scans ALL 4 partitions — 4× the work

3.3 — Index Behavior on Partitioned Tables

Pre-PostgreSQL 11: You had to manually create indexes on each partition individually.

PostgreSQL 11+: Create an index on the master table → automatically propagated to all partitions.

-- One command creates index on all partitions:
CREATE INDEX grades_parts_g_idx ON grades_parts (g);

-- PostgreSQL automatically creates:
-- grades_parts_g_0_35_g_idx     on partition g_0_35
-- grades_parts_g_35_60_g_idx    on partition g_35_60
-- grades_parts_g_60_80_g_idx    on partition g_60_80
-- grades_parts_g_80_100_g_idx   on partition g_80_100

-- Verify:
\d grades_parts_g_0_35
-- Shows: automatically created index on g column

Important: The master table itself has no real index (it has no data). The indexes exist only on the child partition tables.

3.4 — Size Comparison: Partitioned vs Non-Partitioned

From the demo (10 million rows, grades 0–100):

Non-partitioned table:
  grades_original:          ~362 MB (table)
  grades_original_g_idx:    ~69 MB  (index on g column)

Partitioned (4 partitions: 0-35, 35-60, 60-80, 80-100):
  grades_parts_g_0_35:      ~126 MB (most rows fall here — grades cluster low)
  grades_parts_g_35_60:     ~smaller
  grades_parts_g_60_80:     ~smaller
  grades_parts_g_80_100:    ~smallest

  Per-partition index:      ~24 MB  (vs 69 MB for full table index)
  → ~3× smaller index per partition

Why smaller indexes matter for performance: A 24 MB index fits entirely in the buffer pool (RAM). Every index traversal is in-memory — 0 disk I/Os for the B+ Tree traversal. A 69 MB index may not fully fit, causing cache misses and disk reads during index traversal.

The demo didn't show a dramatic speed difference because:

  • The Docker container had access to ~16 GB RAM
  • Neither the table (362 MB) nor the index (69 MB) caused memory pressure
  • In production with memory constraints (500 MB limit, or a table/index of GBs), the difference becomes significant

3.5 — Planner Decision: Index Scan vs Sequential Scan on Partitions

The query planner makes cost-based decisions. Partitions make these decisions easier and often better:

Scenario: Query returns 30% of rows in a partition

On a 1B row table (non-partitioned):
  Planner: "30% of 1B = 300M rows via scattered index reads"
  Decision: Sequential scan (full table scan) — 1B rows scanned
  Cost: Enormous

On a 10M row partition:
  Planner: "30% of 10M = 3M rows"
  Decision: May still choose sequential scan, but now scans only 10M rows
  Cost: 100× less than the full table scan

Furthermore, the planner may switch from sequential scan to index scan because
the partition is small enough that even a scattered index scan is acceptable.

4. Partitioning vs Sharding

This is one of the most commonly confused distinctions in system design interviews.

Side-by-Side Comparison

AspectPartitioningSharding
LocationSame database, same serverMultiple database servers, different machines
Client awarenessClient is agnostic (queries master table)Client must know which shard to target
Table nameDifferent names per partitionSame table name, different servers
TransactionsFull ACID (single database)Cross-shard transactions extremely complex (2PC)
JoinsNormal SQL JOINs workCross-shard JOINs require application-level merging
ComplexityLow-medium (database handles it)High (application/proxy must route)
Use caseSingle-server scale improvementHorizontal scale beyond one machine's capacity
Failure domainSingle server is still a single point of failureOne shard failure affects only that shard's users
 
PARTITIONING (same server):

Client → SQL Query → PostgreSQL Server

                    ┌─────▼─────────────────────────────┐
                    │  Query Planner (routes internally) │
                    └─────┬──────────────┬──────────────┘
                          │              │
                   ┌──────▼──────┐ ┌────▼────────┐
                   │Partition 1  │ │Partition 2  │
                   │(same disk)  │ │(same disk)  │

    └──────────────────┘        └────────────────────┘

5. Pros and Cons of Partitioning

Pros

1. Query Performance Improvement

Partition pruning reduces the effective dataset from billions of rows to millions. Smaller B+ Tree indexes fit in RAM. Index-only scans become feasible.

2. Planner Makes Better Decisions

On a smaller partition, the planner more accurately chooses between sequential scan and index scan. A sequential scan on 10M rows is fast; on 1B rows it's catastrophic.

3. Easy Bulk Loading

You can create a standalone table with data, then attach it to the partitioned master table in one DDL statement — near-zero downtime for bulk data ingestion.

-- Bulk load into a staging table
CREATE TABLE orders_2024_q1_staging (LIKE orders INCLUDING INDEXES);
\COPY orders_2024_q1_staging FROM '/data/orders_q1.csv' CSV HEADER;

-- Attach as partition (very fast — just metadata change)
ALTER TABLE orders ATTACH PARTITION orders_2024_q1_staging
    FOR VALUES FROM ('2024-01-01') TO ('2024-04-01');

MySQL-specific advantage: MySQL supports CSV as a storage engine. Point a table at a CSV file and it becomes immediately queryable — attach it to a partitioned table with no data copy needed.

4. Cheap Archival (Tiered Storage)

Old partitions that are no longer queried can be moved to a slower, cheaper tablespace (spinning HDD, cold storage). Active partitions stay on fast NVMe SSD.

-- Move old partition to slow/cheap tablespace
ALTER TABLE orders_2019 SET TABLESPACE slow_hdd_tablespace;

-- Or detach entirely and archive to object storage
ALTER TABLE orders DETACH PARTITION orders_2019;
-- Export to S3/GCS, query later with Athena/BigQuery if needed

5. Partition-Level Operations

VACUUM, REINDEX, and CLUSTER can be run on a single partition without locking the whole table.

-- Vacuum only the most active partition (recent data)
VACUUM ANALYZE orders_2024_q4;

-- Reindex a single partition
REINDEX TABLE CONCURRENTLY orders_2024_q4;

-- Drop old partition in milliseconds (vs DELETE on billions of rows which takes hours)
ALTER TABLE orders DETACH PARTITION orders_2019;
DROP TABLE orders_2019;  -- Instant
-- vs: DELETE FROM orders WHERE created_at < '2020-01-01'; -- Hours, massive WAL

Cons

1. Updates That Cross Partition Boundaries Are Slow

If an UPDATE changes the partition key value, the row must move from one partition to another. This is actually a DELETE + INSERT across two tables — significantly more expensive than a simple in-place update.

-- In the grades example: updating a grade from 20 to 65 moves the row
-- from partition g_0_35 to partition g_35_60
UPDATE grades_parts SET g = 65 WHERE id = 12345;  -- Was g=20

-- What actually happens internally:
-- 1. DELETE from g_0_35 where id = 12345
-- 2. INSERT into g_35_60 with id=12345, g=65
-- = 2× I/O, 2× index updates, 2× WAL writes

-- Lesson: Never partition on a column that is frequently updated
-- Good partition keys: immutable or append-only (id, created_at)
-- Bad partition keys: status, grade, category (things that change)

2. Queries Without Partition Key Scan All Partitions

A query that can't prune partitions is worse than querying a single table — it scans N partition tables sequentially, each with its own I/O overhead.

-- GOOD: Partition key (id) in WHERE → prunes to 1 partition
SELECT * FROM customers WHERE id = 5000000;

-- BAD: No partition key in WHERE → scans all 100 partitions
SELECT * FROM customers WHERE name = 'Alice';

-- WORSE: Range condition that spans all partitions
SELECT * FROM customers WHERE id > 1;  -- Hits all 100 partitions

3. Schema Changes Are More Complex

Adding a column or index propagates to all partitions — this usually works automatically in modern PostgreSQL but is not guaranteed across all databases and all operations.

-- PostgreSQL: Adding a column to master propagates to all partitions
ALTER TABLE customers ADD COLUMN email TEXT;  -- Works on PostgreSQL 11+

-- But some operations require per-partition execution
-- Always test schema changes on partitioned tables in staging first

4. Partition Overhead for Small Tables

If the table is small enough to fit in the buffer pool anyway, partitioning adds overhead (metadata lookups, planner complexity) with zero benefit. Only partition tables that are genuinely large.

6. The Demo: Step-by-Step SQL Walkthrough

Step 1: Create Unpartitioned Table and Insert 10M Rows

-- Create table (PostgreSQL inside Docker)
CREATE TABLE grades_original (
    id SERIAL NOT NULL,    -- SERIAL = auto-increment integer
    g  INTEGER NOT NULL    -- grade 0-99
);

-- Insert 10 million rows with random grades 0-99
INSERT INTO grades_original (g)
SELECT FLOOR(RANDOM() * 100)   -- RANDOM() returns 0.0-0.999... × 100 = 0-99.9 → FLOOR = 0-99
FROM GENERATE_SERIES(1, 10000000);  -- Call this 10M times

-- Create index AFTER bulk insert (faster — no index maintenance during insert)
CREATE INDEX grades_original_idx ON grades_original (g);

Why create index after insert:
During bulk insert with an existing index, every inserted row updates the B+ Tree — potential page splits on every insert. By inserting first and indexing after, PostgreSQL builds the B+ Tree in a single sorted pass (bulk build), which is dramatically faster.

Step 2: Create Partitioned Table

-- Master/parent table: PARTITION BY RANGE declares the partition key
CREATE TABLE grades_parts (
    id SERIAL NOT NULL,
    g  INTEGER NOT NULL
) PARTITION BY RANGE (g);

-- Create 4 child partition tables
CREATE TABLE g_0_35   (LIKE grades_parts INCLUDING INDEXES);
CREATE TABLE g_35_60  (LIKE grades_parts INCLUDING INDEXES);
CREATE TABLE g_60_80  (LIKE grades_parts INCLUDING INDEXES);
CREATE TABLE g_80_100 (LIKE grades_parts INCLUDING INDEXES);

-- LIKE ... INCLUDING INDEXES: copies column definitions and index definitions
-- (no data is copied, just structure)

Step 3: Attach Partitions to Master Table

-- Attach each partition with its value range
ALTER TABLE grades_parts ATTACH PARTITION g_0_35
    FOR VALUES FROM (0) TO (35);     -- g: 0 ≤ g < 35

ALTER TABLE grades_parts ATTACH PARTITION g_35_60
    FOR VALUES FROM (35) TO (60);    -- g: 35 ≤ g < 60

ALTER TABLE grades_parts ATTACH PARTITION g_60_80
    FOR VALUES FROM (60) TO (80);    -- g: 60 ≤ g < 80

ALTER TABLE grades_parts ATTACH PARTITION g_80_100
    FOR VALUES FROM (80) TO (100);   -- g: 80 ≤ g < 100

-- Note: TO value is EXCLUSIVE (< not ≤)
-- A grade of exactly 35 goes to g_35_60, not g_0_35

Step 4: Populate Partitioned Table

-- Copy data from original table into partitioned table
-- Database automatically routes each row to the correct partition based on g value
INSERT INTO grades_parts
SELECT * FROM grades_original;

-- What happens internally:
-- For each row: check g value → determine partition → insert into that partition
-- g=20 → g < 35 → goes to g_0_35
-- g=50 → 35 ≤ g < 60 → goes to g_35_60
-- g=75 → 60 ≤ g < 80 → goes to g_60_80
-- g=90 → 80 ≤ g < 100 → goes to g_80_100

Step 5: Create Indexes on Partitioned Table (PostgreSQL 11+)

-- Create index on master → automatically propagates to ALL partitions
CREATE INDEX grades_parts_g_idx ON grades_parts (g);

-- Equivalent to (what PostgreSQL does automatically):
-- CREATE INDEX ON g_0_35 (g);
-- CREATE INDEX ON g_35_60 (g);
-- CREATE INDEX ON g_60_80 (g);
-- CREATE INDEX ON g_80_100 (g);

-- Verify index was created on a partition:
\d g_0_35
-- Shows: Indexes: "g_0_35_g_idx" btree (g)

Step 6: Query and Verify Partition Pruning

-- Query through master table (partition pruning active)
EXPLAIN ANALYZE SELECT count(*) FROM grades_parts WHERE g = 30;

-- Expected output:
-- Index Scan using g_0_35_g_idx on g_0_35
-- (Only partition g_0_35 is touched — others pruned)

-- Compare with non-partitioned:
EXPLAIN ANALYZE SELECT count(*) FROM grades_original WHERE g = 30;
-- Bitmap Index Scan on grades_original_idx (full index traversal)

-- Range query:
EXPLAIN ANALYZE SELECT count(*) FROM grades_parts WHERE g BETWEEN 30 AND 35;
-- May hit g_0_35 AND g_35_60 (range spans partition boundary)

-- Check partition pruning is ON:
SHOW enable_partition_pruning;
-- → on

-- What happens if OFF:
SET enable_partition_pruning = OFF;
EXPLAIN SELECT count(*) FROM grades_parts WHERE g = 30;
-- Scans ALL 4 partitions — completely defeats the purpose of partitioning!
SET enable_partition_pruning = ON;  -- Always restore!

Step 7: Check Object Sizes

-- Compare sizes of table and indexes
SELECT
    relname AS object_name,
    pg_size_pretty(pg_relation_size(oid)) AS size,
    relkind
FROM pg_class
WHERE relname LIKE 'grade%'
ORDER BY pg_relation_size(oid) DESC;

-- Expected results (approximate for 10M rows):
-- grades_original        362 MB   (table)
-- grades_original_idx     69 MB   (index — large)
-- g_0_35                 126 MB   (partition — grades cluster 0-35 statistically)
-- g_0_35_g_idx            24 MB   (partition index — ~3× smaller than full index)
-- g_35_60                 ...

7. The Scripts: Code Walkthrough

create_partitions.mjs — Automated Partition Creation

import pg from 'pg';

async function run() {
    // Step 1: Connect to default 'postgres' database
    const dbClientPostgres = new pg.Client({
        user: "postgres",
        password: "postgres",
        host: "localhost",       // Change from "husseinmac" to your hostname
        port: 5432,
        database: "postgres"
    });

    await dbClientPostgres.connect();

    // Step 2: Create the 'customers' database
    // CAUTION: Uncomment DROP only for fresh setup
    // await dbClientPostgres.query("DROP DATABASE IF EXISTS customers");
    await dbClientPostgres.query("CREATE DATABASE customers");
    await dbClientPostgres.end();  // Close connection to postgres DB

    // Step 3: Connect to the newly created 'customers' database
    const dbClientCustomers = new pg.Client({
        user: "postgres",
        password: "postgres",
        host: "localhost",
        port: 5432,
        database: "customers"    // Now targeting 'customers' DB
    });

    await dbClientCustomers.connect();

    // Step 4: Create master partitioned table
    // PARTITION BY RANGE (id) — declares id as the partition key
    await dbClientCustomers.query(`
        CREATE TABLE customers (id SERIAL, name TEXT)
        PARTITION BY RANGE (id)
    `);

    // Step 5: Create 100 partitions in a loop
    // Assuming: 1B total customers, 10M per partition → 100 partitions
    for (let i = 0; i < 100; i++) {
        const idFrom = i * 10_000_000;            // 0, 10M, 20M, ...
        const idTo   = (i + 1) * 10_000_000;     // 10M, 20M, 30M, ...

        const partitionName = `customers_${idFrom}_${idTo}`;
        // e.g.: customers_0_10000000, customers_10000000_20000000, ...

        // Create the child table with same structure as master
        // 'INCLUDING INDEXES' copies any index definitions from master
        const createSQL = `
            CREATE TABLE ${partitionName}
            (LIKE customers INCLUDING INDEXES)
        `;

        // Attach child table to master with its value range
        // NOTE: TO is exclusive — customers_0_10M covers id 0 to 9,999,999
        const attachSQL = `
            ALTER TABLE customers
            ATTACH PARTITION ${partitionName}
            FOR VALUES FROM (${idFrom}) TO (${idTo})
        `;

        await dbClientCustomers.query(createSQL);
        await dbClientCustomers.query(attachSQL);

        console.log(`Created partition: ${partitionName} (${idFrom} → ${idTo - 1})`);
    }

    await dbClientCustomers.end();
    console.log("Done. 100 partitions created.");
}

run();

Line-by-line key points:

  • idTo - 1 boundary: FOR VALUES FROM (0) TO (10000000) means 0 ≤ id < 10000000 — id=10,000,000 goes to the NEXT partition
  • LIKE customers INCLUDING INDEXES: inherits column structure AND any index definitions already on the master table
  • Two DB connections needed: first to create the customers database (must be done from postgres DB), then to work within customers

populate_customers.mjs — Bulk Data Insert

import pg from 'pg';

async function run() {
    const dbClientCustomers = new pg.Client({
        user: "postgres",
        password: "postgres",
        host: "localhost",
        port: 5432,
        database: "customers"
    });

    await dbClientCustomers.connect();

    // Insert 100 batches × 10M rows = 1B total customers
    for (let i = 0; i < 100; i++) {
        // Each iteration inserts 10M rows
        // GENERATE_SERIES(1, 10000000) calls generate_series 10M times
        // RANDOM() generates a float 0.0-1.0, cast to TEXT for the name field
        const insertSQL = `
            INSERT INTO customers (name)
            SELECT random()::TEXT
            FROM GENERATE_SERIES(1, 10000000)
        `;

        await dbClientCustomers.query(insertSQL);
        console.log(`Inserted batch ${i + 1}/100 (${(i + 1) * 10}M customers total)`);
    }

    await dbClientCustomers.end();
    console.log("Done. 1B customers inserted.");
}

run();

Key behavior: The INSERT targets the master customers table. PostgreSQL's partition routing logic automatically directs each row to the correct child partition based on the auto-generated id value (SERIAL). No application-level routing needed.

8. Important Interview Questions

Basic

Q: What is database partitioning?
A: Partitioning physically splits a large table into multiple smaller sub-tables (partitions) while the database presents them as a single unified table. Queries against the master table are automatically routed to the correct partition(s) based on the partition key in the WHERE clause, reducing the data scanned and enabling smaller, more cache-friendly indexes.

Q: What is partition pruning?
A: Partition pruning is the query planner's ability to eliminate irrelevant partitions before query execution. If a query's WHERE clause includes the partition key, the planner identifies which partitions can contain matching rows and skips all others. In PostgreSQL, enable_partition_pruning must be ON (default) for this to work.

Q: What is the difference between horizontal and vertical partitioning?
A: Horizontal partitioning splits rows — each partition has all columns but a subset of rows (e.g., customers id 0-10M in partition 1, 10M-20M in partition 2). Vertical partitioning splits columns — rarely-accessed columns (e.g., BLOBs) are moved to a separate table/tablespace, keeping the main table compact and cache-friendly.

Q: What is the difference between partitioning and sharding?
A: Partitioning splits data across sub-tables within a single database on a single server. The client is completely unaware — queries target the master table and the database routes internally. Sharding splits data across completely separate database servers. The client (or a proxy) must know which shard to target. Sharding enables horizontal scaling beyond a single machine but adds significant complexity: no cross-shard transactions, no cross-shard JOINs, harder schema migrations.

Q: What are the three types of partitioning in PostgreSQL?
A: Range (values fall within continuous ranges — IDs, dates), List (values match discrete known values — country, status), and Hash (a hash function distributes rows evenly across partitions — good for even distribution without natural ranges).

Intermediate

Q: Why should you avoid using frequently-updated columns as partition keys?
A: Updating the partition key value causes the row to move from one partition to another. Internally, this is a DELETE from the source partition followed by an INSERT into the target partition — two full write operations with index updates on both sides. If done frequently at scale, this causes excessive I/O, index fragmentation, and potential SSD wear. Use immutable or append-only columns (id, created_at) as partition keys.

Q: In the demo, why wasn't a significant performance difference seen between the partitioned and non-partitioned table?
A: The test machine had 16GB RAM and Docker had no memory limit. The full table (362MB) and its index (69MB) both fit comfortably in the buffer pool. With all data in memory, there are no disk I/Os — both queries run at memory speed. The performance difference becomes visible when: (1) the table/index is significantly larger than available RAM, (2) the system is I/O bound (spinning disk, limited SSD throughput), or (3) at much higher row counts (100M+, 1B+ rows).

Q: Why does creating an index on the master partition table propagate to all child partitions (PostgreSQL 11+)?
A: PostgreSQL 11 introduced "declarative partitioning" improvements where DDL operations on the parent table (CREATE INDEX, ALTER TABLE ADD COLUMN) automatically apply to all attached child partitions. Before PostgreSQL 11, you had to create indexes on each partition manually, which was error-prone and tedious at scale.

Q: What happens if enable_partition_pruning is set to OFF?
A: The query planner stops performing partition elimination. Every query against the partitioned master table scans ALL partitions — even if the WHERE clause clearly indicates only one partition is relevant. This can make a partitioned table perform worse than a single non-partitioned table, because the database is now scanning N tables with N separate index lookups instead of 1. Always verify this setting is ON.

Advanced

Q: Explain the full execution path for SELECT name FROM customers WHERE id = 5000000 on a table partitioned into 100 partitions of 10M rows each.
A: (1) Query planner reads partition metadata (stored in pg_inherits and constraint definitions — in memory, near-zero cost). (2) Planner determines: id=5,000,000 falls in range [0, 10M) → only customers_0_10000000 is relevant, 99 partitions are pruned. (3) Planner generates execution plan: Index Scan on customers_0_10000000_pkey or secondary index. (4) Executor traverses B+ Tree index on customers_0_10000000 (10M entries, ~3-4 levels, likely fits in buffer pool). (5) Finds tuple pointer → fetches heap page from customers_0_10000000. (6) Returns row. Total: effectively 2-3 I/Os max, vs traversing an index over 1B entries.

Q: Design a partitioning strategy for a high-volume order processing system with 5 billion orders/year that needs to: serve real-time order lookups by order_id, support monthly financial reports, archive orders older than 2 years to cold storage.
A:
- Primary partition key: created_at (RANGE, monthly partitions) — aligns with reporting requirements and archival policy
- Secondary for lookups: Index on order_id within each partition; consider a separate lookup table mapping order_id → partition month if cross-partition lookups are frequent
- Partition structure: orders_2024_01, orders_2024_02, ... (monthly)
- Tiered storage: Current quarter partitions → NVMe SSD tablespace. Previous year → SATA SSD tablespace. 2+ years ago → Detach partition, export to S3, query via Athena
- Index strategy: Per-partition index on order_id and customer_id; covering index for common report queries
- Archival: ALTER TABLE orders DETACH PARTITION orders_2022_01; DROP TABLE orders_2022_01; after S3 export — instant vs hours-long DELETE

Q: What are the operational risks of running INSERT INTO partitioned_table SELECT * FROM huge_table in a single transaction?
A: (1) The transaction holds locks for the entire duration — can block concurrent writes if row-level locks conflict with partition routing. (2) Creates millions of WAL entries — WAL files grow rapidly, replication lag increases, WAL archival/shipping can fall behind. (3) If the transaction fails or is cancelled partway through, ROLLBACK must undo all inserted rows — PostgreSQL's lazy MVCC approach means millions of dead tuples accumulate, requiring extensive VACUUM afterward. (4) Risk of running out of WAL disk space. Better approach: batch inserts (INSERT ... LIMIT N in a loop), each as a small transaction with pg_sleep() pauses to allow autovacuum to keep up.

9. Common Mistakes

Beginner Mistakes

  • Querying without the partition key in WHERE: SELECT * FROM orders WHERE customer_name = 'Alice' on a table partitioned by id scans ALL partitions. The partition key MUST appear in the WHERE clause for pruning to work.
  • Forgetting enable_partition_pruning: Assuming partitioning helps without verifying pruning is active. Always confirm with EXPLAIN.
  • Partitioning small tables: Adds metadata overhead and planner complexity with zero benefit. Only partition tables that are genuinely large (100M+ rows, or where index exceeds available RAM).

Backend Engineer Mistakes

  • Choosing a mutable partition key: Using status or grade as partition key when these values are frequently updated. Every update crossing a partition boundary = DELETE + INSERT.
  • Range that spans all partitions in queries: WHERE id > 0 or WHERE created_at IS NOT NULL defeats partition pruning — ALL partitions are scanned.
  • Not accounting for the TO being exclusive: FOR VALUES FROM (0) TO (10000000) — a row with id=10,000,000 goes to the NEXT partition. Off-by-one errors create partition gaps where rows fall into no partition and fail to insert.

Production Mistakes

  • Not creating a DEFAULT partition: If a row's partition key value doesn't match any partition definition, INSERT fails with an error. Always create a default partition as a safety net: 
    CREATE TABLE customers_default PARTITION OF customers DEFAULT;
  • Schema changes without testing: Not all DDL operations propagate to partitions in all databases and versions. Test ALTER TABLE operations thoroughly in staging.
  • Forgetting to partition the index, not just the table: Running CREATE INDEX before the table is partitioned, then partitioning it — the index structure may not correctly map to partitions.
  • Single huge INSERT to populate partitioned table: Causes WAL explosion, replication lag, and massive dead tuples if it fails. Always batch populate.

When Partitioning Helps

ScenarioPartitioning Benefit
Index exceeds RAMPer-partition indexes fit in RAM; full index did not. Eliminates disk I/Os for index traversal
Time-series data with time-bounded queriesDate partition pruning + archival of old partitions
Infrequent bulk archival/deletionDrop a partition in milliseconds vs DELETE over hours
Mixed hot/cold dataHot partitions on fast SSD, cold on HDD/cold storage

When Partitioning Does NOT Help (or Hurts)

ScenarioWhy Partitioning Fails

Index Size Math

Table: 1 billion rows, INTEGER partition key (4 bytes), 8KB pages

Non-partitioned index (1B rows):
  ~1B × 14 bytes/entry = ~14GB
  Tree height: log_585(1B) ≈ 3.4 → 4 levels
  Internal nodes: ~24MB (fits in RAM)
  Leaf nodes: ~13.5GB (does NOT fit in typical RAM)
  → Many leaf reads require disk I/O

Partitioned (100 partitions × 10M rows each):
  Per-partition index: 10M × 14 bytes = ~140MB
  Tree height: log_585(10M) ≈ 2.3 → 3 levels
  Entire index: ~140MB (may fit in RAM → 0 disk I/Os)
  → Dramatic reduction in index traversal I/Os

11. System Design Connections

Partitioning + Tiered Storage

Hot tier   (NVMe SSD):  Last 3 months of orders   → sub-millisecond queries
Warm tier  (SATA SSD):  3-12 months of orders      → millisecond queries
Cold tier  (HDD):       1-2 years of orders        → seconds acceptable
Archive    (S3/GCS):    2+ years                   → Athena/BigQuery on demand

Implementation:
  ALTER TABLE orders_2024_q4 SET TABLESPACE nvme_fast;
  ALTER TABLE orders_2023_q4 SET TABLESPACE sata_warm;
  ALTER TABLE orders_2022_q4 SET TABLESPACE hdd_cold;
  -- Detach 2021 and earlier, export to S3, drop

Partitioning + Read Replicas

Write primary:          All partitions (read + write)
Read replica 1:         Replicated via WAL streaming — all partitions available
Read replica 2:         Same

Optimization: Route analytical queries (range scans, reports) to replicas.
               Keep OLTP point lookups on primary.
               Partitioning on both primary and replica — identical structure.

Partitioning as a Bridge to Sharding

Phase 1: Single server, partitioned table           → handles ~10B rows
Phase 2: Add read replicas                          → scales read throughput
Phase 3: Shard: move some partitions to new servers → handles unlimited scale
         customers_0_500M  → DB Server A
         customers_500M_1B → DB Server B
         (Partitions become the natural shard boundaries)
