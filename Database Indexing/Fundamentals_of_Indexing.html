Database Indexing Fundamentals

Interview Prep & System Design

1. Big Picture

Why Indexing Exists

Every database query ultimately becomes one of two operations: find the exact rows you need, or scan everything and throw away what you don't need. Without indexes, the second option is the only option. Indexes exist to make the first option possible.

An index is a separately maintained data structure that maps column values to their physical locations in the heap (table storage). It allows the query engine to answer "where is row X?" in logarithmic time (B+ tree: O(log n)) instead of linear time (heap scan: O(n)).

This difference is not academic:

Table: 10 million rows
Full heap scan: 625,000 page IOs
B+ tree index lookup: 4 page IOs

Ratio: 156,000x fewer disk reads with an index.

The Phone Book Analogy

The best way to think about an index is a phone book with labeled tabs:

  • You want "Zebra Corp" — you flip directly to Z, then scan a small section.
  • Without the tabs, you read page 1, page 2, page 3... through the entire 1,000-page book.
  • The tabs ARE the index. The phone book itself IS the heap.

Where Indexes Appear in Real Systems

  • E-commerce: Product search by category, price range, name prefix.
  • Banking: Account lookup by account number, transaction lookup by date range.
  • Social media: User lookup by username, post lookup by timestamp.
  • Ride-sharing: Driver lookup by geolocation bucket, trip lookup by status.
  • Reporting systems: Aggregations filtered by date range, region, product.

Why Backend Engineers Must Deeply Understand Indexing

  • Indexes are the single biggest lever for query performance optimization.
  • The wrong index strategy can make queries 1,000x slower — invisibly, silently, in production.
  • The query planner may IGNORE your index if you write the query incorrectly (LIKE with wildcards, implicit type conversions, non-SARGable predicates).
  • Index creation on large production tables blocks writes if done incorrectly.
  • Over-indexing makes writes slow — every insert/update/delete must update every index.

2. Core Concepts

What Is an Index?

An index is a separate data structure built on top of one or more columns of a table. It:

  • Is stored as its own set of disk pages (separate from the heap).
  • Contains only the indexed column values (the key) plus pointers back to the heap.
  • Is maintained automatically by the database on every INSERT, UPDATE, and DELETE.
  • Is smaller than the heap (fewer columns, often compressed) → fits in memory more easily.
  • Is organized for fast search — B+ tree by default, allowing O(log n) lookups.
Index on employee.id:

B+ Tree structure (conceptual):
                     [500 | 1000 | 5000]         ← Root node (1 page)
                    /         |          \
          [100|200|300]  [600|700|800]  [1500|2000|3000]   ← Internal nodes
               /                  \
[id=1,p=0,r=1] ... [id=500,p=62,r=3]  [id=1000,p=125,r=7] ...   ← Leaf nodes
     ↑                                      ↑
  (page=0, row=1 in heap)            (page=125, row=7 in heap)

The Two Primary Index Types

The lectures cover two main index structures:

B-Tree / B+ Tree (dominant structure):

  • Balanced tree with sorted keys at every level.
  • Leaf nodes contain key values + heap pointers, linked in a doubly-linked list.
  • Supports: equality, range queries, ORDER BY, GROUP BY, prefix matching.
  • Used by: PostgreSQL, MySQL InnoDB, SQL Server, Oracle (all B+ tree internally).

LSM Tree (Log-Structured Merge Tree):

  • Optimized for write-heavy workloads.
  • Writes go to an in-memory structure (MemTable), flushed to disk as sorted files (SSTables).
  • Background compaction merges SSTables.
  • Used by: RocksDB, Cassandra, LevelDB, HBase, ScyllaDB.
  • Read performance is worse than B+ tree but write throughput is far higher.

Hash Index (SQL Server / in-memory tables):

  • Array of buckets, each holding a linked list of key entries.
  • Hash function maps key → bucket.
  • Excellent for exact equality lookups (O(1) average).
  • Cannot support range queries, ORDER BY, or prefix scans.
  • SQL Server supports this for memory-optimized tables.
  • Bucket count must be configured at creation time (ideally 1-2x number of distinct values).

Scan Types — The Query Execution Vocabulary

Understanding scan types is essential for reading EXPLAIN output and diagnosing slow queries.

Sequential Scan (Full Table Scan)

Operation: Read EVERY page of the heap, check every row against the WHERE clause.
Cost:      O(N) — proportional to total table size, not result size.
When used: No index on the WHERE clause column, or index is not selective enough.

EXPLAIN output (PostgreSQL):
  -> Parallel Seq Scan on employees (workers: 2)
     Filter: (ssn = '666-66-6666')
     Rows Removed by Filter: 10,999,999

The "Rows Removed by Filter" line is critical — it shows how many rows were read but thrown away. A high number here screams "you need an index."

Index Scan

Operation: Traverse B+ tree to find candidate rows (by key value), then fetch each
           matching row from the heap (a separate IO per row).
Cost:      O(log N + K) where K = number of matching rows.
When used: Index exists on WHERE clause column, and the query is selective.

EXPLAIN output (PostgreSQL):
  -> Index Scan using employees_id_idx on employees
     Index Cond: (id = 10000)
     Heap Fetches: 1  ← went to heap to get non-indexed columns

The heap fetch is the additional cost — for each result row the index finds, the database must go back to the heap to retrieve columns not stored in the index.

Index Only Scan

Operation: Traverse B+ tree, return results DIRECTLY from the index.
           The heap is never touched.
Cost:      O(log N) — the sweetest possible query.
When used: ALL columns in the SELECT and WHERE clause exist in the index.

EXPLAIN output (PostgreSQL):
  -> Index Only Scan using students_grade_idx on students
     Index Cond: (grade BETWEEN 80 AND 95)
     Heap Fetches: 0  ← never touched the heap

Heap Fetches: 0 is the goal. This is the most efficient query type.

Important PostgreSQL detail: Even an "Index Only Scan" may show Heap Fetches > 0 if the visibility map is out of date (after recent writes without VACUUM). Run VACUUM to fix this.

Bitmap Index Scan

Operation: Scan the index to build a bitmap of matching page numbers, then
           fetch heap pages in sorted order (batching multiple row fetches from
           the same page into one IO).
Cost:      O(log N + pages_with_matches) — better than index scan for many matches.
When used: Query matches many rows (medium selectivity). PostgreSQL chooses this
           automatically over index scan when result set is larger.

EXPLAIN output (PostgreSQL):
  -> Bitmap Heap Scan on employees
       -> Bitmap Index Scan on employees_name_idx
          Index Cond: (name = 'XZ')

Why this matters: If you query 10,000 rows via a secondary index, those 10,000 rows may be scattered across 10,000 different heap pages (worst case: 10,000 random IOs). A bitmap scan sorts the heap page accesses first, reducing random IO to sequential IO.

The SARGability Concept

SARGable (Search ARGument-able) means the query predicate CAN use an index. This is one of the most important and most violated concepts in practical SQL writing.

SARGable predicates (CAN use index):

WHERE id = 10000                          -- Equality: uses index ✅
WHERE grade BETWEEN 80 AND 95            -- Range: uses index ✅
WHERE name = 'Alice'                     -- Equality on indexed col: uses index ✅
WHERE salary > 50000                     -- Inequality: uses index ✅
WHERE name LIKE 'Ali%'                   -- Prefix match: uses index ✅
WHERE created_at > '2024-01-01'          -- Date range: uses index ✅

Non-SARGable predicates (CANNOT use index):

WHERE name LIKE '%Ali%'                  -- Leading wildcard: full scan ❌
WHERE name LIKE '%Ali'                   -- Leading wildcard: full scan ❌
WHERE UPPER(name) = 'ALICE'             -- Function on column: full scan ❌
WHERE salary * 1.1 > 55000             -- Expression on column: full scan ❌
WHERE YEAR(created_at) = 2024           -- Function on column: full scan ❌
WHERE id != 10000                       -- NOT EQUAL: usually full scan ❌

Why non-SARGable queries bypass indexes:

The index stores the raw column values. If you wrap the column in a function (UPPER(name)) or expression (salary * 1.1), the index values don't match the transformed predicate. The planner has no choice but to scan the table and evaluate the function on every row.

The fix:

-- BAD: Non-SARGable
WHERE UPPER(name) = 'ALICE'

-- GOOD: Push the function to the constant side
WHERE name = LOWER('ALICE')

-- OR: Create a functional index (PostgreSQL supports this)
CREATE INDEX idx_name_upper ON employees (UPPER(name));
WHERE UPPER(name) = 'ALICE'  -- now SARGable ✅

3. Index Types — Complete Reference

Simple (Single-Column) Index

-- PostgreSQL / MySQL
CREATE INDEX idx_employees_name ON employees (name);

-- SQL Server
CREATE NONCLUSTERED INDEX IX_employees_name ON employees (name);

Covers equality and range queries on the name column. Cannot help queries that only reference other columns.

Composite (Multi-Column) Index

A composite index on (col_a, col_b, col_c) is ordered first by col_a, then by col_b within each col_a value, then by col_c within each col_b value.

The left-prefix rule: A composite index can only be used if the query provides values for a prefix of the indexed columns, starting from the leftmost.

CREATE INDEX idx_name_dept ON employees (last_name, department, hire_date);

-- Uses index: ✅
WHERE last_name = 'Smith'
WHERE last_name = 'Smith' AND department = 'Engineering'
WHERE last_name = 'Smith' AND department = 'Engineering' AND hire_date > '2020-01-01'

-- Uses index partially (only last_name): ⚠️
WHERE last_name = 'Smith' AND hire_date > '2020-01-01'
-- (department was skipped — index can filter on last_name, then heap scan for hire_date)

-- Does NOT use index: ❌
WHERE department = 'Engineering'    -- leftmost column not provided
WHERE hire_date > '2020-01-01'     -- leftmost column not provided

Column order strategy for composite indexes (SQL Server / general):

  1. Equality columns first (columns used with = in WHERE)
  2. Range columns last (columns used with >, <, BETWEEN)
  3. Then covered/included columns (columns only in SELECT, not WHERE)

This maximizes the index's ability to filter rows before reaching the range scan portion.

Covering Index (Index with Included Columns)

A covering index is one that contains all columns needed to answer a query — both the WHERE clause columns (key columns) and the SELECT clause columns (either as key or as non-key included columns). When covered, the heap is never accessed → Index Only Scan.

Key vs Non-Key (Included) Columns:

PropertyKey ColumnNon-Key (Included) Column
Stored atAll B+ tree levelsLeaf level only
Used forSearch/filteringData retrieval only
Affects sort orderYesNo
Counts toward key size limitYes (max 900 bytes / 16 cols in SQL Server)No
Can be large typesLimitedYes
 
-- PostgreSQL syntax
CREATE INDEX idx_grade_covering ON students (grade) INCLUDE (id);
-- Key column: grade (for WHERE grade BETWEEN 80 AND 95)
-- Non-key included: id (for SELECT id — avoids heap fetch)

-- SQL Server syntax (identical semantics)
CREATE NONCLUSTERED INDEX IX_Address_PostalCode
ON Person.Address (PostalCode)
INCLUDE (AddressLine1, AddressLine2, City, StateProvinceID);
-- Only PostalCode in the B+ tree key (small, efficient for searching)
-- Address fields only at leaf level (for SELECT — no heap fetch)

The design principle: Put only the columns you search/filter on as key columns. Put everything you SELECT (but don't filter on) as included/non-key columns.

The cost: Every non-key column you include increases the index size. A larger index means more pages → more IOs to read the index → potentially doesn't fit in buffer pool. Only include columns that appear frequently enough in queries to justify the space.

Before non-key index:
  Query: SELECT id, grade FROM students WHERE grade BETWEEN 80 AND 95 ORDER BY grade DESC
  Plan: Index Scan (grade index) → 90,000 heap fetches for id → SLOW

After CREATE INDEX ON students (grade) INCLUDE (id):
  Plan: Index Only Scan → 0 heap fetches → FAST

Unique Index

CREATE UNIQUE INDEX idx_employees_ssn ON employees (ssn);

Guarantees no two rows have the same value in the indexed column(s). Benefits beyond data integrity:

  • Query optimizer advantage: The optimizer knows this index will return at most ONE row for an equality predicate. It can choose a more efficient execution plan.
  • Implicit sort guarantee: Unique columns carry cardinality information that helps join and aggregation planning.

PRIMARY KEY and UNIQUE constraints automatically create unique indexes. There is no functional difference between CREATE UNIQUE INDEX and CREATE UNIQUE CONSTRAINT.

Filtered Index (Partial Index)

A filtered index indexes only a subset of rows that match a predicate. It is smaller, cheaper to maintain, and can be more selective than a full-table index.

-- PostgreSQL (called Partial Index)
CREATE INDEX idx_bills_with_enddate ON production.bill_of_materials (component_id, start_date)
WHERE end_date IS NOT NULL;
-- Only indexes the 199 rows where end_date is not null (out of 2,679 total)
-- 199-row index vs 2,679-row full index — 13x smaller, 13x cheaper to maintain

-- SQL Server (called Filtered Index)
CREATE NONCLUSTERED INDEX FIProductAccessories
ON Production.Product (ProductSubcategoryID, ListPrice)
INCLUDE (Name)
WHERE ProductSubcategoryID >= 27 AND ProductSubcategoryID <= 36;

Ideal use cases for filtered indexes:

ScenarioExample
Sparse columns (mostly NULL)WHERE status IS NOT NULL
Soft-deleted recordsWHERE deleted_at IS NULL
Status-based queriesWHERE order_status = 'PENDING'
Heterogeneous data categoriesWHERE product_type = 'ELECTRONICS'
Recent data access patternsWHERE created_at > '2024-01-01'

Critical rule: For the optimizer to use a filtered index, the query's WHERE clause must be compatible with (a superset of) the index's filter predicate. A query with WHERE end_date > '2020-01-01' would NOT use an index filtered WHERE end_date IS NOT NULL because the predicates are different — the query must reference the filter column.

Columnstore Index (SQL Server)

A columnstore index stores data in column-oriented format rather than row-oriented. SQL Server supports both clustered and nonclustered columnstore indexes.

Architecture:

  • Data is divided into rowgroups (up to 1,048,576 rows each).
  • Each rowgroup is divided into column segments (one per column).
  • Each column segment is compressed independently.
  • Small writes go to a deltastore (row-format B-tree) until enough rows accumulate for compression into the columnstore.
  • The tuple-mover background process compresses closed deltastore rowgroups.
Columnstore Index Physical Layout:

Rowgroup 1 (1M rows):                 Rowgroup 2 (1M rows):
  [id segment: compressed]              [id segment: compressed]
  [salary segment: compressed]          [salary segment: compressed]
  [dept segment: compressed]            [dept segment: compressed]
  ...                                   ...

Deltastore (B-tree, rowstore):
  Small inserts accumulate here until 102,400+ rows → tuple-mover compresses

When to use columnstore (SQL Server):

-- Create on existing rowstore table for mixed OLTP+OLAP
CREATE NONCLUSTERED COLUMNSTORE INDEX idx_sales_columnstore
ON sales (product_id, quantity, price, sale_date);

-- Optimizer automatically chooses columnstore for analytics:
SELECT product_id, SUM(price * quantity) as revenue
FROM sales
GROUP BY product_id;
-- Reads only product_id, price, quantity columns → 3x fewer IOs

Hash Index (SQL Server In-Memory Tables)

-- Only on memory-optimized tables
ALTER TABLE orders_inmem
ADD INDEX ix_hash_orderid UNIQUE
HASH (order_id) WITH (BUCKET_COUNT = 1000000);

Architecture:

  • Array of 8-byte bucket pointers.
  • Hash function maps key → bucket number.
  • Each bucket points to a linked list of matching entries.
  • Multiple keys can hash to the same bucket (hash collision).

Optimal bucket count:

  • Target: 1-2x the number of distinct key values.
  • Acceptable: within 10x of distinct values.
  • Too few → long chain lengths → slow equality lookups.
  • Too many → many empty buckets → wasted memory + slower full scans.

Hash index performance rules:

Query TypePerformance
Equality: WHERE key = ?Excellent (O(1) average)
Range: WHERE key BETWEEN ? AND ?Poor (reverts to scan)
Partial key: only first column of 2-col indexPoor (full scan)
ORDER BY on indexed columnNot supported

4. Deep Dive — B+ Tree Internals

B+ Tree Structure

Every B+ tree index in PostgreSQL, MySQL InnoDB, and SQL Server consists of:

Level 0 (Root):       One root page — entry point for ALL lookups.
Level 1..N-1 (Internal): Pages containing key ranges + pointers to lower pages.
Level N (Leaf):       Pages containing actual key values + heap pointers.
                      Leaf pages are doubly-linked → supports range scans.

Leaf page doubly-linked list:

← Page 3 ← [Page 4: keys 400-499] ↔ [Page 5: keys 500-599] ↔ [Page 6: keys 600-699] → Page 7 →

Once the tree finds the start of a range (e.g., 450), it simply walks the linked list forward to collect all values up to 599 — no need to re-traverse the tree. This is why B+ trees are so efficient for range queries.

Tree height for common table sizes:

RowsApproximate B+ Tree Height
1,0002 levels
1,000,0003 levels
1,000,000,0004 levels

Height ≈ log_b(N) where b is the branching factor (typically 100-400 for integer keys). This is why B+ tree lookups cost only 3-4 IOs regardless of whether the table has a million or a billion rows.

Page Fill Factor

SQL Server (and PostgreSQL with fillfactor) allow configuring how full each index page is when built or rebuilt.

-- SQL Server: 80% fill factor
CREATE INDEX IX_orders_date ON orders (order_date)
WITH (FILLFACTOR = 80);
-- Leaves 20% of each page empty for future inserts
-- Reduces page splits on insert-heavy tables
-- Costs slightly more storage upfront

Page split (what happens without enough fill factor):

Page 5 is full: [400, 450, 470, 480, 490, 500]  ← All 6 slots used

New insert: key=460

Step 1: Allocate new page 5b: [460, 470, 480, 490, 500] → page 5b
        Keep in page 5:       [400, 450]

Step 2: Update parent internal node to point to both page 5 and 5b.

Cost: 2 page writes + 1 parent page write = expensive under heavy insert load.
FILLFACTOR = 80 would have left room for 460 in page 5 → no split needed.

Sort Order in Index

Indexes can be created with ascending or descending sort per column:

-- SQL Server: descending on rejection rate, ascending on product
CREATE NONCLUSTERED INDEX IX_PurchaseOrderDetail_RejectedQty
ON Purchasing.PurchaseOrderDetail
(RejectedQty DESC, ProductID ASC, DueDate, OrderQty);

-- This exactly matches the query:
SELECT RejectedQty, ProductID, DueDate
FROM Purchasing.PurchaseOrderDetail
ORDER BY RejectedQty DESC, ProductID ASC;
-- Without matching index: SORT operator in query plan (expensive)
-- With matching index: no SORT needed → query plan cost drops from 86% to ~1%

Key rule: If an ORDER BY clause matches the index key order exactly (or exactly reversed), the database can eliminate the SORT operation from the query plan. This is significant for large result sets where sorting costs dominate.

6. Creating Indexes in Production — CONCURRENTLY

This is one of the most operationally important concepts in the lectures.

The Problem: Standard CREATE INDEX Blocks Writes

-- DANGEROUS in production:
CREATE INDEX idx_employees_name ON employees (name);
-- Acquires SHARE lock on the table.
-- All INSERT, UPDATE, DELETE are BLOCKED until index is built.
-- For 11M rows: several seconds to minutes of write downtime.

On a busy production table, this can cause:

  • Request timeouts across the entire application.
  • Queue buildup in connection pools.
  • Cascading failures in downstream services.

The Solution: CREATE INDEX CONCURRENTLY

-- SAFE in production:
CREATE INDEX CONCURRENTLY idx_employees_name ON employees (name);
-- No table lock. Reads AND writes continue normally.
-- Index build takes longer but production traffic is unaffected.

How CONCURRENTLY works internally:

  1. First pass: Scan the heap and build an initial index snapshot. During this scan, all existing rows are indexed. New writes are tracked but not yet included.
  2. Wait: Database waits for all transactions that started before the first pass to complete (so no transaction sees an inconsistent state).
  3. Second pass: Scan again to pick up changes that occurred during the first pass.
  4. Wait again: Wait for all transactions from second pass to finish.
  5. Mark valid: Index is marked as valid and the planner begins using it.

Costs of CONCURRENTLY:

  • Takes significantly longer (multiple scans vs one scan).
  • Consumes more CPU and memory during build.
  • Cannot be run inside a transaction block.

Can fail: If the index is unique and a duplicate value is inserted while the index is being built, the index ends up in an INVALID state. You must then:

-- Check for invalid indexes
SELECT indexname, indexdef FROM pg_indexes WHERE tablename = 'employees';
-- Or:
SELECT * FROM pg_stat_user_indexes WHERE idx_scan = 0;

-- Drop and recreate
DROP INDEX CONCURRENTLY idx_employees_name;
CREATE INDEX CONCURRENTLY idx_employees_name ON employees (name);

SQL Server Equivalent: ONLINE = ON

-- SQL Server equivalent of CONCURRENTLY:
CREATE NONCLUSTERED INDEX IX_employees_name
ON employees (name)
WITH (ONLINE = ON);
-- Allows concurrent reads AND writes during index creation.
-- Available in SQL Server Enterprise/Developer editions.

7. The Query Planner — How the Database Decides to Use Indexes

The query planner is a component that analyzes a query and determines the most efficient execution strategy. It may choose to use an index, ignore it, or combine multiple indexes.

When the Planner Chooses NOT to Use an Index

Small tables: For a table with 100 rows (a few pages), a heap scan costs 1-2 IOs. A B+ tree index traversal also costs 2-3 IOs (tree height). The index provides no benefit. The planner correctly ignores the index.

Low selectivity: An index on a column with 2 distinct values (e.g., gender = 'M' or 'F') matches 50% of rows. Reading 50% of the heap randomly is more expensive than a sequential heap scan. The planner ignores the index.

Non-SARGable predicates: As covered — functions or expressions on the indexed column prevent index use.

Large result sets without LIMIT: If your query returns 80% of the rows, a sequential scan is more efficient than 8,000,000 individual heap page fetches via an index.

Stale statistics: The planner uses table statistics (maintained by ANALYZE / VACUUM ANALYZE) to estimate row counts and choose plans. Stale statistics lead to wrong cardinality estimates → wrong plan choices. Always run ANALYZE after bulk loads.

Forcing Index Use (Hints)

-- PostgreSQL: disable seq scan to force index use (for testing/debugging)
SET enable_seqscan = OFF;
SELECT * FROM employees WHERE id = 10000;
SET enable_seqscan = ON;  -- Always reset!

-- SQL Server: index hint
SELECT * FROM employees WITH (INDEX(IX_employees_name))
WHERE name = 'Alice';

-- MySQL: force index
SELECT * FROM employees FORCE INDEX(idx_name)
WHERE name = 'Alice';

Warning: Index hints override the planner. If the planner was right to ignore your index, you'll make things worse. Use hints only for debugging, never in production code without understanding the full impact.

8. Clustered vs Non-Clustered Indexes — Full Reference

Clustered Index

PropertyDescription
Physical organizationTable rows ARE the B+ tree leaf nodes
Leaf nodes containFull row data
Count per tableOne only (one physical ordering)
Range query performanceExcellent — rows are consecutive on disk
Insert performanceDepends on key distribution (see UUID problem)
Key recommendationSequential, narrow, unique, not frequently updated
MySQL behaviorPRIMARY KEY is always the clustered index
PostgreSQL behaviorNo native clustered index; use CLUSTER command to physically reorder (one-time)
SQL Server behaviorOptional; CREATE CLUSTERED INDEX or implied by PRIMARY KEY
Oracle behaviorHeap by default; IOT (Index Organized Table) for clustered

What columns make a good clustered index key (SQL Server guidance):

  • Frequently used in range queries (BETWEEN, >, <)
  • High uniqueness (many distinct values)
  • Accessed sequentially (auto-increment ID, timestamp)
  • Defined as IDENTITY / SERIAL
  • Short key length (secondary indexes store the clustering key → wide key = larger all secondary indexes)

What columns make a BAD clustered index key:

  • Randomly distributed (UUID v4) → page split on every insert
  • Frequently updated → whole row must physically move
  • Wide composite key → all secondary indexes bloat proportionally

Non-Clustered (Secondary) Index

PropertyDescription
Physical organizationSeparate B+ tree; leaf nodes contain key + pointer
Leaf nodes containKey value + row locator (RID in heaps, clustering key in clustered tables)
Count per tableMany (limited by maintenance overhead, not a hard architectural limit)
Range query performanceEach result row may require a separate heap page fetch (random IO)
Heap penaltyEach key match requires an additional IO to the heap for non-indexed columns
PostgreSQL specificsALL indexes are secondary; leaf nodes store ctid (physical row location)

The double-lookup in MySQL non-clustered indexes:

MySQL: table has clustered index on id, secondary index on last_name.

Secondary index leaf node: {last_name='Smith', primary_key=42}
                                                   ↓
Lookup: secondary index → find PK=42 → clustered index lookup → find actual row data

Two B+ tree traversals for one row lookup.
(PostgreSQL: index → ctid → heap page — only one extra lookup, not two tree traversals)

SQL Server Row Locator Logic

If table has a clustered index:
  Nonclustered index leaf node stores: key columns + clustering key value
  Lookup path: nonclustered B+ tree → clustering key → clustered B+ tree → row data

If table is a heap (no clustered index):
  Nonclustered index leaf node stores: key columns + RID (file:page:slot)
  Lookup path: nonclustered B+ tree → RID → heap page → row data

9. EXPLAIN ANALYZE — Reading Query Plans

The most important tool for understanding index behavior is EXPLAIN ANALYZE.

PostgreSQL EXPLAIN ANALYZE

EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT id, grade FROM students
WHERE grade BETWEEN 80 AND 95
ORDER BY grade DESC;

Reading the output:

Index Only Scan Backward using students_grade_id_idx on students
  (cost=0.56..15234.12 rows=6000000 width=8)
  (actual time=0.042..4123.456 rows=5987234 loops=1)
  Index Cond: ((grade > 80) AND (grade < 95))
  Heap Fetches: 0
  Buffers: shared hit=20345 read=602
Planning Time: 0.834 ms

Decoding each field:

JDBC — Demonstrating Scan Type Differences

import java.sql.*;

public class IndexDemonstration {

    private final Connection conn;

    /**
     * Demonstrates the full table scan — no index on the WHERE column.
     * For 11M rows, this takes ~3 seconds.
     */
    public void demonstrateFullTableScan(String nameValue) throws SQLException {
        // EXPLAIN output will show: Parallel Seq Scan
        // Rows Removed by Filter: ~10,999,999
        String sql = "SELECT id FROM employees WHERE name = ?";
        long start = System.currentTimeMillis();
        try (PreparedStatement stmt = conn.prepareStatement(sql)) {
            stmt.setString(1, nameValue);
            ResultSet rs = stmt.executeQuery();
            int count = 0;
            while (rs.next()) count++;
            System.out.printf("Full scan: found %d rows in %dms%n",
                count, System.currentTimeMillis() - start);
        }
    }

    /**
     * Demonstrates index scan — index on id, fetching name (not in index).
     * For 11M rows, this takes ~2.5ms (index found id, heap fetch for name).
     */
    public void demonstrateIndexScan(long employeeId) throws SQLException {
        // EXPLAIN output will show: Index Scan, Heap Fetches: 1
        String sql = "SELECT name FROM employees WHERE id = ?";
        long start = System.currentTimeMillis();
        try (PreparedStatement stmt = conn.prepareStatement(sql)) {
            stmt.setLong(1, employeeId);
            ResultSet rs = stmt.executeQuery();
            if (rs.next()) {
                System.out.printf("Index scan: found '%s' in %dms%n",
                    rs.getString("name"), System.currentTimeMillis() - start);
            }
        }
    }

    /**
     * Demonstrates index only scan — fetching only the indexed column.
     * For 11M rows, this takes ~0.1ms (no heap fetch needed).
     */
    public void demonstrateIndexOnlyScan(long employeeId) throws SQLException {
        // EXPLAIN output will show: Index Only Scan, Heap Fetches: 0
        String sql = "SELECT id FROM employees WHERE id = ?";
        long start = System.currentTimeMillis();
        try (PreparedStatement stmt = conn.prepareStatement(sql)) {
            stmt.setLong(1, employeeId);
            ResultSet rs = stmt.executeQuery();
            if (rs.next()) {
                System.out.printf("Index only scan: found id=%d in %dms%n",
                    rs.getLong("id"), System.currentTimeMillis() - start);
            }
        }
    }

    /**
     * Demonstrates the covering index benefit.
     * Before: SELECT id, grade WHERE grade BETWEEN 80 AND 95 → 16s (heap fetches for id)
     * After:  CREATE INDEX ON students(grade) INCLUDE (id) → 4s (index only)
     */
    public List<Map<String, Object>> demonstrateCoveringIndex(int minGrade, int maxGrade)
            throws SQLException {
        // With covering index: Index Only Scan, Heap Fetches: 0
        // Without covering index: Index Scan, Heap Fetches: ~6,000,000
        String sql = "SELECT id, grade FROM students WHERE grade > ? AND grade < ? ORDER BY grade DESC";
        List<Map<String, Object>> results = new ArrayList<>();
        try (PreparedStatement stmt = conn.prepareStatement(sql)) {
            stmt.setInt(1, minGrade);
            stmt.setInt(2, maxGrade);
            ResultSet rs = stmt.executeQuery();
            while (rs.next()) {
                results.add(Map.of(
                    "id", rs.getLong("id"),
                    "grade", rs.getInt("grade")
                ));
            }
        }
        return results;
    }

    /**
     * The SARGability trap — demonstrates when LIKE breaks index usage.
     */
    public void demonstrateSargability(String name) throws SQLException {
        // Case 1: SARGable — uses index (prefix match)
        String sargable = "SELECT id FROM employees WHERE name LIKE ?";
        // LIKE 'Ali%' → index scan ✅

        // Case 2: Non-SARGable — full table scan despite having an index
        String nonSargable = "SELECT id FROM employees WHERE name LIKE ?";
        // LIKE '%Ali%' → parallel seq scan ❌

        try (PreparedStatement sarg = conn.prepareStatement(sargable)) {
            sarg.setString(1, name + "%");  // Prefix: SARGable
            // This uses the index.
        }

        try (PreparedStatement nonsarg = conn.prepareStatement(nonSargable)) {
            nonsarg.setString(1, "%" + name + "%");  // Leading wildcard: non-SARGable
            // This does a full table scan despite having an index on name.
        }
    }
}

Spring JPA Repository — Index-Aware Query Design

import org.springframework.data.jpa.repository.*;
import org.springframework.data.repository.query.Param;

public interface StudentRepository extends JpaRepository<Student, Long> {

    // BAD: Forces full table scan — no index on name, non-SARGable
    @Query("SELECT s FROM Student s WHERE UPPER(s.name) = UPPER(:name)")
    List<Student> findByNameCaseInsensitive(@Param("name") String name);
    // Wrapping name in UPPER() makes this non-SARGable even if name is indexed.

    // GOOD: Store names in normalized case, query directly
    @Query("SELECT s FROM Student s WHERE s.name = :name")
    List<Student> findByName(@Param("name") String name);
    // SARGable: uses index on name column ✅

    // GOOD: Range query on indexed column — uses index
    @Query("SELECT s.id, s.grade FROM Student s WHERE s.grade > :min AND s.grade < :max ORDER BY s.grade DESC")
    List<Object[]> findGradesInRange(@Param("min") int min, @Param("max") int max);
    // If index exists on grade (or grade INCLUDE id): Index Only Scan ✅

    // GOOD: Pagination with index — avoids large result sets
    Page<Student> findByGradeBetween(int minGrade, int maxGrade, Pageable pageable);
    // Spring adds LIMIT/OFFSET. Index limits rows scanned. ✅

    // GOOD: Atomic increment — no read-then-write race
    @Modifying
    @Query("UPDATE Student s SET s.grade = s.grade + :increment WHERE s.id = :id")
    int incrementGrade(@Param("id") Long id, @Param("increment") int increment);
}

11. Index Design Strategy — The Decision Framework

Step 1: Understand Your Queries

Before creating any index, answer these questions:

What columns appear in WHERE clauses?     → Key columns of the index
What columns appear in ORDER BY / GROUP BY? → Key columns (enables sort elimination)
What columns appear in SELECT?            → Included/non-key columns
Are the predicates equality or range?     → Equality first, range last in composite index
How many rows does the query typically return? → High selectivity → good index candidate
How often does the query run?             → Frequency × cost = priority

Step 2: Selectivity Analysis

-- PostgreSQL: estimate selectivity of a column
SELECT COUNT(DISTINCT grade) AS distinct_values,
       COUNT(*) AS total_rows,
       round(COUNT(DISTINCT grade)::numeric / COUNT(*) * 100, 2) AS selectivity_pct
FROM students;

-- Low selectivity (< 1% distinct values): index rarely helpful for this column alone
-- High selectivity (> 10% distinct values): index very useful
-- 100% selectivity (all unique): perfect index candidate

Step 3: The Index Design Template

Query:  SELECT [col_a, col_b, col_c]
        FROM table
        WHERE col_x = ? AND col_y BETWEEN ? AND ?
        ORDER BY col_y DESC

Index design:
  Key columns:      (col_x, col_y DESC)   ← search columns, equality before range
  Included columns: (col_a, col_b, col_c) ← select columns, no key overhead

CREATE INDEX idx_table_xy ON table (col_x, col_y DESC) INCLUDE (col_a, col_b, col_c);

Step 4: The Trade-Off Matrix

FactorMore IndexesFewer Indexes
SELECT performanceBetterWorse
INSERT performanceWorseBetter
UPDATE performanceWorse (all indexes updated)Better
DELETE performanceWorse (all indexes updated)Better
StorageHigherLower
VACUUM timeHigherLower
Write-heavy OLTP❌ Over-indexing hurts✅ Keep indexes minimal
Read-heavy analytics✅ Many indexes help❌ Under-indexing hurts

General rules (SQL Server guidelines):

  • OLTP tables: keep indexes narrow and few. Over-indexing is as bad as under-indexing.
  • Read-only / DSS tables: as many indexes as needed. No write cost.
  • Rebuild/reorganize fragmented indexes regularly (ALTER INDEX ... REORGANIZE/REBUILD).

12. Important Interview Questions

Basic Questions

Q: What is a database index?
A: An index is a separate data structure (usually a B+ tree) built on one or more columns of a table. It stores column values along with pointers to the heap pages where the corresponding rows live. This allows the database to find rows matching a WHERE clause in O(log n) time instead of O(n) time (full table scan).

Q: What is the difference between Index Scan, Index Only Scan, and Sequential Scan?
A: Sequential scan reads every page of the heap in order (O(n)). Index scan uses the B+ tree to find matching rows, then fetches each row from the heap (additional IO per row). Index Only Scan uses the B+ tree only — never touches the heap — because all needed columns are already in the index. Index Only Scan is the fastest; it requires all SELECT and WHERE columns to exist in the index.

Q: What is a covering index?
A: A covering index contains all columns referenced by a query — both the search (key) columns and the retrieval (SELECT) columns. This allows the query to be answered entirely from the index without accessing the heap, resulting in an Index Only Scan and dramatically fewer IOs.

Q: What is a SARGable predicate?
A: A SARGable (Search ARGument-able) predicate can use an index to speed up query execution. Predicates using =, >, <, BETWEEN, and LIKE 'prefix%' are SARGable. Predicates that wrap the indexed column in a function (UPPER(name), YEAR(date)) or expression (salary * 1.1) are non-SARGable and force a full table scan.

Q: Why is LIKE '%Alice%' slow even with an index?
A: The leading wildcard means the B+ tree cannot be used because the starting character of the value is unknown. The index is sorted by value prefix — %Alice% has no defined prefix, so the entire tree would need to be scanned. The planner correctly bypasses the index and does a full table scan instead.

Intermediate Questions

Q: What is the left-prefix rule for composite indexes?
A: A composite index on (A, B, C) can only be used if the query provides values starting from the leftmost key column. A query on (A) or (A, B) or (A, B, C) uses the index. A query on (B) or (C) or (B, C) does not use the index because the first column A has no constraint.

Q: What is the difference between a key column and an included (non-key) column in an index?
A: Key columns are stored at all levels of the B+ tree and determine the sort order; they enable filtering and sorting. Included columns are stored only at the leaf level and are used only for data retrieval (covering queries). Key columns count toward size limits (900 bytes max in SQL Server); included columns do not. Use included columns for SELECT-only columns to achieve index only scans without bloating the key.

Q: Why does CREATE INDEX block writes, and how do you avoid this in production?
A: Standard CREATE INDEX acquires a SHARE lock on the table, preventing all INSERT, UPDATE, DELETE operations until the index is fully built. For large tables, this can take minutes. CREATE INDEX CONCURRENTLY (PostgreSQL) performs multiple scans, waits for concurrent transactions between scans, and never takes a blocking lock. It takes longer and uses more resources, but allows writes and reads throughout. SQL Server's equivalent is WITH (ONLINE = ON).

Q: What is a filtered index and when should you use one?
A: A filtered index only indexes rows matching a predicate (e.g., WHERE status = 'ACTIVE'). It is smaller, cheaper to maintain, and more selective than a full-table index. Use filtered indexes for: sparse columns (mostly NULL), soft-deleted records, status-based queries on a subset of data, or any column with a "hot" subset that most queries target.

Q: What happens to non-clustered indexes in MySQL vs PostgreSQL when a row is updated?
A: In MySQL (clustered primary key), secondary indexes store the primary key value as the row locator. When a non-indexed column is updated, the clustering key doesn't change, so secondary indexes don't need updating. In PostgreSQL (MVCC, all indexes store ctid), every update creates a new tuple with a new ctid, so ALL indexes on the table must be updated to point to the new ctid — even if the indexed column didn't change. This is the PostgreSQL write amplification problem mitigated partially by the HOT optimization.

Advanced Questions

Q: How does the query planner decide between a sequential scan and an index scan?
A: The planner uses cost-based estimation. It estimates the cost of each plan using table statistics (row count, distinct values, data distribution maintained by ANALYZE). If the index is highly selective (few matching rows), the index scan cost (log(n) + k heap fetches) is lower than the sequential scan cost (all pages). If the query matches >10-20% of rows, a sequential scan may be cheaper because it reads pages sequentially (which is faster than random IO from index-guided heap fetches). Stale statistics (not running ANALYZE) can lead the planner to make wrong choices.

Q: What is the deltastore in a SQL Server columnstore index and why does it exist?
A: The deltastore is a B-tree rowstore that holds newly inserted rows before they can be compressed into the columnstore. Columnstore compression requires large batch sizes (minimum 102,400 rows per rowgroup for good compression). Small individual inserts go to the deltastore first. A background process called the tuple-mover monitors the deltastore and compresses completed rowgroups (up to 1,048,576 rows) into the columnstore. This design allows OLTP-style individual inserts while maintaining columnstore's compression benefits.

Q: What is the Bw-Tree (in-memory nonclustered index in SQL Server) and how does it differ from a B+ Tree?
A: The Bw-Tree (Buzzword Tree) is a lock-free and latch-free variation of a B-Tree used for SQL Server's memory-optimized (Hekaton) nonclustered indexes. Unlike a B+ tree, it never performs in-place updates. Instead, modifications create delta records that are prepended to a linked chain on the affected page. A page mapping table (indirection layer) translates logical page IDs to physical memory addresses. When delta chains grow too long (16+ records), a consolidation merges all deltas into a new page. Page splits and merges are atomic multi-step operations. This design enables lock-free concurrent access because updates never mutate existing pages — they only prepend to chains and atomically update the mapping table pointer.

Q: How do you decide optimal bucket count for a hash index?
A: The target is 1-2x the number of distinct key values. With too few buckets, hash collisions create long linked lists per bucket → O(chain_length) equality lookup instead of O(1). With too many buckets, memory is wasted on empty buckets and full index scans take longer (must traverse all empty buckets). If uncertain, overestimate rather than underestimate — performance degrades more with collisions than empty buckets. The count can be changed with ALTER TABLE ... ALTER INDEX REBUILD.

13. Common Mistakes

Beginner Mistakes

  • Over-indexing every column: "If one index is good, ten must be better." Each extra index adds write overhead on every INSERT/UPDATE/DELETE. A table with 15 indexes requires 15 separate B+ tree updates per insert.
  • Using SELECT * and expecting index only scans: SELECT * fetches all columns, most of which are not in the index. Every index lookup must be followed by a heap fetch. You lose the benefit of the index covering your query.
  • Indexing low-cardinality columns: Indexing a boolean is_active (2 distinct values) on a table where 95% of rows are is_active = true creates an index that matches 95% of rows. The optimizer correctly ignores it for the common case. Use a filtered index instead.
  • Ignoring VACUUM in PostgreSQL: Index Only Scans fall back to heap fetches if the visibility map is stale. After heavy writes, always run VACUUM to update visibility maps.

Backend Engineer Mistakes

  • Non-SARGable queries in application code: Writing ORM queries that call functions on indexed columns (UPPER(name), DATE(created_at), LOWER(email)). These bypass indexes. Normalize data at write time instead of transform at read time.
  • Not running EXPLAIN on production queries: Assuming a query "should" use an index without verifying with EXPLAIN ANALYZE. The planner may make a different choice based on statistics, table size, or predicate structure.
  • Dropping indexes without checking usage first: Removing an index that "looks unused" without checking pg_stat_user_indexes. A nightly batch job that runs once a day at 2am will show low idx_scan counts but removing the index makes that job go from 30s to 4h.
  • Standard CREATE INDEX on a live production table: Taking the table write lock for 5 minutes in production is a critical incident. Always use CONCURRENTLY or ONLINE.
  • Using LIKE '%value%' for search: Leading wildcard = full table scan. For full-text search, use PostgreSQL tsvector + GIN index, or Elasticsearch, not LIKE '%value%'.

Production Mistakes

  • Not monitoring index fragmentation: Over time, B+ tree pages become partially empty due to deletions and page splits. A 90% fragmented index reads twice as many pages for the same query. Rebuild (rewrites entirely) or reorganize (in-place defragment) indexes periodically.
  • Creating unique CONCURRENTLY indexes while duplicates can exist: If a concurrent write introduces a duplicate between the two scans, the index is created in INVALID state — not usable, but still maintained (write overhead with no read benefit). Monitor for INVALID indexes after every CONCURRENTLY build.
  • Hash index bucket count set at table design time, never reviewed: As the table grows, the bucket count becomes proportionally too small → collisions grow → lookup performance degrades. Re-evaluate bucket count when table size doubles.
  • Ignoring index maintenance after bulk loads: After inserting millions of rows, statistics are stale, index fragmentation is high, visibility maps are out of date. Run: VACUUM ANALYZE table_name; after every significant bulk operation.

14. Performance & Scalability Notes

Index Size Estimation

Index size ≈ key_bytes_per_entry × number_of_rows × 1.2 (overhead factor)

Example: Index on (grade INTEGER, id BIGINT) — 4 + 8 = 12 bytes/entry
  50M rows × 12 bytes × 1.2 = 720MB

Example: Covering index on (grade) INCLUDE (id) — same storage, slightly different structure
  Key: 4 bytes (grade) + 6 bytes (ctid) = 10 bytes in internal nodes
  Leaf: 4 bytes (grade) + 8 bytes (id non-key) + 6 bytes (ctid) = 18 bytes in leaf nodes
  Leaf dominates: 50M × 18 × 1.2 ≈ 1GB

Rule of thumb: If the index fits in the buffer pool → lookups are RAM speed.
               If the index exceeds the buffer pool → each lookup may hit disk.

Index Maintenance Cost Under Write Load

Table: 1M row inserts per day, 10 indexes.
Each insert: 1 heap write + 10 index B+ tree updates
Total writes: 11x amplification per inserted row.

With 10 indexes, write throughput is limited to ~1/11 of the raw disk write capacity.

Implication: For high-write-throughput systems (IoT ingestion, clickstream, logging):
  - Minimize indexes (1-2 critical ones max)
  - Consider LSM-tree based stores (RocksDB, Cassandra) which batch writes efficiently
  - Or buffer writes and batch-insert to reduce index maintenance overhead

Buffer Pool Hit Rate and Index Performance

Index Lookup Cost:
  All index levels in buffer pool (cache hit): ~0.1ms (RAM speed)
  One level in buffer pool, one disk read: ~0.2ms
  No levels in buffer pool (cold cache): ~0.4ms (NVMe) to 10ms (HDD) per level

For B+ tree height 3: hot cache → 0.3ms, cold cache → 1.2ms (NVMe) to 30ms (HDD)

Heap fetch cost (one page):
  Buffer pool hit: ~0.05ms
  Disk read: ~0.1ms (NVMe) to 10ms (HDD)

Conclusion: A query that does 90,000 disk reads (index + heap) takes:
  NVMe: 90,000 × 0.1ms = 9 seconds
  HDD:  90,000 × 10ms = 15 minutes (!)

E-Commerce Search Index Strategy

Products table: 100M rows (id, name, category, price, stock, created_at, description)

Required queries:
  1. Product by ID (PK lookup): CREATE INDEX → PK (default)
  2. Browse by category + price: CREATE INDEX ON products (category, price) INCLUDE (name, stock)
  3. New arrivals: CREATE INDEX ON products (created_at DESC) INCLUDE (name, price, category)
  4. Search by name: PostgreSQL tsvector + GIN index (full text search)
  5. In-stock only: CREATE INDEX ON products (category, price) WHERE stock > 0  ← filtered index

  What NOT to index:
  - description (large text, never in WHERE clause)
  - created_at alone (only useful when combined with category)

Financial System — Covering Index for Transaction Queries

transactions table: 5B rows (id, account_id, amount, type, status, created_at, processed_at)

Critical query (account statement):
  SELECT id, amount, type, created_at
  FROM transactions
  WHERE account_id = ? AND created_at > ? AND created_at < ?
  ORDER BY created_at DESC
  LIMIT 100

Index design:
  CREATE INDEX idx_txn_account_date
  ON transactions (account_id, created_at DESC)
  INCLUDE (id, amount, type);

Result:
  - account_id (equality predicate): leftmost key column ✅
  - created_at DESC (range predicate + ORDER BY): second key column, matching sort ✅
  - id, amount, type (SELECT columns): included → Index Only Scan ✅
  - SORT operator eliminated: ORDER BY matches index order ✅

Microservices — Index Strategy Per Service

In microservices, each service owns its database. Index design must be per-service, optimized for that service's specific query patterns:

User Service (PostgreSQL):
  Table: users (id, email, username, created_at, status)
  Indexes:
    UNIQUE (email)            -- Login lookup
    UNIQUE (username)         -- Profile lookup
    (created_at, status)      -- Admin dashboard: new inactive users
    (status) WHERE status = 'PENDING'  -- Filtered: activation queue only

Order Service (MySQL):
  Table: orders (id, user_id, status, total, created_at)
  Indexes:
    (user_id, created_at DESC) INCLUDE (status, total)  -- Order history per user
    (status, created_at) WHERE status = 'PENDING'       -- Order processing queue
    PK (id) → always clustered in MySQL

16. Visual Mental Models

The Complete Index Lookup Path

Query: SELECT name FROM employees WHERE id = 5000

Step 1: B+ Tree traversal (index pages)
  Root node: id=5000 → go right
  Internal node: id=5000 between 4000 and 6000 → go to leaf
  Leaf node: {id=5000, ctid=(page=625, row=3)}  ← found! 3 IOs

Step 2: Heap fetch (one more IO)
  Read heap page 625 → scan to row 3 → extract name='Alice'  ← 1 IO

Total: 4 IOs

vs. Full table scan: 625,000 IOs (10M rows / 16 rows per page)

Index Only Scan — No Heap Touch

Query: SELECT id, grade FROM students WHERE grade BETWEEN 80 AND 95

Covering index: (grade) INCLUDE (id)
  Leaf nodes: {grade=80, id=234567} {grade=81, id=891234} ... {grade=94, id=445678}

B+ tree scan:
  Find grade=80 → walk linked list of leaf pages → grade=95 → stop

All needed values (grade AND id) are in the leaf nodes.
→ Index Only Scan: ZERO heap fetches.
→ Heap stays completely untouched.

SARGable vs Non-SARGable Decision Tree

Does your WHERE clause look like:
                    ↓
        [column] = [value]?              → SARGable ✅ (uses index)
        [column] > / < / BETWEEN ?      → SARGable ✅ (uses index)
        [column] LIKE 'prefix%' ?       → SARGable ✅ (uses index)
        [column] LIKE '%middle%' ?      → NOT SARGable ❌ (full scan)
        FUNCTION([column]) = [value] ?  → NOT SARGable ❌ (full scan)
        [column] * factor = [value] ?   → NOT SARGable ❌ (full scan)
        [column] != [value] ?           → Usually NOT SARGable ❌

Index Type Selection Guide

What type of queries do you run?
  ├── Exact equality (WHERE id = ?)
  │     ├── In-memory table: Hash Index (O(1) lookup)
  │     └── Disk table: B+ Tree Index (O(log n) lookup)
  ├── Range queries (BETWEEN, >, <)
  │     └── B+ Tree Index ✅ (Hash index can't do this)
  ├── Full-text search (LIKE '%word%')
  │     └── GIN index on tsvector (PostgreSQL) / Full-text index
  ├── Spatial data (nearest neighbor, within radius)
  │     └── GiST index / Spatial index
  ├── Only a subset of rows match your filter
  │     └── Filtered / Partial Index (smaller, faster, cheaper)
  ├── Analytics on columns (SUM, AVG, GROUP BY)
  │     └── Columnstore Index (SQL Server) / Column-store engine
  └── Prefix/autocomplete (LIKE 'prefix%')
        └── B+ Tree Index (SARGable prefix) ✅

WHAT IS AN INDEX:
  Separate B+ tree data structure, smaller than heap.
  Leaf nodes: key values + heap pointers (ctid in PG, PK in MySQL).
  Goal: O(log n) row lookup vs O(n) full scan.
  Cost: write amplification — every write updates all indexes.

SCAN TYPES (best to worst):
  1. Index Only Scan  → Heap Fetches: 0 → all columns in index ← GOAL
  2. Index Scan       → Heap Fetches: N → some columns in heap
  3. Bitmap Index Scan → Batches heap fetches → better than index scan for many rows
  4. Sequential Scan  → Full table → 625,000 IOs for 10M rows ← AVOID

SARGABILITY (index use requires this):
  ✅ col = val, col > val, col BETWEEN a AND b, col LIKE 'prefix%'
  ❌ FUNC(col) = val, col * factor = val, col LIKE '%middle%', col != val

COMPOSITE INDEX DESIGN:
  Order: equality columns first, range column last
  (account_id, created_at DESC) → equality on account_id, range on date
  Left-prefix rule: queries must include leftmost columns

KEY vs INCLUDED (NON-KEY) COLUMNS:
  Key: used for searching/filtering, stored all levels, counts toward size limit
  Include: SELECT-only columns, leaf level only, doesn't count toward limit
  → Covering index = all SELECT + WHERE columns present → Index Only Scan

FILTERED INDEX:
  Index only a subset: WHERE status = 'PENDING'
  Smaller, faster, cheaper to maintain than full-table index

CREATE INDEX CONCURRENTLY:
  No write lock → production safe
  Takes longer, uses more CPU/RAM
  Can create INVALID index on unique constraint violation → always verify

REAL PERFORMANCE (PostgreSQL, 11M rows):
  Index only scan (id → id):   0.1ms
  Index scan (id → name):      2.5ms (heap fetch cost)
  Full table scan (no index):  3,000ms
  LIKE '%XZ%' (has index):     1,100ms (index ignored!)

CLUSTERED vs SECONDARY:
  Clustered: table IS the index (MySQL PK always, SQL Server optional)
             Range queries excellent, one per table
  Secondary: separate B+ tree with heap pointers
             Extra heap fetch per row, many per table

UUID DANGER in MySQL:
  UUID v4 = random PK = random insertions into clustered index = page splits
  Use: BIGINT AUTO_INCREMENT, UUID v7 (time-ordered), or ULID

POSTGRESQL SPECIFICS:
  All indexes are secondary (ctid pointer)
  UPDATE creates new ctid → ALL indexes updated (write amplification)
  VACUUM needed for Index Only Scans (updates visibility map)
  REPEATABLE READ = CONCURRENTLY safe for index creation

PRODUCTION RULES:
  ✅ EXPLAIN (ANALYZE, BUFFERS) every slow query
  ✅ Use CREATE INDEX CONCURRENTLY on production tables
  ✅ VACUUM ANALYZE after bulk loads
  ✅ Monitor idx_scan = 0 (unused indexes → remove them)
  ✅ Design for covering indexes on critical query paths
  ❌ Never wrap indexed columns in functions (non-SARGable)
  ❌ Never LIKE '%value%' on large tables (use full-text search)
  ❌ Never UUID v4 as clustered index key in MySQL
  ❌ Never over-index write-heavy tables
