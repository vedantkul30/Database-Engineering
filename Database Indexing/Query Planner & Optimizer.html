Query Planner & Optimizer

1. Big Picture

Why the Query Planner Exists

When you write SQL, you describe what you want — not how to get it. The query planner (also called the query optimizer) is the component that figures out the "how." For any given query, there may be dozens of valid execution plans. The planner's job is to estimate the cost of each plan and choose the cheapest one.

This distinction is fundamental: SQL is declarative. The programmer says "give me all students with grade > 90" — not "go to page 4500, read every row, check grade field, keep it if it's greater than 90." The planner translates the declaration into an efficient imperative execution plan.

Without a planner, every query would require the programmer to manually specify which indexes to use, in what order, with what join strategies. Databases would be unusable at scale.

The Problem It Solves

Same query, three possible plans:

Plan A: Full table scan (seq scan)
  → Read all 10M rows, filter in memory
  → Cost: 625,000 IOs

Plan B: Index scan on grade
  → Traverse B+ tree, fetch each matching row from heap
  → For 10,000 matching rows: 10,003 IOs (3 tree + 10,000 heap)

Plan C: Index only scan on grade (covering index)
  → Traverse B+ tree, all data in index
  → 3 IOs

The planner selects Plan C. Without a planner, you'd have to specify this manually every time.

Where the Planner Appears in Daily Backend Work

  • Understanding why a query is slow ("why isn't it using my index?")
  • Explaining performance regressions after data volume increases
  • Diagnosing stale statistics causing wrong plan choices
  • Designing indexes based on how the planner will use them
  • Writing non-SARGable queries that accidentally bypass indexes

Why Backend Engineers Must Understand the Planner

The planner is not perfect. It makes decisions based on statistics — approximations of the data distribution. When statistics are stale, wrong, or when data skew is extreme, the planner makes bad choices. Engineers who understand the planner can:

  • Read EXPLAIN output and immediately diagnose why a query is slow
  • Anticipate when a newly created index won't be used
  • Know when to run ANALYZE / update statistics
  • Avoid query patterns that make plans unpredictable
  • Know when and how to use query hints as a last resort

2. The EXPLAIN Command — Deep Dive

What EXPLAIN Does

EXPLAIN shows the query execution plan the planner has chosen — without actually executing the query. It answers: "If I run this query, what will the database do?"

EXPLAIN ANALYZE executes the query AND shows both the estimated and actual costs.
EXPLAIN (ANALYZE, BUFFERS) adds IO statistics (cache hits vs disk reads).

-- Just the plan (no execution):
EXPLAIN SELECT * FROM grades WHERE id = 10;

-- Plan + actual execution (runs the query):
EXPLAIN ANALYZE SELECT * FROM grades WHERE id = 10;

-- Plan + actual + IO stats (most detailed):
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT) SELECT * FROM grades WHERE id = 10;

The Anatomy of an EXPLAIN Output

Seq Scan on grades  (cost=0.00..289467.20 rows=12500000 width=31)
│              │      ├──────────┘   └──────────┘         └────────┘
│              │      │              │                      │
Node type   Table     Startup cost   Total cost estimate   Row width (bytes)
                        (ms)           (ms)

Breaking down each field:

Node Type (Scan Strategy)

The first word(s) of each line describe what operation the database chose:

Node TypeMeaning
Seq ScanFull table scan — reads every page in the heap
Index ScanB+ tree traversal → heap fetch per row
Index Only ScanB+ tree traversal → no heap access
Bitmap Index ScanBuild bitmap of matching pages from index
Bitmap Heap ScanFetch heap pages using the bitmap
SortSort the result set (triggered by ORDER BY)
Hash Join / Merge JoinJoin strategies
Parallel Seq ScanMulti-threaded sequential scan

Cost Numbers — cost=startup..total

cost=0.00..289467.20
      │       └── Total estimated cost to return ALL rows
      └────────── Startup cost — work done BEFORE returning first row

Startup cost is the time before the first row is returned. It increases when the planner must do pre-fetch work:

  • ORDER BY without a matching index → must sort all rows before returning any (high startup)
  • Aggregations (SUM, COUNT) → must scan everything first
  • Hash builds in Hash Joins

Total cost is the planner's estimated cost to return the complete result set. The unit is arbitrary — it is not milliseconds. It is a dimensionless cost unit used to compare plans relative to each other. Larger = more expensive.

No ORDER BY:  cost=0.00..289467.20
              ↑ Zero startup — immediately returns first row from heap

ORDER BY name (no index):  cost=1023456.00..1456789.00
                            ↑ Must sort EVERYTHING before returning row 1
                            High startup → bad for LIMIT queries

Why startup cost matters for LIMIT queries:

-- Without index on name, ORDER BY name forces sort of all rows:
SELECT * FROM grades ORDER BY name LIMIT 10;
-- Startup cost: ~1,000,000 (must sort 200M rows to return top 10)

-- With index on name, B+ tree is pre-sorted:
SELECT * FROM grades ORDER BY name LIMIT 10;
-- Startup cost: ~0.4 (walk to first leaf node, return 10 rows)

If startup cost is high and you're doing LIMIT, you're paying the full sort cost even to return just one row. An index on ORDER BY columns eliminates the sort entirely.

rows — Estimated Row Count

rows=12500000

This is the planner's estimate of how many rows this node will return. It comes from table statistics — the histogram, distinct value counts, and correlation data maintained by ANALYZE. This number is CRITICAL:

  • If rows estimate ≈ actual rows → planner has accurate statistics → good plan.
  • If rows estimate << actual rows → planner underestimates, may choose wrong plan.
  • If rows estimate >> actual rows → planner overestimates, may avoid index unnecessarily.

width — Row Width in Bytes

width=31

The average number of bytes per returned row. This is the sum of the average sizes of all selected columns.

EXPLAIN SELECT id FROM grades;           -- width=4   (INTEGER = 4 bytes)
EXPLAIN SELECT name FROM grades;         -- width=19  (TEXT average, varies by data)
EXPLAIN SELECT grade FROM grades;        -- width=8   (DOUBLE = 8 bytes)
EXPLAIN SELECT id, grade, name FROM ...; -- width=31  (4 + 8 + 19)

Why width matters:

  • Larger width → more bytes transferred over network (TCP packets)
  • Larger width → more memory used to hold the result set
  • SELECT only what you need — large BLOBs in SELECT * are especially wasteful

Reading Multi-Node Plans (Inner to Outer)

PostgreSQL's EXPLAIN output is a tree of nodes. You read it from the innermost (most indented) node upward — each node feeds its output to the node above it.

EXPLAIN SELECT * FROM grades ORDER BY name;
Gather Merge  (cost=... rows=...)         ← 4. Merge results from all workers
  Workers Planned: 2
  ->  Sort  (cost=... rows=...)           ← 3. Each worker sorts its partition
        Sort Key: name
        ->  Parallel Seq Scan on grades   ← 2. Each worker scans a portion
              (cost=0.00..218000 rows=...)
                                          ← 1. START HERE (innermost)

Reading order: bottom-up, innermost first.

Step 1: Parallel Seq Scan — 2 workers each read a portion of the heap
Step 2: Each worker sorts its portion (Sort node)
Step 3: Gather Merge — merges the sorted streams from all workers
Step 4: Final result returned to client

EXPLAIN ANALYZE — Actual vs Estimated

EXPLAIN ANALYZE adds actual execution statistics next to the estimates:

EXPLAIN ANALYZE SELECT name FROM grades WHERE id = 10;
Index Scan using grades_pkey on grades
  (cost=0.43..8.45 rows=1 width=19)
  (actual time=0.052..0.054 rows=1 loops=1)
  Index Cond: (id = 10)
  Heap Fetches: 1
Planning Time: 0.234 ms
Execution Time: 0.089 ms

New fields in ANALYZE output:

FieldMeaning

rows=1 (second one)Actual rows returned

Diagnosing plan problems with ANALYZE:

Estimated: rows=100
Actual:    rows=9,500,000

→ Planner severely underestimated row count
→ May have chosen index scan (thinking 100 rows) when seq scan would be better
→ Solution: Run ANALYZE to update statistics

EXPLAIN (ANALYZE, BUFFERS) — IO Statistics

EXPLAIN (ANALYZE, BUFFERS) SELECT id, grade FROM students
WHERE grade > 10 AND grade < 20 LIMIT 10000;
Index Only Scan Backward using students_grade_id_idx on students
  Buffers: shared hit=20 read=602
Buffer StatMeaning
shared hit=2020 pages served from the buffer pool (RAM) — no disk IO
shared read=602602 pages read from disk (physical IO operations)
shared dirtiedPages written (even reads can dirty pages in PostgreSQL — hint bits)
local hit/readTemporary tables buffer stats

Why this matters:

  • High shared hit + low shared read → good cache utilization → fast query.
  • High shared read → lots of physical disk IO → potentially slow, especially on HDD.
  • When you run the same query twice and it gets faster: shared read becomes shared hit because the OS/database page cache now holds those pages in RAM.

This explains the "second run is faster" phenomenon from the lecture:

First run:  shared hit=0, shared read=90000  → 90,000 disk reads → 11 seconds
Second run: shared hit=90000, shared read=0  → 0 disk reads → 1 second

The data is now in the OS page cache → RAM speed instead of disk speed.

3. Scan Types — Complete Reference

Sequential Scan (Full Table Scan)

Strategy: Read every page of the heap in order.
Trigger:  No usable index, or too many rows match to justify index overhead.
Cost:     O(N) — proportional to total table size.
EXPLAIN SELECT * FROM grades;
-- Seq Scan on grades (cost=0.00..289467.20 rows=12500000 width=31)

When the planner chooses sequential scan over index:

  • No index exists on the WHERE column.
  • Query matches too many rows (high selectivity → index overhead exceeds scan cost).
  • Table is small (few pages → scan costs same as index traversal).
  • Non-SARGable predicate (function on column, leading wildcard LIKE).
  • Statistics say the index will return > ~10-20% of rows.

Parallel Sequential Scan:
PostgreSQL can spin up multiple background workers to scan portions of the table in parallel, then merge results with Gather or Gather Merge.

Gather (cost=1000...)
  Workers Planned: 2
  -> Parallel Seq Scan on grades
       Filter: (name = 'XZ')
       Rows Removed by Filter: 6249890

The Rows Removed by Filter line shows how much work was wasted. Here, 6.25M rows were read and discarded — the signature of a missing index.

Index Scan

Strategy: Traverse B+ tree to find matching rows, then fetch each matching row
          from the heap (one heap page access per matching row).
Trigger:  Index exists on WHERE column, query is selective (few matching rows).
Cost:     O(log N + K) where K = number of matching rows.
EXPLAIN SELECT name FROM grades WHERE id = 1000;
-- Index Scan using grades_pkey on grades (cost=0.43..8.45 rows=1 width=19)
--   Index Cond: (id = 1000)
--   Heap Fetches: 1

The heap fetch cost: Each row found in the index requires a SEPARATE fetch to the heap to retrieve columns not stored in the index. For K matching rows, this is K random IOs.

Index scan breakdown for 99 rows:
  B+ tree traversal: 3 IOs
  99 heap page fetches: up to 99 IOs (if each row is on a different page)
  Total: ~102 IOs

vs. Sequential scan (small table):
  All pages: say 10 IOs
  Winner: Sequential scan for small tables or low-selectivity queries.

This is why the planner switches to sequential scan when many rows match — the 99+ heap fetches become more expensive than just scanning the whole table sequentially.

Index Only Scan

Strategy: Traverse B+ tree. All needed columns are in the index.
          The heap is NEVER accessed.
Trigger:  All SELECT + WHERE columns are covered by the index.
Cost:     O(log N + K) — but the K heap fetches are eliminated.
Goal:     Heap Fetches: 0 — this is the performance ideal.
EXPLAIN SELECT id FROM grades WHERE id = 10;
-- Index Only Scan using grades_pkey on grades (cost=0.43..4.45 rows=1 width=4)
--   Index Cond: (id = 10)
--   Heap Fetches: 0

PostgreSQL-specific: why Heap Fetches can be > 0 even on Index Only Scan:

PostgreSQL's MVCC uses a visibility map — a per-page bitmap recording whether all rows on a page are visible to all transactions. For an Index Only Scan, PostgreSQL checks the visibility map before returning a row. If the page is marked "all visible," no heap fetch is needed. If not (e.g., after recent writes or before VACUUM), PostgreSQL must fetch the heap page to determine row visibility.

Heap Fetches: 847 on an Index Only Scan
→ Visibility map is out of date
→ Solution: VACUUM students; (updates visibility map → Heap Fetches drops to 0)
-- Fix index only scan heap fetches:
VACUUM students;                        -- Quick, non-blocking
VACUUM VERBOSE students;               -- Shows details of what was cleaned
VACUUM ANALYZE students;               -- Vacuum + update statistics

Bitmap Index Scan + Bitmap Heap Scan

This is PostgreSQL's clever middle ground between index scan and sequential scan. It is the most subtle and most misunderstood scan type.

The Problem It Solves

Index Scan weakness: K random heap page IOs for K matching rows.
If K = 10,000 rows scattered across 10,000 different pages → 10,000 random IOs.

Sequential Scan weakness: reads ALL pages including irrelevant ones.
If only 10% of rows match → 90% of IO is wasted.

Bitmap Scan: "I'll scan the index to find WHICH pages I need,
              then fetch only those pages in sorted order."

How Bitmap Index Scan Works — Step by Step

Query: SELECT name FROM grades WHERE grade > 95

Step 1: Bitmap Index Scan on grade index
  Walk the B+ tree for grade > 95.
  For each matching entry:
    - Don't immediately fetch the heap row.
    - Instead, record the HEAP PAGE NUMBER in a bitmap.

  Bitmap (one bit per heap page, total pages ≈ 625,000):
  Bit 0: 0 (no matching rows on page 0)
  Bit 1: 0
  ...
  Bit 9: 1  ← page 9 has at least one grade > 95 row
  Bit 10: 0
  Bit 11: 1 ← page 11 has at least one grade > 95 row
  ...

Step 2: Sort bitmap by page number
  Pages to fetch (sorted): [9, 11, 47, 83, 102, ...]

Step 3: Bitmap Heap Scan
  Read heap pages in ORDER (page 9, then 11, then 47, ...)
  For each page, check ALL rows on that page against grade > 95.
  (Some rows on the page may not match → Recheck Cond)
  Return matching rows.
EXPLAIN SELECT name FROM grades WHERE grade > 95;
Bitmap Heap Scan on grades
  Recheck Cond: (grade > 95)
  -> Bitmap Index Scan on idx_grade
       Index Cond: (grade > 95)

Why "Recheck Cond"? When a bitmap page is marked "1," it means "at least one row on this page matches." But the page may contain OTHER rows that don't match. After fetching the page, PostgreSQL rechecks every row against the predicate to filter out non-matches. This recheck is cheap because the page data is already in memory from the fetch.

Bitmap AND / Bitmap OR — Multi-Index Magic

The most powerful feature of bitmap scans: multiple indexes can be combined.

Query: SELECT name FROM grades WHERE grade > 95 AND id < 10000

PostgreSQL can use BOTH indexes:
  Bitmap 1: from grade index → pages with grade > 95 rows
  Bitmap 2: from id index → pages with id < 10000 rows

AND operation (BitmapAnd):
  Bit 0: Bitmap1=0, Bitmap2=1 → 0 AND 1 = 0 (skip this page)
  Bit 9: Bitmap1=1, Bitmap2=1 → 1 AND 1 = 1 (fetch this page)
  Bit 11: Bitmap1=1, Bitmap2=0 → 1 AND 0 = 0 (skip this page)

Result: Only pages where BOTH conditions might be true get fetched.
EXPLAIN SELECT name FROM grades WHERE grade > 95 AND id < 10000;
Bitmap Heap Scan on grades
  Recheck Cond: ((grade > 95) AND (id < 10000))
  -> BitmapAnd
       -> Bitmap Index Scan on idx_grade
            Index Cond: (grade > 95)
       -> Bitmap Index Scan on grades_pkey
            Index Cond: (id < 10000)

Why this is genius: Before bitmap AND, 9,000 rows matched grade > 95. Before bitmap AND, hundreds of thousands matched id < 10000. After the AND operation, only the intersection (~394 rows in the lecture demo) needed to be fetched from the heap. Neither condition alone was selective enough — but together they are.

BitmapAnd result: rows to fetch reduced from ~9,000 + ~100,000 → ~394
Heap IO reduction: ~99.7% fewer random heap page reads

OR operation (BitmapOr):

EXPLAIN SELECT name FROM grades WHERE grade > 95 OR id < 10000;
Bitmap Heap Scan on grades
  -> BitmapOr
       -> Bitmap Index Scan on idx_grade
       -> Bitmap Index Scan on grades_pkey

OR creates the union of both bitmaps — more pages to fetch than AND, but still better than two separate sequential scans or two separate index scans without combining.

Scan Type Decision Summary

Planner decides which scan to use based on estimated row count and statistics:

Very few rows (high selectivity):
  → Index Scan (traverse tree, fetch each row individually)
  → If all cols in index: Index Only Scan

Medium rows (medium selectivity):
  → Bitmap Index Scan + Bitmap Heap Scan
  → Multiple indexes can be BitmapAnd / BitmapOr combined

Many rows (low selectivity) or no index:
  → Sequential Scan (Parallel Seq Scan if parallel_workers > 0)

The threshold is approximate: typically:
  < ~1% of table → Index Scan
  1% - 10% of table → Bitmap Scan
  > 10% of table → Sequential Scan
(These percentages are heuristic; actual thresholds depend on table size, random_page_cost,
effective_cache_size, and other planner cost parameters.)

4. The Query Optimizer — How Index Selection Works

The Core Question

When multiple indexes exist on a table, the optimizer must decide:

  1. Which index(es) to use?
  2. Should it use one, both, or none?
  3. In what order (for multi-index queries)?

Optimizer Input: Table Statistics

The optimizer cannot look at every row to make its decision. Instead it uses statistics — metadata about the table's data distribution maintained automatically (or on demand) by the database.

What statistics contain:

StatisticDescriptionUsed For
n_distinctNumber of distinct values per columnSelectivity estimate
correlationPhysical order correlation (0=random, 1=sorted)Sequential vs random IO preference
null_fracFraction of NULLsNULL predicate cost
avg_widthAverage column width in bytesNetwork/memory cost
n_live_tupEstimated live row countIndex vs scan threshold
histogram_boundsValue distribution histogram (100 buckets)Range query selectivity
most_common_valsMost frequent values + their frequenciesEquality predicate selectivity
 
-- View column statistics (PostgreSQL)
SELECT attname, n_distinct, correlation, null_frac, avg_width
FROM pg_stats
WHERE tablename = 'grades';

-- View histogram and most common values
SELECT attname, most_common_vals, most_common_freqs, histogram_bounds
FROM pg_stats
WHERE tablename = 'grades' AND attname = 'grade';

The Three Index Selection Cases

Case 1: Use Both Indexes (BitmapAnd / BitmapOr)

When: AND query, both indexes are moderately selective.
      Neither index alone filters enough, but together they drastically reduce the result set.

Example:
  Table: 10M rows
  grade > 95: returns 500,000 rows (5%) → borderline for index use
  id < 100000: returns 100,000 rows (1%) → worth indexing
  grade > 95 AND id < 100000: returns ~5,000 rows → very selective

  Plan: BitmapAnd(idx_grade, idx_id) → 5,000 heap fetches
  vs single index: 100,000-500,000 heap fetches
  vs seq scan: 625,000 page IOs

Case 2: Use One Index, Filter on the Other

When: AND query, one index is MUCH more selective than the other.
      Using the highly selective index + filtering in memory is cheaper than using both.

Example:
  id = 42 (primary key): returns EXACTLY 1 row
  grade > 50: returns 5,000,000 rows (50%) → index useless

  Plan: Index Scan on id → 1 row → check grade > 50 in memory → done
  Using both: PK lookup returns 1 row immediately, 0 reason to use grade index.

OR example:
  Primary key lookup almost always wins in AND conditions.
  The optimizer sees "id = 42 AND anything" → use PK, filter everything else.

Case 3: Ignore All Indexes (Sequential Scan)

When: Query returns too many rows for any index to help.
      OR statistics say both conditions are low-selectivity.

Example:
  status IN ('active', 'inactive') → covers 100% of rows → useless index
  grade BETWEEN 0 AND 100 → covers 100% of rows → useless

  Plan: Seq Scan (Parallel Seq Scan if workers available)
  Reason: Index traversal + heap fetch overhead exceeds simple page scan cost.

Practical rule: If the optimizer estimates > ~10-20% of rows will match,
                sequential scan often wins over index scan.

The Skew Problem — When Statistics Lie

The most dangerous scenario for optimizer decisions:

Table: customers (10M rows)
Column: state VARCHAR
Index: idx_state on (state)
Data distribution: California = 9,900,000 rows (99%), Texas = 50,000, Florida = 1, ...

Statistics: most_common_vals = ['California', 'Texas', ...]
            most_common_freqs = [0.99, 0.005, ...]

Query A: WHERE state = 'California'
  Optimizer: estimates 9,900,000 rows → IGNORES index → Sequential Scan ✅ (correct)

Query B: WHERE state = 'Florida'
  Optimizer: estimates... it depends on histogram accuracy.
  If statistics are accurate: estimates 1 row → uses index ✅ (correct, very fast)
  If statistics are stale: may estimate 100,000 rows → may do seq scan ❌ (wrong, slow)

The California problem: An index on a heavily skewed column is useless for the majority value. It's only valuable for the rare values. This is why:

  • Low-cardinality columns (status: active/inactive) rarely benefit from plain B-tree indexes.
  • Filtered/partial indexes (WHERE state != 'California') are perfect for rare-value queries.

The Stale Statistics Problem — A Production Danger

Scenario: Fresh table created. Statistics: rows=0.
  You insert 300M rows via bulk load.
  You immediately run queries WITHOUT running ANALYZE.

  Statistics still say: rows=0 (or rows=3 from last analyze)
  Optimizer thinks: "This table is tiny → full scan is fine."
  Query plan: Seq Scan on 300M rows → pulls all 300M rows.
  Time: 5 minutes instead of milliseconds.

This scenario is real and common after ETL loads, migrations, and bulk inserts.

Always run ANALYZE after significant data changes:

-- PostgreSQL
ANALYZE grades;                        -- Update statistics for one table
ANALYZE;                              -- Update statistics for all tables
VACUUM ANALYZE grades;                -- Clean dead tuples + update stats

-- Oracle
EXEC DBMS_STATS.GATHER_TABLE_STATS('schema', 'table');

-- SQL Server
UPDATE STATISTICS table_name;
-- Or rebuild indexes (which also updates stats):
ALTER INDEX ALL ON table_name REBUILD;

-- MySQL
ANALYZE TABLE grades;

5. Composite Index Best Practices — Full Analysis

The Test Setup (from lectures)

Table: test (a INTEGER, b INTEGER, c INTEGER)
Rows: ~12 million

Test queries:
  Q1: SELECT c FROM test WHERE a = 70
  Q2: SELECT c FROM test WHERE b = 100
  Q3: SELECT c FROM test WHERE a = 100 AND b = 100
  Q4: SELECT c FROM test WHERE a = 100 OR b = 100

Configuration 1: Two Separate Indexes (a) + (b)

CREATE INDEX idx_a ON test (a);
CREATE INDEX idx_b ON test (b);
QueryPlanResult
WHERE a = 70Bitmap Index Scan on idx_a → Bitmap Heap Scan~250ms, 9,000 rows
WHERE b = 100Bitmap Index Scan on idx_b → Bitmap Heap Scan~250ms
WHERE a=100 AND b=100BitmapAnd(idx_a, idx_b) → Bitmap Heap ScanFast, ~6 rows
WHERE a=100 OR b=100BitmapOr(idx_a, idx_b) → Bitmap Heap ScanSlower, more rows

For the AND case: PostgreSQL builds two bitmaps, ANDs them, fetches only pages where BOTH conditions might be satisfied. Very efficient.

For the OR case: PostgreSQL builds two bitmaps, ORs them, fetches the union. More rows to fetch than AND, but still avoids two separate table scans.

Configuration 2: Composite Index (a, b) Only

CREATE INDEX idx_ab ON test (a, b);
QueryPlanResult
WHERE a = 70Bitmap Index Scan on idx_ab (left prefix works)~250ms
WHERE b = 100Parallel Seq Scan (no left prefix — index ignored)SLOW
WHERE a=100 AND b=100Index Scan on idx_ab0.5ms — blazing fast
WHERE a=100 OR b=100Seq Scan (OR with composite: difficult to use)Slow

The left-prefix rule in action:

Composite index idx_ab on (a, b):

B+ Tree leaf nodes: [(a=1,b=5), (a=1,b=8), (a=2,b=3), (a=2,b=7), (a=3,b=1), ...]
                     ↑
                     Sorted by a first, then by b within each a value.

Searching WHERE a = 70:
  Can use index: scan to where a=70 starts, walk forward until a≠70. ✅
  All a=70 entries are contiguous in the B+ tree.

Searching WHERE b = 100:
  Cannot use index efficiently: b=100 is scattered throughout the tree.
  b=100 appears in (a=1,b=100), (a=2,b=100), (a=5,b=100), etc. — everywhere.
  Would require scanning the ENTIRE index to find all b=100 entries.
  → Planner correctly chooses sequential scan instead. ❌

AND performance with composite index:
When querying a=100 AND b=100, the composite index is perfect:

  • Navigate to the exact leaf node where a=100 AND b=100.
  • This is a point lookup — O(log n) to find exactly those rows.
  • Result: 0.5ms vs 250ms with two separate indexes. 500x faster for the AND case.

Configuration 3: Composite (a, b) + Separate (b) — The Best of Both

CREATE INDEX idx_ab ON test (a, b);  -- For a alone, and a+b together
CREATE INDEX idx_b  ON test (b);     -- For b alone
QueryPlanResult
WHERE a = 70Bitmap Index Scan on idx_ab✅ Fast
WHERE b = 100Bitmap Index Scan on idx_b✅ Fast (now has its own index)
WHERE a=100 AND b=100Index Scan on idx_ab✅ Blazing (0.5ms)
WHERE a=100 OR b=100BitmapOr(idx_ab, idx_b) → Bitmap Heap Scan✅ Now works!

For the OR case with Configuration 3:

  • a=100: uses idx_ab (left prefix, a is first column) → builds bitmap 1.
  • b=100: uses idx_b (dedicated b index) → builds bitmap 2.
  • BitmapOr merges bitmaps → fetches union.

Without the dedicated idx_b, the OR query on b=100 would force a full table scan.

The engineering decision:

Ask: Which queries need to be fast?
  a alone?           → idx_ab covers this (left prefix)
  b alone?           → needs idx_b
  a AND b?           → idx_ab is perfect
  a OR b?            → needs both idx_ab AND idx_b
  All four?          → Configuration 3: idx_ab + idx_b

Cost: Two indexes = two structures to maintain on every write.
      More inserts = more write overhead.
      Pick only the indexes that real queries use.

6. Planner Cost Parameters — What Drives the Decisions

The planner's cost model uses configurable parameters that model the relative cost of different operations on your specific hardware:

-- View current cost parameters:
SHOW random_page_cost;    -- Default: 4.0 (relative cost of random disk IO)
SHOW seq_page_cost;       -- Default: 1.0 (relative cost of sequential disk IO)
SHOW cpu_tuple_cost;      -- Default: 0.01 (cost per row processed)
SHOW effective_cache_size; -- Default: 4GB (estimate of OS cache size)

The most important: random_page_cost / seq_page_cost ratio

Default: random_page_cost = 4.0, seq_page_cost = 1.0
Ratio: 4:1 — random IO is 4x more expensive than sequential IO

For NVMe SSDs (random ≈ sequential): set random_page_cost = 1.1 (nearly equal)
For HDD (random MUCH slower): leave at 4.0 or increase to 8.0

Effect:
  Higher random_page_cost → optimizer avoids index scans with many heap fetches
                          → prefers sequential scans and bitmap scans
  Lower random_page_cost  → optimizer more willing to use index scans
                          → set to 1.1 on SSDs to unlock more index usage
-- Tune for SSD storage:
SET random_page_cost = 1.1;
-- or in postgresql.conf:
-- random_page_cost = 1.1

7. Practical Java Implementation

Reading EXPLAIN Output Programmatically

import java.sql.*;

public class QueryPlanAnalyzer {

    private final Connection conn;

    /**
     * Retrieves and logs the query execution plan for a given SQL statement.
     * Use this during development to verify index usage.
     */
    public String explainQuery(String sql, Object... params) throws SQLException {
        StringBuilder plan = new StringBuilder();
        String explainSql = "EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT) " + sql;

        try (PreparedStatement stmt = conn.prepareStatement(explainSql)) {
            for (int i = 0; i < params.length; i++) {
                stmt.setObject(i + 1, params[i]);
            }
            try (ResultSet rs = stmt.executeQuery()) {
                while (rs.next()) {
                    plan.append(rs.getString(1)).append("\n");
                }
            }
        }
        return plan.toString();
    }

    /**
     * Checks whether a query uses a sequential scan (potential performance issue).
     * In production: trigger an alert or log a warning when seq scans appear
     * on large tables in critical code paths.
     */
    public boolean usesSeqScan(String sql, Object... params) throws SQLException {
        String plan = explainQuery(sql, params);
        return plan.contains("Seq Scan") || plan.contains("Parallel Seq Scan");
    }

    /**
     * Checks whether the query achieves an Index Only Scan (ideal).
     */
    public boolean isIndexOnlyScan(String sql, Object... params) throws SQLException {
        String plan = explainQuery(sql, params);
        return plan.contains("Index Only Scan");
    }

    /**
     * Extracts actual execution time from EXPLAIN ANALYZE output.
     * Useful for automated performance regression testing.
     */
    public double getExecutionTimeMs(String sql, Object... params) throws SQLException {
        String plan = explainQuery(sql, params);
        // Extract "Execution Time: X.XXX ms" from the plan
        for (String line : plan.split("\n")) {
            if (line.startsWith("Execution Time:")) {
                return Double.parseDouble(
                    line.replace("Execution Time:", "").replace("ms", "").trim()
                );
            }
        }
        return -1;
    }
}

Statistics Refresh After Bulk Load

import org.springframework.jdbc.core.JdbcTemplate;

@Service
public class BulkDataService {

    private final JdbcTemplate jdbc;

    /**
     * Bulk inserts data and updates statistics afterward.
     * CRITICAL: Without ANALYZE after bulk insert, the optimizer may use
     * stale statistics and choose wrong query plans for subsequent queries.
     */
    @Transactional
    public void bulkInsertAndRefreshStats(List<Student> students) {
        // Bulk insert
        jdbc.batchUpdate(
            "INSERT INTO students (id, name, grade) VALUES (?, ?, ?)",
            students,
            1000,
            (ps, student) -> {
                ps.setLong(1, student.getId());
                ps.setString(2, student.getName());
                ps.setInt(3, student.getGrade());
            }
        );

        // CRITICAL: Update statistics after bulk load
        // Without this, optimizer may choose full table scan on the fresh data
        jdbc.execute("ANALYZE students");

        log.info("Bulk inserted {} rows and updated statistics.", students.size());
    }

    /**
     * For very large bulk loads, VACUUM ANALYZE is more thorough.
     * Cleans dead tuples + updates visibility map + updates statistics.
     * Needed for Index Only Scan to work at Heap Fetches: 0.
     */
    public void fullMaintenance(String tableName) {
        // Cannot run VACUUM inside a transaction
        jdbc.execute("VACUUM ANALYZE " + tableName);
        log.info("Vacuum analyze completed for table: {}", tableName);
    }
}

8. Important Interview Questions

Basic Questions

Q: What is the query planner and why does it exist?
A: The query planner (optimizer) is the database component that takes a SQL statement and determines the most efficient execution plan. SQL is declarative — it describes what data to retrieve, not how. The planner evaluates multiple possible plans (different indexes, join orders, scan strategies), estimates the cost of each using table statistics, and executes the cheapest one.

Q: What is the difference between cost=startup..total in EXPLAIN output?
A: The startup cost is work done before returning the first row — high for ORDER BY without an index (must sort everything first), aggregations, and hash joins. The total cost is the estimated work to return all rows. Both are in arbitrary cost units, not milliseconds. For LIMIT queries, a high startup cost is particularly damaging because you pay the full pre-processing cost even for a small result.

Q: What does rows= in EXPLAIN output represent?
A: The planner's estimated number of rows this node will return, derived from table statistics. If the estimate is very different from actual rows (visible with EXPLAIN ANALYZE), the statistics are stale. Run ANALYZE table_name to fix. Wrong row estimates lead to wrong plan choices (e.g., choosing index scan for a query that will return 5M rows).

Q: What causes "Rows Removed by Filter" in EXPLAIN output?
A: This appears when the database reads rows from the heap or index but they fail the WHERE clause filter and are discarded. A high number means a lot of wasted IO. The typical cause is a missing index on the filter column, or a non-SARGable predicate that prevents index use.

Intermediate Questions

Q: What is a Bitmap Index Scan and why does PostgreSQL use it instead of a regular Index Scan?
A: A Bitmap Index Scan is used when a query matches enough rows that random heap fetches per row (Index Scan) would be slower than a sequential scan, but not so many rows that a full sequential scan is preferred. The bitmap scan first scans the index to build a bitmap of matching page numbers, then fetches those heap pages in sorted (sequential) order. This converts expensive random IO into cheaper sequential IO. It also allows BitmapAnd/Or operations to combine multiple indexes before touching the heap.

Q: Explain the left-prefix rule for composite indexes.
A: A composite index on (a, b) sorts entries first by a, then by b within each a value. A query filtering only on a can use this index because all matching entries are contiguous in the B+ tree (starting from where a equals the search value). A query filtering only on b cannot use this index efficiently because b values are spread throughout the tree, interleaved with different a values. The index is useful only when a left-side prefix of the indexed columns is in the query's WHERE clause.

Q: When does the optimizer choose a sequential scan over an index scan, even when an index exists?
A: The planner estimates the fraction of rows matching the query. When many rows match (roughly > 10-20% of the table), the cost of K random heap fetches via the index exceeds the cost of a single sequential scan. The planner also considers random_page_cost vs seq_page_cost configuration. On SSDs, lowering random_page_cost to 1.1 tells the planner that random and sequential IO have similar cost, encouraging more index use.

Q: What are stale statistics and how do they cause production problems?
A: Statistics are metadata about table data distribution (row counts, distinct values, value histograms) used by the planner to estimate how many rows a predicate will match. After bulk data loads, statistics may be stale — the planner thinks the table has 3 rows when it actually has 300 million. It then chooses a sequential scan thinking it's fast (3 rows), but actually scans 300M rows. Fix: always run ANALYZE after significant data changes. Auto-analyze runs in the background but may lag behind large bulk operations.

Advanced Questions

Q: Explain how BitmapAnd reduces IO compared to two separate index scans.
A: With two separate indexes (idx_a and idx_b) for a query WHERE a=100 AND b=100, a naive approach would scan idx_a → find 10,000 matching rows → for each, fetch the heap page and check b=100. BitmapAnd instead scans both indexes to build two page bitmaps, then ANDs the bitmaps to find pages where BOTH conditions might be satisfied. If idx_a points to 1,000 pages and idx_b points to 500 pages, and their intersection is only 50 pages, we fetch only 50 heap pages instead of 1,000. The bitmap AND is a simple bitwise operation on the page bitmaps — essentially free — while eliminating most random heap IO.

Q: How does the visibility map affect Index Only Scan performance in PostgreSQL, and how do you fix it?
A: PostgreSQL uses MVCC — each row has xmin/xmax visibility metadata. An Index Only Scan would ideally return data without touching the heap. But PostgreSQL must verify that each row is visible to the current transaction. The visibility map records, per heap page, whether all rows on that page are known visible to all transactions (meaning no live MVCC row version ambiguity). If a page is marked "all visible," the Index Only Scan can skip the heap fetch. If not (stale visibility map after writes), it must fetch the heap page — appearing as Heap Fetches > 0 even on an Index Only Scan. Running VACUUM updates the visibility map, setting all-visible bits, and eliminates the extra heap fetches.

Q: How does random_page_cost affect the planner's index vs scan decision?
A: The planner computes the estimated cost of each plan using cost constants. An index scan on K rows costs approximately K × random_page_cost for heap fetches. A sequential scan costs total_pages × seq_page_cost. PostgreSQL defaults to random_page_cost=4.0 and seq_page_cost=1.0 (4:1 ratio — reflecting HDD characteristics). On NVMe SSDs where random and sequential IO have nearly identical latency, this ratio makes index scans appear 4x more expensive than they actually are, causing the planner to prefer sequential scans when it shouldn't. Setting random_page_cost=1.1 corrects this for SSD environments, enabling the optimizer to choose index scans for a much larger fraction of queries.

9. Common Mistakes

Reading EXPLAIN Wrong

  • Reading top-to-bottom instead of bottom-up. The leaf nodes (most indented) execute first. Read from the innermost node outward to understand execution order.
  • Treating cost numbers as milliseconds. The cost= values are not milliseconds. They are relative cost units. Only EXPLAIN ANALYZE shows actual milliseconds in the actual time= field.
  • Ignoring the startup cost. Engineers often focus on total cost and miss the startup cost. For LIMIT queries, a high startup cost (caused by sorting or aggregation) means the database does ALL the expensive work before returning even one row.

Statistics and Planning Mistakes

  • Not running ANALYZE after bulk loads. This is the most common production mistake. Fresh tables after ETL have zero or minimal statistics. Every query on them may choose the wrong plan until ANALYZE runs.
  • Forgetting that EXPLAIN (without ANALYZE) doesn't execute the query. The rows= and cost= values in plain EXPLAIN are estimates, not measurements. They can be significantly wrong. Always use EXPLAIN ANALYZE to see actual numbers.
  • Trusting one execution of EXPLAIN ANALYZE. The first run may show high IO because data is cold (not cached). The second run is faster due to OS page cache. Profile with cache cleared (pg_prewarm extension or testing with pg_stat_reset) for realistic estimates.

Index Design Mistakes (From Planner Perspective)

  • Creating a composite index on (a, b) and expecting it to help queries on b alone. As demonstrated: the left-prefix rule means WHERE b = ? will do a full table scan with composite index (a, b). Always add a separate index on b if you query b alone.
  • Expecting the planner to use an index on a low-cardinality column. An index on status with values 'active'/'inactive' (50% each) will be ignored for equality predicates. The planner knows a seq scan is cheaper. Use filtered/partial indexes for these cases.
  • Creating indexes and not running ANALYZE. A newly created index is visible to the planner but the statistics haven't been updated to reflect the new index's selectivity information. Run ANALYZE table_name after creating indexes to ensure optimal plan generation.

Production Mistakes

  • Using query hints in application code permanently. Hints override the planner. If the data distribution changes (new scale, different data patterns), the hint may force a suboptimal plan. Hints are debugging tools, not architectural solutions. Fix the underlying statistics or index design instead.
  • Not monitoring for sequential scans on large tables. 
    -- Find tables with high sequential scan ratios (missing indexes)
SELECT relname, seq_scan, idx_scan,
       round(seq_scan::numeric/(seq_scan+idx_scan+1)*100,1) as seq_pct
FROM pg_stat_user_tables
WHERE seq_scan > 1000
ORDER BY seq_scan DESC;
  • Assuming the planner is always right. The planner's cost model has blind spots: correlated columns (WHERE city='NYC' AND state='NY' — not independent), complex join cardinality with many tables, and cases where n_distinct is wrong. When you KNOW your data, judicious hints can outperform the planner.

10. Performance & Scalability Notes

Cost of Wrong Plan Selection

Table: 50M rows
Correct plan: Index Only Scan → 4 seconds
Wrong plan (stale stats): Seq Scan → 21 seconds

5x performance regression from a single missing ANALYZE call.

At scale (1B rows):
Correct plan: ~8 seconds
Wrong plan: ~7 minutes

87x performance regression.

Parallel Query Scaling

Workers: 1 (serial) → baseline
Workers: 2 → ~1.7x speedup (not linear — merge overhead)
Workers: 4 → ~3.2x speedup
Workers: 8 → ~5-6x speedup (diminishing returns)

Configure in PostgreSQL:
  max_parallel_workers_per_gather = 4  (default: 2)
  min_parallel_table_scan_size = 8MB  (tables smaller than this don't parallelize)
  parallel_setup_cost = 1000          (overhead of spawning workers)

Memory for Sort and Hash Operations

sort_mem (work_mem in PostgreSQL): Memory for each sort/hash operation per connection.
  Default: 4MB
  Too low: Spills sort to disk → very slow ("external sort")
  
For a query sorting 10M rows × 31 bytes/row = 310MB:
  work_mem = 4MB → sort spills to disk → 10x slower
  work_mem = 512MB → sort stays in memory → fast
  
SET work_mem = '256MB';  -- Per-session tuning for heavy analytical queries
-- Or in postgresql.conf for global default

11. System Design Connections

Query Plan Monitoring in Production

// Spring Boot actuator endpoint to expose slow query stats
@Component
public class SlowQueryMonitor {

    private final JdbcTemplate jdbc;
    private final MeterRegistry metrics;

    @Scheduled(fixedDelay = 60000)  // Every minute
    public void checkForMissingIndexes() {
        // pg_stat_statements extension required
        List<Map<String, Object>> slowQueries = jdbc.queryForList("""
            SELECT query, calls, total_exec_time/calls as avg_ms,
                   rows/calls as avg_rows
            FROM pg_stat_statements
            WHERE total_exec_time/calls > 100     -- avg > 100ms
            AND calls > 100                        -- called frequently
            ORDER BY total_exec_time DESC
            LIMIT 10
            """);

        for (Map<String, Object> q : slowQueries) {
            double avgMs = (double) q.get("avg_ms");
            String query = (String) q.get("query");

            metrics.gauge("db.slow_query.avg_ms",
                Tags.of("query", query.substring(0, Math.min(50, query.length()))),
                avgMs);

            if (avgMs > 1000) {
                log.warn("SLOW QUERY DETECTED (avg {}ms): {}", avgMs, query);
            }
        }
    }

    @Scheduled(fixedDelay = 300000)  // Every 5 minutes
    public void refreshStaleStatistics() {
        // Find tables where auto-analyze may be lagging
        jdbc.queryForList("""
            SELECT relname, n_dead_tup, n_live_tup, last_analyze
            FROM pg_stat_user_tables
            WHERE n_dead_tup > n_live_tup * 0.1   -- >10% dead tuples
            OR last_analyze < NOW() - INTERVAL '1 hour'
            """).forEach(table -> {
                String tableName = (String) table.get("relname");
                log.info("Triggering ANALYZE on stale table: {}", tableName);
                jdbc.execute("ANALYZE " + tableName);
            });
    }
}

Read Replica — Planner Implications

In a primary-replica setup, each node has its own planner and its own statistics. If a read replica's statistics are behind the primary (e.g., replica hasn't run ANALYZE recently), it may choose different query plans for the same query than the primary does. A query that runs in 10ms on the primary may run in 3 seconds on the replica due to a stale statistics plan choice. Always run ANALYZE on replicas after significant schema or data changes.
