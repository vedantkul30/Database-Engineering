# B-Tree vs B+ Tree in Production Database Systems ## Deep Engineering Notes — Interview & Production Ready --- # 1. Big Picture ## Why This Topic Exists Every index you create in PostgreSQL, MySQL, Oracle, or SQL Server is backed by a **B+ Tree**. Not a B-Tree. Understanding *why* databases chose B+ Tree over B-Tree, and what that means for your schema design, key selection, and query patterns, is fundamental knowledge for any backend or database engineer. The B-Tree was invented by Bayer & McCreight in 1970. The B+ Tree came shortly after as a practical improvement for disk-based systems. Today, virtually every production RDBMS uses B+ Trees — but each database implements them with critical differences that directly affect performance. ## The Core Problem Both Solve A table is a data file on disk. Finding one row means reading that file page by page — potentially millions of I/Os. Both B-Tree and B+ Tree solve this by creating a **sorted, balanced index structure** that lets you find any row in O(log n) I/Os instead of O(n). ## Why Backend Engineers Must Care - **Index design** (which columns, what data types, composite key order) directly determines B+ Tree height and element density - **Primary key type** (INT vs UUID) determines insert performance — especially critical in MySQL's clustered storage - **PostgreSQL vs MySQL** secondary index behavior is a B+ Tree design decision that caused Uber to migrate databases - **Range query performance** (e.g., "all orders between dates") depends entirely on whether your engine has linked leaf nodes --- # 2. Core Concepts ## 2.1 — What Is a B-Tree? A B-Tree of degree M is a self-balancing tree where: - Every node has **at most M children** - Every node has **at most M-1 elements** - All leaves are at the **same depth** (perfectly balanced) - Every element stores a **key** (search value) AND a **value** (data pointer to the row) ``` B-Tree of degree 3 (max 2 elements, max 3 children per node): ┌──────────────────────────┐ │ (4, →row4) (8, →row8) │ ← Root └────┬──────────┬──────────┘ ┌──────────┘ └──────────┐ ┌──────────▼──────────┐ ┌──────────────▼──────┐ │ (2,→row2) (3,→row3) │ │ (9,→row9) (11,→row11)│ └─────────────────────┘ └─────────────────────┘ Key insight: EVERY node stores (key, data_pointer) pairs. Internal nodes carry values you don't need during traversal. ``` ### The Hidden Detail Most Textbooks Miss The original 1970 paper stores key-value pairs in **every node** including internal routing nodes. Most textbooks and papers draw only the keys in diagrams, omitting the value pointers for visual clarity. This creates widespread confusion between B-Trees and B+ Trees. In reality, an internal B-Tree node carries both the key (needed for routing) **and** the data pointer (not needed until you reach your target) — wasting precious page space. ### Node = Disk Page (Always) In any production database, one node = one disk page: ``` PostgreSQL: 8KB per page MySQL: 16KB per page Page budget example (PostgreSQL, 8KB): B-Tree element = key (8 bytes INT) + data pointer (16 bytes UUID) = 24 bytes Elements/page = 8192 / 24 ≈ 341 elements per node B+ Tree internal = key only (8 bytes INT) + child pointer (6 bytes) = 14 bytes Elements/page = 8192 / 14 ≈ 585 elements per node ``` Fewer elements per page = taller tree = more I/Os per query. --- ## 2.2 — B-Tree Limitations ### Limitation 1: Values in Every Node Waste Space Reading an internal node during traversal fetches both keys (needed) and data pointers (not needed yet). The data pointers consume space that could otherwise hold more keys — reducing how many elements fit per page. ``` Impact on tree height: INT key (8B) + data pointer (16B UUID) = 24B per element → 341 elements/page → Tree height for 1B rows: log_341(1B) ≈ 3.3 → 4 levels → 4 I/Os INT key (8B) only = 8B per element → 1024 elements/page → Tree height for 1B rows: log_1024(1B) ≈ 3.3 → still 4 levels But: far more elements pruned per level → faster in practice ``` More importantly: large value types (UUID = 16B, strings = variable) make this worse dramatically. ### Limitation 2: Range Queries Are Slow In a B-Tree, finding all keys between 4 and 9 requires **separate root-to-leaf traversals** for each key: ``` Find keys 4, 5, 6, 7, 8, 9 in a B-Tree: Find 4: Root → child → leaf (3 I/Os) Find 5: Root → child → child → leaf (4 I/Os) Find 6: Root → child → leaf (3 I/Os) Find 7: Root → child (2 I/Os, found in internal node) Find 8: Root (1 I/O, found in root) Find 9: Root → child → leaf (3 I/Os) Total: ~16 I/Os for 6 adjacent keys. ``` Despite keys 4–9 being logically adjacent, the B-Tree has no shortcut between them. ### Limitation 3: Internal Nodes Hard to Keep in Memory Because internal nodes carry both keys and values, they are larger. Fewer fit in the buffer pool. Under memory pressure, hot routing nodes get evicted — turning cached traversals into disk reads. --- ## 2.3 — B+ Tree: The Solution B+ Tree keeps the B-Tree's balanced structure but makes two critical changes: 1. **Internal nodes store keys only** — no data pointers 2. **All data pointers live exclusively in leaf nodes** 3. **Leaf nodes are doubly linked** — each leaf points to the next and previous leaf ``` B+ Tree of degree 3: Internal nodes (keys only — slim): ┌─────────────────┐ │ [3] [7] │ ← Root: keys only └───┬──────┬──────┘ ┌─────────┘ └──────────┐ ┌──────▼──────┐ ┌──────▼──────┐ │ [2] [3] │ │ [7] [9] │ ← Internal: keys only └─────────────┘ └─────────────┘ Leaf nodes (keys + data pointers + linked): ┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐ │(1,→r1) (2,→r2) │←→ │(3,→r3) (5,→r5) │←→ │(7,→r7) (9,→r9) │ └─────────────────┘ └─────────────────┘ └─────────────────┘ ↑ doubly-linked list across all leaf pages ↑ ``` ### Why Key Duplication Is Intentional Some keys (e.g., 3, 7 above) appear in both internal nodes and leaf nodes. This is correct and necessary. The internal copy is a **routing guide only**. The leaf copy carries the actual data pointer. Leaf nodes must contain every key so that range scans never need to go back up the tree. --- ## 2.4 — B+ Tree Advantages Over B-Tree ### Advantage 1: Denser Internal Nodes → Shallower Tree ``` Internal node comparison for 1 billion rows, INT key: B-Tree internal: key (8B) + value (6-16B) = 14-24B/element → 341-585 elem/page B+ Tree internal: key (8B) + child ptr (6B) = 14B/element → 585 elem/page Tree height (B+ Tree, INT PK, 1B rows): Level 0 (root): 585 children Level 1: 585² = 342,225 children Level 2: 585³ = 200M children ← covers 1B rows easily Height: 3 levels + leaf = 4 I/Os worst case With internal nodes cached in RAM: effectively 1 I/O per query (just the leaf page). ``` ### Advantage 2: Range Queries via Leaf Linked List ``` Find keys 4 to 9 in a B+ Tree: Step 1: Traverse root → internal → leaf containing key 4. (2-3 I/Os) Leaf page contains: [3, 4, 5, 6] ← all found here Step 2: Follow next-leaf pointer. (1 I/O) Next leaf: [7, 8, 9, 10] ← found 7, 8, 9. Stop at 9. Total: 3-4 I/Os for all 6 keys. vs B-Tree: ~16 I/Os for the same range. ``` Range query cost: O(log n) to find the start + O(k / elements_per_page) to scan k results. ### Advantage 3: Internal Nodes Fit in RAM ``` 1B row table, INT PK (PostgreSQL, 8KB pages): Leaf pages: 1B / 585 = ~1.71M pages = ~13.3 GB (mostly on disk) Level above leaves: 1.71M / 585 = ~2,923 pages = ~22 MB (fits in RAM easily) Root + one more: ~5 pages = negligible → Caching just ~22MB of internal nodes eliminates ALL routing I/Os. → Every query: 0 I/Os for traversal + 1 I/O for leaf page. → This scales to 10B, 100B rows with minimal additional memory. ``` ### Advantage 4: Leaf Nodes Can Live in Data Files Since all data pointers are in leaf nodes, a database can store leaf node data closer to (or within) the actual row data, enabling additional optimizations like clustered storage (MySQL InnoDB). --- # 3. Deep Dive — Internal Workings ## 3.1 — Page Structure of a B+ Tree Node ``` PostgreSQL B+ Tree page layout (8KB): ┌─────────────────────────────────────────────────────────┐ │ Page Header (24 bytes) │ │ - LSN (Log Sequence Number for WAL) │ │ - Page type (leaf / internal) │ │ - Free space pointers │ ├─────────────────────────────────────────────────────────┤ │ Item pointer array │ │ - Array of (offset, length) for each item on page │ ├─────────────────────────────────────────────────────────┤ │ Free space (grows inward) │ ├─────────────────────────────────────────────────────────┤ │ Index tuples (packed from bottom up) │ │ Internal node: (key, child_page_number) │ │ Leaf node: (key, heap_tid / ctid) │ ├─────────────────────────────────────────────────────────┤ │ Special space (56 bytes at end of page) │ │ - Left sibling page number │ │ - Right sibling page number ← the leaf linked list │ │ - B-link pointer (for concurrent splits) │ │ - Level (0 = leaf, 1+ = internal) │ └─────────────────────────────────────────────────────────┘ ``` ## 3.2 — B+ Tree Search: Step by Step ``` Query: SELECT * FROM users WHERE id = 500000; Index: B+ Tree on users.id Tree height: 4 levels (root → 2 internal → leaf) Step 1: Read root page (almost always in buffer pool — 0 disk I/O) Root contains: [100K, 300K, 600K, 900K] 500K is between 300K and 600K → follow 3rd child pointer Step 2: Read level-2 internal page (likely cached — 0 disk I/O) Contains: [420K, 460K, 490K, 510K, 550K] 500K is between 490K and 510K → follow pointer Step 3: Read level-3 internal page (may or may not be cached) Contains: [497K, 499K, 500K, 501K, 503K] 500K found → follow to leaf (or go right if equal, per B+ Tree rules) Step 4: Read leaf page (1 disk I/O if not cached) Contains: [(499998, ctid_a), (499999, ctid_b), (500000, ctid_c)] Found 500000 → ctid_c = (page 8847, offset 14) Step 5 (PostgreSQL heap fetch): Read heap page 8847 → extract row at offset 14 → return to client (MySQL InnoDB: leaf already contains full row — no extra read needed) Total I/Os (worst case): 3 internal + 1 leaf + 1 heap = 5 Total I/Os (best case, internal nodes cached): 0 + 1 leaf + 1 heap = 2 ``` ## 3.3 — Page Split: Full Cost Analysis A page split is the most expensive operation on a B+ Tree. It happens when inserting into a full page. ``` Insert key=42 into full leaf page [38, 40, 44, 46] (max 4 keys): 42 must go between 40 and 44. Page is full. Split required. Before: Page 55: [38(ptr), 40(ptr), 44(ptr), 46(ptr)] After split: Page 55: [38(ptr), 40(ptr)] ← original page, trimmed Page 891: [42(ptr), 44(ptr), 46(ptr)] ← new page allocated Operations required: 1. Allocate new page 891 on disk 2. Copy entries [42, 44, 46] to page 891 3. Update page 55 right-sibling pointer: 55 → 891 4. Set page 891 left-sibling pointer: 891 → 55 5. Set page 891 right-sibling pointer: 891 → (old right neighbor of 55) 6. Update old right neighbor's left-sibling pointer to 891 7. Promote split key (42) up to parent internal node 8. If parent is also full → cascade split upward (can reach root) 9. Write WAL entries for ALL modified pages (durability) 10. Mark all modified pages dirty in buffer pool Worst case cascade: leaf split → internal split → root split → new root → log(n) pages modified and written to WAL ``` **Why random inserts (UUID v4) cause constant splits:** Random keys land in random positions throughout the tree. Most insertions target pages that are already full (statistically, at high load). Sequential inserts always land at the rightmost leaf — which has space reserved and rarely splits. ## 3.4 — How Leaf Linking Enables Range Scans ``` Range query: WHERE created_at BETWEEN '2024-01-01' AND '2024-01-31' Without leaf linking (B-Tree behavior): Find '2024-01-01': root → internal → internal → leaf (4 I/Os) Find '2024-01-02': root → internal → internal → leaf (4 I/Os, possibly same path) Find '2024-01-03': root → ... (4 I/Os each) ... 31 days × 4 I/Os = 124 I/Os minimum With leaf linking (B+ Tree behavior): Find '2024-01-01': root → internal → internal → leaf (4 I/Os) Follow next pointer → next leaf (1 I/O) Follow next pointer → next leaf (1 I/O) ... until past '2024-01-31' = 4 I/Os (initial traversal) + N I/Os (one per leaf page covering the range) = dramatically fewer I/Os ``` --- # 4. Production Database Implementations ## 4.1 — PostgreSQL **Storage model:** Heap-organized table. The table is a separate, unordered heap file. The primary key is just another B+ Tree index — it does NOT determine physical row order. **Secondary index values:** Point directly to the **tuple ID (ctid)** = (page_number, slot_offset), 6 bytes. ``` Secondary index leaf entry in PostgreSQL: [indexed_key_value | ctid (6 bytes)] e.g.: ["alice@example.com" | (page=447, slot=3)] To fetch the row: 1. Traverse secondary B+ Tree → get ctid 2. Fetch heap page 447, slot 3 → full row Total: 2 B+ Tree traversals at most (if PK also needed) ``` **Write amplification problem:** When a row is UPDATEd, PostgreSQL writes a new version to a new heap location (new ctid). All secondary indexes pointing to the old ctid become stale. PostgreSQL must update every secondary index entry. ``` Table with 5 secondary indexes: UPDATE orders SET status = 'shipped' WHERE id = 12345; Writes: 1. New row version in heap (new ctid) 2. Update idx_orders_status: old entry removed, new entry inserted 3. Update idx_orders_user: same 4. Update idx_orders_merchant: same 5. Update idx_orders_date: same 6. Update idx_orders_amount: same (if amount changed) = 1 heap write + up to 5 × 2 index writes = up to 11 writes ``` **HOT update optimization:** If the updated column is NOT indexed AND the new row version fits on the same heap page, PostgreSQL does a Heap-Only Tuple (HOT) update — no index updates at all. This is why keeping fill factor below 100% helps write-heavy tables. **Leaf pointer support:** ✅ Full doubly-linked leaf nodes. Range queries are efficient. **Concurrent B+ Tree (B-Link Tree):** PostgreSQL uses the Lehman & Yao B-Link Tree variant. Each page has a "right-link" pointer. During a split, the right-link lets concurrent readers find the new page without needing the root-to-leaf lock. Enables high-concurrency inserts without excessive locking. --- ## 4.2 — MySQL InnoDB **Storage model:** Clustered index (Index-Organized Table). The table IS the primary key B+ Tree. Leaf nodes of the primary index contain **the full row** (all columns). ``` MySQL InnoDB primary B+ Tree (clustered): Internal nodes: [PK keys for routing only] Leaf nodes: full rows ordered by PK ┌──────────────────────────────────────────────────────────────┐ │ pk=1001 │ name="Alice" │ email="a@b.com" │ status="active" … │ └──────────────────────────────────────────────────────────────┘ ┌──────────────────────────────────────────────────────────────┐ │ pk=1002 │ name="Bob" │ email="b@b.com" │ status="pending" … │ └──────────────────────────────────────────────────────────────┘ Physical storage order on disk = PK order. ``` **Consequence of clustered storage:** INSERT order matters enormously. Sequential PK (BIGINT AUTO_INCREMENT) = rows always appended to the rightmost leaf page = minimal splits. Random PK (UUID v4) = every insert causes a page split somewhere in the middle of the physical table = catastrophic write performance at scale. **Secondary index values:** Point to the **primary key value** (not a physical location). ``` Secondary index leaf entry in MySQL: [indexed_key_value | primary_key_value] e.g.: ["alice@example.com" | pk=1001] To fetch the row: 1. Traverse secondary B+ Tree → get PK value (1001) 2. Traverse primary B+ Tree → get full row at PK=1001 = Double lookup (two B+ Tree traversals) ``` **Why secondary indexes point to PK (not physical location):** When a row is UPDATEd in a clustered table, the row stays at the same PK position (the PK doesn't change). Secondary indexes remain valid — no write amplification on update. If secondary indexes pointed to physical locations (like PostgreSQL), every row movement would require updating all secondary indexes. **UUID v4 as PK in MySQL — Why It's Catastrophic:** ``` Table: orders (id UUID PRIMARY KEY, ...) ← worst possible MySQL design Problem 1: Clustered random inserts Every INSERT with a new random UUID must find its sorted position in the physical table (= primary B+ Tree). Page splits happen constantly. Buffer pool thrashes trying to keep random pages cached. Problem 2: Bloated secondary indexes Every secondary index entry carries the UUID PK (16 bytes) instead of a BIGINT (4-8 bytes). 5 secondary indexes × 1B rows × extra 12 bytes = 60GB extra index storage More storage = less fits in buffer pool = more disk I/Os = slower queries. Problem 3: Double lookup with large values Secondary index → UUID (16 bytes) → primary B+ Tree lookup Each leaf entry in every secondary index is 12 bytes heavier than with BIGINT PK. ``` **Leaf pointer support:** ✅ Full doubly-linked leaf nodes. --- ## 4.3 — MongoDB (WiredTiger Engine) WiredTiger is a B+ Tree storage engine used by MongoDB. **Notable design decision:** WiredTiger does **not** implement leaf node linking by default for its B+ Trees. ``` Reason: "Range queries are not a primary MongoDB use case." Document databases are designed for key lookups, not ordered range scans across large sets. Removing leaf links: - Reduces write overhead (no sibling pointer updates on split) - Saves 8-16 bytes per page (left/right sibling pointers) - Acceptable trade-off given MongoDB's access patterns ``` This is a deliberate engineering trade-off: optimize for the common case (point lookups), accept worse range scan performance. MongoDB also supports an LSM-Tree variant (used in some configurations), which is a completely different structure optimized for write-heavy workloads. --- ## 4.4 — Oracle **Storage model:** Two options: 1. **Heap-organized tables** (default): Similar to PostgreSQL. Rows unordered. Secondary indexes store physical rowid. 2. **Index-Organized Tables (IOT)**: Same as MySQL's clustered index. The table is the primary B+ Tree. **Secondary index behavior (heap tables):** Point to physical rowid (similar to PostgreSQL's ctid). Same write amplification trade-off. **Secondary index behavior (IOT):** Secondary indexes store a **logical rowid** which includes the primary key. Double lookup required (same as MySQL). --- ## 4.5 — SQL Server **Storage model:** Two options: 1. **Heap tables**: Rows unordered (no clustered index defined). 2. **Clustered tables** (default when PK defined): Same as MySQL InnoDB's clustered structure. **Nonclustered (secondary) indexes:** Store the clustered key value for lookup — same double-lookup pattern as MySQL. **Unique consideration:** SQL Server allows you to choose *which* index is clustered, not necessarily the primary key. This gives more flexibility than MySQL. --- # 5. The PostgreSQL vs MySQL Secondary Index Trade-off This is one of the most important practical distinctions in production database engineering. ## Summary Table | Aspect | PostgreSQL (→ ctid) | MySQL InnoDB (→ PK) | |---|---|---| | Secondary index entry size | Small — ctid is 6 bytes | PK-dependent (INT=4B, UUID=16B) | | Secondary index lookup | Direct heap fetch (1 extra I/O) | Double B+ Tree traversal (2 extra I/Os) | | Write amplification on UPDATE | High — all secondary indexes update when row moves | Low — PK doesn't change on UPDATE | | UUID as PK impact | Damages PK index only | Damages table storage + ALL secondary indexes | | Row movement on UPDATE | Always (new heap slot) | Rare (only if PK column updated) | | Best for | Read-heavy, infrequent updates | Write-heavy tables with stable PKs | ## The Uber Migration Context Uber moved from PostgreSQL to MySQL for their main trip database. One contributing factor: - Uber's trip table had frequent UPDATEs (status changes: requested → accepted → in-progress → completed) - PostgreSQL's ctid-based secondary indexes meant every status UPDATE required updating all secondary index entries pointing to the old ctid - At Uber's write volume (millions of trips/day), this write amplification was a meaningful overhead - MySQL's PK-based secondary indexes meant a status UPDATE (non-PK column) did not require updating secondary indexes **Important nuance:** This was one of several factors, not the sole reason. WAL volume, replication behavior, and connection handling were also cited. --- # 6. Key Selection and B+ Tree Performance ## Data Type Impact on Index Performance | Key Type | Size | Elements/Page (8KB) | Tree Height (1B rows) | Notes | |---|---|---|---|---| | INT (4B) | 4 bytes | ~730 | 3 levels | Ideal | | BIGINT (8B) | 8 bytes | ~585 | 3-4 levels | Standard choice | | UUID v4 (binary, 16B) | 16 bytes | ~360 | 4 levels | Acceptable if ordered insert | | UUID v4 (string, 36B) | 36 bytes | ~195 | 4-5 levels | Avoid as index key | | VARCHAR(50) | 50B avg | ~120 | 5 levels | Only index if necessary | | VARCHAR(255) | 255B avg | ~28 | 7+ levels | Serious performance issue | ## Sequential vs Random Insert Patterns ``` Sequential inserts (BIGINT AUTO_INCREMENT, ULID, UUID v7): Time → [1001] [1002] [1003] [1004] [1005] ... All inserts go to the SAME rightmost leaf page. That page stays HOT in buffer pool. Page splits happen only when rightmost page fills up. New page allocated at the right edge. Clean. Buffer pool: 1 hot page handles thousands of inserts. Splits: Rare, predictable, at the right edge only. Random inserts (UUID v4): Time → [f47ac...] [550e8...] [b2c3d...] [a716-...] ... Each insert targets a random position in the tree. Each insert fetches a different leaf page from disk. Each fetched page evicts a different cached page. High probability the evicted page is needed next. Buffer pool: Constant cache misses. Thrashing. Splits: Frequent, unpredictable, scattered throughout tree. ``` ## MySQL-Specific: UUID as PK ``` MySQL InnoDB, UUID v4 PRIMARY KEY: Physical table (clustered B+ Tree by PK): Insert "550e8400..." → must go between existing UUIDs Insert "f47ac10b..." → must go somewhere else entirely Insert "b2c3d479..." → somewhere else again Every insert potentially splits a page in the MIDDLE of the physical table. The physical table becomes fragmented. Sequential scans become random I/Os. Benchmark impact (approximate, varies by workload): Sequential BIGINT PK: 50,000-100,000 inserts/second UUID v4 PK (MySQL): 5,000-15,000 inserts/second at scale (Performance gap widens as table grows beyond buffer pool capacity) ``` --- # 7. Fill Factor — Controlling Page Split Frequency Fill factor controls how full each B+ Tree page is when first built or rebuilt. Leaving free space reduces future page splits. ```sql -- PostgreSQL: Create index with 70% fill factor -- Leaves 30% of each page empty for future inserts CREATE INDEX idx_orders_user ON orders(user_id) WITH (fillfactor = 70); -- Rebuild existing index with new fill factor REINDEX INDEX CONCURRENTLY idx_orders_user; -- Set fill factor at table level (helps HOT updates) ALTER TABLE orders SET (fillfactor = 70); VACUUM orders; -- Rebuilds pages with new fill factor ``` **Trade-off:** ``` fillfactor = 100 (default): Pages fully packed → maximum storage efficiency First insert into a full page → immediate split Bad for write-heavy tables fillfactor = 70: Pages 70% full → 30% free space for new inserts Many inserts absorbed without splits Cost: ~43% more pages (100/70 = 1.43×) → slightly larger index Worth it for write-heavy tables ``` --- # 10. Important Interview Questions ## Basic **Q: What is the difference between a B-Tree and a B+ Tree?** A: B-Tree stores key-value (data pointer) pairs in ALL nodes including internal nodes. B+ Tree stores only keys in internal nodes; all data pointers live in leaf nodes only. B+ Tree leaf nodes are also doubly linked, enabling efficient range scans. B+ Trees are used by virtually all production databases because internal nodes are slimmer (more keys per page, fewer I/Os) and range queries are faster (follow leaf links instead of re-traversing from root). **Q: Why is one B+ Tree node equal to one disk page?** A: Reading from disk happens in fixed-size chunks (pages). Making one node equal one page means a single disk I/O retrieves the entire node. If a node spanned multiple pages, one traversal step would require multiple I/Os — wasting I/O budget on a single routing decision. **Q: Why does a larger key data type slow down index performance?** A: Larger keys mean fewer key-value pairs fit in each internal node page. Fewer elements per node means more nodes needed to index the same data, making the tree taller. A taller tree requires more I/Os per traversal. Also, larger keys mean fewer nodes fit in the buffer pool (RAM), reducing cache hit rates. **Q: What is a page split and why is it expensive?** A: A page split occurs when inserting into a full B+ Tree page. The full page is split into two half-full pages, a key is promoted to the parent internal node, doubly-linked list pointers are updated, WAL entries are written for all modified pages, and the split may cascade up the tree. It involves multiple I/Os, multiple WAL writes, and pointer updates — significantly more expensive than a normal insert. **Q: Why does the B+ Tree's leaf linked list help range queries?** A: After finding the first key in a range (O(log n) traversal), subsequent keys in the range can be found by simply following the next-leaf pointer — one I/O per leaf page — without returning to the root. This reduces a range scan from O(k × log n) to O(log n + k/page_capacity). ## Intermediate **Q: How does PostgreSQL's secondary index differ from MySQL's? What are the consequences?** A: PostgreSQL secondary indexes store the physical tuple ID (ctid = page + offset, 6 bytes). MySQL InnoDB secondary indexes store the primary key value. PostgreSQL: smaller entries, direct heap access, but write amplification when rows move (all secondary indexes must update with new ctid). MySQL: no write amplification on non-PK updates, but requires double lookup (secondary B+ Tree → PK value → primary B+ Tree), and large PKs (UUID) bloat all secondary indexes. **Q: Why is UUID v4 as a primary key in MySQL significantly worse than in PostgreSQL?** A: MySQL InnoDB is clustered — the table IS the primary B+ Tree, ordered by PK. UUID v4's random values cause every insert to target a random page in the physical table, causing constant page splits throughout the entire table storage, buffer pool thrashing, and progressive performance degradation. Additionally, all secondary indexes store the 16-byte UUID PK value, making every secondary index entry 12 bytes heavier than with a BIGINT PK. In PostgreSQL, the table is an unordered heap — UUID PK inserts just append to the heap. Only the PK index suffers from random splits, not the entire table structure. **Q: Why does MongoDB's WiredTiger B+ Tree not have leaf node links? Is this a bug?** A: It's a deliberate design decision. MongoDB's primary access pattern is key lookups (find document by _id or indexed field), not range scans. Removing leaf links reduces write overhead (no sibling pointer updates during splits) and saves page space. The trade-off — worse range scan performance — is acceptable for document database workloads. Engineers only include features they absolutely need, not features that impose overhead for rare use cases. **Q: How does keeping internal B+ Tree nodes in memory reduce effective I/Os to nearly 1?** A: For a 1-billion-row table with an 8-byte integer PK (PostgreSQL, 8KB pages): leaf nodes occupy ~13GB (too large for typical RAM), but the internal node layer immediately above the leaves is only ~23MB (easily fits in shared_buffers). With all internal nodes cached, every query follows the traversal entirely in memory and requires only 1 disk I/O — for the leaf page containing the actual data pointer. This scales to tens of billions of rows with minimal extra memory for internal nodes. ## Advanced **Q: Walk through everything that happens — I/Os, WAL writes, pointer updates — when a B+ Tree leaf page splits.** A: (1) New page allocated on disk; (2) Half the entries copied to new page; (3) Original page's right-sibling pointer updated to new page; (4) New page's left-sibling pointer set to original, right-sibling set to original's old right neighbor; (5) Old right neighbor's left-sibling pointer updated to new page; (6) Split boundary key promoted to parent internal node; (7) If parent internal node is also full, cascade split upward (repeating steps 1-6); (8) WAL entry written for every modified page (original leaf, new leaf, parent, potentially root) before any page is written to disk; (9) All modified pages marked dirty in buffer pool. Worst case: O(tree height) pages modified and WAL-logged for a single insert. **Q: Explain why Uber's migration from PostgreSQL to MySQL was influenced by B+ Tree secondary index design.** A: Uber's trip table underwent frequent UPDATEs (trip status changes). In PostgreSQL, every UPDATE creates a new row version at a new heap location (new ctid). All secondary indexes pointing to the old ctid must be updated. With multiple secondary indexes on the trips table and millions of updates per day, this write amplification was significant — every logical UPDATE translated to many physical index writes. MySQL's secondary indexes point to the stable PK value; a non-PK UPDATE (like status change) doesn't require any secondary index updates. This reduced write amplification was one factor favoring MySQL for Uber's write-heavy workload. **Q: What is the HOT (Heap-Only Tuple) optimization in PostgreSQL and how does it relate to B+ Tree maintenance?** A: When a PostgreSQL UPDATE modifies a non-indexed column and the new row version fits on the same heap page as the old version, PostgreSQL performs a HOT update. The index entry is NOT updated — it still points to the old row's ctid, which contains a forwarding pointer to the new version. This completely avoids B+ Tree write amplification for that update. The conditions required: (1) the updated column must not be indexed, (2) there must be free space on the same page. Setting fillfactor below 100% increases the probability of free space being available, making HOT updates more frequent. --- # 11. Common Mistakes ## Beginner Mistakes - **Confusing B-Tree with B+ Tree**: Textbooks draw B-Trees without value pointers, making them look identical. Remember: B+ Tree = keys only in internal nodes + linked leaves. - **Thinking "more indexes = always better"**: Each index is a separate B+ Tree that must be maintained on every INSERT/UPDATE/DELETE. - **Indexing low-cardinality columns alone**: An index on a boolean column with 2 values returns 50% of the table — the query planner ignores it and does a full scan instead. ## Backend Engineer Mistakes - **UUID v4 as MySQL primary key**: The single most common MySQL performance mistake. Causes progressive write degradation as the table grows. - **Not understanding composite index prefix rule**: Index on `(user_id, status)` cannot serve queries filtering only on `status`. The leftmost column must always be in the WHERE clause. - **Ignoring secondary index bloat from large PKs**: In MySQL, choosing UUID over BIGINT as PK makes every secondary index entry 12 bytes heavier — multiplied across millions of rows and multiple indexes. ## Production Mistakes - **Running REINDEX on a large table without CONCURRENTLY**: `REINDEX INDEX idx_name` takes an exclusive lock, blocking all reads and writes. Always use `REINDEX INDEX CONCURRENTLY` in production (PostgreSQL 12+). - **Not monitoring index bloat**: Heavily updated tables accumulate dead index entries that are never cleaned without VACUUM. Use `pg_stat_user_indexes` to monitor index size vs table size ratio. - **Using string UUIDs instead of binary UUIDs in MySQL**: A string UUID (`VARCHAR(36)`) uses 36 bytes per entry. Binary UUID (`BINARY(16)`) uses 16 bytes — 2.25× more compact. If you must use UUID in MySQL, at minimum store it as binary. ```sql -- Worse: string UUID id VARCHAR(36) PRIMARY KEY -- 36 bytes per entry in every secondary index -- Better: binary UUID (still not ideal for clustered, but less bad) id BINARY(16) PRIMARY KEY -- 16 bytes per entry -- Best: ordered binary UUID (UUID v7) or BIGINT id BIGINT AUTO_INCREMENT PRIMARY KEY -- 4-8 bytes, sequential, clustered-friendly ``` --- # 12.
