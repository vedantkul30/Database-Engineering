Database Sharding

Production Engineering Notes

1. Big Picture

Why Sharding Exists

A single database server has physical limits: CPU cores, RAM, disk I/O bandwidth, network throughput, and connection limits. As data grows and write throughput demands exceed what one machine can handle, you need to distribute both data and load across multiple machines. This is sharding.

But sharding is the last resort — not the first solution. Before sharding, you have a well-defined escalation path of progressively complex (but less complex than sharding) solutions.

"The last thing you want to do is shard your database. There are so many other things you can do before complicating your life with sharding."

The Scaling Problem Chain

Problem: "My reads are slow"
   ↓
Solution 1: Add indexes on queried columns
   ↓ (indexes grow large too, queries still slow at scale)
Solution 2: Add partitioning (same server, smaller indexes)
   ↓ (server CPU/connection limits hit)
Solution 3: Add read replicas (scale reads horizontally)
   ↓ (write throughput saturates the single master)
Solution 4: Multi-master replication (US-East / US-West)
   ↓ (write conflicts become unmanageable, need true horizontal write scale)
Solution 5: SHARD ← you are here. Final option.

Where Sharding Is Used

CompanySharding Use Case
YouTube/GoogleMySQL sharding via Vitesse — managed app-level sharding at scale
InstagramPostgreSQL sharding via Citus — user_id based shards
UberMySQL sharding for trip data
WhatsAppErlang + custom sharding for message storage
ShopifyMySQL sharding per merchant — each merchant is a shard
StripeShard by merchant_id

2. Core Concepts

2.1 — What Is Sharding?

Sharding splits a large table across multiple independent database servers. Each server holds a shard — a subset of the rows. The table name stays the same, but the server (host + port) changes.

Non-sharded (1 server, 1M URLs):
┌──────────────────────────────────────┐
│  DB Server (port 5432)               │
│  ┌──────────────────────────────┐   │
│  │  url_table (1,000,000 rows)  │   │
│  └──────────────────────────────┘   │
└──────────────────────────────────────┘

Sharded (3 servers, ~333K URLs each):
┌─────────────────────┐  ┌─────────────────────┐  ┌─────────────────────┐
│  Shard 1 (port 5432)│  │  Shard 2 (port 5433)│  │  Shard 3 (port 5434)│
│  url_table          │  │  url_table          │  │  url_table          │
│  ~333K rows         │  │  ~333K rows         │  │  ~333K rows         │
│  (url_id: set A)    │  │  (url_id: set B)    │  │  (url_id: set C)    │
└─────────────────────┘  └─────────────────────┘  └─────────────────────┘
        ↑                        ↑                        ↑
  Client MUST know which shard to hit. No automatic routing.

Critical difference from partitioning:

PartitioningSharding
ServersSingle serverMultiple servers
Table namesDifferent (customers_0_10M, customers_10M_20M)Same (url_table on every shard)
Client awarenessNone — database routes automaticallyRequired — client/proxy routes to correct server
TransactionsFull ACIDNo cross-shard ACID transactions
JOINsNormal SQL JOINs workCross-shard JOINs essentially impossible

2.2 — Consistent Hashing

Consistent hashing is the algorithm used to determine which shard a given input should go to. The requirement: given the same input string, always return the same shard — deterministically and consistently.

Why Not Simple Modulo?

Simple modulo approach:
  hash("wikipedia.com/sharding") % 3 = shard 2

Problem: Add a 4th shard:
  hash("wikipedia.com/sharding") % 4 = shard 1  ← DIFFERENT!

With 3 shards → 4 shards, ~75% of all keys map to different shards.
Every cached/stored record must be relocated. Catastrophic.

The Hash Ring

Consistent hashing places both shard nodes and data keys on a virtual circular ring (0 → 2^32 or similar range). Each key maps to the nearest node clockwise on the ring.

Hash Ring (0 → MAX):

                    0 / MAX
                      │
           Shard A ───┤──── Shard B
          /           │           \
    key3 ●            │            ● key1
        |      ring   │     ring   |
    key2 ●            │            ● key4
          \           │           /
           Shard C ───┤
                      │

Rules:
  - Each key maps to the nearest shard clockwise
  - key1 → Shard B (nearest clockwise)
  - key2 → Shard C (nearest clockwise)
  - key3 → Shard A (nearest clockwise)

Adding a new Shard D: only keys between Shard D's position and
the previous node need to be relocated. ~1/N keys affected (vs 75% with modulo).

How It Works in Practice

Input: "wikipedia.com/sharding"
Step 1: SHA-256 hash → large hex string
Step 2: Take first N bytes → a number in [0, MAX)
Step 3: Find nearest shard clockwise on ring → Shard 2 (port 5433)
Step 4: Always returns port 5433 for this exact input

Deterministic: same input → same output, always.

docker ps

Each container has the same url_table schema, same table name, different port. That is the essence of sharding.

Step 1: Shard Configuration

import org.springframework.jdbc.core.JdbcTemplate;
import org.springframework.jdbc.datasource.DriverManagerDataSource;

import java.util.HashMap;
import java.util.Map;

/**
 * ShardManager holds one JdbcTemplate per shard.
 * Key = port number (String) → Value = JdbcTemplate for that shard.
 *
 * In the Node.js version:
 *   const clients = {
 *     "5432": new Client({port: 5432}),
 *     "5433": new Client({port: 5433}),
 *     "5434": new Client({port: 5434})
 *   }
 */
public class ShardManager {

    // Map of port → JdbcTemplate (one connection pool per shard)
    private final Map<String, JdbcTemplate> shards = new HashMap<>();

    public ShardManager() {
        // Initialize 3 shard connections
        shards.put("5432", createJdbcTemplate("localhost", 5432));
        shards.put("5433", createJdbcTemplate("localhost", 5433));
        shards.put("5434", createJdbcTemplate("localhost", 5434));
    }

    private JdbcTemplate createJdbcTemplate(String host, int port) {
        DriverManagerDataSource ds = new DriverManagerDataSource();
        ds.setDriverClassName("org.postgresql.Driver");
        ds.setUrl("jdbc:postgresql://" + host + ":" + port + "/postgres");
        ds.setUsername("postgres");
        ds.setPassword("postgres");
        return new JdbcTemplate(ds);
    }

    /**
     * Get the JdbcTemplate for a given port string (returned by consistent hash).
     */
    public JdbcTemplate getShardClient(String port) {
        return shards.get(port);
    }
}

Step 2: Consistent Hashing

import java.nio.charset.StandardCharsets;
import java.security.MessageDigest;
import java.util.*;

/**
 * ConsistentHashRing maps arbitrary string keys to one of N shard ports.
 *
 * Algorithm:
 *   1. Place virtual nodes for each shard on a sorted ring (TreeMap).
 *   2. To find a shard for key K: hash K → position on ring → find nearest
 *      shard clockwise (ceiling entry in TreeMap).
 *   3. Same key always returns same shard (deterministic).
 *   4. Adding/removing shards only remaps ~1/N of keys.
 *
 * In the Node.js version:
 *   const hr = new HashRing();
 *   hr.add("5432"); hr.add("5433"); hr.add("5434");
 *   const server = hr.get(urlId); // returns "5432", "5433", or "5434"
 */
public class ConsistentHashRing {

    // TreeMap: ring position (long) → shard port (String)
    // TreeMap keeps entries sorted by key → enables ceiling() lookup
    private final TreeMap<Long, String> ring = new TreeMap<>();

    // Number of virtual nodes per physical shard.
    // More virtual nodes = more even distribution across the ring.
    private final int virtualNodes;

    public ConsistentHashRing(int virtualNodes) {
        this.virtualNodes = virtualNodes;
    }

    /**
     * Add a shard to the ring.
     * Creates 'virtualNodes' positions for this shard across the ring.
     *
     * @param shardPort e.g. "5432"
     */
    public void addShard(String shardPort) {
        for (int i = 0; i < virtualNodes; i++) {
            // Each virtual node has a unique key: "5432-vnode-0", "5432-vnode-1", ...
            long position = hash(shardPort + "-vnode-" + i);
            ring.put(position, shardPort);
        }
    }

    /**
     * Given an input key, find which shard it maps to.
     * Returns the shard port (e.g. "5433").
     *
     * @param key e.g. a URL ID "5fJkQ"
     */
    public String getShard(String key) {
        if (ring.isEmpty()) {
            throw new IllegalStateException("No shards registered");
        }

        long position = hash(key);

        // Find the nearest shard clockwise from 'position'
        // ceilingEntry: smallest key >= position
        Map.Entry<Long, String> entry = ring.ceilingEntry(position);

        // If no entry >= position, wrap around to the first entry on the ring
        if (entry == null) {
            entry = ring.firstEntry();
        }

        return entry.getValue();
    }

    /**
     * SHA-256 hash of a string, returns a long for ring positioning.
     * Same as Node.js: crypto.createHash('sha256').update(input).digest('base64')
     */
    private long hash(String key) {
        try {
            MessageDigest md = MessageDigest.getInstance("SHA-256");
            byte[] digest = md.digest(key.getBytes(StandardCharsets.UTF_8));
            // Take first 8 bytes and convert to long (ring position)
            long result = 0;
            for (int i = 0; i < 8; i++) {
                result = (result << 8) | (digest[i] & 0xFF);
            }
            return result;
        } catch (Exception e) {
            throw new RuntimeException("Hashing failed", e);
        }
    }
}

Step 3: URL Shortener Service

import org.springframework.jdbc.core.JdbcTemplate;
import java.nio.charset.StandardCharsets;
import java.security.MessageDigest;
import java.util.Base64;
import java.util.List;
import java.util.Map;

/**
 * UrlShortenerService handles:
 *   1. POST /shorten?url=... → generate 5-char code, determine shard, write to shard DB
 *   2. GET  /{urlId}         → determine shard from urlId, read from shard DB
 */
public class UrlShortenerService {

    private final ConsistentHashRing hashRing;
    private final ShardManager shardManager;

    public UrlShortenerService(ConsistentHashRing hashRing, ShardManager shardManager) {
        this.hashRing = hashRing;
        this.shardManager = shardManager;
    }

    /**
     * POST /shorten?url=https://wikipedia.com/sharding
     *
     * Node.js equivalent:
     *   app.post("/", async (req, res) => {
     *     const url = req.query.url;
     *     const hash = crypto.createHash('sha256').update(url).digest('base64');
     *     const urlId = hash.substring(0, 5);
     *     const server = hr.get(urlId);
     *     const client = clients[server];
     *     await client.query("INSERT INTO url_table(url, url_id) VALUES($1,$2)", [url, urlId]);
     *   });
     */
    public Map<String, String> shortenUrl(String url) throws Exception {
        // Step 1: SHA-256 hash the URL → Base64 string
        MessageDigest md = MessageDigest.getInstance("SHA-256");
        byte[] hashBytes = md.digest(url.getBytes(StandardCharsets.UTF_8));
        String hash = Base64.getEncoder().encodeToString(hashBytes);

        // Step 2: Take first 5 characters as the short code
        // Same as Node.js: hash.substring(0, 5)
        String urlId = hash.substring(0, 5);

        // Step 3: Consistent hash the urlId → get which shard port to use
        // Same as Node.js: const server = hr.get(urlId);
        String shardPort = hashRing.getShard(urlId);

        // Step 4: Get the JdbcTemplate for that shard
        // Same as Node.js: const client = clients[server];
        JdbcTemplate jdbcTemplate = shardManager.getShardClient(shardPort);

        // Step 5: Insert into the correct shard
        // Same as Node.js: await client.query("INSERT INTO url_table(url, url_id) VALUES($1,$2)")
        // Use parameterized query — NEVER use string concatenation (SQL injection risk)
        jdbcTemplate.update(
            "INSERT INTO url_table(url, url_id) VALUES(?, ?)",
            url, urlId
        );

        return Map.of(
            "urlId",  urlId,
            "url",    url,
            "shard",  shardPort   // which shard it went to (for debugging)
        );
    }

    /**
     * GET /{urlId}  →  return original URL
     *
     * Node.js equivalent:
     *   app.get("/:urlId", async (req, res) => {
     *     const id = req.params.urlId;
     *     const server = hr.get(id);
     *     const client = clients[server];
     *     const result = await client.query("SELECT * FROM url_table WHERE url_id=$1", [id]);
     *     if (result.rowCount > 0) res.send(result.rows[0]);
     *     else res.sendStatus(404);
     *   });
     */
    public Map<String, Object> getUrl(String urlId) {
        // Step 1: Consistent hash urlId → find which shard holds this record
        // SAME hash function, SAME input → SAME shard every time
        String shardPort = hashRing.getShard(urlId);

        // Step 2: Get JdbcTemplate for that shard
        JdbcTemplate jdbcTemplate = shardManager.getShardClient(shardPort);

        // Step 3: Query that shard
        List<Map<String, Object>> rows = jdbcTemplate.queryForList(
            "SELECT * FROM url_table WHERE url_id = ?",
            urlId
        );

        if (rows.isEmpty()) {
            return null;  // Controller will return 404
        }

        // Add which shard served this request (useful for debugging/observability)
        rows.get(0).put("shard", shardPort);
        return rows.get(0);
    }
}

Step 4: REST Controller

import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;
import java.util.Map;

/**
 * REST API Controller — exposes POST and GET endpoints.
 *
 * POST /shorten?url=...     → shorten a URL (writes to correct shard)
 * GET  /{urlId}             → retrieve original URL (reads from correct shard)
 */
@RestController
public class UrlShortenerController {

    private final UrlShortenerService service;

    public UrlShortenerController(UrlShortenerService service) {
        this.service = service;
    }

    @PostMapping("/shorten")
    public ResponseEntity<Map<String, String>> shortenUrl(
            @RequestParam String url) throws Exception {
        Map<String, String> result = service.shortenUrl(url);
        return ResponseEntity.ok(result);
    }

    @GetMapping("/{urlId}")
    public ResponseEntity<?> getUrl(@PathVariable String urlId) {
        Map<String, Object> result = service.getUrl(urlId);
        if (result == null) {
            return ResponseEntity.notFound().build();  // HTTP 404
        }
        return ResponseEntity.ok(result);
    }
}

Step 5: Application Bootstrap

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.context.annotation.Bean;

@SpringBootApplication
public class ShardingApplication {

    public static void main(String[] args) {
        SpringApplication.run(ShardingApplication.class, args);
    }

    /**
     * Wire up the consistent hash ring with our 3 shard ports.
     * 150 virtual nodes per shard = 450 total ring positions = even distribution.
     */
    @Bean
    public ConsistentHashRing consistentHashRing() {
        ConsistentHashRing ring = new ConsistentHashRing(150);
        ring.addShard("5432");  // Shard 1
        ring.addShard("5433");  // Shard 2
        ring.addShard("5434");  // Shard 3
        return ring;
    }

    @Bean
    public ShardManager shardManager() {
        return new ShardManager();
    }

    @Bean
    public UrlShortenerService urlShortenerService(
            ConsistentHashRing ring, ShardManager shardManager) {
        return new UrlShortenerService(ring, shardManager);
    }
}

4. The Scaling Decision Tree

Before sharding, exhaust these options in order:

Level 1: Indexing

Problem: Slow reads on large table
Solution: Add B+ Tree index on queried columns

CREATE INDEX idx_url_table_url_id ON url_table(url_id);

Handles: 10M-100M rows on SSD with < 5ms query latency
Cost: Minimal. Just disk space and insert overhead.
Limit: Index grows with table. Eventually too large for buffer pool.

Level 2: Partitioning

Problem: Index too large for RAM, buffer pool misses
Solution: Horizontal partitioning — split table on same server

CREATE TABLE url_table PARTITION BY RANGE (id);
-- url_table_0_1M, url_table_1M_2M, etc.

Handles: 100M-10B rows, smaller per-partition indexes fit in RAM
Cost: Low. Database handles routing transparently.
Limit: Single server CPU/RAM/connection limits still apply.

Level 3: Read Replicas

Problem: Too many concurrent read connections saturating server
Solution: Replicate to N read replicas; route reads to replicas

Master: handles all writes
Replica 1, 2, 3...: handle reads (load balanced by proxy/Nginx)

Handles: Read-heavy workloads (social media, news sites — 95% reads)
Cost: Medium. Replica lag (eventual consistency for reads).
Limit: Writes still go to single master. Master write capacity unchanged.
Architecture:

Client → Nginx/HAProxy ─┬→ Master (port 5432) [Writes only]
                        ├→ Replica 1           [Reads]
                        ├→ Replica 2           [Reads]
                        └→ Replica 3           [Reads]

Replication: Master → WAL streaming → Replicas (async, near real-time)

Level 4: Multi-Master (Regional)

Problem: Write throughput saturates single master
Solution: Two masters in different regions (US-East, US-West)

US-East customers → US-East master
US-West customers → US-West master
Conflicts: rare (different customer sets) → resolve when they occur

Handles: Write-heavy, geographically segregated data
Cost: Medium-High. Conflict resolution logic needed.
Limit: Geographic overlap or truly global shared state → conflicts.

Level 5: Sharding (Last Resort)

Problem: All of the above exhausted. Write throughput truly exceeded.
Solution: Horizontal sharding across N database servers.

Cost: VERY HIGH.
  - Application routing complexity
  - No cross-shard transactions
  - No cross-shard JOINs
  - Schema changes must hit every shard
  - Resharding is a massive operation

5. Consistent Hashing Deep Dive

Why Standard Hash + Modulo Fails for Resharding

java
// Simple hash approach
int shard = Math.abs(urlId.hashCode()) % numShards;

// With 3 shards: "5fJkQ" → shard 2
// Add 4th shard: "5fJkQ" → shard 1  ← DIFFERENT!
// All existing records now point to wrong shards
// Must relocate ~75% of all data. Catastrophic downtime.

Why Consistent Hashing Solves This

Ring with 3 shards:
  Position 10% → Shard A
  Position 40% → Shard B
  Position 75% → Shard C

Add Shard D at position 55%:
  Old: 40-75% → Shard B
  New: 40-55% → Shard B, 55-75% → Shard D

Only keys in the 40-55% range need to move from B to D.
That's ~1/4 of Shard B's data = ~1/N total data.
vs modulo: ~75% of ALL data moves.

Virtual Nodes — Why They Matter

Without virtual nodes, with 3 shards you have 3 ring positions. The distribution may be very uneven:

Ring without virtual nodes:
  Shard A at position 5%
  Shard B at position 6%
  Shard C at position 90%

Result:
  Shard A: keys 90% → 5%   = 15% of keyspace
  Shard B: keys 5% → 6%    =  1% of keyspace  ← overloaded  
  Shard C: keys 6% → 90%   = 84% of keyspace  ← hot spot

With 150 virtual nodes per shard (450 total positions), distribution becomes statistically even — each shard handles ~33% of the keyspace regardless of the physical ring position assignments.

6. Pros and Cons of Sharding

Pros

1. Horizontal Scalability

Data scalability:   add more shards → store more data
Memory scalability: each shard has its own RAM → no single server memory limit
CPU scalability:    queries distributed → parallel execution across shards
I/O scalability:    separate disks → no I/O bottleneck on one machine

2. Smaller Indexes Per Shard

Non-sharded: url_table (1B rows) → index = 14GB (exceeds RAM, disk I/Os)
Sharded 100 ways: url_table (10M rows each) → index = 140MB (fits in RAM → 0 disk I/Os)

3. Fault Isolation

Shard 2 crashes → only users whose data lives on Shard 2 are affected
vs non-sharded: one DB crash → ALL users affected

4. Security / Access Control

VIP customer data → Shard A (restricted access, extra encryption)
Standard customer data → Shards B, C, D (normal access)
Regulatory data (EU customers) → EU-region shards (GDPR compliance)

Cons

1. Client Complexity

Application must:
  - Maintain connections to all N shard servers
  - Implement or use a consistent hash function
  - Route EVERY query to the correct shard
  - Handle shard unavailability gracefully

This logic is in YOUR application code. Every service that touches the
sharded table must implement this routing.

2. No Cross-Shard ACID Transactions

// IMPOSSIBLE with sharding (different database servers):
// "Transfer $100 from account A (Shard 1) to account B (Shard 2)"

// Pseudo-code showing the problem:
try {
    shard1.begin();
    shard1.update("accounts SET balance = balance - 100 WHERE id = 'A'");

    shard2.begin();
    shard2.update("accounts SET balance = balance + 100 WHERE id = 'B'");

    // What if shard2 crashes HERE?
    // shard1 committed, shard2 didn't → money disappeared

    shard1.commit();
    shard2.commit();
    // Even if both commit, there's a window between them where data is inconsistent.
} catch (Exception e) {
    shard1.rollback();   // Works for Shard 1
    shard2.rollback();   // May not work if Shard 2 already committed
}

// Solutions: 2-Phase Commit (2PC) — complex and slow
//            Saga pattern — eventual consistency
//            Both add massive complexity. Neither is simple.

3. Cross-Shard JOINs Are Essentially Impossible

-- On a non-sharded DB: simple JOIN
SELECT u.name, o.total
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE o.total > 1000;

-- With sharding: users on Shard 1-3, orders on Shard 4-6
-- You CANNOT run this SQL. Different servers don't know about each other.

-- Workaround (ugly):
-- 1. Query all order shards → collect user_ids in application
-- 2. For each user_id → determine shard → query that shard
-- This is O(n) queries instead of O(1) JOIN. Terrible at scale.

4. Resharding Is a Major Operation

Current: 3 shards, each holding 333M rows
Need: 4 shards (one shard is getting too large)

Problem: All existing records were routed via hash(key) % 3 logic.
         With 4 shards, ~25% of records are now on the wrong shard.

Solution: 
  1. Set up Shard 4
  2. For every record: recalculate shard with new N
  3. If new shard ≠ current shard: migrate the record
  4. This can take DAYS on hundreds of millions of rows
  5. Must be done without downtime (shadow writes, dual-read)
  6. Consistent hashing reduces this to ~1/N records, but still significant

This is why Shopify, YouTube etc. used fixed shard counts for years
before building Vitesse or similar middleware.

5. Schema Changes Are Risky

-- Add a column to url_table:
-- Must run this on ALL shards:
ALTER TABLE url_table ADD COLUMN created_at TIMESTAMPTZ DEFAULT NOW();

-- If Shard 2 fails during migration:
-- Shard 1: has created_at column
-- Shard 2: missing created_at column
-- Shard 3: has created_at column
-- → Application crashes on Shard 2 queries for created_at
-- → Rollback is a nightmare across multiple servers

6. Queries Without Shard Key Hit All Shards (Scatter-Gather)

// You have shard key: url_id → know exactly which shard to query
service.getUrl("5fJkQ");  // → 1 shard queried

// You DON'T have shard key: search by url domain
// "Find all URLs from wikipedia.com"
// → Must query ALL shards and merge results
// This is called "scatter-gather" — performance degrades with more shards
for (String port : ALL_SHARD_PORTS) {
    List<Map> results = shardManager.getShardClient(port)
        .queryForList("SELECT * FROM url_table WHERE url LIKE ?", "%wikipedia.com%");
    allResults.addAll(results);
}
// Latency = max(latency of slowest shard) instead of 1 shard

8. Important Interview Questions

Basic

Q: What is database sharding?
A: Sharding splits a large table across multiple independent database servers. Each server holds a shard — a horizontal subset of the rows. The table name stays the same on each shard, but the server changes. Unlike partitioning (which is on a single server with automatic routing), sharding requires the client or a proxy to determine which server to connect to.

Q: What is the difference between sharding and partitioning?
A: Partitioning splits data into multiple sub-tables on the same database server. The client queries the master table and routing is transparent. Sharding splits data across multiple database servers. The client must know (or use a proxy to determine) which server to connect to. Partitioning supports full ACID transactions; sharding does not support cross-shard transactions.

Q: What is consistent hashing and why is it used for sharding?
A: Consistent hashing places shard nodes and data keys on a virtual ring (0 → MAX). Each key maps to the nearest shard clockwise. It's used because simple modulo hashing (key % N) remaps ~(N-1)/N of all keys when N changes (catastrophic for resharding). Consistent hashing remaps only ~1/N of keys when a shard is added or removed.

Q: What is a shard key?
A: The shard key is the column value used to determine which shard a row belongs to. The consistent hash function is applied to this value. It must be included in every query to enable direct shard routing; queries without the shard key must hit all shards (scatter-gather), which is expensive.

Q: Why should sharding be the last resort?
A: Sharding eliminates ACID transactions across shards, makes JOINs extremely difficult, requires complex client routing logic, complicates schema migrations, and makes resharding a massive operation. Many cheaper alternatives (indexing, partitioning, read replicas, multi-master) should be exhausted first.

Intermediate

Q: What happens to existing data when you add a new shard?
A: With simple modulo hashing, ~(N-1)/N of all records map to different shards — requiring massive data migration. With consistent hashing, only ~1/N of records (those between the new shard's ring position and its predecessor) need to be migrated. The migration itself still requires dual-write periods, shadow reads, and careful cutover to avoid data loss or downtime.

Q: A user's shopping cart involves updating inventory (Shard 1) and creating an order (Shard 2) atomically. How do you handle this in a sharded database?
A: Options include: (1) 2-Phase Commit (2PC): coordinator locks both shards, then commits both or aborts both — correct but slow and fragile (coordinator failure leaves locks held). (2) Saga pattern: execute each step independently with compensating transactions for rollback — eventually consistent but complex. (3) Design away the problem: ensure all data for one transaction lives on the same shard (customer-based sharding ensures cart + order + inventory-reservation are on one shard). Option 3 is usually best — choose your shard key to colocate related data.

Q: What is scatter-gather in sharding and when does it happen?
A: Scatter-gather occurs when a query doesn't include the shard key in its WHERE clause. The application must broadcast the query to ALL shards, collect all results, merge them, and return the final result. Latency = latency of the slowest shard. This defeats the purpose of sharding for those query patterns. Design shard keys to match the most frequent access patterns, and consider denormalization or secondary indexes (maintained at application level) for other patterns.

Q: Walk through exactly what happens when the Node.js / Java application handles POST /shorten?url=https://wikipedia.com/sharding.
A: (1) URL is SHA-256 hashed → large hash string. (2) First 5 characters extracted → urlId = "5fJkQ". (3) Consistent hash ring: ring.getShard("5fJkQ") → position on ring → nearest shard clockwise → returns "5433". (4) JDBC connection for port 5433 retrieved from ShardManager. (5) Parameterized INSERT executed on Shard 2 (port 5433): INSERT INTO url_table(url, url_id) VALUES(?, ?). (6) Same urlId always returns "5433" for this input → GET request will always find it on Shard 2.

Advanced

Q: Design the sharding strategy for a global e-commerce platform with 500M users, 10B orders, operating in US, EU, and Asia.
A:
1. Shard key: user_id (hash-based sharding) — colocates all of a user's orders on one shard, enabling transactional operations per user
2. Consistent hashing ring: 100 shards, 150 virtual nodes each → even distribution
3. Geographic sharding: US shards in US-East/West, EU shards in Frankfurt/Dublin (GDPR compliance), Asia shards in Singapore/Tokyo
4. Cross-region queries: Handled by a global metadata service that knows which region a user_id maps to; route to correct region first, then shard within region
5. Transactions: All user operations (checkout, refund, inventory reserve) colocated on user's shard → single-shard ACID works
6. Cross-shard queries (analytics, fraud detection): Routed to separate OLAP system (BigQuery/Redshift) via CDC streaming — never run analytics on sharded OLTP
7. Resharding: Managed via Vitesse or custom middleware; consistent hashing ensures < 1% of records relocated per shard added

Q: Explain why Shopify chose per-merchant sharding and how this influenced their data model.
A: Shopify shards by shop_id (merchant). Each merchant's store, products, orders, and customers all live on the same shard. This design: (1) enables full ACID transactions per merchant (checkout, refund, inventory all on one shard); (2) enables per-merchant schema variations; (3) provides natural data isolation (GDPR compliance, data portability); (4) makes scaling predictable (add shards for more merchants). The tradeoff: cross-merchant queries (platform-wide analytics, trending products) are scatter-gather operations — routed to a separate analytics warehouse instead.

9. Common Mistakes

Beginner Mistakes

  • Sharding too early: Sharding before exhausting indexing, partitioning, and read replicas. Adds massive complexity for problems that could have been solved much more simply.
  • Using mutable data as shard key: If the shard key value changes, the row must move to a different shard — a DELETE + INSERT across servers. Use immutable fields (user_id, order_id, created_at bucket) as shard keys.
  • Simple modulo hashing: key % N — any change in N remaps the majority of data. Always use consistent hashing for production sharding.

Backend Engineer Mistakes

  • No connection pooling per shard: Opening a new TCP connection for every request to each shard is catastrophic. Use a connection pool (HikariCP in Java, pg-pool in Node.js) per shard.
  • Querying without shard key in WHERE clause: Leads to scatter-gather across all shards. Design queries so the shard key is always present in the filter.
  • Ignoring the scatter-gather problem at design time: Discovering that your second most common query pattern doesn't include the shard key after sharding is live. Model your access patterns first, then choose shard key accordingly.

Production Mistakes

  • No resharding plan: Building a sharded system with no plan for when individual shards grow too large. Always design resharding strategy before going live.
  • Schema migrations without a rollout plan: Applying ALTER TABLE on all shards simultaneously can cause brief incompatibilities. Use backward-compatible migrations (add columns before making them required, deploy code that handles both old and new schema).
  • Not monitoring shard hotspots: Consistent hashing distributes keys evenly, but data hotspots (a celebrity user with 100M followers on one shard) can overload individual shards. Monitor per-shard query rates and data sizes.
  • No fallback for shard unavailability: If Shard 2 is down, what happens? Application should handle shard failures gracefully (circuit breaker, fallback response) rather than returning 500 for all users on that shard.

10. Performance & Scalability Notes

Throughput Scaling With Sharding

Single server:
  Max write throughput: ~50,000 writes/sec (modern SSD, simple INSERT)
  Max read throughput:  ~100,000 reads/sec (with buffer pool hits)
  Max connections:      ~1,000 (PostgreSQL default max_connections)

3-shard setup (linear scaling for independent workloads):
  Write throughput: ~150,000 writes/sec
  Read throughput:  ~300,000 reads/sec
  Connections:      ~3,000 total

100-shard setup:
  Write throughput: ~5,000,000 writes/sec
  Read throughput:  ~10,000,000 reads/sec

Caveat: This assumes perfect key distribution (consistent hashing + virtual nodes) and workloads where no cross-shard operations are needed. Cross-shard queries do NOT benefit from sharding.

Index Size Per Shard

Non-sharded (1B rows, INTEGER key):
  Full index: ~14 GB — does not fit in typical server RAM (8-64GB)
  Result: many disk I/Os per query

100 shards (10M rows each, same key):
  Per-shard index: ~140 MB — fits in RAM
  Result: index traversal = 0 disk I/Os (all in buffer pool)

This is why sharding AND partitioning are often used together:
  shard → partition → index → row
