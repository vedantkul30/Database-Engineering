ACID — Consistency, Durability & Practical Examples

PART A — CONSISTENCY

1. Big Picture

Why Does Consistency Exist as an ACID Property?

Consistency is the most philosophically loaded of the four ACID properties. While Atomicity, Isolation, and Durability are purely database-enforced guarantees, Consistency is co-owned by the database and the application developer. The database enforces the rules you define (constraints, foreign keys, triggers). You define what "correct state" means.

Without consistency guarantees, databases become unreliable data stores where the numbers don't add up, orphaned records pile up, and business logic breaks silently.

The Problem It Solves

ProblemWithout ConsistencyWith Consistency
Referential integrityRecords reference non-existent parentsFK constraints prevent orphaned rows
Financial totalsAccount balances don't match transaction logsAtomicity + constraints keep totals in sync
Data model correctnessAggregates diverge from detail recordsDefined rules enforced at commit
Distributed readsReads return stale data from lagging replicasSynchronous replication or read-your-writes

Where It Appears in Real Systems

  • Banking: Total debits must always equal total credits across the ledger.
  • E-commerce (Amazon): Order items must reference valid products; inventory cannot go negative.
  • Instagram/Social Media: Like counts must match actual like records.
  • Ride-sharing (Uber): A trip record must reference a valid driver and rider.
  • Microservices: Even without shared databases, each service must maintain its own referential integrity.

Why Backend Engineers Must Care

  • Consistency bugs are silent — no exception is thrown when a like count drifts from the actual likes table. The bug shows up weeks later in a BI report.
  • In distributed systems, the "C" in CAP theorem refers to a different kind of consistency (linearizability), not ACID consistency. Conflating the two is a classic interview mistake.
  • NoSQL databases do NOT eliminate consistency concerns — they just push enforcement to the application layer.

2. Core Concepts

Consistency Is Twofold

The instructor clearly separates consistency into two distinct types. Understanding this distinction is critical for interviews and system design.

Type 1: Consistency in Data (State Correctness)

Definition: The data persisted on disk is consistent with the rules and invariants defined for the data model. This is the ACID definition of consistency.

Key insight: This is user-defined (by the DBA or application developer). The database has no inherent understanding of what "correct" means for your domain — you teach it through constraints, foreign keys, triggers, and check constraints.

What enforces it:

MechanismDescription
Foreign KeysPrevent orphaned child records
Check ConstraintsEnforce domain rules (e.g., price > 0)
Unique ConstraintsPrevent duplicate entries
NOT NULLPrevent missing required data
TriggersCustom logic on insert/update/delete
AtomicityAll-or-nothing writes prevent partial states
IsolationPrevents reads of partially-written data

Example — Instagram Data Model:

pictures table:
| id  | blob | num_likes |
|-----|------|-----------|
| 1   | ...  | 5         |   ← claims 5 likes
| 2   | ...  | 1         |   ← claims 1 like

likes table:
| user_id | picture_id |
|---------|------------|
| john    | 1          |
| edmund  | 1          |   ← only 2 actual like records for picture 1
| john    | 2          |
| edmund  | 4          |   ← picture 4 doesn't exist!

Inconsistencies spotted:

  1. pictures.num_likes = 5 for picture 1, but likes table has only 2 records → Count mismatch
  2. likes has an entry for picture_id = 4, but picture 4 does not exist → Orphaned record / broken referential integrity

Root causes:

  • The num_likes column was updated separately from the likes table (not atomically).
  • Picture 4 was deleted without cascading the delete to the likes table.

Type 2: Consistency in Reads (Visibility Correctness)

Definition: After a transaction commits a change, any subsequent read by any client should see that change. If a read returns a stale (pre-commit) value, the system is exhibiting read inconsistency.

Key insight: This applies to the system as a whole, especially in distributed systems with multiple database instances (replicas, shards). The data on disk may be perfectly correct, but different clients reading from different nodes may see different versions of reality.

Classic scenario:

Primary DB ──write──▶ [X = 100 committed]
                              │
                    Async replication (lag: 200ms)
                              │
                              ▼
Replica DB              [X = 99 — stale]

Client A writes X=100 to Primary  → Success
Client B reads  X from Replica     → Gets 99 ← Inconsistent read!

Why this happens:

  • Asynchronous replication means the replica is always slightly behind the primary.
  • Reads routed to the replica return the last replicated state, not the latest committed state.
  • This is NOT data corruption — the primary is correct. But from the client's perspective, the system is behaving inconsistently.

Eventual Consistency

Definition: A distributed system property that guarantees that, if no new updates are made, eventually all reads will return the last written value.

Critical distinction (from the lecture):

ScenarioEventual Consistency Applicable?
Type 1 — Data corruption (num_likes=5, actual=2)❌ NO — Data is corrupt. Only a repair job can fix it.
Type 2 — Replica lag (primary has X=100, replica has X=99)✅ YES — Replica will catch up; eventually consistent.

Consistency spectrum:

Strong Consistency          Eventual Consistency
(Synchronous Replication)   (Asynchronous Replication)
        │                           │
  Reads always            Reads may return stale
  return latest           data for a window of time
  committed value         before replicas catch up
        │                           │
   Slower writes              Faster writes
   Higher latency             Lower latency

Production examples:

SystemConsistency Model
PostgreSQL synchronous replicationStrong consistency
MySQL async replication (default)Eventual consistency
Amazon DynamoDB (default reads)Eventual consistency
Amazon DynamoDB (strongly consistent reads)Strong consistency (costs 2x)
Google SpannerExternal consistency (strongest possible)
Cassandra (quorum reads/writes)Tunable consistency

Atomicity and Isolation as Consistency Enablers

The instructor makes an important point: consistency depends on atomicity and isolation.

  • Atomicity ensures partial writes never persist. A failed fund transfer doesn't leave one account debited and the other uncredited.
  • Isolation ensures in-progress reads don't see partially-written data, preventing phantom inconsistencies during concurrent operations.

If you break either A or I, you break C.

3. Deep Dive — Referential Integrity Enforcement

Database-Level vs Application-Level Enforcement

Option 1: Database enforces (FK constraints)
─────────────────────────────────────────────
+ Guaranteed — no code path can bypass it
+ Enforced even for direct DB access (admin tools, scripts)
- Schema changes are harder
- Can slow writes (FK checks on every insert/update/delete)

Option 2: Application enforces (no FK in DB)
─────────────────────────────────────────────
+ Faster writes (no FK check overhead)
+ More flexible schema evolution
- Any bug in application code breaks integrity
- Direct DB access (scripts, other services) can corrupt data
- NoSQL databases often take this path

Production reality:

  • High-scale systems (Uber, Twitter) often drop FK constraints for write performance and enforce referential integrity at the application layer with idempotent jobs that repair inconsistencies.
  • Financial systems (banks, Stripe) keep FK constraints and accept the write overhead because data integrity is non-negotiable.

NoSQL and Consistency

NoSQL databases do NOT eliminate consistency concerns. They simply move enforcement responsibility.

MongoDB example:
  users collection: { _id: 1, name: "John" }
  posts collection: { _id: 101, author_id: 1, text: "..." }

  If user 1 is deleted without deleting their posts:
  posts collection: { _id: 101, author_id: 1, text: "..." }  ← orphaned!

  MongoDB has no FK to prevent this.
  Application code must handle the cascade delete.
  If it doesn't: referential integrity is broken — same problem as SQL, just silent.

4. Practical Backend Engineering Perspective

Stripe — Idempotency and Consistency

Stripe's payment API uses idempotency keys to maintain consistency. If a network timeout causes a retry, the idempotency key ensures the payment is only processed once. This is an application-level consistency mechanism that works around the inherent inconsistency of distributed HTTP communication.

Amazon — Read-Your-Writes Consistency

Amazon DynamoDB offers "strongly consistent reads" that guarantee you see your own writes. This is the "read-your-writes" consistency model — a middle ground between eventual and strong consistency. Used in shopping cart operations where a user adds an item and immediately expects to see it in their cart.

Banking Systems — Double-Entry Bookkeeping

Banks enforce consistency via double-entry bookkeeping: every debit has a matching credit. This is a business-level consistency rule enforced by application code AND database constraints. The sum of all debits must always equal the sum of all credits — an invariant checked by reconciliation jobs.

Instagram — Denormalized Counters

Instagram stores num_likes on the picture record (a denormalized counter) for read performance. Keeping this in sync with the actual likes table requires atomic increments:

-- Atomic: both happen in one transaction
BEGIN;
INSERT INTO likes (user_id, picture_id) VALUES (42, 1);
UPDATE pictures SET num_likes = num_likes + 1 WHERE id = 1;
COMMIT;

If the application does these separately (two round-trips, no transaction), a crash between them produces the exact inconsistency the instructor describes.

PART B — DURABILITY

6. Big Picture

What Is Durability?

Durability is the guarantee that once a transaction commits, its changes are permanent — even in the face of:

  • Database process crash
  • Server power loss
  • OS crash
  • Hardware failure

The key word is committed. Durability only applies to committed transactions. An uncommitted transaction that is lost in a crash is expected behavior — that is atomicity at work (roll it back / never commit it).

Why It Matters

Without durability, a database saying "commit successful" is a lie. The user believes their data is safe, but a crash moments later could erase it. This is catastrophic for:

  • Financial transactions
  • Medical records
  • E-commerce orders
  • Any system where data loss = real-world consequence

The Performance Trade-off

Writing to Memory (RAM):  ~100 nanoseconds
Writing to SSD (NVMe):    ~100 microseconds   (1,000x slower)
Writing to HDD:           ~10 milliseconds    (100,000x slower)

Every durability guarantee requires eventually writing to disk. This is inherently slower than memory. The art of database engineering is making this as fast as possible while maintaining the guarantee.

7. Core Concepts — Durability Techniques

Technique 1: Write-Ahead Log (WAL)

The WAL is the most important durability mechanism in modern relational databases (PostgreSQL, MySQL InnoDB, Oracle, SQL Server all use it).

The core idea:

Before modifying any data structure (table, index, B+ tree), write a compact log entry describing the change. Flush that log entry to disk first. Only then (asynchronously) update the actual data structures.

Why this works:

  • WAL entries are append-only and sequential → extremely fast disk writes (sequential I/O is orders of magnitude faster than random I/O).
  • Data structures (B+ trees, heap pages) require random I/O → slow.
  • If a crash occurs, the WAL is replayed to reconstruct the correct state.
  • The actual data files are eventually updated in the background (checkpointing).

WAL entry anatomy:

WAL Segment File:
┌─────────────────────────────────────────────────────────────┐
│ LSN: 0001 | TXN: 500 | Op: UPDATE | Table: products         │
│           | Row: id=1 | Old: inventory=100 | New: inventory=90│
├─────────────────────────────────────────────────────────────┤
│ LSN: 0002 | TXN: 500 | Op: INSERT | Table: sales            │
│           | Row: (sale_id=1, pid=1, price=999.99, qty=10)   │
├─────────────────────────────────────────────────────────────┤
│ LSN: 0003 | TXN: 500 | Op: COMMIT                          │
└─────────────────────────────────────────────────────────────┘

Crash recovery flow:

Crash occurs mid-operation
         │
         ▼
Database restarts
         │
         ▼
Reads WAL from last checkpoint
         │
         ▼
Replays all committed transactions (REDO)
         │
         ▼
Rolls back all uncommitted transactions (UNDO)
         │
         ▼
Data files restored to consistent committed state ✅

Technique 2: Asynchronous Snapshots (Memory-First)

Used by: Redis (with RDB snapshots), VoltDB, some in-memory databases.

How it works:

  1. All writes go to memory (RAM) → extremely fast.
  2. Periodically (every N seconds, or when a threshold is hit), a background thread snapshots the entire in-memory state to disk.
  3. On crash, the last snapshot is loaded, and any changes since the snapshot are lost.
Timeline:
T=0s   Snapshot written to disk: {X=10}
T=1s   Write X=11 (in memory only)
T=2s   Write X=12 (in memory only)
T=3s   CRASH
T=4s   Restart: loads snapshot from T=0 → X=10
T=3s–T=3s of writes LOST

Trade-off: Speed vs. durability window. Redis calls the configurable snapshot interval the "durability window" — data written in the last N seconds before a crash may be lost.

Technique 3: Append-Only File (AOF / Redo Log)

Used by: Redis AOF mode, some log-structured databases.

How it works:

  • Every write operation is appended to a sequential log file.
  • On crash, the log is replayed from the beginning (or from a checkpoint) to reconstruct state.
  • Very similar to WAL, but typically used in key-value stores rather than relational databases.
AOF log:
SET X 10
SET X 11
SET X 12
INCR counter
...

Redis supports combining RDB snapshots with AOF for a balance of recovery speed and durability granularity.

The OS Cache Problem — fsync

This is one of the most important (and dangerous) internals in database durability.

The problem:

When a database calls write() to the OS, the OS does NOT immediately write to disk. It writes to its page cache (in RAM) and returns success. The actual flush to disk happens asynchronously, in batches, for performance.

Database: "OS, write this WAL segment to disk."
OS:       "Sure! Done!" ← LIES. It wrote to OS page cache (RAM), not disk.
Database: "Great, transaction committed!"

--- CRASH ---

OS page cache: GONE (it was in RAM)
Disk: Old data
Result: Database told client "committed" but data is LOST

The solution — fsync:

fsync() is a system call that forces the OS to flush its page cache to disk synchronously. The database waits for fsync() to return before declaring a transaction committed.

Database: "OS, write WAL segment to disk."
OS:       Writes to page cache.
Database: "OS, fsync() — flush to actual disk NOW."
OS:       Physically writes to disk, waits for confirmation from storage controller.
OS:       Returns success.
Database: "Transaction committed." ← NOW this is a true guarantee.

The trade-off:

Without fsync:
  Write → OS cache → immediate "success" → fast but not durable

With fsync:
  Write → OS cache → fsync() → wait for disk → "success" → slow but durable

fsync latency:

  • NVMe SSD: ~100 microseconds
  • SATA SSD: ~500 microseconds
  • HDD: ~10 milliseconds

This is why every commit in a durable database has a measurable latency floor — it's waiting for the disk.

Redis Durability Modes

Redis gives explicit control over the durability-performance trade-off (unique among popular databases):

ModeSettingDurabilityPerformanceRisk

8. Deep Dive — WAL Internals

WAL Write Path (PostgreSQL)

Client COMMIT
      │
      ▼
WAL record created in WAL buffer (shared memory)
      │
      ▼
WAL writer flushes WAL buffer to WAL segment file on disk
(fsync called here — this is the durability guarantee)
      │
      ▼
"COMMIT" returned to client ✅
      │
      ▼  (asynchronously, in background)
Checkpoint process writes dirty pages from buffer pool to data files
(B+ tree pages, heap pages updated on disk)

WAL and B+ Tree

The actual table data (heap pages, B+ tree index pages) are cached in the buffer pool (shared_buffers in PostgreSQL). These are called dirty pages when they have been modified in memory but not yet flushed to disk.

Buffer Pool (RAM):
  [dirty page: products id=1, inventory=90]  ← not yet on disk
  [dirty page: sales index]                   ← not yet on disk

WAL on Disk:
  [LSN 0001: UPDATE products SET inventory=90 WHERE id=1]
  [LSN 0002: INSERT sales ...]
  [LSN 0003: COMMIT]

Checkpoint (periodic):
  Flushes dirty pages to disk → data files updated
  Advances checkpoint LSN → older WAL segments can be recycled

Key insight: At the moment of commit, only the WAL is flushed to disk — not the actual data pages. The data pages are eventually flushed by the checkpoint process. This is what makes commits fast despite durability.

WAL Compression

WAL records are compact binary records of the delta (change), not full copies of pages. A WAL record for UPDATE products SET inventory = 90 WHERE id = 1 might be:

[RelFileNode: 12345, Block: 0, Offset: 128] [Old: 100] [New: 90]

This is far smaller than writing the entire 8KB page to disk.

9. Practical Backend Engineering Perspective

Stripe — Synchronous Commit

Stripe's payment system cannot afford to lose committed transactions under any circumstances. They use synchronous replication (primary waits for at least one replica to confirm WAL receipt before acknowledging commit to the application). This doubles the commit latency but provides durability even if the primary dies immediately after commit.

Netflix — WAL for Change Data Capture (CDC)

Netflix reads PostgreSQL WAL streams to propagate database changes to Kafka topics in real time. The WAL is the source of truth for what changed. This is called Change Data Capture (CDC). Tools like Debezium tap the WAL to stream row-level changes without impacting the database.

Trading Systems — fsync Tuning

High-frequency trading systems sometimes run PostgreSQL with synchronous_commit = off (giving up the "last few milliseconds" durability guarantee) to achieve sub-millisecond commit latency. The acceptable risk: in a crash, up to wal_writer_delay (default 200ms) of committed transactions could be lost. For trade log data that is simultaneously stored elsewhere, this trade-off is acceptable.

IoT / Logging Systems — Redis AOF "everysec"

IoT telemetry systems using Redis for real-time ingestion often configure appendfsync everysec — accepting the risk of losing up to 1 second of data in a crash. At 10,000 events/second, this means at most 10,000 events lost — acceptable for sensor readings that are also stored in the physical devices.

13. Demonstrating Atomicity

The Business Rule

Selling 10 phones requires TWO database operations:

  1. Deduct 10 from products.inventory
  2. Insert a row into sales

Both must succeed or both must fail. This is atomicity.

SQL Demo (from lecture)

-- BAD: Without transaction — crash between these two statements corrupts data
UPDATE products SET inventory = inventory - 10 WHERE pid = 1;
-- <<< CRASH HERE >>> inventory is 90, but no sale record exists!
INSERT INTO sales (pid, price, quantity) VALUES (1, 999.99, 10);

-- GOOD: With transaction — crash means rollback, inventory stays at 100
BEGIN;

UPDATE products SET inventory = inventory - 10 WHERE pid = 1;
-- If we crash here, rollback restores inventory to 100

INSERT INTO sales (pid, price, quantity) VALUES (1, 999.99, 10);

COMMIT; -- Only now are both changes durable and visible

Simulating a crash (from lecture):

BEGIN;
UPDATE products SET inventory = inventory - 10 WHERE pid = 1;
-- Exit psql WITHOUT committing (Ctrl+C or kill connection)
-- psql exits, TCP connection drops, PostgreSQL auto-rolls back

-- Check: inventory is still 100 ✅
SELECT inventory FROM products WHERE pid = 1;

Java JDBC — Atomicity Implementation

import java.sql.*;

public class SaleService {

    /**
     * Sells a given quantity of a product.
     * Atomically: deducts inventory AND inserts a sale record.
     * Either both happen or neither happens.
     */
    public void sellProduct(int productId, double price, int quantity) throws SQLException {
        Connection conn = null;
        try {
            conn = DatabaseConfig.getConnection();
            conn.setAutoCommit(false); // Begin transaction

            // Step 1: Deduct inventory
            String updateSql = "UPDATE products SET inventory = inventory - ? WHERE pid = ?";
            try (PreparedStatement stmt = conn.prepareStatement(updateSql)) {
                stmt.setInt(1, quantity);
                stmt.setInt(2, productId);
                int rowsAffected = stmt.executeUpdate();
                if (rowsAffected == 0) {
                    throw new SQLException("Product not found: " + productId);
                }
            }

            // If a crash/exception occurs HERE:
            // The transaction is NOT committed. On rollback (catch block),
            // inventory is restored. No sale record was inserted.
            // This is ATOMICITY in action.

            // Step 2: Record the sale
            String insertSql = "INSERT INTO sales (pid, price, quantity) VALUES (?, ?, ?)";
            try (PreparedStatement stmt = conn.prepareStatement(insertSql)) {
                stmt.setInt(1, productId);
                stmt.setDouble(2, price);
                stmt.setInt(3, quantity);
                stmt.executeUpdate();
            }

            conn.commit(); // Both changes committed atomically
            System.out.println("Sale committed: " + quantity + " units of product " + productId);

        } catch (SQLException e) {
            // ROLLBACK: Atomicity guarantees neither change persists
            if (conn != null) {
                try {
                    conn.rollback();
                    System.err.println("Transaction rolled back due to: " + e.getMessage());
                } catch (SQLException rollbackEx) {
                    System.err.println("Rollback failed: " + rollbackEx.getMessage());
                }
            }
            throw e;
        } finally {
            if (conn != null) {
                try {
                    conn.setAutoCommit(true); // Reset for connection pool
                    conn.close();
                } catch (SQLException e) {
                    System.err.println("Connection close failed: " + e.getMessage());
                }
            }
        }
    }
}

Spring Boot Equivalent — @Transactional

import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

@Service
public class SaleServiceSpring {

    private final ProductRepository productRepo;
    private final SaleRepository saleRepo;

    public SaleServiceSpring(ProductRepository productRepo, SaleRepository saleRepo) {
        this.productRepo = productRepo;
        this.saleRepo = saleRepo;
    }

    /**
     * @Transactional wraps this method in BEGIN/COMMIT.
     * Any RuntimeException causes automatic ROLLBACK.
     * This is ATOMICITY at the Spring layer.
     */
    @Transactional
    public void sellProduct(int productId, double price, int quantity) {
        // Step 1: Deduct inventory (atomic decrement — no read-then-write race)
        int updated = productRepo.decrementInventory(productId, quantity);
        if (updated == 0) {
            throw new IllegalArgumentException("Product not found: " + productId);
        }

        // Step 2: Record the sale
        Sale sale = new Sale(productId, price, quantity);
        saleRepo.save(sale);

        // Spring commits on method return.
        // Any exception thrown here → Spring auto-rollbacks both operations.
    }
}

Product Repository with Atomic Decrement

import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.data.jpa.repository.Modifying;
import org.springframework.data.jpa.repository.Query;
import org.springframework.data.repository.query.Param;

public interface ProductRepository extends JpaRepository {

    /**
     * Atomic inventory decrement — let the database do the math.
     * Never: read inventory → subtract in Java → write back (race condition!).
     * Always: UPDATE products SET inventory = inventory - :qty WHERE pid = :pid
     */
    @Modifying
    @Query("UPDATE Product p SET p.inventory = p.inventory - :qty WHERE p.pid = :pid")
    int decrementInventory(@Param("pid") int productId, @Param("qty") int quantity);
}

Thread running sellProduct():

  << JVM crash / OutOfMemoryError / server restart >>

PostgreSQL:
  TCP connection dropped
  Detects open transaction → automatic ROLLBACK
  inventory restored to 100 ✅

On restart:
  Database has inventory=100, no orphaned sale record ✅
  This is ATOMICITY.

Java — Consistency Verification

@Service
public class ConsistencyChecker {

    private final ProductRepository productRepo;
    private final SaleRepository saleRepo;

    @Transactional(readOnly = true)
    public boolean isInventoryConsistent(int productId, int initialInventory) {
        Product product = productRepo.findById(productId)
            .orElseThrow(() -> new IllegalArgumentException("Product not found"));

        int currentInventory = product.getInventory();
        int totalSold = saleRepo.sumQuantityByProductId(productId);

        boolean consistent = (currentInventory + totalSold) == initialInventory;

        if (!consistent) {
            System.err.printf(
                "INCONSISTENCY DETECTED: Product %d — inventory=%d, sold=%d, " +
                "sum=%d but expected=%d%n",
                productId, currentInventory, totalSold,
                currentInventory + totalSold, initialInventory
            );
        }
        return consistent;
    }
}

15. Demonstrating Isolation

The Problem — Without Proper Isolation

Scenario from lecture: A report-generating transaction reads counts and then detailed rows. Concurrently, another transaction commits a new sale. The report becomes internally inconsistent.

-- Session 1: Generate report (READ COMMITTED — default)
BEGIN;
SELECT pid, COUNT(*) FROM sales GROUP BY pid;
-- Sees: pid=1 → 3 sales, pid=2 → 3 sales

    -- Session 2 (concurrent): New sale committed
    INSERT INTO sales (pid, price, quantity) VALUES (1, 999.99, 10);
    COMMIT;

-- Session 1 continues:
SELECT pid, price, quantity FROM sales;
-- Now sees pid=1 → 4 rows! Count=3 but detail=4 → INCONSISTENT REPORT
COMMIT;

The Fix — REPEATABLE READ

-- Session 1: Generate report (REPEATABLE READ)
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT pid, COUNT(*) FROM sales GROUP BY pid;
-- Sees: pid=1 → 3 sales, pid=2 → 3 sales (snapshot taken here)

    -- Session 2 (concurrent): New sale committed
    INSERT INTO sales (pid, price, quantity) VALUES (1, 999.99, 10);
    COMMIT;  -- Session 1 does NOT see this!

-- Session 1 continues:
SELECT pid, price, quantity FROM sales;
-- Still sees only 3 rows for pid=1 ← snapshot preserved ✅
-- Count and detail are consistent ✅
COMMIT;

Java JDBC — Isolation Level Control

public class ReportService {

    /**
     * Generates a consistent sales report using REPEATABLE READ.
     * Without this, concurrent inserts could make count ≠ detail rows.
     */
    public SalesReport generateConsistentReport() throws SQLException {
        Connection conn = null;
        try {
            conn = DatabaseConfig.getConnection();

            // Set isolation level BEFORE beginning transaction
            conn.setTransactionIsolation(Connection.TRANSACTION_REPEATABLE_READ);
            conn.setAutoCommit(false);

            // Query 1: Aggregate counts
            Map countsByProduct = new HashMap<>();
            String countSql = "SELECT pid, COUNT(*) as cnt FROM sales GROUP BY pid";
            try (Statement stmt = conn.createStatement();
                 ResultSet rs = stmt.executeQuery(countSql)) {
                while (rs.next()) {
                    countsByProduct.put(rs.getInt("pid"), rs.getInt("cnt"));
                }
            }

            // Even if another transaction inserts a sale here and commits,
            // Session 1's snapshot is frozen at the start of the transaction.
            // No phantom rows. No non-repeatable reads.

            // Query 2: Detail rows (consistent with count above)
            List details = new ArrayList<>();
            String detailSql = "SELECT pid, price, quantity FROM sales ORDER BY pid";
            try (Statement stmt = conn.createStatement();
                 ResultSet rs = stmt.executeQuery(detailSql)) {
                while (rs.next()) {
                    details.add(new SaleDetail(
                        rs.getInt("pid"),
                        rs.getDouble("price"),
                        rs.getInt("quantity")
                    ));
                }
            }

            conn.commit(); // End of snapshot

            // Counts and details are guaranteed consistent ✅
            return new SalesReport(countsByProduct, details);

        } catch (SQLException e) {
            if (conn != null) conn.rollback();
            throw e;
        } finally {
            if (conn != null) {
                conn.setAutoCommit(true);
                conn.close();
            }
        }
    }
}

public class DurabilityDemo {

    /**
     * Demonstrates durability: insert and commit, then reconnect.
     * After reconnecting to the database (simulating a restart),
     * the committed record must still be present.
     */
    public void demonstrateDurability() throws SQLException {
        int insertedId;

        // Phase 1: Insert and commit
        try (Connection conn = DatabaseConfig.getConnection()) {
            conn.setAutoCommit(false);

            String sql = "INSERT INTO products (name, price, inventory) VALUES (?, ?, ?) RETURNING pid";
            try (PreparedStatement stmt = conn.prepareStatement(sql)) {
                stmt.setString(1, "TV");
                stmt.setDouble(2, 3000.00);
                stmt.setInt(3, 10);
                ResultSet rs = stmt.executeQuery();
                rs.next();
                insertedId = rs.getInt("pid");
            }

            conn.commit();
            System.out.println("Committed product with id: " + insertedId);
        }
        // Connection closed — simulates process restart

        // Phase 2: Reconnect and verify (simulates database restart)
        try (Connection conn = DatabaseConfig.getConnection()) {
            conn.setAutoCommit(true);

            String sql = "SELECT * FROM products WHERE pid = ?";
            try (PreparedStatement stmt = conn.prepareStatement(sql)) {
                stmt.setInt(1, insertedId);
                ResultSet rs = stmt.executeQuery();

                if (rs.next()) {
                    System.out.printf("Durability confirmed: pid=%d, name=%s, inventory=%d%n",
                        rs.getInt("pid"), rs.getString("name"), rs.getInt("inventory"));
                } else {
                    System.err.println("DURABILITY FAILURE: Committed record not found after reconnect!");
                }
            }
        }

/**
 * Demonstrates all four ACID properties in one coherent scenario.
 * Scenario: Selling 10 phones atomically, with consistency checks,
 * isolation verification, and durability confirmation.
 */
@Service
public class AcidDemonstrationService {

    private final ProductRepository productRepo;
    private final SaleRepository saleRepo;

    public AcidDemonstrationService(ProductRepository productRepo, SaleRepository saleRepo) {
        this.productRepo = productRepo;
        this.saleRepo = saleRepo;
    }

    // ─────────────────────────────────────────────
    // ATOMICITY: Both inventory deduction and sale
    // record creation happen atomically.
    // ─────────────────────────────────────────────
    @Transactional(rollbackFor = Exception.class)
    public void sellProductAtomically(int productId, double salePrice, int quantity) {
        // Step 1: Atomic decrement (no read-then-write race)
        int updated = productRepo.decrementInventory(productId, quantity);
        if (updated == 0) {
            throw new IllegalStateException("Product not found — transaction will rollback");
        }

        // Simulate failure between the two operations
        // Uncomment to see atomicity: inventory NOT decremented because of rollback
        // if (true) throw new RuntimeException("Simulated crash!");

        // Step 2: Record sale
        saleRepo.save(new Sale(productId, salePrice, quantity));

        // CONSISTENCY: After commit, inventory + total_sold = initial_inventory
        // DURABILITY: After commit, data persists through restarts
    }

    // ─────────────────────────────────────────────
    // ISOLATION: Consistent report with REPEATABLE READ
    // ─────────────────────────────────────────────
    @Transactional(isolation = Isolation.REPEATABLE_READ, readOnly = true)
    public SalesReport generateIsolatedReport(int productId) {
        // Snapshot created here — concurrent inserts invisible from this point
        long count = saleRepo.countByProductId(productId);
        List details = saleRepo.findByProductId(productId);

        // count == details.size() guaranteed by REPEATABLE READ isolation ✅
        return new SalesReport(count, details);
    }

    // ─────────────────────────────────────────────
    // CONSISTENCY CHECK: Verify data invariants
    // ─────────────────────────────────────────────
    @Transactional(readOnly = true)
    public void verifyConsistency(int productId, int initialInventory) {
        int current = productRepo.findById(productId)
            .orElseThrow().getInventory();
        int sold = saleRepo.sumQuantityByProductId(productId);

        int sum = current + sold;
        if (sum != initialInventory) {
            throw new IllegalStateException(String.format(
                "CONSISTENCY VIOLATION: Product %d — current=%d + sold=%d = %d ≠ %d (initial)",
                productId, current, sold, sum, initialInventory
            ));
        }
        System.out.printf("Consistency OK: %d (current) + %d (sold) = %d (initial)%n",
            current, sold, initialInventory);
    }
}

18. Important Interview Questions

Basic Questions

Q: What are the two types of consistency discussed in the context of ACID?
A: (1) Consistency in data — the persisted data satisfies the rules/constraints defined by the schema and application (referential integrity, foreign keys, check constraints). (2) Consistency in reads — after a commit, all subsequent reads return the committed value, even across replicas.

Q: Is ACID consistency guaranteed purely by the database?
A: No. The "C" in ACID is partially the application's responsibility. The database enforces constraints you define (FK, CHECK, NOT NULL), but the rules themselves — what "valid state" means for your domain — must be specified by the developer/DBA.

Q: What is durability?
A: Durability guarantees that once a transaction commits, its changes persist permanently on non-volatile storage (SSD/HDD), surviving crashes, power loss, and restarts.

Q: What is the WAL?
A: Write-Ahead Log — a sequential append-only log of all database changes. Before modifying data structures, the database writes a compact record of the change to the WAL and flushes it to disk. On crash, the WAL is replayed to restore the committed state.

Q: What is eventual consistency?
A: A property of distributed systems where, given no new updates, all replicas will eventually converge to the same value. It applies to replica lag (Type 2 consistency), not to corrupt data (Type 1) — corrupt data does not "eventually fix itself" without explicit repair.

Intermediate Questions

Q: How does atomicity contribute to consistency?
A: Atomicity ensures partial writes never persist. A fund transfer that crashes midway is rolled back entirely — no half-debit exists on disk. This prevents the data from entering an invalid intermediate state, directly preserving consistency.

Q: What is fsync and why does it matter for durability?
A: fsync() is a system call that forces the OS to flush its write cache to disk. Without it, the OS buffers writes in RAM and lies to the database about their completion. A crash before the OS flushes means data loss despite the database saying "committed." Databases call fsync() before returning "commit" to guarantee true durability.

Q: What is the trade-off between synchronous and asynchronous replication?
A: Synchronous replication: the primary waits for replicas to confirm WAL receipt before acknowledging commit → strong consistency, slower commits, no replica lag. Asynchronous replication: primary acknowledges immediately, replicates in the background → eventual consistency, faster commits, potential replica lag.

Q: Why might a NoSQL database still have consistency problems?
A: NoSQL databases lack foreign key constraints, so referential integrity must be enforced by the application. If the application code fails to do so (e.g., deleting a document without deleting references to it), the same data inconsistency problems occur — they're just harder to detect because the database doesn't enforce the rules.

Advanced Questions

Q: Explain WAL and checkpointing in PostgreSQL. Why are dirty pages not immediately flushed to disk on commit?
A: At commit time, PostgreSQL flushes the WAL record to disk (via fsync) but not the actual data pages. Data pages are cached in the buffer pool (shared_buffers) as "dirty pages." The checkpoint process periodically flushes dirty pages to disk. This design is optimal because: (1) WAL writes are sequential (fast), while data page writes are random (slow). (2) Multiple changes to the same page can be batched into one disk write. (3) Crash recovery uses WAL to reconstruct any dirty pages not yet checkpointed. The checkpoint LSN tells the recovery process where to start replaying WAL.

Q: What is the "group commit" optimization?
A: When many transactions commit simultaneously, the database groups their WAL records and calls fsync() once for the batch rather than once per transaction. This amortizes the expensive fsync cost across many transactions, dramatically improving throughput under concurrent load. PostgreSQL's wal_writer_delay and MySQL's sync_binlog control this behavior.

Q: How does Redis handle durability differently from PostgreSQL?
A: Redis is memory-first. It offers a spectrum: (1) No persistence — purely in-memory, all data lost on crash. (2) RDB snapshots — periodic full snapshots to disk, data within the window may be lost. (3) AOF everysec — fsync every second, up to 1 second of data at risk. (4) AOF always — fsync every write, strongest durability but slowest. PostgreSQL's WAL + fsync on commit provides "durable on commit" semantics with no configurable loss window for committed transactions.

19. Common Mistakes

  • Ignoring consistency in NoSQL: Assuming MongoDB/Cassandra eliminates consistency concerns. It doesn't — it eliminates the database's enforcement and moves it to your code.
  • Conflating ACID consistency with CAP consistency: CAP consistency = linearizability (every read sees the last write across all nodes). ACID consistency = data satisfies invariants. These are completely different concepts.
  • Not using transactions for multi-step operations: Performing two related writes without a transaction. A crash between them leaves the database in an inconsistent state.
  • Assuming setAutoCommit(false) alone is enough: You must also explicitly commit() or rollback(). Forgetting commit() leaves the transaction open, blocking other operations (lock contention, idle transactions).
  • Not configuring isolation level explicitly: Using the ORM default (usually READ COMMITTED) for report generation, then being surprised when reports are internally inconsistent.
  • Disabling fsync in production: fsync=off in PostgreSQL gives a huge performance boost during load testing but must never be in production — a crash can corrupt the entire database.
  • Using synchronous_commit=off without understanding the risk: Commits appear successful but the WAL isn't flushed. A crash within wal_writer_delay (default 200ms) can lose recently committed data.
  • Not calling conn.rollback() in catch blocks: In JDBC, if you catch an exception and don't rollback, the connection returns to the pool with an open transaction. The next user of that connection inherits the uncommitted state — catastrophic.

20. Performance & Scalability Notes

Consistency Trade-offs at Scale

ApproachWrite LatencyConsistencyScalability
Synchronous replicationHigh (waits for replica)StrongLow write throughput
Async replicationLowEventualHigh write throughput
Single-node (no replication)MediumStrongLimited by single node
Distributed consensus (Spanner/CockroachDB)High (Paxos rounds)ExternalScal horizontally

WAL Performance Impact

  • Sequential WAL writes: ~500MB/s on NVMe SSD — extremely fast.
  • fsync latency: The dominant cost per commit. Batching (group commit) reduces per-transaction cost.
  • WAL size: Large transactions create large WAL segments. Very large transactions (bulk loads) can fill WAL too fast, causing max_wal_size pressure and forced checkpoints.
  • WAL archiving: Streaming WAL to S3/GCS for point-in-time recovery adds background I/O. Usually negligible but can cause write amplification on very busy systems.

Buffer Pool and Durability

  • Larger shared_buffers (PostgreSQL) → more dirty pages in memory → longer time between disk flushes → more WAL to replay on crash recovery.
  • Checkpointing checkpoint_completion_target=0.9 spreads I/O across 90% of the checkpoint interval → smoother I/O, less write burst.
