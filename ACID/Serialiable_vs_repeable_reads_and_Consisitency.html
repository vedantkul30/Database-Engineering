Serializable vs Repeatable Read & Eventual Consistency

PART A — SERIALIZABLE vs REPEATABLE READ ISOLATION

1. Big Picture

Why This Topic Exists

Most engineers who learn about isolation levels memorize the table — "SERIALIZABLE prevents all anomalies, REPEATABLE READ prevents non-repeatable reads" — but they never truly understand why SERIALIZABLE needs to exist over and above REPEATABLE READ when there are no dirty reads, no phantom reads, and no non-repeatable reads.

The answer is: there exists a class of concurrency anomalies that cannot be detected by looking at individual row-level changes in isolation. These anomalies arise from the interaction pattern between two transactions — specifically read-write dependencies — even when no single row is touched by both transactions.

This is the gap SERIALIZABLE fills.

The Problem It Solves

Two transactions can both execute perfectly, touching entirely different rows, and yet produce a result that could never have occurred if those transactions ran one after another in any serial order.

This is called a write skew (the general category) or a non-serializable execution. It is invisible to REPEATABLE READ because REPEATABLE READ only guards rows that were read — it cannot guard against the combination of reads and writes forming a logical cycle.

Where It Appears in Real Systems

  • Double-booking systems: Two agents both check seat availability (read: 1 seat left), both confirm a booking (write: seat assigned) → two bookings for one seat.
  • Doctor on-call: Both doctors check "is at least one on call?" → both go off call → nobody on call.
  • Bank transfer race: Both transfers read sufficient balance, both debit → overdraft.
  • Order book matching: Two orders both read "unmatched sell order exists" → both consume the same sell order → over-fill.

Why Backend Engineers Must Care

  • REPEATABLE READ is not enough for all business invariants. If your business rule involves checking a condition and then acting on it, you may need SERIALIZABLE.
  • Serialization failures are expected and must be handled. Application code must implement retry logic. This is not optional.
  • Performance cost is real. SERIALIZABLE is significantly more expensive. Use only where the business domain demands it.

2. Core Concepts

The Foundational Example — The Swap Problem

Consider a table with one column containing rows: A, A, B, B.

  • Transaction T1: UPDATE test SET field = 'B' WHERE field = 'A' → changes A's to B's
  • Transaction T2: UPDATE test SET field = 'A' WHERE field = 'B' → changes B's to A's

Both transactions run concurrently.

What Happens Under REPEATABLE READ

Initial state: [A, A, B, B]

T1 starts, reads: rows 1 and 2 (value = A)
T2 starts, reads: rows 3 and 4 (value = B)

T1 writes: row 1 → B, row 2 → B
T2 writes: row 3 → A, row 4 → A

T1 commits.
T2 commits.

Final state: [B, B, A, A]

Key observation: T1 and T2 touched completely different rows. From the database's perspective under REPEATABLE READ:

  • T1 read rows 1 and 2 — those values didn't change while T1 was reading them ✅
  • T2 read rows 3 and 4 — those values didn't change while T2 was reading them ✅
  • No dirty reads, no non-repeatable reads, no phantom reads.

REPEATABLE READ sees no problem here and commits both transactions.

Result: [B, B, A, A]

What Serial Execution Would Have Produced

Serial order T1 → T2:

Start: [A, A, B, B]
T1:    [B, B, B, B]   ← all A's become B's
T2:    [A, A, A, A]   ← all B's become A's
End:   [A, A, A, A]

Serial order T2 → T1:

Start: [A, A, B, B]
T2:    [A, A, A, A]   ← all B's become A's
T1:    [B, B, B, B]   ← all A's become B's
End:   [B, B, B, B]

The only valid serial outcomes are: all A's OR all B's.

The REPEATABLE READ outcome [B, B, A, A] is neither. It could not have been produced by any serial execution of T1 and T2. This is a non-serializable execution — a correctness violation that REPEATABLE READ does not detect.

What Happens Under SERIALIZABLE

T1 starts (SERIALIZABLE).
T2 starts (SERIALIZABLE).

T1 reads rows 1 and 2 (field = A).
T2 reads rows 3 and 4 (field = B).

T1 writes rows 1 and 2 → B.
T2 writes rows 3 and 4 → A.

T1 commits: ✅ SUCCESS.

T2 tries to commit:
PostgreSQL detects a read-write dependency cycle between T1 and T2.
ERROR: could not serialize access due to read/write dependencies among transactions
DETAIL: Reason code: Canceled on identification as a pivot, during commit attempt.
HINT:   The transaction might succeed if retried.

T2 is ROLLED BACK.

Application retries T2. No concurrent transactions remain. T2 commits successfully.

Final state: All A's or all B's — a valid serial outcome. ✅

The Formal Concept: Read-Write Dependency Cycles

PostgreSQL's SERIALIZABLE implementation (called Serializable Snapshot Isolation — SSI) works by tracking read-write anti-dependencies between transactions.

An anti-dependency exists when:

  • T1 reads a value that T2 writes (T1 read something T2 will change)
  • T2 reads a value that T1 writes (T2 read something T1 will change)

When such a cycle is detected, the execution cannot correspond to any serial order. PostgreSQL aborts one transaction (the "pivot") to break the cycle.

Anti-dependency cycle in the swap example:

T1 reads rows {1,2} (A's)  →  T2 writes rows {3,4} (A's) [T2 changes what T1 might read next]
T2 reads rows {3,4} (B's)  →  T1 writes rows {1,2} (B's) [T1 changes what T2 might read next]

Cycle: T1 →(rw)→ T2 →(rw)→ T1

PostgreSQL detects this cycle and aborts T2.

Pessimistic Alternative: SELECT FOR UPDATE

The instructor mentions that you can achieve the same serializable guarantee pessimistically using SELECT FOR UPDATE under READ COMMITTED:

-- T1: Locks the rows it's about to modify
BEGIN;  -- READ COMMITTED
SELECT * FROM test WHERE field = 'A' FOR UPDATE;  -- Acquires exclusive locks on rows 1,2
UPDATE test SET field = 'B' WHERE field = 'A';
COMMIT;  -- Locks released

-- T2: Cannot read the locked rows until T1 commits
BEGIN;
SELECT * FROM test WHERE field = 'B' FOR UPDATE;
-- BLOCKS until T1 commits. Then reads the committed state.
-- T1 changed rows 1,2 to B. T2 now sees rows 1,2,3,4 as B's → changes all to A.
UPDATE test SET field = 'A' WHERE field = 'B';
COMMIT;
-- Result: [A, A, A, A] ✅

Trade-off:

ApproachMechanismOn ConflictThroughput
SERIALIZABLE (SSI)Optimistic, conflict detection at commitABORT + RETRYHigher (no waiting)
SELECT FOR UPDATEPessimistic, explicit row locksWAITLower (blocking)

3. Deep Dive — PostgreSQL SSI Internals

How SSI Tracks Dependencies

PostgreSQL maintains a SIREAD lock (predicate lock) for every row, page, or relation that a SERIALIZABLE transaction reads. These are not blocking locks — they are tracking locks. They record "T1 read this data."

When another transaction writes data that was predicate-locked by T1, PostgreSQL records an anti-dependency: T2 wrote something T1 read.

PostgreSQL then checks: does this create a cycle (T1→T2→T1)? If yes → abort the pivot transaction.

Predicate lock tracking (conceptual):
  T1 reads rows where field='A'  → SIREAD lock on {row1, row2}
  T2 reads rows where field='B'  → SIREAD lock on {row3, row4}
  T1 writes rows {row1, row2}    → Anti-dep recorded: T2→T1 (T2 read what T1 wrote)
  T2 writes rows {row3, row4}    → Anti-dep recorded: T1→T2 (T1 read what T2 wrote)
  Cycle detected: T1 ↔ T2       → Abort T2 (later committer is the pivot)

Performance Implications of SSI

  • Memory overhead: SIREAD locks are held in memory. Heavy SERIALIZABLE workloads with many transactions and many rows read can pressure the lock table.
  • False positives: SSI sometimes aborts transactions that would not actually have conflicted if run serially. This is conservative but safe.
  • Granularity promotion: If too many row-level SIREAD locks accumulate for one transaction, PostgreSQL promotes to a relation-level (table-level) SIREAD lock. This increases false aborts.
  • max_pred_locks_per_transaction: Controls the threshold before promotion. Tuning this is important for SERIALIZABLE-heavy workloads.

The SQLSTATE 40001 Error

When PostgreSQL aborts a SERIALIZABLE transaction due to a conflict, it returns:

ERROR:  could not serialize access due to read/write dependencies among transactions
SQLSTATE: 40001

This is the standard SQLSTATE for serialization failures. Applications must catch this and retry the transaction. It is not an error in the traditional sense — it is the expected mechanism of SSI.

4. Practical Java Implementation

JDBC — SERIALIZABLE with Retry Loop

import java.sql.*;

public class SerializableTransactionService {

    private static final int MAX_RETRIES = 5;
    private static final String SERIALIZATION_FAILURE = "40001";

    /**
     * Executes a transaction at SERIALIZABLE isolation level with automatic retry.
     * SERIALIZABLE transactions may be aborted by the database when a
     * read/write dependency cycle is detected (SQLSTATE 40001).
     * The application MUST retry these — it is expected behavior.
     */
    public void executeSerializable(TransactionWork work) throws SQLException {
        int attempt = 0;
        while (true) {
            attempt++;
            Connection conn = null;
            try {
                conn = DatabaseConfig.getConnection();
                conn.setTransactionIsolation(Connection.TRANSACTION_SERIALIZABLE);
                conn.setAutoCommit(false);

                work.execute(conn);

                conn.commit();
                System.out.println("Committed on attempt " + attempt);
                return;

            } catch (SQLException e) {
                if (conn != null) {
                    try { conn.rollback(); } catch (SQLException ignored) {}
                }

                // SQLSTATE 40001 = serialization failure → retry
                if (SERIALIZATION_FAILURE.equals(e.getSQLState()) && attempt < MAX_RETRIES) {
                    System.out.printf("Serialization conflict on attempt %d, retrying...%n", attempt);
                    try {
                        // Exponential backoff: 50ms, 100ms, 200ms...
                        Thread.sleep(50L * (1L << (attempt - 1)));
                    } catch (InterruptedException ie) {
                        Thread.currentThread().interrupt();
                        throw new SQLException("Interrupted during retry backoff", ie);
                    }
                } else {
                    throw e; // Max retries exceeded or non-retriable error
                }

            } finally {
                if (conn != null) {
                    try {
                        conn.setAutoCommit(true);
                        conn.close();
                    } catch (SQLException ignored) {}
                }
            }
        }
    }

    @FunctionalInterface
    public interface TransactionWork {
        void execute(Connection conn) throws SQLException;
    }
}

The Swap Problem in Java

public class SwapDemo {

    private final SerializableTransactionService txService;

    // PROBLEMATIC: REPEATABLE READ — produces non-serializable result
    public void swapRepeatableRead(String fromValue, String toValue) throws SQLException {
        Connection conn = DatabaseConfig.getConnection();
        conn.setTransactionIsolation(Connection.TRANSACTION_REPEATABLE_READ);
        conn.setAutoCommit(false);
        try {
            String sql = "UPDATE test SET field = ? WHERE field = ?";
            try (PreparedStatement stmt = conn.prepareStatement(sql)) {
                stmt.setString(1, toValue);
                stmt.setString(2, fromValue);
                stmt.executeUpdate();
            }
            conn.commit();
            // If two threads call this concurrently: T1(A→B) and T2(B→A)
            // Result may be [B,B,A,A] — neither all-A nor all-B ← NON-SERIALIZABLE
        } catch (SQLException e) {
            conn.rollback();
            throw e;
        }
    }

    // CORRECT: SERIALIZABLE — guarantees all-A or all-B
    public void swapSerializable(String fromValue, String toValue) throws SQLException {
        txService.executeSerializable(conn -> {
            String sql = "UPDATE test SET field = ? WHERE field = ?";
            try (PreparedStatement stmt = conn.prepareStatement(sql)) {
                stmt.setString(1, toValue);
                stmt.setString(2, fromValue);
                stmt.executeUpdate();
            }
            // If conflict detected at commit: SQLSTATE 40001 thrown
            // executeSerializable() catches it and retries automatically
        });
    }
}

Spring Boot — SERIALIZABLE with @Transactional and Retry

import org.springframework.retry.annotation.Backoff;
import org.springframework.retry.annotation.Retryable;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Isolation;
import org.springframework.transaction.annotation.Transactional;
import org.springframework.dao.CannotSerializeTransactionException;

@Service
public class SerializableService {

    private final TestRepository testRepo;

    /**
     * @Retryable catches CannotSerializeTransactionException (Spring wraps SQLSTATE 40001)
     * and retries up to 5 times with exponential backoff.
     * This is the production pattern for SERIALIZABLE transactions in Spring.
     */
    @Retryable(
        retryFor = CannotSerializeTransactionException.class,
        maxAttempts = 5,
        backoff = @Backoff(delay = 50, multiplier = 2, maxDelay = 1000)
    )
    @Transactional(isolation = Isolation.SERIALIZABLE)
    public void swapValues(String fromVal, String toVal) {
        testRepo.updateFieldWhereEquals(toVal, fromVal);
        // Spring commits on method return.
        // If SQLSTATE 40001: Spring throws CannotSerializeTransactionException
        // @Retryable catches it, waits backoff, calls this method again.
        // Each retry gets a FRESH transaction with a new snapshot.
    }
}

Critical note: With @Retryable + @Transactional, the retry wrapper must be in a separate bean or called through the Spring proxy. Self-invocation (calling the method from within the same class) bypasses Spring's AOP proxy and @Transactional won't apply on retry.

// WRONG: self-invocation — @Transactional ignored on retry
@Service
public class BadService {
    @Retryable(retryFor = CannotSerializeTransactionException.class, maxAttempts = 3)
    public void outer() {
        inner();  // direct call, bypasses proxy
    }

    @Transactional(isolation = Isolation.SERIALIZABLE)
    public void inner() { /* ... */ }
}

// CORRECT: separate beans
@Service
public class RetryWrapper {
    private final InnerService innerService;

    @Retryable(retryFor = CannotSerializeTransactionException.class, maxAttempts = 3,
               backoff = @Backoff(delay = 50, multiplier = 2))
    public void swapWithRetry(String from, String to) {
        innerService.swap(from, to);
    }
}

@Service
public class InnerService {
    @Transactional(isolation = Isolation.SERIALIZABLE)
    public void swap(String from, String to) {
        repo.updateFieldWhereEquals(to, from);
    }
}

5. Important Interview Questions — Serializable vs Repeatable Read

Basic Questions

Q: What is the difference between REPEATABLE READ and SERIALIZABLE?
A: REPEATABLE READ guarantees that rows a transaction has read will not change during the transaction. SERIALIZABLE additionally guarantees that the concurrent execution of transactions produces a result equivalent to some serial (one-after-another) execution. REPEATABLE READ can produce results that are impossible under any serial order — SERIALIZABLE cannot.

Q: When would you choose SERIALIZABLE over REPEATABLE READ?
A: When the business invariant depends on a condition-then-action pattern: "check if X is true, then do Y." If two transactions both check the same condition concurrently and both act on it, REPEATABLE READ won't prevent conflicting actions. Classic cases: booking systems (check seat availability → book), financial (check balance → debit), doctor on-call (check if anyone on call → go off call).

Q: What error does PostgreSQL return on a serialization conflict?
A: ERROR: could not serialize access due to read/write dependencies among transactions with SQLSTATE 40001. The application must catch this and retry the entire transaction.

Intermediate Questions

Q: In the swap example (A→B concurrent with B→A), why does REPEATABLE READ allow a non-serializable result?
A: T1 reads A-rows and T2 reads B-rows — completely different rows. REPEATABLE READ only protects rows that were read by a transaction. Since neither transaction re-reads the other's rows, no non-repeatable read occurs. REPEATABLE READ has no mechanism to detect that the combination of the two transactions' reads and writes forms a logical cycle.

Q: What is an anti-dependency (read-write dependency) in the context of SSI?
A: An anti-dependency from T1 to T2 exists when T1 reads a value that T2 subsequently writes. If T2 to T1 anti-dependencies also exist (T2 reads something T1 writes), a cycle forms. A cycle means the two transactions cannot be reordered into any serial execution — SSI detects this and aborts one.

Q: What is the pessimistic alternative to SERIALIZABLE isolation?
A: SELECT FOR UPDATE under READ COMMITTED. The transaction explicitly locks the rows it intends to modify before reading them. Other transactions attempting to lock the same rows block and wait. This is serializable in effect but uses waiting (blocking) rather than abort-and-retry.

Advanced Questions

Q: How does PostgreSQL implement SERIALIZABLE isolation (SSI) at a high level?
A: PostgreSQL uses Serializable Snapshot Isolation (SSI). Each SERIALIZABLE transaction gets a snapshot (like REPEATABLE READ). Additionally, PostgreSQL tracks SIREAD (predicate) locks on all data read by SERIALIZABLE transactions. When a write occurs on data covered by a SIREAD lock, PostgreSQL records a read-write anti-dependency. If a cycle of anti-dependencies is detected between transactions, the later-committing transaction is aborted with SQLSTATE 40001. This is an optimistic approach — no blocking, just conflict detection at commit time.

Q: What is a "false positive abort" in SSI?
A: SSI is conservative. To limit memory and CPU overhead, it sometimes promotes fine-grained (row-level) SIREAD locks to coarser (page-level or relation-level) locks. This can cause PostgreSQL to detect a "conflict" where none actually exists (two transactions that would not have conflicted under exact tracking). The abort is safe (it never aborts a correct serializable execution) but increases the retry rate unnecessarily. Tuning max_pred_locks_per_transaction can reduce false positives.

Q: Why must SERIALIZABLE retry logic use exponential backoff?
A: Under high contention, many transactions may conflict simultaneously. Without backoff, all retried transactions re-execute at the same time → another wave of conflicts → retry storm. Exponential backoff (50ms, 100ms, 200ms...) staggers retries, reducing collision probability and allowing the system to recover.

6. Common Mistakes

  • Not implementing retry logic. Treating SQLSTATE 40001 as a fatal error instead of retrying. This silently drops writes in high-contention scenarios.
  • Using SERIALIZABLE for read-only transactions. Read-only transactions do not need SERIALIZABLE. They cannot cause write skew. Use REPEATABLE READ or even READ COMMITTED with readOnly=true for better performance.
  • Assuming SERIALIZABLE is always slow. Under low contention, SSI has minimal overhead over REPEATABLE READ. The cost spikes under high contention due to aborts and retries.
  • Not resetting the connection's isolation level after use. If using JDBC with a connection pool, the isolation level may persist across transactions. Always reset to the default after setting a non-default level.
  • Self-invocation with @Transactional and @Retryable in Spring. As shown above, calling a @Transactional method from within the same bean bypasses the proxy. The transaction won't apply. Use separate beans.
  • Holding SERIALIZABLE transactions open for a long time. The longer a SERIALIZABLE transaction runs, the more SIREAD locks it accumulates, and the higher the probability of a spurious abort. Keep SERIALIZABLE transactions as short as possible.

7. Visual Mental Models

Why REPEATABLE READ Fails on the Swap

Initial: Row1=A, Row2=A, Row3=B, Row4=B

T1 (REPEATABLE READ):          T2 (REPEATABLE READ):
  Snapshot: {R1=A,R2=A,R3=B,R4=B}  Snapshot: {R1=A,R2=A,R3=B,R4=B}
  Reads R1,R2 (=A) ✅             Reads R3,R4 (=B) ✅
  Writes R1=B, R2=B               Writes R3=A, R4=A
  COMMIT ✅                        COMMIT ✅

Final: [B,B,A,A]  ← NOT equivalent to any serial execution ❌

Serial T1→T2 would give: [A,A,A,A]
Serial T2→T1 would give: [B,B,B,B]
[B,B,A,A] is WRONG.

How SSI Detects the Cycle

SIREAD locks (who read what):
  T1: SIREAD on {R1, R2}  (rows where field=A)
  T2: SIREAD on {R3, R4}  (rows where field=B)

Anti-dependencies (who wrote what the other read):
  T1 writes R1,R2 (B) → T2's SIREAD covered R3,R4 only — but T2 also
    will read the result of T1's write if T2 ran after T1
  T2 writes R3,R4 (A) → T1's SIREAD covered R1,R2 only — but T1 also
    will read the result of T2's write if T1 ran after T2

Cycle: T1 --(rw dep)--> T2 --(rw dep)--> T1

SSI: ABORT T2 (pivot). T2 must retry.

Pessimistic vs Optimistic for Serializability

PESSIMISTIC (SELECT FOR UPDATE):

T1: SELECT FOR UPDATE → [LOCK R1,R2] ──────────────────────── UPDATE → COMMIT → UNLOCK
T2:                       SELECT FOR UPDATE → [WAIT for T1's lock] → READ committed → UPDATE → COMMIT

Result: T2 sees T1's committed changes → truly serial ✅
Cost: T2 WAITS (blocking, latency added)

OPTIMISTIC (SERIALIZABLE SSI):

T1: READ → ────────────────── WRITE → COMMIT ✅
T2: READ → WRITE → COMMIT → ABORT (SQLSTATE 40001) → RETRY → COMMIT ✅

Result: One transaction retries ✅
Cost: T2 does WASTED WORK then retries (CPU cost)

PART B — EVENTUAL CONSISTENCY (DEEP DIVE)

8. Big Picture

Why Eventual Consistency Exists

Eventual consistency is not a NoSQL invention. It is a consequence of distribution. The moment you have data in more than one place — whether that is a primary-replica pair in PostgreSQL, a Redis cache in front of MySQL, or a globally distributed Cassandra cluster — you have introduced the possibility that different nodes hold different versions of the same data at the same point in time.

The CAP theorem tells us that in the presence of a network partition, a distributed system must choose between Consistency and Availability. Most systems choose Availability (serve requests even if the data might be slightly stale) and accept Eventual Consistency.

The Marketing Problem With "Eventual Consistency"

The instructor makes a critical observation: "eventual consistency" is overloaded as a term and is often misused as a marketing phrase for NoSQL databases. Understanding what it actually means is essential for system design interviews and production architecture decisions.

9. Core Concepts — Two Types of Consistency (Revisited)

Type 1: Consistency in Data (Correctness of Persisted State)

What it means: The data stored in the database satisfies all defined invariants and referential integrity rules.

Example:

pictures table:    id=1, num_likes=5   ← claims 5 likes
likes table:       only 2 rows for picture 1

INCONSISTENCY: num_likes=5 ≠ count(likes where picture_id=1)=2

Causes of Type 1 inconsistency:

  • Violated atomicity: multi-table transaction partially committed
  • Violated isolation: concurrent writes produced an invalid state
  • Missing foreign key constraints: orphaned records after delete
  • Application bug: update one table but not the related table

Can eventual consistency fix Type 1?

No. Corrupt data does not heal itself. If num_likes=5 but only 2 like records exist, no amount of time or replication will fix this. The data is permanently wrong unless a repair job explicitly corrects it.

"There is no eventual consistency coming out of that. You just corrupted your database."
— Instructor

Type 2: Consistency in Reads (Visibility of Committed Changes)

What it means: After a transaction commits a change, any subsequent read — from any client, on any node — returns the new value, not the pre-commit value.

This is where eventual consistency applies.

Classic scenario:

Architecture:
  [Primary DB] ──async replication──▶ [Replica 1]
                                  └──▶ [Replica 2]
  [Load Balancer] routes reads to replicas for scalability.

Sequence:
  T=0ms: Client A writes X=100 to Primary. COMMIT. ✅
  T=1ms: Replication in progress (async lag: ~50ms typical)
  T=2ms: Client B reads X from Replica 1 → gets X=99 (stale!) ← INCONSISTENT

  T=50ms: Replication completes. Replica 1 now has X=100.
  T=51ms: Client B reads X from Replica 1 → gets X=100 ✅

  → System was EVENTUALLY consistent. It converged.

Can eventual consistency fix Type 2?

Yes — if the underlying data is correct (Type 1 is fine), the replica will eventually receive and apply the replication stream and become current.

Caching Introduces Inconsistency

"The moment you introduce a cache, you are inconsistent."
— Instructor

This is an absolute rule. When you put Redis in front of your database:

Without cache:
  Client → Database: Read X → 100 ✅ (always current)

With cache:
  Client → Redis: Read X → 99 (stale!) ← INCONSISTENT
  Client → Redis (miss) → Database → 100 (current)

Cache becomes stale immediately after any write to the database.
The system is inconsistent until the cache entry is invalidated or expires.

This means: Every system that uses caching is eventually consistent by definition. The only question is whether that window of inconsistency is acceptable.

synchronous_standby_names = 'replica1'
# Primary waits for replica1 to confirm WAL receipt before commit.
# One slow replica = slow commits for everyone.

# Quorum-based (N of M replicas must confirm):
synchronous_standby_names = 'ANY 2 (replica1, replica2, replica3)'

10. Practical Backend Engineering Perspective

Banking — Zero Tolerance for Both Types of Inconsistency

Banks cannot tolerate Type 1 (corrupt balance data) AND must carefully handle Type 2 (replica lag). Standard approach:

  • All writes go to the primary.
  • Balance checks (debit authorization) also go to the primary (bypassing replicas).
  • Reporting/statement queries may go to replicas (acceptable lag for statements).
  • Synchronous replication to at least one replica (often in a different data center).

Amazon — Read-Your-Writes Consistency

DynamoDB offers "strongly consistent reads" that always read from the leader node. Standard reads are eventually consistent (cheaper, 2x read capacity). The application chooses per-request:

// Eventually consistent (cheaper, may return stale data)
GetItemRequest eventualRead = GetItemRequest.builder()
    .tableName("accounts")
    .key(Map.of("id", AttributeValue.builder().s("acc-123").build()))
    .consistentRead(false)  // default
    .build();

// Strongly consistent (reads from leader, always current)
GetItemRequest strongRead = GetItemRequest.builder()
    .tableName("accounts")
    .key(Map.of("id", AttributeValue.builder().s("acc-123").build()))
    .consistentRead(true)  // 2x read cost
    .build();

Instagram — Tolerating Like Count Lag

Instagram's like counter intentionally tolerates eventual consistency. The displayed count may be slightly stale (a few seconds behind). This is acceptable because:

  • The exact like count at any instant is not a business-critical invariant.
  • Strong consistency on every like would require synchronous writes to all cache layers — too expensive at scale.
  • Eventual consistency is explicitly accepted as a product trade-off.

Uber — Surge Pricing Reads

Uber's surge pricing reads driver availability from caches and replicas. A slight lag in the data (driver locations up to a few seconds stale) is acceptable. The system is eventually consistent — all nodes converge — but a tiny window of inconsistency exists. This is a deliberate trade-off for the latency performance required at global scale.

Cache Invalidation Strategy in Java/Spring

@Service
public class ProductService {

    private final ProductRepository repo;
    private final RedisTemplate<String, Product> cache;

    /**
     * Write-Through Cache Pattern:
     * Write to DB AND cache atomically (within same method).
     * Cache is always in sync immediately after write.
     * Still eventually consistent during the write itself (brief window).
     */
    @Transactional
    public void updateProduct(int productId, double newPrice) {
        // 1. Update database (primary source of truth)
        repo.updatePrice(productId, newPrice);
        // DB committed here

        // 2. Update cache immediately (minimize staleness window)
        String cacheKey = "product:" + productId;
        Product updated = repo.findById(productId).orElseThrow();
        cache.opsForValue().set(cacheKey, updated, Duration.ofMinutes(10));

        // Window of inconsistency: between DB commit and cache update.
        // Very brief but non-zero. A read in that window returns stale cache.
    }

    /**
     * Cache-Aside with Invalidation Pattern:
     * On write, delete (invalidate) cache entry.
     * Next read will miss cache, load from DB (current), repopulate cache.
     * Safer than write-through but larger staleness window on read.
     */
    @Transactional
    public void updateProductCacheAside(int productId, double newPrice) {
        repo.updatePrice(productId, newPrice);

        // Invalidate cache — next read will fetch fresh from DB
        String cacheKey = "product:" + productId;
        cache.delete(cacheKey);
    }
}

11. Important Interview Questions — Eventual Consistency

Basic Questions

Q: What is eventual consistency?
A: A property of distributed systems guaranteeing that, if no new updates are made, all replicas will eventually converge to the same value. During the convergence window, reads from different nodes may return different values. It applies specifically to read visibility (Type 2 consistency), not to data corruption (Type 1).

Q: Is eventual consistency a NoSQL-only concept?
A: No. Any relational database with asynchronous read replicas exhibits eventual consistency for reads routed to those replicas. PostgreSQL with async streaming replication, MySQL with standard replication — both are eventually consistent on replica reads. Eventual consistency is a consequence of distribution, not of the data model.

Q: What is the difference between strong consistency and eventual consistency?
A: Strong consistency guarantees every read returns the most recently committed write, regardless of which node handles the request. Eventual consistency allows a window where reads may return stale data, but guarantees convergence over time. Strong consistency typically requires synchronous replication or consensus protocols (Paxos/Raft) and incurs higher write latency.

Intermediate Questions

Q: Can eventual consistency fix corrupt data (Type 1 inconsistency)?
A: No. If atomicity is violated and a multi-table update partially commits (e.g., balance debited but credit never written), the data is corrupt. No replication or eventual convergence will repair this — all replicas will eventually converge on the corrupt state. Eventual consistency only heals visibility lag — it does not heal wrong data.

Q: How does introducing a cache affect consistency?
A: A cache creates a second copy of data. After any write to the database, the cache holds stale data until invalidated or expired. This makes the system eventually consistent by definition. The cache invalidation strategy (write-through, write-behind, cache-aside) determines the size of the inconsistency window and the risk of serving stale data.

Q: What strategies exist to achieve "read-your-writes" consistency with async replication?
A: Several approaches:
(1) Route writes and subsequent reads from the same user session to the primary.
(2) After a write, store the write timestamp client-side; route reads to the primary until replication lag is confirmed to have caught up.
(3) Use synchronous replication so replicas are always current.
(4) Use database features like PostgreSQL's pg_stat_replication to detect lag and steer.

Advanced Questions

Q: Explain the CAP theorem and how it relates to eventual consistency.
A: The CAP theorem states that a distributed system can guarantee at most two of three properties: Consistency (every read sees the latest write), Availability (every request gets a response), and Partition Tolerance (system continues despite network failures). Since network partitions are unavoidable in distributed systems, the real choice is C vs A during a partition. Systems that choose Availability (respond even during partition) by definition cannot guarantee consistency in that window — they become eventually consistent. Systems that choose Consistency may refuse requests during a partition (unavailable).

Q: What is the PACELC extension of CAP and why is it more useful in practice?
A: PACELC extends CAP: If Partition (P), choose Availability (A) or Consistency (C); Else (E), even without partition, choose Latency (L) or Consistency (C). This captures the practical trade-off: even without failures, strong consistency requires coordination (Paxos rounds, sync replication) which adds latency. Most real systems face the L vs C trade-off far more often than the P vs C trade-off. DynamoDB by default chooses L (low latency reads) over C; DynamoDB with strongly consistent reads chooses C at higher latency.

Q: How does Cassandra implement tunable consistency?
A: Cassandra allows per-operation consistency levels. With N total replicas, a write/read at quorum level requires floor(N/2)+1 replicas to acknowledge. If W + R > N, the system is strongly consistent (at least one node must have seen the latest write). At CONSISTENCY ONE, only one replica must respond — eventual consistency. This tunability allows operators to choose the right trade-off per operation type.

12. System Design Connections

Eventual Consistency in Microservices — The Saga Pattern

In microservices, each service has its own database. A business operation spanning multiple services (e.g., place order → reserve inventory → charge payment) cannot use a single ACID transaction. Instead, the Saga pattern is used: each step is a local transaction; compensating transactions handle failures.

Order Saga:
  Step 1: OrderService.createOrder()       → local TX, commits
  Step 2: InventoryService.reserve()       → local TX, commits
  Step 3: PaymentService.charge()          → local TX, commits
              │
              └── If Step 3 fails:
                  Step 3 rollback: PaymentService.refund()
                  Step 2 rollback: InventoryService.release()
                  Step 1 rollback: OrderService.cancelOrder()

The system is eventually consistent: at some point after the saga completes (or its compensations complete), all services converge to a consistent state. There is a window where the order exists but inventory has not been reserved — temporary inconsistency by design.

Architecture Decision: Where to Read

Write path:  Client → [Load Balancer] → Primary DB (strong consistency)
Read path:   Client → [Load Balancer] → Replica DB (eventual consistency)

Balance reads (financial): route to Primary
Recent transaction reads:  route to Primary (read-your-writes)
Historical reports:        route to Replica (lag acceptable)
Dashboard analytics:       route to Replica (slight lag fine)
Cache reads (Redis):        accept staleness per TTL

14. Common Mistakes

  • Saying "NoSQL is eventually consistent, SQL is strongly consistent." Wrong. Any SQL database with async replication is eventually consistent on replica reads.
  • Thinking eventual consistency fixes data corruption. It does not. It only heals replica lag. Corrupt data remains corrupt across all replicas.
  • Not distinguishing between CAP consistency and ACID consistency in interviews. They are completely different. CAP consistency = linearizability. ACID consistency = invariant preservation. Conflating them is an immediate red flag to interviewers.
  • Assuming caches are always in sync. Every cache creates eventual consistency. Every single cache, always.
  • Not accounting for replica lag in SLA design. If your SLA says "balance must always reflect the latest deposit," you cannot route balance reads to async replicas.
  • Using eventually consistent reads for financial operations. Withdrawals, balance checks, payment authorization — these must read from the primary (strongly consistent source).

  REPEATABLE READ:
    • Prevents: dirty reads, non-repeatable reads, lost updates
    • Does NOT prevent: write skew (non-serializable executions)
    • Example: T1(A→B) || T2(B→A) both commit → [B,B,A,A] (impossible serially)

  SERIALIZABLE:
    • Prevents ALL anomalies including write skew
    • PostgreSQL uses SSI (optimistic): detects read-write dependency cycles
    • On conflict: SQLSTATE 40001 → application MUST retry
    • Pessimistic alternative: SELECT FOR UPDATE (blocking, not aborting)

  When to use SERIALIZABLE:
    • Check-then-act patterns (check seat, book seat)
    • Any business invariant spanning multiple rows
    • Financial operations requiring true isolation

  Java retry pattern:
    catch(CannotSerializeTransactionException e) → retry with backoff
    SQLSTATE 40001 = expected, not a bug

EVENTUAL CONSISTENCY:
  Two types of consistency:
    Type 1 (Data):  Is persisted data correct? (referential integrity)
                    → Enforced by: atomicity, isolation, FK constraints
                    → Eventual consistency DOES NOT FIX corrupt data
    Type 2 (Reads): Does a read return latest committed value?
                    → Affected by: replica lag, caching, sharding
                    → Eventual consistency DOES FIX replica lag

  Eventual consistency ≠ NoSQL-only:
    Any SQL DB with async replicas = eventually consistent on replicas

  Caching = always eventual consistency:
    Two copies of data = inconsistent until cache invalidated

  Strong vs Eventual:
    Strong → sync replication → slower writes, no lag
    Eventual → async replication → faster writes, lag exists

  Do not route to replicas:
    ❌ Balance checks, payment auth, financial reads
  Safe to route to replicas:
    ✅ Analytics, reports, historical queries, like counts

  CAP ≠ ACID:
    CAP C = linearizability (latest write seen globally)
    ACID C = data satisfies invariants
    COMPLETELY DIFFERENT — never conflate in interviews
