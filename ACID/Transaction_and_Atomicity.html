Transactions and Atomicity

1. Big Picture

Why Transactions Exist

Databases store structured data across multiple tables and rows. In real applications, a single business operation usually requires:

  • Multiple reads
  • Multiple updates
  • Multiple inserts/deletes

A single SQL query is often not enough. Example:

  • Bank transfer
  • Order placement
  • Payment processing
  • Inventory update
  • Ride booking
  • Trading execution

All these operations require multiple SQL statements that must behave as ONE logical operation. That is why transactions exist.

Problem Transactions Solve

Without transactions:

  • Partial updates happen
  • Data becomes inconsistent
  • Money disappears
  • Orders get duplicated
  • Inventory becomes incorrect
  • Distributed systems become unreliable

Example: Deduct money from Account A successfully, but crash before adding money to Account B.

Result:

  • Money lost forever
  • Corrupted database state

Transactions prevent this.

Why Backend Engineers Must Care

Transactions are foundational to:

  • Banking systems
  • Payment gateways
  • E-commerce
  • Distributed systems
  • Event-driven systems
  • Microservices
  • Inventory systems
  • Trading platforms

Backend engineers constantly work with:

  • ORMs
  • JDBC transactions
  • Connection pools
  • Retry mechanisms
  • Locking
  • Deadlocks
  • Consistency guarantees

Understanding transactions separates junior engineers from strong backend/database engineers.

2. Core Concepts

What is a Transaction?

A transaction is:

A collection of SQL queries treated as ONE unit of work.

If all queries succeed:

  • COMMIT

If any query fails:

  • ROLLBACK

Transaction Lifecycle

Step 1 — BEGIN

BEGIN;

Tells database:

  • Start a new transaction context
  • Group upcoming queries together

Step 2 — Execute Queries

Example:

SELECT balance FROM account WHERE id = 1; UPDATE account SET balance = balance - 100 WHERE id = 1; UPDATE account SET balance = balance + 100 WHERE id = 2;

Step 3 — COMMIT

COMMIT;

Means:

  • Persist changes permanently
  • Make changes visible to others
  • Mark transaction successful

Step 4 — ROLLBACK (If Needed)

ROLLBACK;

Means:

  • Undo all changes
  • Restore previous consistent state

Important Insight

A transaction is logically indivisible. Either:

  • Everything happens
    OR
  • Nothing happens

Example: Money Transfer

Initial State

Account IDBalance
11000
2500

Transfer: $100 from Account 1 → Account 2

Correct Transaction

BEGIN; SELECT balance FROM account WHERE id = 1; UPDATE account SET balance = balance - 100 WHERE id = 1; UPDATE account SET balance = balance + 100 WHERE id = 2; COMMIT;

Final State:

Account IDBalance
1900
2600

Consistent.

Implicit Transactions

Even if you do NOT write BEGIN;, most databases automatically create transactions internally.

Example:

UPDATE users SET name='John' WHERE id=1;

Internally becomes:

BEGIN; UPDATE users SET name='John' WHERE id=1; COMMIT;

This is called: Autocommit mode

Read-Only Transactions

Transactions are not only for writes. Example: Generating a financial report.

You want:

  • Consistent snapshot of data
  • Same view throughout report generation

Without transaction:

  • Concurrent updates may change results mid-report

With transaction:

  • Snapshot remains stable

Atomicity

Atomicity is one of the ACID properties. ACID:

  • Atomicity
  • Consistency
  • Isolation
  • Durability

Definition

Atomicity means: All operations inside a transaction succeed OR all fail.

No partial execution allowed.

Example of Atomicity Failure

Suppose:

BEGIN; UPDATE account SET balance = balance - 100 WHERE id = 1; -- DATABASE CRASHES HERE UPDATE account SET balance = balance + 100 WHERE id = 2; COMMIT;

After restart:

Account IDBalance
1900
2500

Money vanished. This violates atomicity.

3. Deep Dive

What Happens Internally During a Transaction?

Internal Transaction Flow

Client | v Connection Pool | v Database Session | v Transaction Manager | +--> Buffer Pool | +--> WAL (Write Ahead Log) | +--> Lock Manager | +--> Storage Engine

Step-by-Step Internal Execution

BEGIN

Database:

  • Creates transaction ID
  • Allocates transaction metadata
  • Starts tracking changes

Query Execution

Suppose:

UPDATE account SET balance = balance - 100 WHERE id = 1;

Database internally:

  1. Finds row using index/B+Tree
  2. Loads page into memory (buffer pool)
  3. Acquires lock
  4. Modifies row in memory
  5. Writes undo/redo information to WAL
  6. Marks page dirty

Dirty Pages

Dirty page:

  • Memory page modified
  • Not yet flushed to disk

WAL (Write Ahead Logging)

Critical database concept. Before modifying actual disk pages:

  • Database writes log record first

Why? Crash recovery.

WAL Example

Transaction T1: Old Balance = 1000 New Balance = 900

WAL stores:

T1 updated account 1: 1000 -> 900

If crash happens: Database replays or undoes changes

Undo Logs vs Redo Logs

Undo Log

Used for rollback. Example:

Previous value = 1000

Allows restoring old state.

Redo Log

Used for crash recovery. Example:

Apply update: 1000 -> 900

Allows replaying committed transactions.

Commit Internals

When COMMIT happens, Database:

  1. Flushes WAL to durable storage
  2. Marks transaction committed
  3. Releases locks
  4. Makes changes visible

Why Commits Can Be Slow

Commit often requires:

  • fsync()
  • Disk flush
  • WAL persistence

Disk sync is expensive.

Database Design Tradeoffs

Strategy 1 — Eager Persistence

Write changes immediately.

  • Pros: Fast commit, Better crash recovery
  • Cons: Heavy I/O during transaction

Example: PostgreSQL

Strategy 2 — Deferred Persistence

Keep changes mostly in memory.

  • Pros: Fast query execution
  • Cons: Slow commit, Riskier crashes

Rollback Internals

Rollback means:

  • Use undo logs
  • Reverse changes

Example: 900 -> 1000

Large rollback:

  • Heavy CPU
  • Heavy disk I/O
  • Long recovery time

Crash Recovery

If database crashes mid-transaction, on restart:

  1. Scan WAL
  2. Detect incomplete transactions
  3. Undo uncommitted changes
  4. Redo committed changes if necessary

This process is called: Recovery / Crash recovery

Why Long Transactions Are Dangerous

Long transactions:

  • Hold locks longer
  • Increase WAL size
  • Increase rollback cost
  • Increase memory usage
  • Increase recovery time

Production systems avoid them.

4. Practical Backend Engineering Perspective

Banking Systems

Example: Money transfer. Must guarantee:

  • No money loss
  • No double spending

Transactions are mandatory. Used in: Core banking, ATM systems, Payment settlement.

E-commerce

Example: Order placement. Steps:

  1. Create order
  2. Reduce inventory
  3. Charge payment
  4. Generate invoice

If inventory update fails: Entire order must rollback.

Stripe-like Payment Systems

Payment processing involves:

  • Ledger entries
  • Wallet balances
  • Fraud checks
  • Transaction records

Atomicity prevents financial corruption.

Uber-like Systems

Ride completion transaction:

  1. Deduct rider balance
  2. Credit driver earnings
  3. Update trip state

Must happen atomically.

Trading Systems

Trade execution:

  1. Debit buyer funds
  2. Credit seller funds
  3. Transfer asset ownership

Any inconsistency can cause massive financial loss.

Netflix-like Reporting Systems

Read-only transactions provide:

  • Stable analytics snapshots
  • Consistent reporting

Without snapshot isolation: Metrics fluctuate during report generation.

5. Important Interview Questions

Basic Questions

What is a transaction?

A collection of SQL operations treated as one logical unit of work.

What is atomicity?

Either all operations succeed or all fail.

Difference between COMMIT and ROLLBACK?

COMMITROLLBACK
Persist changesUndo changes
Makes changes permanentRestores old state

What is autocommit?

Database automatically wraps each query in a transaction.

Intermediate Questions

Why are transactions needed?

To maintain consistency across multiple operations.

Why are long transactions bad?

  • Lock contention
  • Large rollback cost
  • Memory growth
  • Recovery overhead

What happens during database crash recovery?

Database:

  • Replays committed changes
  • Undoes incomplete transactions

Using WAL/redo/undo logs.

What is WAL?

Write Ahead Log. Database writes changes to log before data pages.

Advanced Questions

Why is COMMIT expensive?

Requires durable disk synchronization.

Explain crash recovery internals.

  1. Read WAL
  2. Identify committed transactions
  3. Redo committed updates
  4. Undo incomplete transactions

Why does PostgreSQL commit quickly?

Because it aggressively writes WAL during transaction execution, reducing commit-time work.

What are dirty pages?

Modified memory pages not yet persisted to disk.

6. Common Mistakes

Beginner Mistakes

Forgetting transactions

Executing related updates separately. Very dangerous.

Not handling rollback

Ignoring exceptions in application code.

Huge transactions

Processing thousands of operations in one transaction.

Backend Engineer Mistakes

Holding transaction open during API calls

BAD:

BEGIN Call external service Wait 5 seconds COMMIT

Locks remain active entire time.

User interaction inside transaction

Never wait for user input while transaction is open.

Mixing business logic and transaction boundaries poorly

Leads to: Deadlocks, Lock contention, Timeouts.

Production Mistakes

Long-running reports on primary DB

Can: Block writes, Increase MVCC cleanup lag.

Ignoring WAL growth

Large transactions generate huge WAL files.

Massive rollback events

Can overload recovery system.

8. Database Internals Notes

Pages

Databases read/write fixed-size pages: Usually 8KB or 16KB. Updates modify pages in memory first.

B+ Trees

Indexes are usually B+ Trees. Used for: Fast row lookup, Efficient range scans.

Buffer Pool

Memory cache for pages. Queries operate mostly on memory pages first.

WAL

Guarantees durability and recovery. Critical for ACID compliance.

Locks

Transactions acquire locks to prevent corruption. Examples:

  • Row locks
  • Table locks
  • Shared locks
  • Exclusive locks

MVCC (Multi-Version Concurrency Control)

Used by databases like: PostgreSQL, Oracle.

Allows:

  • Readers without blocking writers
  • Snapshot isolation

Query Planner

Determines: Index usage, Scan strategy, Join algorithms.

Transaction isolation can affect query planning.

9. Additional Important Knowledge (Beyond the Lecture)

ACID Overview

A — Atomicity

All or nothing.

C — Consistency

Valid state transitions only.

I — Isolation

Concurrent transactions behave correctly.

D — Durability

Committed changes survive crashes.

Savepoints

Partial rollback support. Example:

SAVEPOINT step1; ROLLBACK TO step1;

Useful in complex workflows.

Distributed Transactions

In microservices: One DB transaction is not enough. Need:

  • Two-phase commit
  • Saga pattern
  • Outbox pattern

Eventual Consistency

Modern distributed systems often relax strict consistency for scalability. Example:

  • Distributed caches
  • Multi-region databases

Idempotency

Critical for retry-safe transaction handling. Example: Payment retries should not double-charge customers.

Transaction Isolation Levels

Will be covered later but important:

  • Read Uncommitted
  • Read Committed
  • Repeatable Read
  • Serializable

10. Practical SQL Examples

Money Transfer

BEGIN; SELECT balance FROM account WHERE id = 1; UPDATE account SET balance = balance - 100 WHERE id = 1; UPDATE account SET balance = balance + 100 WHERE id = 2; COMMIT;

Safer Version with Validation

BEGIN; SELECT balance FROM account WHERE id = 1 FOR UPDATE; UPDATE account SET balance = balance - 100 WHERE id = 1 AND balance >= 100; UPDATE account SET balance = balance + 100 WHERE id = 2; COMMIT;

Java JDBC Transaction Example

Connection conn = dataSource.getConnection(); try { conn.setAutoCommit(false); PreparedStatement debit = conn.prepareStatement( "UPDATE account SET balance = balance - ? WHERE id = ?" ); PreparedStatement credit = conn.prepareStatement( "UPDATE account SET balance = balance + ? WHERE id = ?" ); debit.setInt(1, 100); debit.setInt(2, 1); debit.executeUpdate(); credit.setInt(1, 100); credit.setInt(2, 2); credit.executeUpdate(); conn.commit(); } catch (Exception e) { conn.rollback(); } finally { conn.close(); }

11. System Design Connections

Transactions and Distributed Systems

Transactions are central to:

  • Service consistency
  • Reliable messaging
  • Inventory correctness
  • Payment correctness

Microservices Challenge

Single DB transaction cannot span: Multiple services, Multiple databases.

Requires:

  • Saga orchestration
  • Event-driven consistency

Replication

Committed transactions replicate to replicas using WAL/binlog.

High Availability

Crash recovery + WAL replay enable:

  • Failover
  • Point-in-time recovery

Caching

Transactions complicate cache consistency. Need:

  • Cache invalidation
  • Write-through strategies
  • CDC pipelines

13. Lecture Summary

Key Takeaways

  • Transactions group multiple queries into one logical operation.
  • Atomicity guarantees all-or-nothing execution.
  • WAL is essential for crash recovery.
  • Commits are expensive because of durability guarantees.
  • Rollbacks can be extremely costly.
  • Long transactions are dangerous in production systems.
  • Read-only transactions are important for consistent snapshots.

Must Remember Points

  • Every SQL statement runs inside a transaction.
  • COMMIT makes changes durable.
  • ROLLBACK restores consistency.
  • Crash recovery depends on WAL/undo logs.
  • Partial updates are unacceptable in production systems.

Interview Nuggets

  • PostgreSQL optimizes heavily around WAL and commit behavior.
  • Large transactions can slow recovery dramatically.
  • Atomicity is enforced even during crashes.
  • MVCC allows snapshot-based reads.
