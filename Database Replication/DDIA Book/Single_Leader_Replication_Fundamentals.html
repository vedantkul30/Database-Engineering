Single-Leader Replication & Zero-Disk Architectures

Engineering Mastery Notes (Source: DDIA Chapter 6 Extract)

1. Big Picture

Why Replication Exists

In a single-node system, your database is highly vulnerable. Every write needs to be securely stored, but if that single node fails, your system experiences downtime or catastrophic data loss. Replication exists to copy that data across multiple nodes via a network.

Problems Single-Leader Replication Solves

  1. High Availability: If a node crashes, others have the data.
  2. Read Scalability: By separating read operations from write operations, you can scale read-heavy applications infinitely by simply adding more read-only nodes.
  3. Disaster Recovery: Maintaining off-site or cross-region copies of data.

Where Single-Leader Replication is Used in Production

Almost everywhere. It is the default, most proven replication topology in the industry:

  • Relational DBs: PostgreSQL, MySQL, Oracle Data Guard, SQL Server (Always On).
  • Document DBs: MongoDB, DynamoDB.
  • Message Brokers: Kafka, RabbitMQ (Quorum Queues).
  • Consensus Systems: Raft-based systems like CockroachDB, TiDB, and etcd.

Note: In older literature, this was called "master-slave", but the industry standard is now Primary/Backup, Active/Passive, or Leader/Follower.

Why Backend Engineers Must Care: You will almost certainly build systems on top of a Single-Leader DB. If you do not understand how data moves from the Leader to the Follower, you will design systems that serve stale data, fail catastrophically during network partitions, or lose user data during leader crashes.

2. Core Concepts

  • Replica: Any node that stores a copy of the database.
  • Leader (Primary / Source / Active): The single designated node that accepts all writes. It writes to its local storage first, then streams changes to others.
  • Follower (Read Replica / Secondary / Hot Standby): Replicas that consume the leader's replication log/change stream and apply the writes locally in the exact same order. They serve reads only.
  • Synchronous Replication: The leader waits for the follower to confirm it has securely written the data before the leader acknowledges a successful write to the client.
  • Asynchronous Replication: The leader sends the data to the follower and immediately acknowledges the client's write without waiting for the follower's confirmation.
  • Semi-Synchronous Replication: A hybrid approach where exactly one follower is synchronous, and all others are asynchronous.
  • Zero-Disk Architecture (ZDA): A modern pattern where compute nodes are stateless, and all durable state (including WAL) is pushed to Object Storage (like Amazon S3).

3. Deep Dive: Mechanisms & Pipelines

Internal Working: The Replication Log

When a client updates a database, the leader doesn't just update the raw data files. It appends the change to a Replication Log. Followers read this log and replay it.

Step-by-Step: Setting Up a New Follower (Without Downtime)

You cannot simply copy database files using cp or rsync while the database is running; the files are constantly mutating, resulting in a corrupted, inconsistent copy. To add a replica safely:

  1. Snapshot: Take a consistent snapshot of the Leader's DB at a specific point in time (e.g., using Percona XtraBackup for MySQL or pg_basebackup for Postgres).
  2. Copy: Move the snapshot to the new Follower.
  3. Connect & Request: The Follower connects to the Leader and asks for all changes since the snapshot's exact position.
    • PostgreSQL uses: Log Sequence Number (LSN)
    • MySQL uses: Binlog Coordinates or Global Transaction Identifiers (GTIDs)
  4. Catch Up: The Follower processes the backlog. Once caught up, it streams changes in real-time.

Deep Dive: Databases Backed by Object Storage (S3/GCS)

Modern systems are decoupling storage from compute using Object Stores (S3).

  • Tiered Storage: Hot data stays on NVMe/SSDs; cold/archived data goes to S3.
  • Zero-Disk Architecture (ZDA): Nodes have no persistent local state. Disks are only used for caching. All WAL and data files live in S3.
    • Examples: WarpStream, Confluent Freight, Redpanda Serverless (Kafka-compatible ZDA), Turbopuffer, SlateDB.
  • How it works: DBs use S3's Conditional Writes (Compare-and-Set / CAS) to safely implement transactions and leader elections without local disks.

Performance Tradeoffs of S3 DBs:

  • Pros: Insanely cheap, built-in multi-AZ durability, stateless compute nodes.
  • Cons: High API per-call costs, high latency (necessitates heavy write-batching), lack of standard POSIX filesystem features (random writes are expensive).

4. Distributed Systems Perspective: Failure Scenarios

Scenario What Happens System Impact SRE Resolution
Sync Follower Crashes Leader cannot get write confirmation. Catastrophic Write Outage. System blocks all writes. A timeout demotes the sync follower. An async follower is dynamically promoted to be the new sync follower.
Async Follower Crashes Reads might hit fewer nodes (capacity drop). Negligible. Writes continue uninterrupted. Node restarts, reads its LSN/GTID, and catches up automatically.
Leader Crashes (Fully Async setup) Un-replicated writes living only on the Leader's RAM/disk are stranded. Data Loss. Confirmed writes to users vanish if a failover happens. Accept the data loss for availability, or use Semi-Sync replication.

5. Replication Models

Single-Leader Replication (Primary-Backup)

  • Architecture: 1 Leader accepts all writes. N Followers apply the replication log.
  • Write Path: Client → Leader → Local Disk → Follower.
  • Read Path: Client → Leader (for strict consistency) OR Follower (for eventual consistency).
  • Sharding Integration: If a DB is sharded (e.g., 10 partitions), each shard has its own independent Leader.
  • Limitations: The Leader is a strict bottleneck for write throughput.

(Note: While Multi-Leader and Leaderless replication exist, Single-Leader is the foundational baseline used by 90% of traditional systems).

6. CAP and Consistency Tradeoffs

Single-Leader replication is a masterclass in evaluating the trade-off between Latency, Durability, and Availability.

  1. Fully Synchronous: Strong Consistency & Durability. Poor Latency & Availability. (If the follower hiccups, your whole system goes down).
  2. Fully Asynchronous: Incredible Latency & Availability. Poor Durability. (Fast writes, but you will lose data if the leader burns down).
  3. Semi-Synchronous (The Golden Standard): Leader + 1 Sync Follower + N Async Followers. Ensures data is always on at least two machines without blocking on the entire fleet.

7. Practical Backend Engineering Perspective

How top systems operate in production:

  • Amazon Aurora / Neon Safekeepers: Decouples the WAL. Instead of local followers, the WAL is written synchronously to a distributed storage layer (EBS/Safekeepers), acting as a highly available, specialized follower.
  • Kafka / RabbitMQ: Use leader-based replication under the hood for partitions/queues. Kafka uses In-Sync Replicas (ISR) which act as a quorum for synchronous replication.
  • Disaster Recovery (Litestream / WAL-G): Instead of just sending logs to another Postgres instance, the log is continuously shipped to an S3 bucket. You can spawn a new follower entirely from S3.

8. Important Interview Questions

Basic

Q: What is the difference between a synchronous and asynchronous replica?
A: A sync replica blocks the leader from confirming a write to a client until the replica successfully stores it. An async replica does not block the leader; the leader confirms the write immediately after saving it locally.

Intermediate

Q: How do you add a new read replica to a production MySQL database without taking the application offline?
A: Do not copy the raw files directly. Take a consistent snapshot, restore it on the new node, and configure the new node to connect to the leader using the exact Binlog Coordinates/GTID from the snapshot to stream and catch up on the missed transactions.

Advanced

Q: How does a Zero-Disk Architecture (ZDA) system manage leader election if it doesn't have local persistent state or a Zookeeper cluster?
A: ZDA systems leverage Object Storage (like S3). They use S3's atomic Conditional Write (Compare-and-Set / CAS) functionality. Multiple nodes can attempt to write a lease file to S3 concurrently; the object store ensures only one succeeds, effectively acting as the consensus mechanism for leader election.

9. Common Mistakes

  • Beginner Mistake: Thinking a raw cp or rsync of /var/lib/postgresql/data is enough to create a backup or replica. Result: Corrupt data.
  • Backend Engineer Mistake: Assuming "Asynchronous" means eventual consistency but guaranteed durability. Truth: If the leader dies before the network packet hits the async follower, that write is gone forever.
  • Production SRE Mistake: Configuring all replicas as Synchronous. Truth: This guarantees downtime. A single network blip to one replica will halt your entire production write path.

10. Performance & Scalability Notes

  • Write Bottlenecks: In Single-Leader, you cannot scale writes by adding nodes. You must vertically scale the leader (bigger CPU/RAM) or implement Database Sharding.
  • Object Storage Latency: When migrating to a ZDA/S3-backed database, write latency shifts from microsecond local-disk speeds to 20-50ms HTTP API speeds. Systems must aggressively batch writes to S3 to offset API per-call costs and network overhead.

11. Distributed Systems Internals

  • Consensus-based Leaders: Systems like etcd or TiDB use Raft. Raft is mathematically a Single-Leader architecture, but the leader is dynamically elected via Quorum (majority vote). If the leader dies, the cluster detects the missing heartbeats and elects a new leader automatically.
  • Quorum Writes: A variation of semi-sync where the leader waits for a majority of replicas to acknowledge the write (e.g., 3 out of 5) rather than a single specific sync follower.

12. Beyond the Lecture (Extra Mile Insights)

  • Split Brain: If a network partition occurs, a Follower might think the Leader is dead and promote itself. Now you have two Leaders accepting writes. Fix: Fencing tokens or strict consensus (ZooKeeper).
  • Replication Lag: Because async replicas process logs sequentially, heavy parallel write loads on the leader will cause the replica to fall behind (lag). Reading from it will return stale data (violating read-after-write consistency).

13. Practical Code / System Design Architecture

Routing logic in a Backend Application (Java/Spring):

public class DBRouter {
    // Writes strictly go to the Leader
    @Transactional
    public void createUser(User user) {
        setDataSource(DataSourceType.LEADER); 
        userRepository.save(user);
    }

    // Reads can be distributed among Followers
    @Transactional(readOnly = true)
    public User getUser(String id) {
        setDataSource(DataSourceType.FOLLOWER_ROUND_ROBIN);
        return userRepository.findById(id);
    }
}

Zero-Disk Architecture Mental Model:
Instead of App → Leader → Follower, modern data systems use S3 as the ultimate source of truth:

App -> [Stateless Compute Node] --(Batched WAL)--> [Amazon S3]
App <- [Stateless Compute Node] <--(Read Data)---- [Amazon S3]

14. System Design Connections

  • Tiered Storage: Crucial for massive scale (e.g., Snowflake, ClickHouse). Hot data stays in memory/NVMe (the "Leader" concept locally), while cold data and backups stream to S3.
  • Cost Engineering: Cloud providers charge heavily for cross-AZ network traffic. S3-backed databases bypass cross-AZ networking fees because S3 handles the multi-zone replication internally.

16. Failure Analysis Section

Component Failure Risk Level SRE Recovery / Mitigation
Sync Follower Complete crash. Critical (Writes Stop) Auto-downgrade to async, promote another node to sync.
Leader Disk Disk full / failure. High (Downtime) Initiate Failover. Promote Sync Follower to Leader.
Network Leader separated from clients but active. Critical (Split Brain) Fencing mechanisms invalidate the old leader's writes.
S3 API AWS S3 goes down. High (ZDA Systems) ZDA databases will completely halt. Rely on AWS SLAs.

17. Lecture Summary

  • Key Takeaway 1: Single-Leader replication separates writes (Leader) from reads (Followers) but introduces a strict write bottleneck.
  • Key Takeaway 2: You must choose between latency and durability. Synchronous replication guarantees durability at the cost of availability; Asynchronous guarantees speed at the risk of data loss. Semi-synchronous is the industry compromise.
  • Key Takeaway 3: The future of DB architecture is moving toward Zero-Disk Architectures (ZDA), treating Object Storage (S3) as the primary transaction log and consensus layer, drastically simplifying state management.

18. Revision Notes (2-Minute Review Sheet)

  • Leader: Accepts writes. Follower: Accepts reads, copies log.
  • Replication Log: Abstract stream of DB changes (LSN in Postgres, GTID in MySQL).
  • Sync vs Async: Sync waits for follower (safe, slow). Async doesn't wait (unsafe, fast).
  • Semi-Sync: 1 Leader + 1 Sync Follower + N Async Followers. (Best practice).
  • Adding a Node: Snapshot → Copy → Request LSN → Catch up. (Never raw file copy).
  • Zero-Disk Architecture (ZDA): Compute is stateless. Disks are caches. State and WAL live in Amazon S3. Uses Compare-and-Set (CAS) for consensus. High latency, massive cost savings.
