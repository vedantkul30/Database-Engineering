Leaderless Replication (Dynamo-Style)

📖 Engineering Mastery Notes (Source: DDIA Chapter 6 Extract)

1. Big Picture

Why Leaderless Replication Exists

In 2007, Amazon faced a crisis: during the holiday shopping season, minor database node failures or network hiccups were causing the "Add to Cart" functionality to fail. In a single-leader database, if the leader is dead, writes stop entirely until a failover completes. Amazon realized that rejecting a customer's write means losing money.

They published the Dynamo Paper, introducing a paradigm shift: Abandon the Leader. If any node in the cluster is alive, it should accept writes.

Problems Leaderless Replication Solves

  1. Zero Downtime for Writes: There is no "failover" pause. If a node burns down, the client seamlessly writes to the surviving nodes.
  2. Gray Failures & Tail Latency: If a node isn't dead but is "limping" (e.g., slow disk, GC pause), a leader-based system chokes. A leaderless system masks this via Request Hedging (taking the fastest responses from a pool of replicas).
  3. Multi-Active Geo-Distribution: Inherently supports accepting writes anywhere in the world simultaneously.

Where it is Used in Production

  • Cassandra / ScyllaDB: The industry standards for massive, high-write throughput (used by Netflix, Uber, Apple).
  • Riak: Heavily used in telecommunications and shopping carts.
  • Amazon Dynamo (Original): The internal system that inspired the architecture. (Note: AWS DynamoDB today uses a different, single-leader Paxos architecture).

Why Backend Engineers Must Care:
Leaderless systems are High Availability (HA) monsters. But they push the complexity of consistency and conflict resolution directly onto you, the backend engineer. If you don't understand quorums and version vectors, your application will serve resurrected deleted data, drop user updates, and fail silently under concurrent load.

2. Core Concepts

  • Coordinator Node: A replica (or a separate proxy) that receives the client's request and fans it out to the other replicas holding the data partition.
  • Quorum: The minimum number of votes (acknowledgments) required to consider a read or write successful.
  • N (Replication Factor): Total number of replicas storing a specific piece of data.
  • W (Write Quorum): Number of nodes that must acknowledge a write.
  • R (Read Quorum): Number of nodes that must respond to a read.
  • Read Repair: A mechanism where a client reading a stale value from a node actively writes the newer value back to that node to fix it.
  • Hinted Handoff: If Node A is dead, Node B accepts its writes temporarily ("hints") and pushes them to Node A when it boots back up.
  • Anti-Entropy: A background daemon that constantly scans all replicas, compares data (usually via Merkle Trees), and syncs missing pieces.
  • Happens-Before Relation: The mathematical definition of causality. Event A happened before Event B if B knew about A, depended on A, or built upon A.
  • Version Vector: A collection of version numbers from all replicas (e.g., [Node1: v2, Node2: v1]) used to track causality and detect concurrent writes.

3. Deep Dive: Internal Workings of Leaderless Systems

The Write Path (No Failovers)

  1. Client sends a write to any node (the Coordinator).
  2. Coordinator hashes the partition key to determine which N nodes own this data.
  3. Coordinator sends the write to all N nodes in parallel.
  4. It waits for W nodes to reply "OK".
  5. If W is met, it returns 200 OK to the client. It completely ignores if the remaining N - W nodes crashed or timed out.

The Read Path & Conflict Resolution

  1. Client asks the Coordinator for a key.
  2. Coordinator requests the key from all N nodes in parallel.
  3. It waits for R responses.
  4. Because some nodes might have missed previous writes, the Coordinator might receive different versions (e.g., Version 6 from Node 3, Version 7 from Nodes 1 & 2).
  5. The Coordinator uses version numbers/timestamps to pick the freshest data and returns it to the client.
  6. Read Repair: The Coordinator asynchronously sends Version 7 to Node 3 to fix its stale state.

Catch-Up Mechanisms

How does a node that was offline for an hour get its missing data?

  1. Read Repair (Reactive): Fixed only when a client happens to read the data. Great for hot data, useless for cold data.
  2. Hinted Handoff (Proactive Short-term): Neighboring nodes store a "hint" file for the dead node. When it wakes up, neighbors stream the hints. (Fails if the dead node is gone for days, as neighbors will delete hints to save disk space).
  3. Anti-Entropy (Proactive Long-term): A background process that synchronizes replicas. It uses cryptography (Merkle Trees) to compare datasets efficiently across the network without sending the actual data.

4. Distributed Systems Perspective: The "Happens-Before" Relation

Relativity and Concurrency

In distributed systems, time does not matter. Because of NTP clock skew, comparing physical timestamps is dangerous. Instead, we use logical causality: The Happens-Before relationship.

  • Scenario 1: Client A inserts X. Client B reads X, then increments it to X+1. B happened-before A. B's write safely overwrites A's.
  • Scenario 2: Client A and Client B both update X to different values. Neither read the other's update. They are concurrent.
  • The Rule: Two events are concurrent if neither happened before the other. They do not need to overlap in physical milliseconds. If the network is split, they could happen hours apart and still be mathematically "concurrent."

5. Replication Models: Quorum Mathematics

The entire safety of a leaderless system rests on the formula:

W + R > N

If the sum of your write quorum and read quorum is strictly greater than your replication factor, your read and write sets must overlap by at least one node. The read will mathematically guarantee seeing the latest write.

Common Configurations:

  • N=3, W=2, R=2: The industry standard. Tolerates 1 node failure. Fast reads, fast writes.
  • N=5, W=3, R=3: High durability. Tolerates 2 node failures.
  • N=3, W=3, R=1: Optimized for heavy reads. Extremely fast reads, but if 1 node dies, the database becomes strictly Read-Only (Availability is compromised).

6. CAP and Consistency Tradeoffs: The Illusion of Strong Consistency

Crucial Engineering Insight: Even if W + R > N, Dynamo-style databases DO NOT guarantee Strong Consistency (Linearizability) in edge cases.

Why W + R > N reads can still be stale:

  1. Partial Write Failures: A write hits 1 node and fails on the other 2. W=2 is not met. The DB returns an error to the client. However, the write on the 1 node is NOT rolled back. A subsequent read might accidentally hit that 1 node and return the "failed" data.
  2. Clock Skew (LWW): If Cassandra uses Last-Write-Wins timestamps to resolve conflicts, a node with a broken, fast clock will permanently overwrite newer data.
  3. Node Restoration: Node A crashes and loses disk data. It is restored from a 3-day-old backup. The number of nodes holding the newest write quietly drops below W, silently breaking the quorum overlap.

7. Practical Backend Engineering Perspective

How top companies utilize these properties:

  • Netflix (Cassandra): Netflix stores user viewing history in Cassandra. They use a sloppy quorum (Consistency Level ANY) for writes. If a region goes down, they just want the telemetry written somewhere. If a user watches a movie, it is critical the data isn't dropped, even if it takes 5 minutes to appear on their profile.
  • Multi-Region Routing: In ScyllaDB/Cassandra, you don't do blind cross-region writes. A client hits a Coordinator in us-east. That coordinator writes to local nodes, and sends one packet to a coordinator in eu-west, which then fans out locally. This saves massive cross-atlantic bandwidth.
  • Hedging: When an API must respond in <50ms, Backend Engineers send the read query to 3 Cassandra nodes simultaneously. The first one to reply is used. The other two are ignored. This eliminates tail latency caused by JVM Garbage Collection pauses on individual DB nodes.

8. Important Interview Questions

Basic

Q: Explain the quorum formula W + R > N.
A: It ensures that the number of nodes required to acknowledge a write (W) and the number required to serve a read (R) overlap by at least one node out of the total replicas (N). This mathematically guarantees the read will consult at least one node holding the freshest data.

Intermediate

Q: A leaderless database has a node go offline for 5 minutes, then it reboots. How does it get the writes it missed?
A: Through three mechanisms: Hinted Handoff (neighbor nodes stored the writes temporarily and push them to the rebooted node), Read Repair (when a client reads a stale value from the node, it pushes the updated value back), and Anti-Entropy (a background daemon comparing Merkle trees to sync missing data).

Advanced

Q: Does W + R > N guarantee strict linearizability (strong consistency)? Why or why not?
A: No, it only guarantees eventual consistency with a high probability of reading the latest write. Linearizability can be broken by partial write failures (data is not rolled back on success < W), physical clock skew if using LWW, and concurrent interleaved read/write operations.

9. Common Mistakes

  • Beginner Mistake: Assuming W=2 means "wait for 2 specific nodes." It actually means "wait for the fastest 2 out of N nodes."
  • Backend Engineer Mistake: Using Cassandra as a relational database. Cassandra has no ACID transactions and no cross-partition joins. If you try to model a banking ledger in Cassandra without application-level distributed locks, you will destroy your financial data due to concurrent writes.
  • Production SRE Mistake: Disabling Anti-Entropy repairs to save CPU. Over time, "deleted" data (tombstones) will expire inconsistently across nodes, causing deleted items to resurrect permanently (the "Zombie Data" problem).

10. Performance & Scalability Notes

  • Write Amplification: A single client write of 1MB results in N network hops of 1MB, plus hinted handoffs, plus anti-entropy checks. Leaderless systems demand massive internal network bandwidth.
  • Sloppy Quorums: If a major network partition happens, you can configure the DB to accept writes on any node in the entire cluster (even nodes that don't own the partition). This guarantees 100% Write Availability, but destroys Read Consistency until the partition heals.

11. Distributed Systems Internals: Capturing Causality

To safely merge concurrent data without a leader, the database must track causality using Version Vectors.

The Shopping Cart Walkthrough (Crucial Mental Model)

  • Goal: Two clients adding to a cart concurrently without losing data.
  • Mechanism: The client must read before it writes, and pass the version number back.
  1. C1 Adds Milk: Server assigns v1. Returns [Milk], v1.
  2. C2 Adds Eggs (Concurrent): C2 doesn't know about Milk. Sends Eggs. Server sees a conflict. Assigns v2. State is Siblings: [Milk] and [Eggs]. Returns both to C2.
  3. C1 Adds Flour: C1 bases its write on v1. Sends [Milk, Flour], v1. Server knows v1 is overwritten, but Eggs (v2) is concurrent. State is now [Milk, Flour] v3 and [Eggs] v2.
  4. C2 Adds Ham: C2 previously saw both Milk and Eggs. It merges them: [Milk, Eggs, Ham]. It passes back the version indicating it merged them. The server safely overwrites the old states.

Version Vectors (Multi-Node):
Instead of a single v1, a Version Vector looks like [ReplicaA: 2, ReplicaB: 1]. If Node C receives a write with [ReplicaA: 2, ReplicaB: 2], it can mathematically prove this new write "happened after" the old state, and safely overwrite it.

12. Beyond the Lecture (Extra Mile Insights)

  • Consistency Levels (Cassandra):
    • ONE: Fast, highly available, stale reads likely.
    • QUORUM: The standard W+R>N.
    • LOCAL_QUORUM: Quorum achieved only within the local datacenter, ignoring cross-region latency. (The SRE gold standard for multi-region).
    • EACH_QUORUM: Strict quorum in every region. (Very slow).
  • Merkle Trees: Anti-entropy uses Merkle trees (hash trees). Instead of sending 1TB of data over the network to see if two nodes match, they send a 32-byte hash of the root tree. If it matches, the 1TB is identical. If it differs, they traverse the tree to find the exact 1MB chunk that differs.

13. Practical Code / Database Examples

Routing and Consistency in Java (Cassandra Datastax Driver):

// A Senior Engineer's configuration for high-availability writes
SimpleStatement writeStatement = SimpleStatement.builder("INSERT INTO users (id, name) VALUES (?, ?)")
    .addPositionalValues(userId, name)
    // Accept write if a majority of nodes in the LOCAL datacenter ack it.
    // This protects against AWS regional outages (us-east to eu-west severed).
    .setConsistencyLevel(DefaultConsistencyLevel.LOCAL_QUORUM)
    .build();

session.execute(writeStatement);

// Read-Repair in action (Implicit in driver):
// When we query with QUORUM, if the driver detects 1 node is stale, 
// the Cassandra Coordinator automatically triggers a Read Repair in the background.
SimpleStatement readStatement = SimpleStatement.builder("SELECT * FROM users WHERE id = ?")
    .addPositionalValues(userId)
    .setConsistencyLevel(DefaultConsistencyLevel.LOCAL_QUORUM)
    .build();

14. System Design Connections

  • High-Volume Telemetry Systems: Designing an Uber metrics ingestion service? Use Leaderless Replication. You care about handling 10 million writes per second with zero downtime. You do not care if a read is 2 seconds stale.
  • Microservices Resilience: If your microservice depends on a leader-based DB, a DB failover causes a 30-second spike in HTTP 500s. With a leaderless DB, the microservice experiences zero downtime, masking the infrastructure failure from the end-user completely.

16. Failure Analysis Section

Failure Mode Internal System Action Consequence & SRE Resolution
1 Replica Crashes (N=3, W=2) Coordinator writes to 2 surviving replicas. No impact. Writes succeed. Hinted handoff queues data for the dead node.
2 Replicas Crash (N=3, W=2) Coordinator hits 1 replica. Quorum fails. Write Downtime. App receives HTTP 500. Must wait for node recovery.
Network Partition (Split Brain) Nodes cannot talk to each other. Stale Reads / Siblings. Nodes accept concurrent writes. App must resolve siblings via CRDTs upon healing.
Fast Clock Skew (LWW on) Node A's clock is 5 mins fast. Permanent Data Loss. Node A's old writes overwrite Node B's genuinely new writes.

17. Lecture Summary

  • Key Takeaway 1: Leaderless architectures (Dynamo-style) trade Strong Consistency for extreme Write Availability and low tail latency. There is no such thing as "Failover" downtime.
  • Key Takeaway 2: To ensure you don't read stale data, you must tune Quorums using W + R > N. However, edge cases (partial failures, clock skew) mean this is still fundamentally an Eventually Consistent system.
  • Key Takeaway 3: Concurrent writes will happen. You cannot use physical time to sort them. You must use Logical Clocks (Version Vectors) to track causality and let the application deterministically merge Siblings (or use CRDTs).

18. Revision Notes (2-Minute Review Sheet)

  • Architecture: No Leader. Any node accepts writes. Clients/Coordinators write in parallel.
  • Quorums: N (Replicas), W (Write Acks), R (Read Acks). Overlap formula: W + R > N.
  • Catch-Up:
    1. Read Repair: Fixes stale data instantly on read.
    2. Hinted Handoff: Neighbors hold data for temporarily dead nodes.
    3. Anti-Entropy: Background Merkle Tree sync for deep recovery.
  • Concurrency: Two events are concurrent if neither happened before the other.
  • Version Vectors: Logical clocks [N1:v2, N2:v1] used to track causality, identify siblings, and safely merge concurrent offline writes.
  • Tradeoffs: High availability, handles gray failures beautifully via Hedging. Downside: Complex conflict resolution, write amplification, eventual consistency edge cases.
