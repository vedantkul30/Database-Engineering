Dealing with Conflicting Writes

📖 Engineering Mastery Notes (Source: DDIA Chapter 6 Extract)

1. Big Picture

The Inevitability of Conflicts

If you adopt a Multi-Leader or Leaderless replication architecture, you have inherently traded strong consistency for high availability and low latency. The price you pay for this trade-off is Write Conflicts. When two nodes accept writes for the same piece of data independently—without coordinating with each other first—those writes will eventually collide during the asynchronous replication process.

The True Definition of "Concurrent"

In distributed systems, "concurrent" does not mean two events happened at the exact same physical millisecond. Physical time is largely irrelevant. Two writes are concurrent if neither was aware of the other when the write occurred. If User A and User B both update a wiki page, and neither had seen the other's update prior to saving, those writes are concurrent—even if they were submitted hours apart (e.g., while offline on an airplane).

Why Backend Engineers Must Care

If you handle write conflicts poorly, your system will suffer from silent data corruption, resurrected deleted data, or completely broken application invariants (like double-booked hotel rooms or negative bank balances).

2. Core Concepts

  • Conflict: When two concurrent operations mutate the same state in ways that cannot be trivially merged.
  • Concurrent Writes: Writes that occur independently, where neither operation "happened-before" the other.
  • Conflict Avoidance: Architecting the routing layer so that concurrent writes to the same record are mathematically impossible.
  • Last Write Wins (LWW): A crude resolution strategy that picks the write with the highest timestamp and forcefully deletes the others.
  • Siblings: Storing all conflicting versions of a record in the database simultaneously and pushing the responsibility of merging them to the application or user.
  • Strong Eventual Consistency: A guarantee that all replicas that have received the same set of updates will eventually converge to the exact same state, regardless of the order in which the updates arrived.
  • CRDT (Conflict-Free Replicated Data Type): Mathematical data structures that handle merges automatically without a central server.
  • OT (Operational Transformation): An algorithm that mathematically transforms the index/position of operations to merge them safely (used heavily in real-time text editing).

3. Deep Dive: Conflict Resolution Strategies

How do we fix the collision when Leader 1 says the title is "B" and Leader 2 says the title is "C"?

A. Conflict Avoidance (The SRE Favorite)

  • Internal Working: Instead of solving the conflict, you prevent it. You partition your data such that all writes for a specific record are always routed to the same leader.
  • Example: User 123's profile updates are exclusively routed to the us-east leader.
  • The Flaw: It breaks down during a failover. If us-east goes down and you reroute User 123 to eu-west, a delayed write from the old leader might suddenly appear, creating the exact conflict you tried to avoid.

B. Last Write Wins (LWW) - Discarding Writes

  • Internal Working: Every write is tagged with a physical timestamp. The database compares timestamps: If (Write_A.time > Write_B.time) { keep(A); drop(B); }
  • The Flaw: Physical clocks drift (NTP Clock Skew). "Last Write" actually means "the write from the machine whose clock happened to be furthest ahead."
  • The Cost: Silent Data Loss. It is acceptable only for highly immutable data or caches.

C. Manual Resolution (Siblings)

  • Internal Working: The database detects the conflict and saves both values. When the client reads the record, the database returns a list: ["Title B", "Title C"].
  • The Flaw: The application developer must write complex merge logic, or the user is forced to click "Resolve Conflict" (like a Git merge conflict).

D. Automatic Resolution

  • Internal Working: Algorithms (like CRDTs or OTs) that track the exact intent of the operations (Insertions/Deletions/Increments) rather than the final state, ensuring that state naturally converges to a mathematically identical state on all nodes.

4. Distributed Systems Perspective: The Amazon Shopping Cart Anomaly

When you ask the application to manually merge siblings, naïve merge logic causes catastrophic bugs.

The Scenario:
Amazon originally used a Multi-Leader architecture for shopping carts. To merge concurrent carts, their application logic took the Set Union of the items.

  • State 0: Cart contains [Soap, Book, DVD].
  • Device A: User removes Book. Cart A is now [Soap, DVD].
  • Device B (Concurrent): User removes DVD. Cart B is now [Soap, Book].
  • The Merge: The system merges the siblings by combining all items present in either list. [Soap, DVD] U [Soap, Book] = [Soap, Book, DVD].
  • The Bug: Both deleted items reappeared in the customer's cart!

The Distributed Systems Fix: You cannot just track the current state. You must track the operations (Tombstones). A proper algorithm tracks that Book and DVD were explicitly deleted, resolving the cart to [Soap].

5. System Internals: CRDTs vs. Operational Transformation (OT)

If two users concurrently edit the string "ice":

  • User 1 prepends n → "nice"
  • User 2 appends ! → "ice!"

How does the system correctly arrive at "nice!"?

Operational Transformation (OT)

  • Used in: Google Docs.
  • Mechanism: Relies on index positions.
    • Op 1: Insert n at Index 0.
    • Op 2: Insert ! at Index 3.
  • The Problem: If Op 1 applies first, the string is nice. If we blindly apply Op 2 (Insert ! at Index 3), we get nic!e (Wrong!).
  • The Transformation: The OT server detects that Op 1 shifted the string by 1 character. It mathematically transforms Op 2 to: Insert ! at Index 4. Result: nice!.

Conflict-Free Replicated Data Types (CRDTs)

  • Used in: Riak, Redis Enterprise, Cosmos DB, Figma, Automerge.
  • Mechanism: Abandons indexes entirely. Every character gets a globally unique, immutable ID (e.g., i = 1A, c = 2A, e = 3A).
  • The Operation: Insert n before 1A. Insert ! after 3A.
  • The Merge: Because operations reference immutable IDs instead of shifting array indexes, they can be applied in any order across any number of replicas, and they will perfectly converge without needing a centralized transformation server.

6. Practical Backend Engineering Perspective

  • Riak & Cassandra (LWW vs Siblings): Cassandra defaults to strict Last Write Wins (based on microsecond timestamps). You will lose data if you aren't careful. Riak allows you to configure allow_mult=true, which returns Siblings to your app to resolve manually.
  • Counters (Likes/Views): Never use UPDATE table SET likes = 5. If two nodes do this concurrently, you lose a like. Use a CRDT PN-Counter (Positive-Negative Counter), or log the operations: UPDATE table SET likes = likes + 1. The database will easily merge mathematical increments.
  • Meeting Room Booking (Multi-Row Conflicts): The hardest conflict to detect. If User A books Room 1 at 10 AM, and User B books Room 1 at 10 AM on different leaders. Neither modifies the same row (they are inserting new booking rows). But they violate a global constraint ("No overlapping bookings"). Multi-Leader databases cannot natively prevent this. You must use a Single-Leader for inventory/booking systems.

7. Important Interview Questions

Basic

Q: What is the difference between Conflict Avoidance and Conflict Resolution?
A: Conflict avoidance routes all writes for a specific record to a single leader to prevent concurrent writes. Conflict resolution accepts that concurrent writes will happen and uses strategies like LWW, Siblings, or CRDTs to merge them after the fact.

Intermediate

Q: Explain why "Last Write Wins" (LWW) is dangerous in distributed databases.
A: LWW relies on physical server clocks (timestamps) to determine which write "wins." Because of NTP clock skew, a server with a fast clock will overwrite data from a server with an accurate clock, leading to permanent, silent data loss.

Advanced

Q: How did Amazon's original Dynamo shopping cart resurrect deleted items, and how would you fix it?
A: It resurrected items because the merge logic took the set-union of two concurrent cart states (Siblings). If Cart A deleted X, and Cart B deleted Y, the union restored both X and Y. To fix it, you must use a CRDT (like an Observed-Remove Set or OR-Set) that explicitly stores "Tombstones" for deleted items so the merge algorithm knows to keep them deleted.

8. Common Mistakes

  • Beginner Mistake: Assuming physical timestamps (updated_at) are a reliable way to sort and merge distributed events.
  • Backend Engineer Mistake: Writing manual merge logic that only looks at the current payload without understanding the delta/operations that resulted in that payload.
  • System Design Mistake: Recommending a Multi-Leader or Leaderless architecture for an application with strict global constraints (e.g., username uniqueness, bank account balances, meeting room bookings). These require strong consistency (Single-Leader or Consensus).

9. Practical Code / Database Examples

Handling Siblings in Application Logic (Java/Spring):

// When fetching data from a database that returns Siblings (e.g., Riak/CouchDB)
public UserProfile resolveProfileConflicts(List<UserProfile> siblings) {
    if (siblings.size() == 1) return siblings.get(0);
    
    // BAD MERGE (Data Loss):
    // return siblings.get(siblings.size() - 1); 
    
    // GOOD MERGE (Deterministic Application-Level Resolution):
    UserProfile merged = new UserProfile();
    for (UserProfile sibling : siblings) {
        // Example logic: For lists, use a union. For strings, take the longest.
        merged.getTags().addAll(sibling.getTags()); 
        
        if (sibling.getBio() != null && sibling.getBio().length() > merged.getBio().length()) {
            merged.setBio(sibling.getBio());
        }
    }
    
    // Important: Write the resolved version back to the DB immediately!
    db.saveResolvedState(merged); 
    return merged;
}

10. System Design Connections

  • Event Sourcing: CRDTs and Operational Transformation share a philosophical core with Event Sourcing. If you store facts/operations ("User clicked Like", "User removed item") instead of state ("Likes = 5", "Cart = [Soap]"), you can resolve almost any conflict deterministically.
  • Microservices: If you have active-active microservices across regions, any shared state must use automatic conflict resolution. Relying on synchronous cross-region calls will destroy your SLAs.

      \                                        /

Initial State:  [1A: 'i', 2A: 'c', 3A: 'e']

Node 1 Ops: Insert 'n' BEFORE 1A
Node 2 Ops: Insert '!' AFTER 3A

Merge Execution (CRDT):
- Nodes exchange operations.
- Because operations reference absolute IDs, no index shifting is needed.
- Result converges perfectly: [New1: 'n', 1A: 'i', 2A: 'c', 3A: 'e', New2: '!']

12. Failure Analysis Section

Failure Mode Root Cause Impact Architecture Fix
Silent Data Loss Using LWW with NTP drift. User writes are permanently lost. Use logical clocks (Version Vectors) or CRDTs.
Deleted Items Reappear Naïve Set-Union sibling merge. Application state corruption. Track Deletes via Tombstones (OR-Set CRDT).
Broken Unique Constraint Multi-leader routing for user registrations. Two users claim the same @username. Conflict Avoidance: Force all registrations to a Single Primary Leader.
OT Index Corruption Concurrent text edits without a central OT server. Gibberish text (nic!e). Use CRDTs for decentralized peer-to-peer sync, or centralize the OT server.

13. Lecture Summary

  • Key Takeaway 1: In Multi-Leader systems, write conflicts are not an edge case; they are a fundamental reality of the architecture.
  • Key Takeaway 2: "Last Write Wins" (LWW) is actually "Highest Timestamp Wins," which guarantees data loss due to clock skew. Use it only when data loss is acceptable.
  • Key Takeaway 3: Automatic resolution (CRDTs / OT) is the modern gold standard for building collaborative software (Figma, Google Docs) and distributed state synchronization.

14. Revision Notes (2-Minute Review Sheet)

  • Concurrent Writes: Neither write knew about the other. Time is irrelevant.
  • Conflict Avoidance: Route same-record writes to a single leader. Fails if routing changes.
  • LWW (Last Write Wins): Relies on timestamps. Dangerous. Destroys data.
  • Siblings: DB stores all conflicting versions. App must resolve them.
  • Amazon Cart Bug: Set-union of siblings resurrects deleted items. Fix requires tombstones.
  • OT (Operational Transformation): Transforms array indexes. Used by Google Docs. Requires a central server to dictate order.
  • CRDT (Conflict-Free Replicated Data Types): Math structures using unique IDs instead of indexes. Automatically converges state. Decentralized.
  • Constraint Violations: Multi-leader cannot enforce global uniqueness (e.g., double-booked rooms). Use Single-Leader for invariants.
