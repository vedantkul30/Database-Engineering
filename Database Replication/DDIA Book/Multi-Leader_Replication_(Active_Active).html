Multi-Leader Replication (Active-Active)

📖 Engineering Mastery Notes (Source: DDIA Chapter 6 Extract)

1. Big Picture

The Limitation of Single-Leader

Single-Leader replication has a fatal flaw: It has a single point of write failure and a strict geographic bottleneck. If your leader is in us-east-1 (Virginia) and a user in Mumbai wants to write to your database, they must suffer ~200ms of network latency (Round Trip Time - RTT) crossing the Pacific and Atlantic oceans. If the submarine cables are severed, they cannot write at all.

Why Multi-Leader (Active-Active) Exists

Multi-Leader (or Bidirectional / Active-Active) replication promotes multiple nodes to accept writes simultaneously. Each leader acts as a follower to the other leaders, creating a mesh of data synchronization.

It solves two massive engineering challenges:

  1. Global Write Latency (Geo-Distribution): Users write to the region physically closest to them, masking inter-region latency.
  2. Offline-First / Disconnected Operation: The "network partition" is intentional (e.g., you edit a Google Doc on an airplane; your laptop is the "Leader").

Where Multi-Leader is Used in Production

  • Global SaaS (Figma, Notion, Google Docs): Real-time collaborative editing uses sync engines where every user's browser is technically a "Leader."
  • Uber / Lyft: Active-Active multi-region architectures ensure that if AWS us-east-1 goes down, rides in New York aren't dropped; traffic shifts to us-west-2 and those leaders accept writes.
  • Mobile Apps (Calendars, Note-taking): Local-first software where your phone's local SQLite/Realm DB is a leader syncing with cloud leaders.

The SRE Reality Check: Multi-leader is notorious for causing data corruption (conflicts). Top engineering teams view it as dangerous territory. You only use it when the business absolutely mandates global write availability or offline capabilities.

2. Core Concepts

  • Multi-Leader (Active-Active): More than one node accepts writes.
  • Synchronous vs. Asynchronous Multi-Leader: Synchronous multi-leader is practically useless (it acts exactly like Single-Leader, blocking if any network link breaks). Production multi-leader is always asynchronous.
  • Geo-Distribution: Deploying databases across physical geographic regions to survive total regional outages and reduce latency.
  • Replication Topology: The communication path defining how writes flow between multiple leaders.
  • Sync Engine: A software library (like Automerge, Yjs, PouchDB) that manages the asynchronous, multi-leader replication between client devices (laptops/phones) and the server.
  • Local-First Software: Applications designed to work entirely locally first, with network sync moved to a background process.

3. Deep Dive: Architectural Trade-offs of Multi-Region

If you deploy a multi-region Active-Active cluster, here is what changes:

Metric Single-Leader Multi-Leader
Write Performance Poor. Cross-globe writes take 100-300ms. Excellent. Writes are local (<10ms). Inter-region sync is hidden from the user.
Regional Outage Tolerance Medium. Requires a failover to a new region, blocking writes during transition. High. Survives seamlessly. Healthy regions continue independently. Offline region catches up later.
Network Partition Tolerance Poor. If connection to the leader breaks, writes fail. High. Regions operate in isolated bubbles until the link is restored.
Data Consistency Strong. Easy to enforce constraints (e.g., "Username must be unique"). Weak / Nightmare. Massive risk of conflicting writes and invariant violations.

4. Distributed Systems Perspective: Topologies

How do leaders actually pass messages to each other?

A. Circular Topology

  • Mechanism: L1 → L2 → L3 → L1.
  • Risk: High. A single node failure (L2) breaks the entire replication chain.
  • Loop Prevention: Each write is tagged with the IDs of nodes it has passed through to prevent infinite loops.

B. Star (Tree) Topology

  • Mechanism: A root node forwards to all sub-nodes.
  • Risk: High. The root node is a Single Point of Failure (SPOF) for replication routing.

C. All-to-All Topology (The Production Standard)

  • Mechanism: Every leader connects directly to every other leader.
  • Risk: Causality Violations. Because network paths have different latencies, messages can "overtake" each other.

5. The Causality Anomaly (Out-of-Order Writes)

The biggest failure mode in an All-to-All topology is Causality Violation.

Scenario:

  1. Client A inserts a row on Leader 1. (Event A)
  2. Client B sees the row, and updates it on Leader 3. (Event B)
  3. Leader 2 receives the replication streams from L1 and L3.
  4. Due to network congestion, L2 receives the UPDATE (Event B) before it receives the INSERT (Event A).

What happens? Leader 2 throws an error ("Cannot update row that does not exist") or drops the write. The databases diverge silently.
The Fix: You cannot use physical timestamps (NTP clock skew makes them unreliable). You must use Version Vectors (logical clocks) to explicitly order dependencies.

6. Practical Backend Engineering Perspective

When to absolutely avoid Multi-Leader

  • Financial Ledgers / Banking: You cannot allow Leader A and Leader B to simultaneously withdraw $100 from an account that only has $100.
  • Unique Constraints: If two users register the username @bob at the exact same millisecond on Leader US and Leader EU, both writes succeed locally. You now have a corrupted global state.

Dangerous Pitfalls (The SRE Minefield)

  • Auto-Incrementing IDs: If L1 and L2 both generate ID=5, they will collide upon syncing. Fix: Use UUIDs/ULIDs, or configure offsets (e.g., L1 generates odds, L2 generates evens).
  • Triggers: If a replicated write fires a local DB trigger, and that trigger generates another write, you risk a cascade of infinite replication loops.
  • Foreign Key Constraints: Out-of-order causality (Insert vs Update/Delete) will violently break FK constraints on replicas.

7. Beyond the Lecture: Conflict Resolution & Collaboration (Extra Mile)

This section addresses the elephant in the room regarding Sync Engines/Google Docs.

How do Google Docs or Figma actually resolve concurrent edits without destroying the file?

  1. Operational Transformation (OT): Used by Google Docs. The server mathematically transforms incoming concurrent operations based on their index positions. (Complex and relies on a central server).
  2. CRDTs (Conflict-Free Replicated Data Types): Used by Figma, Automerge, Yjs. Mathematical data structures where concurrent operations are commutative (order doesn't matter). If User A adds "X" and User B adds "Y", a CRDT guarantees both nodes converge to the exact same final state without a central server.

8. Important Interview Questions

Basic

Q: What is the main advantage of Multi-Leader replication over Single-Leader?
A: Drastically reduced write latency for geo-distributed users, and the ability to continue accepting writes even during regional network partitions or offline scenarios.

Intermediate

Q: You decide to use a multi-leader architecture for a global app. You notice that occasionally, updates to rows are failing on some leaders because the row "doesn't exist yet." What is happening?
A: This is a causality violation in an all-to-all topology. The network link carrying the UPDATE was faster than the link carrying the initial INSERT. The replica received the events out of order.

Advanced

Q: How do you handle primary key generation in a Multi-Leader database?
A: Never use standard auto-incrementing integers. They will collide when leaders sync. Use UUIDs (v4 or v7), ULIDs, or configure the database sequence generators to use distinct offsets and increments (e.g., Region A uses keys ending in 1, Region B uses keys ending in 2).

9. Common Mistakes

  • Beginner Mistake: Assuming Multi-Leader is just "Single Leader but better." It is a fundamentally different paradigm that abandons strong consistency.
  • Backend Engineer Mistake: Using physical timestamps (created_at) to figure out which concurrent write should "win" during a conflict. NTP Clocks drift. Server A's clock might be 5ms ahead of Server B's. You will incorrectly overwrite valid data.
  • System Design Mistake: Suggesting Multi-Leader for an e-commerce checkout or inventory system in an interview. (Inventory strictly requires single-leader or consensus to prevent double-spending).

10. Sync Engines & Local-First Architecture

The Paradigm Shift: Instead of treating the browser/mobile app as a "dumb client" that makes REST API calls to a DB, treat the client as a full database replica (Leader).

  • Architecture: UI ↔ Local DB (Sync Engine) ↔ Background Thread ↔ Cloud Servers.
  • Pros:
    • Zero UI latency (renders in <16ms).
    • True offline capability.
    • Declarative programming (no need to write complex retry-logic or error handling for HTTP 500s; the sync engine handles the network transparently).
  • Cons:
    • Cannot be used for massive datasets (you can't sync the whole Amazon catalog to an iPhone).
    • Heavy battery usage if background syncing is aggressive.

12. Failure Analysis Section

Failure Mode Impact Recovery / System Architecture Defense
Inter-Region Partition Trans-atlantic cable severed. US and EU cannot talk. Both regions accept writes safely. Sync engines queue changes locally. Automatically merges upon reconnection.
All-to-All Causality Drop Update arrives before Insert. System must use Version Vectors to buffer the Update in memory until the requisite Insert arrives.
Circular Link Failure Node 2 in a Ring topology crashes. Replication halts globally. Use All-to-All topologies to provide redundant routing paths.
Concurrent Edit (Conflict) Two users modify the exact same text locally while offline. System relies on CRDTs, Operational Transformation, or prompts the user to resolve the merge manually (like Git).

13. Lecture Summary

  • Key Takeaway 1: Multi-Leader is the ultimate solution for High Availability and low-latency Geo-distribution, but it completely breaks ACID constraints across the global system.
  • Key Takeaway 2: Topology matters. All-to-all is fault-tolerant but suffers from out-of-order network anomalies. Circular/Star topologies enforce order but are fragile.
  • Key Takeaway 3: The future of frontend web-apps is moving toward Local-First Software, which is fundamentally an extreme version of Multi-Leader replication using Sync Engines and CRDTs to manage conflicts.

14. Revision Notes (2-Minute Review Sheet)

  • Why use Multi-Leader? Multi-region active-active deployments (low latency), offline-first apps (Google Docs).
  • The Golden Rule: It is always Asynchronous. Sync defeats the purpose.
  • Topologies: Circular (fragile), Star (fragile), All-to-all (robust, but causes out-of-order writes).
  • Dangerous Features: Auto-increment IDs (will collide), Triggers (loops), Unique Constraints (cannot enforce globally).
  • Local-First/Sync Engines: Treat the mobile app as a Leader node. UI updates instantly. Sync happens in the background. Uses CRDTs to merge changes.
  • Causality Anomaly: UPDATE arrives before INSERT. Fix using Version Vectors (logical clocks), not timestamps.
