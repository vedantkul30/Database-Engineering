Database Replication

Production Engineering Notes

1. Big Picture

Why Replication Exists

A single database server is a single point of failure and a single point of performance limitation. Replication solves three distinct problems simultaneously:

  1. Reliability — if the primary database crashes, a replica can take over
  2. Fault Tolerance — data survives hardware failure because it exists on multiple machines
  3. Accessibility / Scalability — reads can be distributed across multiple replicas, reducing load on the primary
Single database (no replication):
┌─────────────────────────┐
│  Primary DB             │
│  Reads + Writes         │  ← Single point of failure
│  Single server RAM/CPU  │  ← Single point of bottleneck
└─────────────────────────┘

With replication:
┌─────────────────────────┐
│  Primary (Master)       │  ← Writes go here only
│  All writes             │
└────────────┬────────────┘
             │ WAL streaming
    ┌────────┴────────────────────────┐
    │                                 │
┌───▼─────────────┐      ┌───────────▼───────┐
│  Replica 1      │      │  Replica 2        │
│  (US-East)      │      │  (EU-West)        │
│  Reads only     │      │  Reads only       │
└─────────────────┘      └───────────────────┘

Where Replication Is Used

Company / SystemReplication Use Case
InstagramRead replicas for feed/profile reads (95%+ of traffic); single master for writes
GitHubMySQL master + replicas; failover to replica on master failure
NetflixCassandra multi-region replication for globally consistent content metadata
Banking systemsSynchronous replication to standby for zero data loss (RPO = 0)
YouTubeMySQL master + replicas before Vitesse sharding was added
Amazon RDSAutomated Multi-AZ standby with synchronous replication for RDS instances

2. Core Concepts

2.1 — Master / Standby Replication (Primary / Replica)

Also called: master-backup, master-follower, primary-replica, leader-follower.

The model:

  • One master/primary node accepts ALL writes (INSERT, UPDATE, DELETE, DDL like CREATE TABLE)
  • One or more standby/replica nodes receive those writes from the master and are read-only
  • Clients can read from any replica; they MUST write to the master
                    ┌──────────────────────┐
                    │  PRIMARY (Master)    │
                    │  port: 5432          │
                    │  Accepts: R + W      │
                    │  Sends: WAL stream   │
                    └──────────┬───────────┘
                               │
              ┌────────────────┼────────────────┐
              │                │                │
   ┌──────────▼──────┐ ┌───────▼──────┐ ┌──────▼────────┐
   │ Standby 1       │ │ Standby 2    │ │ Standby 3     │
   │ port: 5433      │ │ US-West      │ │ EU-Frankfurt  │
   │ Read-only       │ │ Read-only    │ │ Read-only     │
   │ "standby1"      │ │ "standby2"   │ │ "standby3"    │
   └─────────────────┘ └──────────────┘ └───────────────┘

Client writing:  → Always to PRIMARY only
Client reading:  → Can go to PRIMARY or any STANDBY
                   (PRIMARY always has latest data)
                   (STANDBY may have slight lag — eventual consistency)

Why this is the preferred model:

  • No conflict resolution needed — only one node ever accepts writes
  • Simple to reason about: one source of truth
  • Easy to scale reads: add more replicas
  • Easy to implement: Postgres WAL streaming, MySQL binlog replication, etc.

2.2 — Multi-Master Replication

Multiple nodes accept writes simultaneously. Each write from any master is propagated to all other masters.

┌──────────────────┐        ┌──────────────────┐
│   Master 1       │ ←────→ │   Master 2       │
│   US-East        │        │   US-West        │
│   Reads + Writes │        │   Reads + Writes │
└──────────────────┘        └──────────────────┘

Client A writes row X to Master 1 simultaneously with
Client B writing row X to Master 2 with different values.
→ CONFLICT. Which value wins?

Why this is avoided:

  • Write conflicts are mathematically unavoidable — two masters can receive conflicting writes for the same row
  • Conflict resolution is complex: last-write-wins? application logic? user prompt?
  • Correct conflict resolution requires understanding business domain semantics — the database can't know which write "should" win
  • Most systems with high write throughput use sharding instead

When multi-master is used:

  • Geographic distribution where network latency between regions makes single-master unacceptable for writes
  • US-East / US-West with segregated customer sets (rare conflicts between regions)
  • Active-active setups with careful application-level conflict avoidance

2.3 — Synchronous vs Asynchronous Replication

This is the most operationally important dimension of replication. It determines the trade-off between durability/consistency and write latency.

Synchronous Replication

The client's write transaction does NOT complete (commit returns to client) until the data has been written on the master AND confirmed by at least N standby replicas.

Client writes:

  Client                  Master              Standby 1
     │                      │                     │
     │── INSERT ──────────→ │                     │
     │                      │── WAL ──────────→  │
     │                      │                     │── write to disk
     │                      │ ←──── ACK ─────────│
     │ ←──── COMMIT ───────│                     │
     │ (unblocked)          │                     │

Client is BLOCKED until standby confirms.
Write latency = time to write master + network RTT + standby write time.

PostgreSQL configuration:

-- In master postgresql.conf or standby recovery config:
-- Wait for at least 1 standby before returning commit to client
synchronous_standby_names = 'FIRST 1 (standby1, standby2, standby3)';

-- Wait for any 2 standbys:
synchronous_standby_names = 'ANY 2 (standby1, standby2, standby3)';

-- Priority order (standby1 preferred, standby2 is fallback):
synchronous_standby_names = 'FIRST 1 (standby1, standby2)';

-- Wait for ALL standbys (maximum durability, maximum latency):
synchronous_standby_names = 'standby1, standby2, standby3';

Pros:

  • Zero data loss (RPO = 0) — even if master dies immediately after commit, standby has the data
  • Strong consistency — clients always read the latest committed data from any replica
  • Required for financial systems, healthcare records, any system where data loss is catastrophic

Cons:

  • Write latency increases proportionally to network distance between master and standby
  • If standby is unreachable, master BLOCKS all writes (configurable behavior)
  • Lower write throughput

Asynchronous Replication

The client's write transaction completes on the master only. The master then propagates the WAL to standbys in the background — the client does not wait.

Client writes:

  Client                  Master              Standby 1
     │                      │                     │
     │── INSERT ──────────→ │                     │
     │ ←──── COMMIT ───────│                     │
     │ (unblocked)          │── WAL ──────────→  │  ← happens asynchronously
                            │                     │    after client is already done

Pros:

  • Write latency = master write time only (no waiting for standbys)
  • Master does not block if standbys are slow or temporarily unavailable
  • High write throughput
  • Standbys can be geographically distant without impacting write performance

Cons:

  • Potential data loss — if master crashes between commit and WAL propagation, committed data is lost
  • Eventual consistency — standbys may lag behind master by milliseconds to seconds
  • RPO > 0 (Recovery Point Objective — some data may be lost on failover)

Comparison Table

DimensionSynchronousAsynchronous
Client write latencyHigher (master + network + standby)Lower (master only)
Write throughputLowerHigher
Data loss on master failureZero (RPO = 0)Possible (RPO > 0)
Read consistencyStrongEventual
Standby lagZero (by definition)Milliseconds to seconds
Master behavior if standby unreachableBlocks writes (configurable)Continues normally
Use casesBanking, healthcare, financialSocial media, analytics, CDN

2.4 — How WAL Streaming Works in PostgreSQL

The Write-Ahead Log (WAL) is the mechanism that makes replication work.

Write path on master:

  Transaction:
  BEGIN
  UPDATE users SET name = 'Alice' WHERE id = 1;
  COMMIT;

  Internally:
  Step 1: WAL record written to WAL buffer (in memory)
  Step 2: WAL record flushed to WAL file on disk (durability on master)
  Step 3: WAL record streamed to standby replica(s) over TCP
  Step 4: Standby applies WAL record to its own data files
  Step 5 (sync): Master waits for standby ACK before returning COMMIT to client
  Step 5 (async): Master returns COMMIT to client immediately; standby applies later

WAL as the unit of replication:

  • WAL records contain the exact byte-level changes to each page
  • Standby replays WAL in order — its data files become byte-for-byte identical to master
  • WAL replay is idempotent: replaying the same WAL twice produces the same result
  • This is also how crash recovery works: on restart, PostgreSQL replays uncommitted WAL
pg_stat_replication view on master — shows connected standbys:

SELECT
    application_name,   -- "standby1"
    state,              -- "streaming"
    sync_state,         -- "sync" or "async"
    sent_lsn,           -- how far master has sent
    write_lsn,          -- how far standby has written
    flush_lsn,          -- how far standby has flushed to disk
    replay_lsn,         -- how far standby has applied
    write_lag,          -- network + write lag
    flush_lag,          -- network + write + flush lag
    replay_lag          -- total lag before standby is current
FROM pg_stat_replication;

3. The Demo — Step by Step

3.1 — Setup: Two PostgreSQL Instances (Master + Standby)

Step 1: Create folder structure

mkdir -p ~/replication
# Will hold: master_data/ and standby_data/ — PostgreSQL data directories

Step 2: Start Master container

docker run \
  --name p_master \
  -p 5432:5432 \
  -v ~/replication/master_data:/var/lib/postgresql/data \
  -e POSTGRES_PASSWORD=postgres \
  -d postgres:13

# Verify running:
docker logs p_master
# Should show: "database system is ready to accept connections"

Step 3: Start Standby container

docker run \
  --name p_standby \
  -p 5433:5432 \
  -v ~/replication/standby_data:/var/lib/postgresql/data \
  -e POSTGRES_PASSWORD=postgres \
  -d postgres:13

At this point: two completely independent PostgreSQL instances with no replication. Their data directories are different — this must be fixed.

Step 4: Stop both instances

docker stop p_master p_standby
# Must stop to safely copy data files

Step 5: Copy master data directory to standby

# Back up original standby data
mv ~/replication/standby_data ~/replication/standby_data_backup

# Copy master data directory to standby location
# -r = recursive (copy entire directory tree)
cp -r ~/replication/master_data ~/replication/standby_data

# Why: Replication requires the standby to start with an IDENTICAL
# copy of the master's data. If the data files are different,
# the WAL stream cannot be applied correctly → replication errors.

Step 6: Configure Master — Allow replication connections

# Edit the pg_hba.conf in master's data directory
vim ~/replication/master_data/pg_hba.conf

# Add this line at the end:
# TYPE    DATABASE      USER        ADDRESS        METHOD
host      replication   postgres    all            md5

# This allows: any host to connect to master AS the postgres user
# in "replication" mode using MD5 password authentication.
# Production: use a dedicated replication user, restrict to specific IP.
# Edit postgresql.conf on master:
vim ~/replication/master_data/postgresql.conf

# Add/uncomment this line:
synchronous_standby_names = 'FIRST 1 (standby1)'

# Meaning: before returning COMMIT to client, wait for at least 1
# of the listed standbys ("standby1") to confirm receipt of WAL.
# "standby1" is the application_name we'll configure in the standby.

Step 7: Configure Standby — Connect to master

# Edit postgresql.conf on STANDBY:
vim ~/replication/standby_data/postgresql.conf

# Add/uncomment this line:
primary_conninfo = 'application_name=standby1 host=localhost port=5432 user=postgres password=postgres'

# application_name=standby1: must match what master expects in synchronous_standby_names
# host=localhost: master is on the same machine (exposed via docker -p 5432:5432)
# port=5432: master's exposed port
# user/password: replication authentication (use dedicated user in production)
# Create the standby.signal file in standby's data directory:
touch ~/replication/standby_data/standby.signal

# This file tells PostgreSQL: "run this instance as a standby replica, not a primary."
# Without this file, PostgreSQL would start as another independent primary.
# The file can be empty — its mere presence triggers standby mode.

Step 8: Start both instances

docker start p_master p_standby

# Check master logs:
docker logs p_master
# Should show: "standby1 is now a synchronous standby with priority 1"

# Check standby logs:
docker logs p_standby
# Should show: "started streaming WAL from the primary"

Step 9: Verify replication is working

# Connect to MASTER:
docker exec -it p_master psql -U postgres

# Check replication status:
SELECT application_name, state, sync_state, replay_lag
FROM pg_stat_replication;
# → Row for "standby1", state="streaming", sync_state="sync"
# Empty result = replication not working
# Connect to STANDBY:
docker exec -it p_standby psql -U postgres

Step 10: Test replication

-- On MASTER: Create table
CREATE TABLE test (id INTEGER, t VARCHAR(200));

-- On STANDBY: Table appears automatically
\d test
-- → Shows the table (received via WAL)

-- On MASTER: Insert data
INSERT INTO test (id, t)
SELECT FLOOR(RANDOM() * 1000), RANDOM()::TEXT
FROM GENERATE_SERIES(1, 10000);

-- On STANDBY: Data appears
SELECT COUNT(*) FROM test;
-- → 10000 (or appearing rapidly)

-- On STANDBY: Try to write (will FAIL — read-only)
CREATE TABLE test2 (id INTEGER);
-- ERROR: cannot execute CREATE TABLE in a read-only transaction

-- On MASTER: Schema change propagates
ALTER TABLE test ADD COLUMN g INTEGER;

-- On STANDBY: Column appears
\d test
-- → Shows id, t, g columns

4. Java Application Integration

Read/Write Routing Pattern

In a replicated setup, the application must route writes to the primary and reads to replicas.

import org.springframework.jdbc.core.JdbcTemplate;
import org.springframework.jdbc.datasource.DriverManagerDataSource;

/**
 * ReplicationAwareDataSource routes queries to the correct node:
 *   - Writes (INSERT, UPDATE, DELETE, DDL) → Primary only
 *   - Reads (SELECT) → Any replica (round-robin load balancing)
 *
 * In production: use a proxy like PgBouncer, HAProxy, or AWS RDS Proxy
 * instead of implementing this in application code.
 */
public class ReplicationAwareDataSource {

    // Primary — accepts all writes and reads
    private final JdbcTemplate primary;

    // Read replicas — reads only, load balanced
    private final List<JdbcTemplate> replicas;
    private final AtomicInteger replicaIndex = new AtomicInteger(0);

    public ReplicationAwareDataSource() {
        // Primary connection (master)
        this.primary = createJdbcTemplate("localhost", 5432);

        // Replica connections (standbys)
        this.replicas = List.of(
            createJdbcTemplate("localhost", 5433),  // Standby 1
            createJdbcTemplate("localhost", 5434)   // Standby 2 (if present)
        );
    }

    /**
     * Get the primary JdbcTemplate for write operations.
     * ALL writes must go here.
     */
    public JdbcTemplate getPrimary() {
        return primary;
    }

    /**
     * Get a replica JdbcTemplate for read operations.
     * Round-robin load balancing across replicas.
     *
     * If no replicas available, falls back to primary.
     */
    public JdbcTemplate getReplica() {
        if (replicas.isEmpty()) {
            return primary;  // Fallback to primary
        }
        // Round-robin: 0, 1, 0, 1, 0, 1, ...
        int index = replicaIndex.getAndIncrement() % replicas.size();
        return replicas.get(index);
    }

    private JdbcTemplate createJdbcTemplate(String host, int port) {
        DriverManagerDataSource ds = new DriverManagerDataSource();
        ds.setUrl("jdbc:postgresql://" + host + ":" + port + "/postgres");
        ds.setUsername("postgres");
        ds.setPassword("postgres");
        return new JdbcTemplate(ds);
    }
}
/**
 * UserService demonstrates write-to-primary, read-from-replica routing.
 */
@Service
public class UserService {

    private final ReplicationAwareDataSource dataSource;

    public UserService(ReplicationAwareDataSource dataSource) {
        this.dataSource = dataSource;
    }

    /**
     * WRITE operation → always uses PRIMARY.
     * If sent to a replica: ERROR "cannot execute INSERT in a read-only transaction"
     */
    public void createUser(String name, String email) {
        dataSource.getPrimary().update(
            "INSERT INTO users(name, email) VALUES(?, ?)",
            name, email
        );
        // Write is:
        // 1. Applied to primary's WAL
        // 2. (Sync) Confirmed by standby before returning
        // 3. (Async) Returned immediately; standby applies later
    }

    /**
     * READ operation → routes to a replica.
     * Replica may be slightly behind primary (async replication lag).
     * Acceptable for: profile views, feeds, search, analytics.
     * NOT acceptable for: "read your own write" scenarios.
     */
    public List<Map<String, Object>> searchUsers(String namePrefix) {
        return dataSource.getReplica().queryForList(
            "SELECT id, name, email FROM users WHERE name LIKE ?",
            namePrefix + "%"
        );
    }

    /**
     * READ YOUR OWN WRITE pattern — use primary for reads immediately
     * after a write to guarantee you see what you just wrote.
     *
     * Example: after updating profile, redirect to profile page —
     * that read should go to primary to avoid seeing stale data.
     */
    public Map<String, Object> getUserAfterUpdate(long userId) {
        // Must read from primary — replica may not have applied the update yet
        return dataSource.getPrimary().queryForMap(
            "SELECT * FROM users WHERE id = ?",
            userId
        );
    }
}

Monitoring Replication Lag in Java

/**
 * ReplicationMonitor checks replication health and lag.
 * Run periodically (every 30 seconds) to detect replication issues.
 */
@Component
public class ReplicationMonitor {

    private final JdbcTemplate primary;

    @Scheduled(fixedRate = 30_000)
    public void checkReplicationHealth() {
        List<Map<String, Object>> replicas = primary.queryForList("""
            SELECT
                application_name,
                state,
                sync_state,
                write_lag,
                flush_lag,
                replay_lag,
                pg_size_pretty(pg_wal_lsn_diff(sent_lsn, replay_lsn)) AS lag_size
            FROM pg_stat_replication
            ORDER BY replay_lag DESC
            """);

        if (replicas.isEmpty()) {
            log.error("CRITICAL: No standbys connected to primary! Replication may be broken.");
            // Alert: PagerDuty / Slack / email
            return;
        }

        for (Map<String, Object> replica : replicas) {
            String name = (String) replica.get("application_name");
            Object replayLag = replica.get("replay_lag");

            log.info("Replica {}: state={}, sync={}, replay_lag={}",
                name,
                replica.get("state"),
                replica.get("sync_state"),
                replayLag
            );

            // Alert if replica is significantly behind
            // In production: parse the interval and threshold
            // Example threshold: lag > 10 seconds → alert
        }
    }
}

5. Pros and Cons of Replication

Pros

1. Horizontal Read Scalability

Without replicas: All reads + writes → single server
  Server handles: 10,000 reads/sec + 1,000 writes/sec

With 3 read replicas: Reads distributed across 4 nodes
  Primary: 1,000 writes/sec + ~2,500 reads/sec (25% of reads)
  Replica 1: ~2,500 reads/sec
  Replica 2: ~2,500 reads/sec
  Replica 3: ~2,500 reads/sec
  Total read capacity: ~10,000 reads/sec on primary → ~40,000 reads/sec

2. Fault Tolerance / High Availability

If the primary fails, a standby can be promoted to primary. Read traffic continues on surviving replicas during failover.

Failure scenario:
  Primary dies at 14:32:05
  Ops/automation detects failure by 14:32:10 (5 seconds)
  Standby 1 promoted to primary by 14:32:15 (10 seconds total)
  Application reconnects to new primary by 14:32:20

Result: ~15 seconds of write unavailability, reads continue on other standbys.

3. Region-Based Query Routing

Architecture:
  Primary:   AWS us-east-1 (Virginia)   — handles writes from all regions
  Standby 1: AWS eu-west-1 (Ireland)    — handles reads for European users
  Standby 2: AWS ap-southeast-1 (Singapore) — handles reads for Asian users

European user reads their profile:
  → Request goes to eu-west-1 standby (< 10ms latency)
  vs → Request goes to us-east-1 primary (> 100ms latency)

Result: 10× better read latency for European users.
Trade-off: European users may see writes committed 50-200ms after they happen
           (asynchronous replication lag across regions).

4. Backup Without Impacting Primary

-- Run pg_dump against a replica, not the primary
-- This avoids placing I/O load from backup on the primary
pg_dump -h replica-host -p 5433 -U postgres mydb > backup.sql

-- Run long analytical queries against replica
SELECT region, SUM(revenue) FROM orders
GROUP BY region;
-- → Runs on replica, doesn't slow down primary serving user traffic

Cons

1. Eventual Consistency

Timeline (asynchronous replication, 50ms lag):

T=0ms:  User submits a post (write to primary)
T=0ms:  Primary commits → returns OK to user
T=50ms: WAL propagated to replica
T=51ms: User refreshes their feed (read from replica)
         → Post NOT visible yet (replica 1ms behind)
T=100ms: User refreshes again
          → Post NOW visible (replica caught up)

This is "eventual consistency." The data will be consistent, eventually.

When this is acceptable:

  • Social media likes, follower counts
  • Product view counts
  • Comment counts
  • Any metric where slight delay is imperceptible

When this is NOT acceptable:

  • Bank balance after a transfer
  • Inventory count after a purchase
  • Order status after payment
  • Any "read your own write" scenario

2. Slower Writes (Synchronous Mode)

Synchronous replication write latency breakdown:

                  Master write:   5ms
            Network to standby:   3ms  (same data center)
               Standby write:     4ms
            ACK back to master:   3ms  (same data center)
                               ──────
              Total for client:  15ms  vs 5ms without sync replication

With cross-region standbys (US-East → EU-West):
              Network (one way):  100ms
              Network (return):   100ms
              Total overhead:    200ms+ per write!

This is why synchronous replication to remote regions is almost never used.
Typical setup: synchronous to same-DC standby, async to remote replicas.

3. Multi-Master Conflict Resolution Complexity

Conflict scenario:

T=0:   User changes email to "alice@gmail.com" on Master 1 (US-East)
T=0:   Same user changes email to "alice@yahoo.com" on Master 2 (US-West)
       (Both happen at the same millisecond — network partition)
T=5ms: Masters sync. CONFLICT: which email is correct?

Resolution strategies:
  Last-Write-Wins (LWW): Use timestamp → but clocks are never perfectly synchronized
  Application logic: Route all writes for user X to Master 1 (defeats the purpose)
  CRDTs: Special data structures that merge automatically (complex, limited types)
  User resolution: Ask the user → unacceptable UX

No perfect solution. This is why multi-master is avoided.

4. Replication Lag Under Heavy Write Load

Problem: Primary is processing 50,000 writes/second.
         Standby can only apply 40,000 WAL records/second.
         Lag grows by 10,000 records/second.

After 1 minute: 600,000 records of lag.
After 1 hour: 36,000,000 records of lag.

The standby is now 1 hour behind the primary.
Reads from standby return data that is 1 hour stale.

Mitigation:
  - Synchronous replication: forces standby to keep up (at cost of write speed)
  - More powerful standby hardware
  - Reduce write load (caching, batching)
  - Multiple standbys each handling a subset of WAL

6. Replication vs Other Scaling Techniques

Complete Scaling Strategy (The Right Order)

Step 1: Single server + indexes
        Handles: read-heavy, < 1M rows, < 1,000 concurrent users

Step 2: Single server + indexes + partitioning
        Handles: > 10M rows, index doesn't fit in RAM

Step 3: Single primary + read replicas
        Handles: read-heavy (90%+ reads), global user base
        Cost: eventual consistency for reads

Step 4: Single primary + replicas + caching (Redis)
        Handles: extremely hot reads (same data read millions of times)
        Cost: cache invalidation complexity

Step 5: Multi-master replication (same region)
        Handles: write-heavy, acceptable conflict resolution
        Cost: conflict resolution logic

Step 6: Sharding
        Handles: write throughput beyond any single server
        Cost: no cross-shard ACID, no JOINs, massive complexity

7. Key Configuration Reference (PostgreSQL)

Master postgresql.conf

# Enable WAL replication
wal_level = replica          # Minimum for streaming replication
                              # Options: minimal, replica, logical

max_wal_senders = 10         # Max number of standby connections
                              # Set to at least the number of standbys + 1

wal_keep_size = 1GB          # Keep at least 1GB of WAL on disk
                              # Prevents standbys from falling too far behind
                              # and losing their replication slot

# Synchronous replication config
# FIRST 1: wait for the highest-priority standby
# ANY 2: wait for any 2 standbys
# '' (empty): asynchronous mode (default)
synchronous_standby_names = 'FIRST 1 (standby1, standby2)'

Master pg_hba.conf

# Allow standby servers to connect for replication
# In production: replace 'all' with the specific standby IP/subnet
host    replication    replication_user    0.0.0.0/0    md5

Standby postgresql.conf

# Connection string to primary
# application_name: must match what master expects in synchronous_standby_names
primary_conninfo = 'application_name=standby1 host=primary-host port=5432 user=replication_user password=secret'

# How far standby can lag before it pauses (optional safety valve)
# recovery_min_apply_delay = '30s'  # Delayed standby (useful for human error recovery)

standby.signal

# Simply create this empty file in the standby's data directory:
touch /var/lib/postgresql/data/standby.signal

# This is the ONLY trigger needed to start PostgreSQL in standby mode.
# Without it, the instance starts as an independent primary.
# When standby is promoted to primary, this file is automatically removed.

Monitoring Queries

-- On PRIMARY: Check connected standbys and their lag
SELECT
    application_name,
    state,          -- 'streaming' = active, 'catchup' = falling behind
    sync_state,     -- 'sync', 'async', 'potential'
    write_lag,      -- Time for WAL to reach standby
    flush_lag,      -- Time for WAL to be flushed to standby disk
    replay_lag,     -- Time for WAL to be applied on standby
    pg_size_pretty(
        pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn)
    ) AS bytes_behind
FROM pg_stat_replication;

8. Important Interview Questions

Basic

Q: What is database replication?
A: Replication is the process of continuously copying data from one database server (the primary/master) to one or more other servers (standbys/replicas). It ensures multiple servers hold the same data, providing redundancy against failure, distributing read load across replicas, and enabling geographic distribution of data.

Q: What is the difference between master-standby and multi-master replication?
A: In master-standby, only the primary accepts writes. Standbys are read-only and receive writes from the primary via WAL streaming. No conflicts are possible — simple and reliable. In multi-master, multiple nodes accept writes simultaneously. Conflicts (same row updated differently on two masters) are unavoidable and require complex conflict resolution logic. Most systems avoid multi-master due to this complexity.

Q: What is the difference between synchronous and asynchronous replication?
A: Synchronous: the client's write transaction doesn't complete until at least one standby confirms receipt of the WAL. Zero data loss, strong consistency, higher write latency. Asynchronous: the client's write completes on the primary immediately; standbys receive WAL in the background. Lower write latency, potential data loss if primary fails before WAL is propagated, eventual consistency on standbys.

Q: What is a standby.signal file in PostgreSQL?
A: An empty file placed in the PostgreSQL data directory that signals the instance to start in standby (read-only replica) mode instead of as an independent primary. Its presence alone triggers standby behavior — the file content doesn't matter. When the standby is promoted to primary, PostgreSQL automatically removes this file.

Q: What is WAL and how does it enable replication?
A: WAL (Write-Ahead Log) is PostgreSQL's durability mechanism. Every data modification is first written to the WAL before being applied to data files. For replication, the master streams its WAL to standbys over a TCP connection. Standbys replay the WAL in order, applying the same changes to their own data files. The result: standbys maintain a byte-for-byte copy of the master's data, slightly delayed by network and processing time.

Intermediate

Q: A client writes a record and immediately reads it back. With asynchronous replication, what can go wrong, and how do you fix it?
A: The read may be routed to a replica that hasn't yet applied the WAL for that write. The client sees stale data — the record they just wrote appears not to exist. This is the "read your own write" problem. Solutions: (1) Route the read to the primary (always up-to-date). (2) Sticky sessions — route all requests from a given user to the same replica for a time window. (3) Include a write timestamp in the response; replica checks if it has applied WAL up to that timestamp before serving the read. Option 1 is simplest and most reliable.

Q: In the demo, why was it necessary to copy the master's data directory to the standby before starting replication?
A: PostgreSQL WAL streaming replication works by the standby applying WAL records on top of an existing identical baseline. The WAL records describe byte-level changes ("write bytes X at page Y"). If the standby's data files don't match the master's baseline, the WAL operations produce incorrect results — like applying a diff to the wrong version of a file. The initial data copy ensures both instances start from the same state. In production, this is done with pg_basebackup instead of a filesystem copy (safe for live systems).

Q: What does synchronous_standby_names = 'FIRST 1 (standby1, standby2)' mean?
A: It tells the master: before returning a commit to the client, wait for at least 1 of the listed standbys (in priority order: standby1 first, standby2 as fallback) to confirm receipt and write of the WAL. FIRST 1 means the highest-priority available standby. If you wrote ANY 2, any 2 of the listed standbys must confirm. The application_name in each standby's primary_conninfo must match the names listed here.

Q: What is replication lag and what causes it?
A: Replication lag is how far a standby is behind the primary in applying WAL changes. Causes: (1) Network latency between primary and standby. (2) Standby disk I/O can't keep up with primary's write rate. (3) Standby CPU overloaded (running heavy queries while applying WAL). (4) Network packet loss or disconnection requiring re-connection. Consequences: stale reads from standbys, potential data loss window on failover (larger lag = more potential data loss).

Advanced

Q: Walk through exactly what happens — at the WAL level — when a transaction commits in synchronous replication mode.
A: (1) Client sends COMMIT. (2) Primary writes WAL record for the transaction to WAL buffer. (3) Primary flushes WAL buffer to WAL file on disk (durability on primary). (4) Primary sends WAL record to standby(s) over the walsender process via TCP. (5) Standby's walreceiver process receives WAL, writes it to standby WAL buffer, flushes to standby WAL file, sends ACK to primary. (6) Primary's walsender receives ACK, updates flush_lsn in pg_stat_replication. (7) Primary checks if the synchronous commit condition is satisfied (e.g., FIRST 1 ACK received). (8) Primary returns COMMIT success to client. Only now is the client unblocked.

Q: Design the replication topology for a global e-commerce platform (write-heavy checkout, read-heavy browse, EU GDPR requirements, 99.99% uptime SLA).
A:
- Primary: AWS us-east-1, handles all writes globally
- Synchronous standby: AWS us-east-1b (same region, different AZ) — zero data loss on primary failure, sub-millisecond lag, auto-failover
- Asynchronous read replicas: us-west-2 (West Coast users), eu-west-1 (EU users — also satisfies GDPR data residency for EU customer reads), ap-southeast-1 (Asia users)
- GDPR compliance: EU customer writes still go to us-east-1 primary; a separate EU-only cluster with its own primary handles EU customer data for full GDPR isolation (data never leaves EU)
- Read routing: Application uses PgBouncer connection pooler; reads for browsing, search, recommendations go to nearest replica; checkout reads (inventory, price) go to primary
- Failover: Patroni (or AWS RDS Multi-AZ) monitors primary; promotes synchronous standby in <30 seconds; DNS failover redirects application connections

Q: What are the database-level implications of promoting a standby to primary?
A: (1) standby.signal file is removed from the data directory — instance transitions from recovery mode to primary mode. (2) PostgreSQL begins accepting write transactions. (3) The old primary (if it recovers) must be fenced — otherwise you have split-brain (two nodes believing they are primary). (4) The promoted standby may have slightly less data than the original primary if async replication had lag — this data is permanently lost. (5) Other standbys must be reconfigured to follow the new primary (update primary_conninfo). (6) In Postgres 12+, pg_promote() function can trigger promotion from within a SQL session. In production: use Patroni, Repmgr, or cloud provider automation (AWS RDS Multi-AZ) to handle this automatically.

9. Common Mistakes

Beginner Mistakes

  • Reading from primary for everything: defeats the purpose of read replicas; all read load still falls on primary
  • Not monitoring replication lag: replicas fall behind silently; users see increasingly stale data without any alert
  • Using the postgres superuser for replication: security risk — create a dedicated replication user with only REPLICATION privilege 
    CREATE USER replicator WITH REPLICATION LOGIN PASSWORD 'strong_password';

Backend Engineer Mistakes

  • Routing writes to replicas: ERROR: cannot execute INSERT in a read-only transaction — entire feature breaks
  • "Read your own write" without sticky primary: user submits a form, immediately sees the old value because the read hit a lagging replica
  • Not handling replica unavailability: if a replica goes down, application must fall back to primary for reads — hardcoded replica connections cause failures
  • Assuming synchronous replication = zero performance impact: synchronous replication to a same-DC standby adds ~5-15ms. To a cross-region standby, it adds 100-300ms per write — absolutely unacceptable for interactive applications

Production Mistakes

  • wal_keep_size too small: if a standby disconnects briefly and reconnects, the WAL segment it needs may have already been recycled — standby must be rebuilt from scratch
  • No automated failover: primary fails at 3 AM; engineers manually promote a standby 45 minutes later — 45 minutes of downtime. Use Patroni, Repmgr, or cloud-managed replication (RDS Multi-AZ, Cloud SQL HA)
  • Not testing failover: replication looks healthy on dashboards but the failover procedure has never been practiced — when primary actually fails, the process breaks
  • Forgetting to fence the old primary: after promoting a standby, the original primary recovers and starts accepting writes again — split-brain, data divergence, potential data loss and corruption
  • Using replication as a backup strategy: replication propagates DELETE, TRUNCATE, and DROP TABLE immediately to all replicas — a human error deleting a table is replicated in milliseconds. Replication ≠ backup. Always maintain point-in-time recovery (PITR) backups separately.

10. Performance & Scalability Notes

Read Scalability with Replicas

Single server capacity (8-core, 64GB RAM, NVMe SSD):
  Read throughput:  ~50,000 queries/second
  Write throughput: ~10,000 writes/second

With 5 read replicas (reads distributed):
  Read throughput:  ~250,000 queries/second (5×)
  Write throughput: ~10,000 writes/second (unchanged — writes still to single primary)
  Primary read load: reduced by ~80% (reads go to replicas)

Important: write throughput does NOT scale with replicas.
For write scale → sharding (much more complex).

Replication Lag Budget

Acceptable lag depends on use case:

Use case                     Max acceptable lag
────────────────────────────────────────────────
Social media feed             5 seconds
Product catalog               1 second
Inventory count (high-value)  100 milliseconds
Bank balance                  0 (use primary)
Order status after payment    0 (use primary)
Analytics dashboard           5 minutes

Design your routing strategy based on acceptable lag per query type.

WAL Volume and Network

Write-heavy workload: 10,000 rows/second
Typical WAL record size: ~100 bytes per row change
WAL generated: ~1 MB/second

Network requirements for WAL streaming:
  Same data center:  1 MB/s easily handled
  Cross-region:      1 MB/s = ~8 Mbps (fine on 1Gbps links)
  
Spike workload: 100,000 rows/second
WAL generated: ~10 MB/second
Cross-region: ~80 Mbps — monitor network utilization

11. System Design Connections

Replication + Load Balancing

Application → PgBouncer/HAProxy → Primary (writes)
                                → Replica 1 (reads)
                                → Replica 2 (reads)
                                → Replica 3 (reads)

HAProxy health check: SELECT pg_is_in_recovery()
→ false = primary (route writes here)
→ true  = standby (route reads here)

Replication + Caching

Hot data path (social media likes):

Write:  App → Primary → [replicated] → Replicas
              Also invalidate Redis cache key

Read:   App → Redis cache → HIT: return cached value (fastest)
                           → MISS: read from Replica → cache in Redis

Result:
  Redis handles 99% of reads (microseconds)
  Replicas handle cache misses (milliseconds)
  Primary handles writes only

Replication + Failover Automation (Patroni)

Patroni topology:

  ┌──────────────────────────────────────────────┐
  │  etcd cluster (distributed configuration)    │
  │  Stores: who is primary, health status        │
  └──────────────────────────────────────────────┘
              ↑ heartbeat every 10s
  ┌───────────┼────────────────────────────┐
  │           │                            │
  ┌───▼─────────────┐         ┌────────────▼───┐
  │  Patroni + PG   │         │  Patroni + PG  │
  │  (Primary)      │         │  (Standby)     │
  └─────────────────┘         └────────────────┘

Primary fails → etcd loses heartbeat after 30s
→ Patroni on standby promotes itself
→ Updates etcd: "I am now primary"
→ HAProxy re-reads etcd → routes writes to new primary
→ Total failover time: 30-60 seconds, automatic
