Transaction Isolation

Complete Learning Notes

1. Big Picture

Why Does Isolation Exist?

Databases are not single-user systems. In any production application, hundreds or thousands of concurrent TCP connections hit the database simultaneously. Every connection runs transactions. When multiple transactions touch the same data at the same time, a fundamental question arises:

Can a transaction see changes made by other in-flight or recently committed transactions?

Without isolation rules, transactions would freely interfere with each other — producing incorrect, inconsistent, and often dangerous results. Isolation is the "I" in ACID and it defines the rules of engagement between concurrent transactions.

The Problem It Solves

Without IsolationWith Isolation
Transactions read each other's uncommitted garbageDefined visibility boundaries
Aggregations produce wrong totals mid-flightConsistent snapshots
Concurrent writes silently overwrite each otherConflict detection and prevention
Reports are internally inconsistentRepeatable, coherent reads

Where It Is Used in Real Systems

  • Financial systems — account balance reads must never include uncommitted debits/credits
  • E-commerce — inventory deductions must not be double-counted during flash sales
  • Reporting pipelines — aggregate reports must not mix data from different points in time
  • Trading systems — order book reads must be consistent within a single decision cycle
  • Ride-sharing (Uber) — driver assignment must not be double-booked by two concurrent requests

Why Backend Engineers Must Care

  • Choosing the wrong isolation level is a silent data corruption bug — no exceptions thrown, no logs, just wrong numbers
  • Isolation directly impacts throughput and latency — the stricter the isolation, the more expensive it is
  • Most ORMs default to READ COMMITTED — understanding what that means prevents nasty surprises
  • Debugging concurrency bugs without isolation knowledge is nearly impossible

2. Core Concepts

The Central Question

Transaction T1 is running.
Transaction T2 makes a change (committed or not).
Should T1 see that change?

The answer is: it depends on the isolation level configured. There is no universal right or wrong — it is a trade-off between consistency and performance.

Read Phenomena (The Bad Side Effects)

Read phenomena are undesirable behaviors that can occur when concurrent transactions interact without sufficient isolation. They are the problems that isolation levels are designed to solve.

  1. Dirty Read
  2. Non-Repeatable Read
  3. Phantom Read
  4. Lost Update

Dirty Read

Definition: A transaction reads data written by another transaction that has not yet committed. That uncommitted data is "dirty" because it may be rolled back.

Example:

Sales Table:
| product_id | quantity | price |
|------------|----------|-------|
| 1          | 10       | 5     |
| 2          | 20       | 4     |

T1 (Report Generator):
  BEGIN;
  SELECT product_id, quantity * price FROM sales;
  -- Reads: product_1 = 50, product_2 = 80

    T2 (Sales Update — CONCURRENT):
      BEGIN;
      UPDATE sales SET quantity = 15 WHERE product_id = 1;
      -- NOT YET COMMITTED

T1 (still running):
  SELECT SUM(quantity * price) FROM sales;
  -- With dirty read: reads quantity=15 for product_1
  -- Result: (15 * 5) + 80 = 155   ← WRONG
  -- Expected: (10 * 5) + 80 = 130

T1:
  COMMIT;

    T2:
      ROLLBACK;  -- The dirty data never actually existed!

Why this is catastrophic: T1 read data that was eventually rolled back. The data never truly existed in the committed state. The report shows 155 — a number that has no basis in reality.

Non-Repeatable Read

Definition: Within the same transaction, the same row is read twice but returns different values because another transaction committed a change between the two reads.

Key distinction from Dirty Read: The interfering transaction actually commits before the second read. So the data is real — but T1's view of the world is inconsistent within itself.

Example:

T1 (Report Generator):
  BEGIN;
  SELECT product_id, quantity * price FROM sales;
  -- Reads: product_1 = 50, product_2 = 80

    T2 (Sales Update — CONCURRENT):
      BEGIN;
      UPDATE sales SET quantity = 15 WHERE product_id = 1;
      COMMIT;  ← Committed before T1's next read

T1 (still running):
  SELECT SUM(quantity * price) FROM sales;
  -- Now reads quantity=15 (committed by T2)
  -- Result: (15 * 5) + 80 = 155   ← WRONG WITHIN T1's context
  -- T1 saw 50 first, now sees 75 via sum — inconsistent
  COMMIT;

Why it matters: T1 read product_1's revenue as 50 in query 1, but the sum in query 2 implicitly re-reads it as 75. These are two different queries, but they read the same underlying data. The inconsistency is subtle and hard to catch.

How databases fix this:

DatabaseApproach
PostgreSQLMVCC — keeps old row versions. T1 always reads its snapshot.
MySQL / OracleUndo log — stores previous values. T1 re-reads original value from undo log.
SQL ServerDepends on isolation level; can use row versioning or locks.

Phantom Read

Definition: A transaction executes a range query twice. Between the two executions, another transaction inserts a new row that satisfies the range condition. The new row "appears from nowhere" on the second read.

Key distinction: You cannot lock a row that doesn't exist yet. Phantom reads involve rows that were never read by T1 — they were inserted after T1's first query.

Example:

T1 (Report Generator):
  BEGIN;
  SELECT quantity, price FROM sales;
  -- Reads: product_1 = (10, 5), product_2 = (20, 4)
  -- Total rows: 2

    T2 (New Product — CONCURRENT):
      BEGIN;
      INSERT INTO sales (product_id, quantity, price) VALUES (3, 10, 1);
      COMMIT;  ← Committed

T1 (still running):
  SELECT SUM(quantity) FROM sales;
  -- Now picks up product_3 (quantity=10)
  -- Result: 10 + 20 + 10 = 40   ← WRONG
  -- Expected: 10 + 20 = 30
  COMMIT;

Why it's different from Non-Repeatable Read:

  • Non-repeatable read: An existing row changed.
  • Phantom read: A new row appeared that satisfies the query's filter.

Why it's hard to prevent:

  • With non-repeatable reads, the database can lock the rows that were read.
  • With phantom reads, you cannot lock a row you've never seen. The row didn't exist when T1 first ran its query.
  • Solution requires range locks or snapshot isolation.

Lost Update

Definition: Two transactions read the same value, both compute a new value based on it, and both write back — the second write silently overwrites the first, making the first update disappear.

Example:

Sales Table: product_1 quantity = 10

T1 (Adds 10 units):           T2 (Adds 5 units):
  BEGIN;                         BEGIN;
  -- Both read quantity = 10     -- Both read quantity = 10
  -- T1 computes: 10 + 10 = 20   -- T2 computes: 10 + 5 = 15
  UPDATE sales                   UPDATE sales
    SET quantity = 20              SET quantity = 15
    WHERE product_id = 1;          WHERE product_id = 1;
  COMMIT;                        COMMIT;
                                 -- Final value: 15 ← WRONG
                                 -- Expected: 25

Why this is devastating: T1's entire update is silently discarded. No error is raised. The database happily accepts both writes. The final value 15 reflects only T2's change.

Solution: Row-level locking with SELECT FOR UPDATE, or use optimistic concurrency control with version checks.

3. Isolation Levels

Isolation levels are the database's answer to "which read phenomena do you want to prevent?" Each level trades off data consistency against performance and concurrency.

The Four Standard SQL Isolation Levels

READ UNCOMMITTED (Lowest / Fastest)

  • What it means: A transaction can read data written by other transactions, even if those transactions have NOT committed.
  • Phenomena allowed: Dirty Read, Non-Repeatable Read, Phantom Read, Lost Update
  • Use case: Almost never. Theoretically fastest. SQL Server supports it; most others don't bother.
  • Why it's dangerous: You can read data that gets rolled back — you are reading lies.
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
BEGIN;
SELECT * FROM sales; -- May return uncommitted garbage
COMMIT;

READ COMMITTED (Most Common Default)

  • What it means: Each query within a transaction only sees data that has been committed before that query runs.
  • Phenomena prevented: Dirty Read ✅
  • Phenomena allowed: Non-Repeatable Read ⚠️, Phantom Read ⚠️, Lost Update ⚠️
  • Default in: PostgreSQL, Oracle, SQL Server, DB2
  • Optimized for: Most database engines are heavily optimized for this workload.
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
BEGIN;
SELECT * FROM sales;         -- Reads committed data at this moment
-- T2 commits an update here
SELECT SUM(quantity) FROM sales; -- May now see T2's change ← non-repeatable
COMMIT;

Production impact: This is the default. If you have a long-running report transaction, it will pick up any committed changes mid-flight. This is often fine, but for financial aggregations it can produce subtly wrong results.

REPEATABLE READ

  • What it means: Any row that a transaction reads will not change for the duration of that transaction. The database guarantees that if you read a row once, re-reading it (or reading it through an aggregation) will yield the same value.
  • Phenomena prevented: Dirty Read ✅, Non-Repeatable Read ✅, Lost Update ✅
  • Phenomena allowed: Phantom Read ⚠️ (in most databases — see PostgreSQL exception below)
  • Default in: MySQL InnoDB

PostgreSQL special behavior: PostgreSQL implements REPEATABLE READ as Snapshot Isolation. This means it also prevents Phantom Reads. PostgreSQL's REPEATABLE READ ≡ SNAPSHOT in most other databases.

SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
BEGIN;
SELECT * FROM sales;         -- T1 sees a consistent snapshot
-- T2 inserts product_3 and commits
SELECT SUM(quantity) FROM sales; -- In MySQL: sees product_3 (phantom!)
                                 -- In PostgreSQL: does NOT see product_3 ✅
COMMIT;

SNAPSHOT ISOLATION

  • What it means: At the start of the transaction, the database takes a "snapshot" of all committed data. Every read within the transaction sees exactly that snapshot — regardless of what other transactions commit during the transaction's lifetime.
  • Phenomena prevented: All read phenomena ✅
  • Note: Not a standard SQL isolation level, but supported by PostgreSQL (as REPEATABLE READ), SQL Server (as SNAPSHOT), and Oracle.
T1 starts at time T=0. Snapshot = state of DB at T=0.
T2 commits at T=5 (inserts/updates).
T1 reads at T=10 — still sees the T=0 snapshot.

Implementation: Uses MVCC (Multi-Version Concurrency Control). Each row has version metadata. Reads always return the version visible to the transaction's snapshot timestamp.

SERIALIZABLE (Highest / Slowest)

  • What it means: Transactions execute as if they were run one after another in serial order — no concurrency at all from the data consistency perspective.
  • Phenomena prevented: All read phenomena ✅ (including write-write conflicts)
  • Implementation options:
    • Pessimistic: Actual locking — one transaction blocks all others on conflicting rows.
    • Optimistic (SSI — Serializable Snapshot Isolation): Transactions proceed concurrently but the database detects conflicts at commit time and aborts the loser with a serialization error.
  • Used when: Financial ledgers, distributed coordination, anything where "wrong answer" is unacceptable.
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
BEGIN;
SELECT * FROM orders WHERE status = 'pending';
UPDATE orders SET status = 'processing' WHERE id = 42;
COMMIT;
-- If a conflict is detected: ERROR: could not serialize access due to concurrent update
-- Application must RETRY the transaction

4. Deep Dive

Concurrency Control Approaches

Databases implement isolation levels using two fundamental strategies:

Pessimistic Concurrency Control (Locking)

Philosophy: "Conflicts are likely. Lock proactively."

The database obtains locks before accessing data and holds them until the transaction completes.

Lock Granularities:

Lock TypeWhat It LocksCost
Row LockA single rowLow (but many locks = high overhead)
Page LockA disk page (contains many rows)Medium
Table LockEntire tableHighest — blocks all concurrent access

Lock Escalation (Production Danger):

When a transaction holds too many row locks (e.g., 700,000 rows), the database may automatically escalate to a table lock to reduce memory pressure. This blocks ALL other transactions on that table — a production killer.
T1 locks row #1, row #2, ... row #700,000
Database: "Too many row locks, escalating to TABLE LOCK"
T2: UPDATE sales SET price = 10 WHERE product_id = 999;
T2 → WAITING... (even though it touches a completely different row)

How row-level locking prevents Lost Updates:

-- T1
BEGIN;
SELECT quantity FROM sales WHERE product_id = 1 FOR UPDATE; -- Acquires row lock
-- T2 tries to SELECT FOR UPDATE on same row → BLOCKS and WAITS
UPDATE sales SET quantity = quantity + 10 WHERE product_id = 1;
COMMIT; -- Lock released

-- T2 now unblocks and reads the committed value (20), then adds 5 → 25 ✅

Optimistic Concurrency Control (OCC)

Philosophy: "Conflicts are rare. Proceed without locks, detect conflicts at commit time."

How it works:

  1. Transaction reads data without acquiring locks.
  2. Transaction computes changes.
  3. At commit time, the database checks: "Has anything I read changed since I read it?"
  4. If yes → ABORT the transaction and return a serialization error.
  5. Application must RETRY.
T1 reads product_1 quantity = 10 (no lock)
T2 reads product_1 quantity = 10 (no lock)
T2 commits: quantity = 15

T1 tries to commit: quantity = 20
Database: "product_1 changed since you read it — ABORT"
T1 gets: ERROR: could not serialize access (SQLSTATE 40001)
Application retries T1, reads quantity = 15, writes 25 ✅

Advantages of OCC:

  • No lock management overhead
  • No waiting — transactions never block each other
  • Better throughput under low contention

Disadvantages:

  • High contention → many aborts and retries → worse throughput than pessimistic locking
  • Application must handle retry logic

Who uses it:

  • PostgreSQL's Serializable Snapshot Isolation (SSI)
  • Most NoSQL databases (DynamoDB, Cassandra via lightweight transactions)
  • CockroachDB

MVCC (Multi-Version Concurrency Control)

MVCC is the engine behind Snapshot Isolation and Repeatable Read. Instead of overwriting rows in place, the database keeps multiple versions of each row.

PostgreSQL MVCC:

Original row: product_1, quantity=10, xmin=100, xmax=NULL

T2 (txid=200) updates quantity to 15:
  Old row: product_1, quantity=10, xmin=100, xmax=200   ← marked dead
  New row: product_1, quantity=15, xmin=200, xmax=NULL  ← current

When T1 (txid=150) reads, it uses its snapshot timestamp (150) to filter:

  • New row has xmin=200 — 200 > 150 → T1 cannot see this row
  • Old row has xmax=200 — not yet dead at T1's snapshot → T1 reads quantity=10 ✅

MySQL / Oracle MVCC (Undo Log):

Current row in table: product_1, quantity=15 (latest)
Undo log entry: product_1, old quantity=10, pointer → older version

When T1 needs the old version, it follows the undo log chain to reconstruct the row's historical value. For long-running transactions with many changes, this chain can be very long, making reads expensive.

5. Practical Backend Engineering Perspective

READ UNCOMMITTED in Production

Never use this for anything involving money, user data, or business logic.

Possible use: Approximate analytics dashboards where a few milliseconds of lag in consistency is acceptable and read throughput is paramount. Even then, READ COMMITTED is almost always a better choice.

READ COMMITTED in Production

Amazon DynamoDB Transactions: Uses a form of read committed for standard reads — you see committed data at read time, not at transaction start.

Stripe Payment Processing: Order of operations matters. Stripe uses READ COMMITTED level but structures transactions carefully (short-lived, idempotency keys) to avoid non-repeatable read issues.

E-commerce Flash Sales: During a flash sale, two users may see the same inventory count (say 1 remaining), both proceed to checkout. READ COMMITTED doesn't prevent this race. Solution: SELECT FOR UPDATE or optimistic locking with version fields.

REPEATABLE READ in Production

Banking — Balance Checks: A transaction that reads your balance, verifies sufficient funds, and then debits needs REPEATABLE READ (or higher) to guarantee the balance hasn't changed between the check and the debit.

Netflix Billing: Monthly billing runs aggregate reads across subscription records. If a plan change commits mid-run, REPEATABLE READ ensures the billing job sees a consistent snapshot.

MySQL InnoDB Default: MySQL defaults to REPEATABLE READ with gap locks to prevent phantom reads. Gap locks lock the "gaps" between index entries to prevent inserts in a range.

SERIALIZABLE in Production

Trading Systems — Order Matching: An order matching engine cannot afford lost updates or phantom reads. Two buy orders must not both match against the same sell order. SERIALIZABLE (or careful SELECT FOR UPDATE) is mandatory.

Banking — Fund Transfers: TRANSFER(A → B, $100) must be fully serializable. Any interleaving that allows a double-spend or a phantom balance must be prevented.

PostgreSQL SSI: Modern PostgreSQL implements SERIALIZABLE using Serializable Snapshot Isolation — a sophisticated algorithm that provides full serializability with minimal locking, using read-write conflict tracking and automatic abort on detected cycles.

Practical Pattern: Preventing Lost Updates in Java

// BAD: Read-then-write without locking (Lost Update vulnerability)
public void addSale(long productId, int unitsToAdd) {
    // Transaction 1 and Transaction 2 both read quantity=10
    int currentQty = repo.findQuantity(productId);  // Both read 10
    repo.updateQuantity(productId, currentQty + unitsToAdd);
    // T1 writes 20, T2 writes 15 — one update is lost!
}

// GOOD: Use SELECT FOR UPDATE (pessimistic locking)
@Transactional(isolation = Isolation.REPEATABLE_READ)
public void addSalePessimistic(long productId, int unitsToAdd) {
    // SELECT quantity FROM sales WHERE product_id = ? FOR UPDATE
    Sale sale = repo.findByIdWithLock(productId);  // Row is now locked
    sale.setQuantity(sale.getQuantity() + unitsToAdd);
    repo.save(sale);
    // Lock released on commit — other transactions now proceed with updated value
}

// GOOD: Optimistic locking with @Version
@Entity
public class Sale {
    @Id private Long id;
    private int quantity;
    @Version private Long version;  // JPA auto-increments, detects conflicts
}

@Transactional
public void addSaleOptimistic(long productId, int unitsToAdd) {
    try {
        Sale sale = repo.findById(productId).orElseThrow();
        sale.setQuantity(sale.getQuantity() + unitsToAdd);
        repo.save(sale);  // Throws OptimisticLockingFailureException if version mismatch
    } catch (OptimisticLockingFailureException e) {
        // Retry logic
        addSaleOptimistic(productId, unitsToAdd);
    }
}

// BEST (atomic): Let the database do the math — no read needed
@Transactional
public void addSaleAtomic(long productId, int unitsToAdd) {
    // UPDATE sales SET quantity = quantity + ? WHERE product_id = ?
    repo.incrementQuantity(productId, unitsToAdd);
    // Database handles this atomically — no lost update possible
}

Spring Transaction Isolation in Java

@Transactional(isolation = Isolation.READ_UNCOMMITTED)  // Rarely used
@Transactional(isolation = Isolation.READ_COMMITTED)    // Most common
@Transactional(isolation = Isolation.REPEATABLE_READ)   // For consistent aggregations
@Transactional(isolation = Isolation.SERIALIZABLE)      // For critical financial ops

JDBC direct control:

Connection conn = dataSource.getConnection();
conn.setAutoCommit(false);
conn.setTransactionIsolation(Connection.TRANSACTION_REPEATABLE_READ);
// ... execute queries ...
conn.commit();
conn.setTransactionIsolation(Connection.TRANSACTION_READ_COMMITTED); // reset

6. Important Interview Questions

Basic Questions

Q: What is transaction isolation?
A: Isolation determines how and when changes made by one transaction become visible to other concurrent transactions. It prevents transactions from interfering with each other in ways that produce incorrect results.

Q: What are the four standard SQL isolation levels?
A: READ UNCOMMITTED, READ COMMITTED, REPEATABLE READ, and SERIALIZABLE (in order from least to most strict).

Q: What is a dirty read?
A: Reading data written by a transaction that has not yet committed. The data may be rolled back, making the read invalid.

Q: What is the default isolation level in PostgreSQL? MySQL?
A: PostgreSQL defaults to READ COMMITTED. MySQL InnoDB defaults to REPEATABLE READ.

Q: What is a phantom read?
A: When a transaction runs a range query twice and gets different rows because another transaction inserted a new row that satisfies the range condition between the two reads.

Intermediate Questions

Q: What is the difference between Non-Repeatable Read and Phantom Read?
A: Non-repeatable read occurs when an existing row changes between two reads. Phantom read occurs when a new row appears (via INSERT by another transaction) that satisfies the query's filter. The distinction matters for implementation: non-repeatable reads can be prevented by locking the rows read; phantom reads require range locks or snapshot isolation because you cannot lock a row that doesn't exist yet.

Q: How does PostgreSQL prevent phantom reads at REPEATABLE READ level?
A: PostgreSQL implements REPEATABLE READ as Snapshot Isolation using MVCC. Every transaction gets a snapshot of the database at its start time. New rows inserted by other transactions are invisible to the snapshot because they have a higher transaction ID, which is filtered out by the snapshot's visibility rules.

Q: What is the difference between pessimistic and optimistic concurrency control?
A: Pessimistic uses locks — it assumes conflicts will happen and prevents them proactively. Optimistic doesn't use locks — it assumes conflicts are rare, lets transactions proceed, and aborts the loser if a conflict is detected at commit time. Pessimistic is better under high contention; optimistic is better under low contention.

Q: What is a Lost Update and how do you prevent it?
A: A lost update happens when two transactions read the same value, both compute a new value based on it, and both write back — the last write overwrites the first. Prevention strategies: (1) SELECT FOR UPDATE for pessimistic locking, (2) @Version / optimistic locking with retry, (3) Atomic SQL updates (UPDATE x SET qty = qty + 1).

Q: What happens when a serializable transaction is aborted?
A: The application receives a serialization error (SQLSTATE 40001). The application is responsible for retrying the entire transaction. This is expected behavior with optimistic concurrency control.

Advanced Questions

Q: How does MVCC work in PostgreSQL vs MySQL?
A:
- PostgreSQL: Keeps old row versions in the heap (same table). Each row has xmin (transaction that created it) and xmax (transaction that deleted/updated it). A transaction's snapshot determines which versions are visible. Vacuum eventually removes dead row versions.
- MySQL InnoDB: Stores only the latest row in the clustered index. Old versions are stored in the undo log (a separate structure). Reading an old version requires following a chain of undo log records. Long-running transactions with frequent updates create long undo chains, degrading read performance.

Q: What is Serializable Snapshot Isolation (SSI)?
A: SSI is PostgreSQL's implementation of SERIALIZABLE isolation. It tracks read-write dependencies between concurrent transactions. If a cycle of dependencies is detected (indicating that the serial order assumed by both transactions is impossible), one transaction is aborted. SSI provides true serializability without the throughput cost of full locking. It uses optimistic concurrency control under the hood.

Q: What is lock escalation and why is it dangerous?
A: When a transaction holds too many fine-grained locks (e.g., row locks), the database may automatically upgrade to a coarser lock (page lock or table lock) to reduce lock manager memory overhead. This is called lock escalation. It's dangerous because a table lock blocks ALL transactions on that table, turning a localized conflict into a global bottleneck. Mitigation: avoid long-running transactions on large tables, use optimistic locking instead.

Q: Can you explain why NoSQL databases prefer optimistic concurrency control?
A: NoSQL databases are designed for high write throughput across distributed nodes. Pessimistic locking requires coordination between nodes to check lock state — this is expensive over the network and reduces availability. Optimistic CC allows each node to proceed independently and detect conflicts only at commit time, aligning with the BASE consistency model. DynamoDB's conditional writes and Cassandra's lightweight transactions are examples of OCC in distributed NoSQL.

7. Common Mistakes

Beginner Mistakes

  • Assuming the default is safe for all use cases. READ COMMITTED is the default in PostgreSQL, but it allows non-repeatable reads and phantom reads. Beginners often assume default = safe.
  • Confusing Non-Repeatable Read and Phantom Read. These are different phenomena with different prevention mechanisms. Phantom read is specifically about inserted rows, not changed ones.
  • Using SERIALIZABLE everywhere. This is unnecessarily slow. Use the minimum isolation level that correctly satisfies the business requirement.
  • Not implementing retry logic for SERIALIZABLE transactions. Serialization errors are expected and must be handled by the application.

Backend Engineer Mistakes

  • Read-then-write without locking. Classic race condition. Always use SELECT FOR UPDATE or atomic SQL updates when the new value depends on the current value.
  • Long-running transactions at high isolation levels. A REPEATABLE READ transaction that runs for 5 minutes on a busy write table creates a long undo chain (MySQL) or accumulates dead tuples (PostgreSQL), degrading performance significantly.
  • Not setting isolation level explicitly and relying on ORM defaults. Spring JPA defaults to the database default. If you need REPEATABLE READ, you must set it explicitly with @Transactional(isolation = Isolation.REPEATABLE_READ).
  • Mixing isolation levels within the same connection pool. Ensure connections are properly reset between transactions.

Production Mistakes

  • Using READ UNCOMMITTED on financial data. Even for approximate dashboards, this can expose uncommitted intermediate states that are completely invalid.
  • Not monitoring for lock contention. Lock escalation and long-running blocked transactions are invisible without monitoring. Use pg_locks, SHOW ENGINE INNODB STATUS, or equivalent.
  • Assuming REPEATABLE READ prevents all anomalies. In MySQL it doesn't prevent phantom reads. In PostgreSQL it does. Know your database.
  • Using SELECT FOR UPDATE on table scans. This locks entire ranges of rows or even the whole table, creating massive contention.
  • Ignoring serialization retry in microservices. In a microservices environment, transaction boundaries often span HTTP calls. A serialization failure deep in the call chain must propagate back and be retried at the right level.

8. Performance & Scalability Notes

READ UNCOMMITTED

  • Performance: Theoretically fastest — no lock management, no version tracking.
  • Reality: Rarely implemented. The performance gain is marginal and the correctness cost is catastrophic.

READ COMMITTED

  • Performance: Excellent. Most databases are optimized for this level.
  • MVCC cost: Minimal — only tracks the latest committed version per query, not per transaction.
  • Scalability: Scales well; readers and writers don't block each other (in MVCC databases).

REPEATABLE READ

  • Performance: Moderate overhead.
  • PostgreSQL: Snapshot at transaction start. Read performance is good; no extra undo log reading. But dead tuples accumulate for long transactions (bloat).
  • MySQL: May need to traverse long undo log chains for old versions — gets expensive for long transactions with many concurrent writes.
  • Memory: PostgreSQL holds row versions in heap — more disk I/O for vacuum. MySQL holds undo log in the InnoDB tablespace.

SERIALIZABLE

  • Pessimistic (locking): Low concurrency. Throughput drops dramatically under contention. One slow transaction blocks everyone.
  • Optimistic (SSI): Better concurrency, but CPU cost for conflict tracking. High abort rates under heavy contention (wasted work).
  • Network: In distributed databases, SERIALIZABLE requires cross-node coordination — significant network latency.

Lock Management Cost

  • Row locks in memory: ~60-100 bytes per lock.
  • 700,000 row locks ≈ 42-70 MB just for lock metadata.
  • Lock escalation threshold: SQL Server escalates at 5,000 locks per table. PostgreSQL doesn't escalate but limits total locks via max_locks_per_transaction.

Throughput vs. Isolation Trade-off

READ UNCOMMITTED  →  Maximum throughput, zero correctness guarantees
READ COMMITTED    →  High throughput, prevents dirty reads
REPEATABLE READ   →  Moderate throughput, consistent per-transaction view
SERIALIZABLE      →  Low throughput (pessimistic) or moderate with retries (optimistic)

9. Database Internals Notes

Pages

  • Database data is stored in fixed-size pages (PostgreSQL: 8KB, MySQL: 16KB).
  • Page locks are coarser than row locks but finer than table locks.
  • When using clustered indexes (MySQL), rows in the same page are physically close — a page lock effectively locks a range of rows.

B+ Trees and Isolation

  • Index scans traverse B+ tree leaf pages. Gap locks (MySQL) lock the space between index entries to prevent phantom inserts.
  • PostgreSQL uses predicate locks in SERIALIZABLE mode — tracks which ranges were read.

WAL (Write-Ahead Log)

  • All changes are written to WAL before being applied to data pages.
  • On rollback, the database uses WAL/undo information to reverse changes.
  • MVCC old versions in PostgreSQL are in the heap, not WAL. WAL records the change for durability, not visibility.

Buffer Pool

  • The buffer pool (shared_buffers in PostgreSQL, innodb_buffer_pool_size in MySQL) caches hot pages.
  • MVCC old versions that haven't been vacuumed (PostgreSQL) or undo log entries (MySQL) must be readable — they are kept in or near the buffer pool for performance.
  • Long-running transactions prevent vacuuming in PostgreSQL, causing table bloat.

MVCC Internals

PostgreSQL row header fields:

xmin  — Transaction ID that created this row version
xmax  — Transaction ID that deleted/updated this row version (0 if still live)
ctid  — Physical location (page, offset) of the current version

Visibility check pseudo-code:

is_visible(row, snapshot):
  if row.xmin > snapshot.xmax: return false  -- created after snapshot
  if row.xmin in snapshot.in_progress: return false  -- not committed yet
  if row.xmax == 0: return true  -- not deleted
  if row.xmax > snapshot.xmax: return true  -- deleted after snapshot
  if row.xmax in snapshot.in_progress: return true  -- delete not committed
  return false  -- deleted before snapshot

Locks in PostgreSQL

PostgreSQL tracks locks in pg_locks:

-- See active locks
SELECT pid, locktype, relation::regclass, mode, granted
FROM pg_locks
JOIN pg_stat_activity USING (pid)
WHERE NOT granted;

-- See blocking queries
SELECT blocked.pid, blocked.query, blocking.pid, blocking.query
FROM pg_stat_activity blocked
JOIN pg_stat_activity blocking
  ON blocking.pid = ANY(pg_blocking_pids(blocked.pid));

Transactions Internals

Each transaction gets a Transaction ID (XID) — a monotonically increasing 32-bit integer in PostgreSQL (with wraparound handling). This XID is the basis for MVCC visibility decisions.

In MySQL InnoDB, transactions use an internal trx_id. The undo log stores old row versions keyed by trx_id.

10. Practical SQL Examples

Setting Isolation Levels

-- PostgreSQL / MySQL
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
BEGIN;
-- ... queries ...
COMMIT;

-- Or set for the session
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- PostgreSQL alternative
BEGIN ISOLATION LEVEL SERIALIZABLE;
-- ... queries ...
COMMIT;

Demonstrating Lost Update and the Fix

-- Setup
CREATE TABLE sales (product_id INT PRIMARY KEY, quantity INT, price DECIMAL);
INSERT INTO sales VALUES (1, 10, 5.00), (2, 20, 4.00);

-- ❌ BAD: Vulnerable to lost update (two sessions)
-- Session 1:
BEGIN;
SELECT quantity FROM sales WHERE product_id = 1; -- reads 10
-- (Session 2 also reads 10 and commits quantity=15)
UPDATE sales SET quantity = 10 + 10 WHERE product_id = 1; -- writes 20
COMMIT; -- But session 2 already wrote 15. Lost!

-- ✅ GOOD: SELECT FOR UPDATE prevents lost update
-- Session 1:
BEGIN;
SELECT quantity FROM sales WHERE product_id = 1 FOR UPDATE; -- Acquires row lock
-- Session 2 is now BLOCKED trying to get the same lock
UPDATE sales SET quantity = quantity + 10 WHERE product_id = 1;
COMMIT; -- Lock released. Session 2 now reads 20 and adds 5 → 25 ✅

-- ✅ BEST: Atomic increment — no read needed
UPDATE sales SET quantity = quantity + 10 WHERE product_id = 1;
-- Database handles atomically. No race condition possible.

Observing Isolation Behavior in PostgreSQL

-- Terminal 1: Start transaction, read data
BEGIN ISOLATION LEVEL READ COMMITTED;
SELECT * FROM sales; -- product_1: qty=10

-- Terminal 2: Update and commit
BEGIN;
UPDATE sales SET quantity = 15 WHERE product_id = 1;
COMMIT;

-- Terminal 1: Re-read (READ COMMITTED sees the change)
SELECT * FROM sales; -- product_1: qty=15 ← non-repeatable read!
COMMIT;

-- Now test REPEATABLE READ
-- Terminal 1:
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT * FROM sales; -- product_1: qty=15

-- Terminal 2:
BEGIN;
UPDATE sales SET quantity = 20 WHERE product_id = 1;
COMMIT;

-- Terminal 1:
SELECT * FROM sales; -- product_1: qty=15 ← still 15! Snapshot held.
COMMIT;

Phantom Read Demonstration

-- Terminal 1 (REPEATABLE READ in MySQL — phantoms possible)
BEGIN ISOLATION LEVEL REPEATABLE READ; -- MySQL
SELECT * FROM sales; -- 2 rows

-- Terminal 2:
INSERT INTO sales VALUES (3, 10, 1.00);
COMMIT;

-- Terminal 1:
SELECT SUM(quantity) FROM sales; -- MySQL: 3 rows summed (phantom!)
                                 -- PostgreSQL: 2 rows (no phantom)
COMMIT;

Serializable with Retry in Java

@Service
public class SaleService {

    private static final int MAX_RETRIES = 3;

    @Autowired
    private SalesRepository repo;

    public void processSale(long productId, int units) {
        int attempts = 0;
        while (attempts < MAX_RETRIES) {
            try {
                executeSaleTransaction(productId, units);
                return; // success
            } catch (TransactionSystemException e) {
                if (isSerializationFailure(e) && attempts < MAX_RETRIES - 1) {
                    attempts++;
                    log.warn("Serialization conflict on attempt {}, retrying...", attempts);
                    continue;
                }
                throw e;
            }
        }
        throw new RuntimeException("Failed after " + MAX_RETRIES + " attempts");
    }

    @Transactional(isolation = Isolation.SERIALIZABLE)
    private void executeSaleTransaction(long productId, int units) {
        Sale sale = repo.findById(productId).orElseThrow();
        if (sale.getQuantity() < units) {
            throw new InsufficientInventoryException();
        }
        sale.setQuantity(sale.getQuantity() - units);
        repo.save(sale);
    }

    private boolean isSerializationFailure(TransactionSystemException e) {
        // PostgreSQL SQLSTATE 40001 = serialization_failure
        return e.getMessage() != null && e.getMessage().contains("40001");
    }
}

11. System Design Connections

Scalability

  • Higher isolation = lower concurrency = lower scalability.
  • Horizontal scaling of databases (sharding) doesn't eliminate isolation concerns — distributed transactions are even harder. Tools like Google Spanner use TrueTime for distributed SERIALIZABLE isolation.
  • Most scalable systems use READ COMMITTED with careful application-level conflict handling.

High Availability

  • In active-passive replication, reads from the replica may lag behind the primary — a form of eventual consistency. This is weaker than READ COMMITTED.
  • In active-active setups (e.g., Galera Cluster), write conflicts are resolved at commit time — similar to optimistic CC.

Distributed Systems

  • CAP Theorem: SERIALIZABLE isolation in a distributed system conflicts with availability during partitions. Most distributed databases choose READ COMMITTED or SNAPSHOT isolation.
  • Two-Phase Commit (2PC): Enables SERIALIZABLE across distributed nodes but is slow and blocks on coordinator failure.
  • CockroachDB / Google Spanner: Implement distributed SERIALIZABLE isolation using hybrid logical clocks and Paxos consensus.

Microservices

  • Microservices rarely share a database. "Distributed transactions" across services typically use the Saga pattern (eventual consistency) rather than ACID isolation.
  • Within a single service and its database, isolation levels apply normally.
  • Outbox pattern + idempotency keys compensate for the lack of distributed isolation.

Caching

  • Caches (Redis, Memcached) do NOT provide isolation. A cache hit may return stale data that a concurrent transaction has already updated.
  • Cache invalidation must be carefully timed relative to transaction commits to avoid serving dirty or stale data.
  • Write-through caches update the cache and DB atomically; write-behind caches introduce eventual consistency.

Databases at Scale

  • Amazon Aurora: Uses a distributed storage layer with READ COMMITTED by default. Supports REPEATABLE READ and SERIALIZABLE. Uses distributed MVCC.
  • TiDB: MySQL-compatible distributed database. Implements Snapshot Isolation using MVCC with distributed timestamps via Placement Driver.
  • CockroachDB: PostgreSQL-compatible distributed database. Default isolation is SERIALIZABLE (SSI). Uses Hybrid Logical Clocks.

Pessimistic vs Optimistic Concurrency

PESSIMISTIC (SELECT FOR UPDATE):

T1: READ → [LOCK ROW] → COMPUTE → WRITE → [UNLOCK]
T2:                  → WAIT... ─────────────────────→ READ → COMPUTE → WRITE

OPTIMISTIC (OCC / @Version):

T1: READ(v=1) ─────────────────── WRITE(v=2, check v=1) → SUCCESS
T2: READ(v=1) ──── WRITE(v=2, check v=1) → CONFLICT → ABORT → RETRY

Isolation Level Decision Tree

Is correctness critical (finance, medical, legal)?
├── YES → SERIALIZABLE (with retry logic)
└── NO
    ├── Do you run aggregations across rows in one transaction?
    │   ├── YES → REPEATABLE READ (or SNAPSHOT)
    │   └── NO
    │       ├── Do you just need to avoid reading uncommitted garbage?
    │       │   ├── YES → READ COMMITTED (default, usually fine)
    │       │   └── NO → READ UNCOMMITTED (analytics only, never financial)

13. Lecture Summary

Key Takeaways

  1. Isolation is about concurrent transaction visibility. It defines what one transaction can see of another's in-progress or committed changes.
  2. There are four read phenomena: Dirty Read (read uncommitted data), Non-Repeatable Read (same row, different value in same TX), Phantom Read (new rows appear in range queries), Lost Update (concurrent writes overwrite each other).
  3. There are four standard isolation levels (plus Snapshot Isolation): READ UNCOMMITTED, READ COMMITTED, REPEATABLE READ, SERIALIZABLE.
  4. READ COMMITTED is the most common default. It prevents dirty reads but allows non-repeatable reads and phantom reads.
  5. PostgreSQL's REPEATABLE READ = Snapshot Isolation. It prevents phantom reads, unlike MySQL's REPEATABLE READ.
  6. Two implementation strategies: Pessimistic (locks) and Optimistic (conflict detection at commit). Each has trade-offs.
  7. MVCC is the engine behind snapshot isolation. PostgreSQL keeps old row versions in the heap; MySQL uses an undo log.
  8. Higher isolation = lower throughput. Choose the minimum isolation level that satisfies correctness requirements.

Must Remember Points

  • READ UNCOMMITTED = dirty reads possible. Almost never use.
  • READ COMMITTED = dirty reads impossible; NRR and phantom still possible.
  • REPEATABLE READ = NRR impossible; phantom still possible (in MySQL).
  • PostgreSQL REPEATABLE READ = Snapshot = phantom impossible.
  • SERIALIZABLE = all phenomena prevented; slowest.
  • Lost Update prevention: use SELECT FOR UPDATE, @Version, or atomic SQL.
  • Serializable transactions may abort — application MUST retry.
  • Lock escalation (too many row locks → table lock) is a production killer.

Interview Nuggets

  • "What is the default isolation level of PostgreSQL?" → READ COMMITTED
  • "Does PostgreSQL's REPEATABLE READ prevent phantom reads?" → YES (Snapshot Isolation)
  • "Does MySQL's REPEATABLE READ prevent phantom reads?" → Partially (gap locks help, but not fully equivalent to Snapshot)
  • "How do you prevent lost updates?" → SELECT FOR UPDATE, atomic updates, or @Version optimistic locking
  • "What does SERIALIZABLE error 40001 mean?" → Serialization conflict; retry the transaction
  • "Why do NoSQL databases prefer optimistic concurrency?" → Lock management is expensive in distributed systems; OCC scales better under low contention

15. Beyond the Lecture

Write Skew (Not Covered)

Write skew is a subtle anomaly not fully addressed by standard isolation levels. It occurs when two transactions read overlapping data, each makes a decision based on what it read, and the combination of their writes violates a constraint — even though neither transaction individually violated anything.

Classic example:

Constraint: At least one doctor must be on call.
Doctor A and Doctor B are both on call.

T1: reads "both on call" → Doctor A takes sick leave → writes A=off_call
T2: reads "both on call" → Doctor B takes sick leave → writes B=off_call
Result: Both doctors are off call → constraint violated!

Neither T1 nor T2 alone violated the constraint. But together, they did. This is NOT prevented by Snapshot Isolation — only by SERIALIZABLE.

Gap Locks (MySQL InnoDB)

MySQL prevents phantom reads in REPEATABLE READ using gap locks. A gap lock locks the space between two index entries, preventing inserts into that range.

-- T1: Reads rows where quantity BETWEEN 5 AND 20
SELECT * FROM sales WHERE quantity BETWEEN 5 AND 20 FOR UPDATE;
-- Gap lock acquired on the range (5, 20) in the index
-- T2 tries: INSERT INTO sales VALUES (4, 8, 2.0); → BLOCKED (8 falls in range)

Gap locks can cause unexpected blocking and deadlocks if not understood.

Two-Phase Locking (2PL)

The formal theory behind pessimistic isolation. 2PL states: a transaction may not acquire any new lock after it has released any lock. This guarantees serializability. Strong 2PL holds all locks until commit.

Deadlocks

When two transactions each hold a lock the other needs:

T1 holds lock on row A, wants lock on row B
T2 holds lock on row B, wants lock on row A
→ DEADLOCK

Databases detect deadlocks and abort one transaction (the victim). Application must retry. Prevention: always acquire locks in the same order across all transactions.

SELECT FOR SHARE vs SELECT FOR UPDATE

SELECT * FROM sales WHERE product_id = 1 FOR SHARE;   -- Shared lock: allows other readers, blocks writers
SELECT * FROM sales WHERE product_id = 1 FOR UPDATE;  -- Exclusive lock: blocks all other readers and writers

Use FOR SHARE when you want to read a row and prevent it from being modified while you're working, but you don't mind other readers. Use FOR UPDATE when you're about to modify the row.
